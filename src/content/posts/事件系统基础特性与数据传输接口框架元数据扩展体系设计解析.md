# 事件系统基础特性与数据传输接口框架元数据扩展体系设计解析

游戏框架的事件系统是模块间通信的核心基础设施。一个设计良好的事件系统需要同时满足：**类型安全、零耦合、高性能、易扩展**。本文深入解析框架事件系统的接口规范、元数据驱动的分发机制以及各组件的设计取舍。

---

## 一、事件系统的职责边界

框架的 `EventSystem` 承担三类不同语义的通信：

| 类型 | 用途 | 语义 |
|------|------|------|
| **Publish/Subscribe** | 模块间松耦合通知 | 一对多广播，发布者不关心谁在监听 |
| **Invoke** | 带类型的精确分发 | 一对一调用，类似有类型约束的函数指针 |
| **生命周期分发** | Awake/Update/Destroy 等 | 框架驱动，非业务代码直接调用 |

这三种通信模式覆盖了游戏中 95% 以上的模块间交互需求。

## 二、数据传输接口设计

### 2.1 IEvent —— 事件数据载体

```csharp
// 所有事件数据结构必须实现此接口
public interface IEvent
{
    // 标记接口：用于泛型约束和反射识别
    // 无方法体，只做类型约束
}

// 示例：道具使用事件
public struct UseItemEvent : IEvent
{
    public long ItemId;
    public int Count;
    public long UserId;
}
```

使用 `struct` 而非 `class`：
- 避免堆分配（栈上分配或内联）
- 不可变语义（传值而非引用）
- 适合高频触发的小数据包

### 2.2 IEventHandler —— 事件处理器

```csharp
// 泛型事件处理器接口
public interface IEventHandler<in T> where T : IEvent
{
    // 同步处理
    void Run(T args);
}

// 异步事件处理器
public interface IEventHandlerAsync<in T> where T : IEvent
{
    ETTask RunAsync(T args);
}
```

### 2.3 IInvoke —— 精确调用接口

```csharp
// Invoke 的参数接口
public interface IInvokeArgs
{
    // 标记接口
}

// Invoke 处理器
public interface IInvoke<in T> where T : IInvokeArgs
{
    void Handle(T args);
}
```

`Invoke` 与 `Publish` 的核心区别：Publish 允许零个或多个处理器，Invoke 要求**恰好一个**处理器响应，类似方法调用。

## 三、Attribute 元数据体系

### 3.1 EventAttribute —— 事件绑定声明

```csharp
[AttributeUsage(AttributeTargets.Class, AllowMultiple = true)]
public class EventAttribute : Attribute
{
    public SceneType SceneType { get; }

    public EventAttribute(SceneType sceneType = SceneType.All)
    {
        this.SceneType = sceneType;
    }
}

// 使用方式：
[Event(SceneType.Battle)]  // 只在战斗场景生效
public class UseItemEventHandler : IEventHandler<UseItemEvent>
{
    public void Run(UseItemEvent args)
    {
        // 处理道具使用
    }
}
```

`SceneType` 过滤机制是框架的关键设计——同一个事件可以在不同场景有不同的处理逻辑，而无需在处理器内写 if 判断。

### 3.2 InvokeAttribute —— Invoke 绑定声明

```csharp
[AttributeUsage(AttributeTargets.Class)]
public class InvokeAttribute : Attribute
{
    public int Type { get; }

    public InvokeAttribute(int type)
    {
        this.Type = type;
    }
}

// 使用方式：
[Invoke(InvokeType.LoadConfig)]
public class LoadConfigInvoker : IInvoke<LoadConfigArgs>
{
    public void Handle(LoadConfigArgs args)
    {
        ConfigManager.Load(args.ConfigPath);
    }
}
```

## 四、事件注册与分发机制

### 4.1 启动时扫描注册

```csharp
// EventSystem 初始化时，扫描所有带 EventAttribute 的类
public void InitEventSystem(Dictionary<string, Type> allTypes)
{
    foreach (var (typeName, type) in allTypes)
    {
        // 找到所有 IEventHandler 实现
        if (!typeof(IEventHandler<>).IsAssignableFrom(type)) continue;

        var attrs = type.GetCustomAttributes<EventAttribute>();
        foreach (var attr in attrs)
        {
            // 按 (EventType, SceneType) 建立多维索引
            var eventType = GetEventType(type);
            this.Register(eventType, attr.SceneType, Activator.CreateInstance(type));
        }
    }
}
```

### 4.2 Publish 分发

```csharp
public void Publish<T>(Scene scene, T args) where T : IEvent
{
    // 1. 精确匹配：当前 SceneType
    var handlers = this.GetHandlers(typeof(T), scene.SceneType);
    foreach (var handler in handlers)
        ((IEventHandler<T>)handler).Run(args);

    // 2. 全局匹配：SceneType.All 的处理器
    var globalHandlers = this.GetHandlers(typeof(T), SceneType.All);
    foreach (var handler in globalHandlers)
        ((IEventHandler<T>)handler).Run(args);
}
```

### 4.3 异步 Publish

```csharp
public async ETTask PublishAsync<T>(Scene scene, T args) where T : IEvent
{
    var tasks = new List<ETTask>();

    var handlers = this.GetAsyncHandlers(typeof(T), scene.SceneType);
    foreach (var handler in handlers)
        tasks.Add(((IEventHandlerAsync<T>)handler).RunAsync(args));

    await ETTask.WhenAll(tasks);
}
```

## 五、SceneType 过滤的工程价值

```csharp
public enum SceneType
{
    All     = 0,  // 所有场景
    Main    = 1,  // 主城
    Battle  = 2,  // 战斗
    Login   = 3,  // 登录
    // ...
}
```

**实际场景示例：**

```
PlayerDamageEvent 事件
├── [Event(SceneType.Battle)] BattleHPBarHandler     → 战斗场景更新血条 UI
├── [Event(SceneType.Battle)] BattleDeathCheckHandler → 战斗场景判断死亡
├── [Event(SceneType.All)]    AchievementHandler       → 所有场景记录成就数据
└── [Event(SceneType.Main)]   MainHPBarHandler          → 主城场景更新血条（不同 UI）
```

同一个事件在战斗场景和主城场景触发完全不同的处理逻辑，而发布者只需：

```csharp
EventSystem.Instance.Publish(scene, new PlayerDamageEvent { Damage = 100 });
```

框架自动根据当前 Scene 的 SceneType 路由到正确的处理器集合。

## 六、Invoke 的使用模式

Invoke 适合"需要类型安全的策略模式"场景：

```csharp
// 定义调用类型枚举
public static class InvokeType
{
    public const int Reload     = 1;
    public const int LoadConfig = 2;
    public const int SaveData   = 3;
}

// 定义参数结构
public struct ReloadArgs : IInvokeArgs
{
    public string ModuleName;
}

// 注册处理器（通过 Attribute，无需手动注册）
[Invoke(InvokeType.Reload)]
public class ReloadInvoker : IInvoke<ReloadArgs>
{
    public void Handle(ReloadArgs args)
    {
        ModuleManager.Reload(args.ModuleName);
    }
}

// 调用
EventSystem.Instance.Invoke(InvokeType.Reload, new ReloadArgs { ModuleName = "Battle" });
```

与直接调用方法相比：
- **可热更**：Invoke 处理器可以在热更包中覆盖
- **可替换**：不同版本/配置可以注册不同的处理器
- **可测试**：测试时可以 mock InvokeType 对应的处理器

## 七、性能优化细节

### 7.1 处理器缓存

```csharp
// 避免每次 Publish 都进行反射查找
// 启动时预热所有处理器字典：
// Dictionary<(Type eventType, SceneType), List<IEventHandler>>
private readonly Dictionary<long, List<object>> handlers = new();

private long GetKey(Type type, SceneType sceneType)
    => ((long)type.GetHashCode() << 32) | (long)sceneType;
```

### 7.2 struct 事件数据的零装箱传递

```csharp
// 泛型方法避免 struct 装箱
public void Publish<T>(Scene scene, T args) where T : IEvent
{
    // T 是 struct，不装箱，直接传值
    // 非泛型版本 Publish(Type, object) 会装箱，性能较差
}
```

## 八、小结

框架事件系统的设计核心是**元数据驱动 + 类型安全**：

1. **接口约束**：`IEvent`/`IInvoke` 提供编译期类型安全
2. **Attribute 声明**：解耦注册与实现，处理器无需知道 EventSystem 的存在
3. **SceneType 过滤**：一次 Publish，框架自动路由到正确场景的处理器
4. **泛型分发**：避免装箱，保持高频事件的性能

这套设计使得添加新事件处理器只需新增一个类并加 `[Event]` 标注，无需修改任何已有代码，真正实现了"对扩展开放，对修改关闭"的开闭原则。
