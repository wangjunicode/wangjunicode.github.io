# 事件系统基础特性与数据传输接口：框架元数据扩展体系设计解析

## 概述

ECS 框架中有一批轻量级但极为重要的"基础设施接口"，它们不直接承载业务逻辑，而是定义了框架的扩展协议：如何标注一个类是 ECS 系统、如何在实体间传递状态、如何把实体数据反序列化还原。本文聚焦 `BaseAttribute`、`BaseInvokeArg`、`ISerializeToEntity`、`ITransfer` 这四个核心类型。

## BaseAttribute：所有框架特性的公共基类

```csharp
[AttributeUsage(AttributeTargets.Class)]
public abstract class BaseAttribute : Attribute
{
    // 空基类：仅作为类型标记
}
```

看似空洞，但意义深远：

**1. 统一扫描锚点**  
框架启动时通过反射扫描所有程序集，只需 `type.IsDefined(typeof(BaseAttribute), false)` 即可快速筛选出所有"框架管理的类"，无需为每种 Attribute 单独写扫描逻辑。

**2. 继承层次约束**  
所有框架 Attribute 都继承自 `BaseAttribute`：

```
BaseAttribute
├── ObjectSystemAttribute      // 标注 ISystem 实现类
├── EventAttribute             // 标注事件处理类
├── InvokeAttribute            // 标注 Invoke 处理类
├── EntitySystemAttribute      // ECS 实体系统标注
├── PlaceholderAttribute       // 代码占位标注
└── ComponentViewAttribute     // 调试视图标注（部分框架版本）
```

**3. 工具链支持**  
IDE 插件、代码生成工具可通过识别 `BaseAttribute` 子类来提供智能补全、跳转、文档生成等功能。

## ObjectSystemAttribute：系统类注册标注

```csharp
[AttributeUsage(AttributeTargets.Class)]
public class ObjectSystemAttribute : BaseAttribute
{
    // 标注该类是一个 ISystem 实现，需要被 EventSystem 注册
}
```

使用示例：

```csharp
[ObjectSystem]
public class PlayerAwakeSystem : AwakeSystem<PlayerComponent>
{
    public override void Awake(PlayerComponent self)
    {
        self.HP = 100;
        self.Name = "Player";
    }
}
```

框架启动时：
```csharp
// EventSystem 初始化阶段
foreach (Type type in allTypes)
{
    if (!type.IsDefined(typeof(ObjectSystemAttribute), false)) continue;
    
    var system = Activator.CreateInstance(type) as ISystemType;
    if (system == null) continue;
    
    // 按系统类型分类注册
    RegisterSystem(system);
}
```

## BaseInvokeArg：Invoke 系统的参数基类

`Invoke` 是框架中"点对点精准调用"（区别于 `Event` 的广播）的底层机制，`BaseInvokeArg` 是所有调用参数的公共基类：

```csharp
public abstract class BaseInvokeArg
{
    // 公共参数基类，通常为空，子类扩展具体字段
}
```

为什么需要这个基类？

**1. 类型安全的分发**  
框架用 `Type → Handler` 的字典做 Invoke 路由，参数类型本身就是"路由键"：

```csharp
// InvokeAttribute 标注
[Invoke(typeof(TimerInvokeArg))]
public class TimerInvokeHandler : AInvokeHandler<TimerInvokeArg>
{
    public override void Handle(TimerInvokeArg args)
    {
        // 处理定时器回调
    }
}

// 调用侧
public class TimerInvokeArg : BaseInvokeArg
{
    public long TimerId;
    public int TimerType;
}

// 触发
EventSystem.Instance.Invoke(new TimerInvokeArg { TimerId = id, TimerType = type });
```

**2. 对象池复用**  
继承自 `BaseInvokeArg` 的参数对象可统一纳入对象池管理：

```csharp
// 高频定时器调用场景：复用参数对象避免 GC
var arg = ObjectPool.Fetch<TimerInvokeArg>();
arg.TimerId = timerId;
arg.TimerType = timerType;
EventSystem.Instance.Invoke(arg);
ObjectPool.Recycle(arg);
```

**3. 调试与日志**  
基类可提供统一的 `ToString()` 覆写，方便日志系统输出调用信息，无需每个子类单独实现。

## ISerializeToEntity：实体反序列化协议

```csharp
public interface ISerializeToEntity
{
    void Serialize();
    void Deserialize();
}
```

这个接口解决了一个关键问题：**ECS 的 Component 在网络同步或热更载入后，如何从原始数据"还原"到运行时状态**。

### Serialize / Deserialize 的语义

- `Serialize()`：把运行时状态写入可持久化字段（通常是 MemoryPack 序列化的字段），准备发送或存储
- `Deserialize()`：从持久化字段重建运行时状态，通常在数据接收或载入后调用

示例：

```csharp
[MemoryPackable]
public partial class InventoryComponent : Entity, ISerializeToEntity
{
    // 序列化字段（MemoryPack 能序列化）
    [MemoryPackInclude]
    private List<ItemData> serializedItems = new();
    
    // 运行时字段（字典，高效查找，但不序列化）
    [MemoryPackIgnore]
    private Dictionary<long, ItemData> itemDict = new();
    
    public void Serialize()
    {
        // 从运行时字典转为序列化列表
        serializedItems.Clear();
        serializedItems.AddRange(itemDict.Values);
    }
    
    public void Deserialize()
    {
        // 从序列化列表重建运行时字典
        itemDict.Clear();
        foreach (var item in serializedItems)
        {
            itemDict[item.Id] = item;
        }
    }
}
```

### 框架调用时机

```csharp
// DeserializeSystem：框架自动调用 Deserialize
[ObjectSystem]
public class InventoryComponentDeserializeSystem 
    : DeserializeSystem<InventoryComponent>
{
    public override void Deserialize(InventoryComponent self)
    {
        self.Deserialize();  // 委托给接口方法
    }
}
```

与 `IDeserializeSystem<T>` 配合，ECS 框架在组件实体被反序列化还原后自动触发，开发者无需手动调用。

## ITransfer：实体跨场景迁移协议

```csharp
public interface ITransfer
{
    // 标记接口：表示该 Entity/Component 支持跨场景迁移
    // 在场景转移时，实现此接口的组件会被框架特殊处理（随 Unit 一起迁移）
}
```

### 为什么需要迁移接口？

在分布式服务器架构中，玩家从一个地图（Scene）移动到另一个地图时，玩家实体及其所有组件需要"跨场景迁移"（Transfer）。但并非所有组件都应随迁移，例如：
- ✅ `PlayerInfoComponent`（基础信息）— 应迁移
- ✅ `InventoryComponent`（背包）— 应迁移  
- ❌ `PathFindingComponent`（寻路状态）— 不应迁移（目标场景的路径无意义）
- ❌ `AnimationCacheComponent`（动画缓存）— 不应迁移（纯视图数据）

`ITransfer` 就是"随迁移走"的标记：

```csharp
[MemoryPackable]
public partial class PlayerInfoComponent : Entity, ISerializeToEntity, ITransfer
{
    [MemoryPackInclude]
    public string PlayerName;
    [MemoryPackInclude]
    public int Level;
    // ...
}

// 寻路组件不实现 ITransfer，不会随玩家迁移
public class PathFindingComponent : Entity
{
    public Vector3 Target;
    public List<Vector3> Path;
}
```

### 框架处理流程

```csharp
// 场景迁移时，框架遍历实体所有组件
public static void TransferEntity(Entity entity, Scene targetScene)
{
    // 序列化所有实现 ITransfer 的组件
    var transferData = new List<byte[]>();
    foreach (var component in entity.Components.Values)
    {
        if (component is ITransfer transferable)
        {
            if (transferable is ISerializeToEntity serializable)
                serializable.Serialize();
            transferData.Add(MemoryPackSerializer.Serialize(component.GetType(), component));
        }
    }
    
    // 在目标场景反序列化还原
    var newEntity = targetScene.AddChildWithId<Entity>(entity.Id);
    foreach (var data in transferData)
    {
        var comp = MemoryPackSerializer.Deserialize(/* type */, data) as Entity;
        if (comp is ISerializeToEntity des) des.Deserialize();
        newEntity.AddComponent(comp);
    }
}
```

## 四个接口的协作全景

```
框架元数据层
├── BaseAttribute ────────── 所有系统/事件/处理器的注册标注基类
│   ├── ObjectSystemAttribute ── 系统类注册
│   └── EventAttribute ───────── 事件处理器注册
│
├── BaseInvokeArg ────────── Invoke 点对点调用的参数基类
│   └── TimerInvokeArg, NetworkInvokeArg 等具体参数
│
├── ISerializeToEntity ───── 组件序列化/反序列化协议
│   └── 配合 MemoryPack + DeserializeSystem 自动触发
│
└── ITransfer ────────────── 实体跨场景迁移标记接口
    └── 仅实现此接口的组件才随实体迁移
```

## 设计哲学总结

这四个看似简单的类型，体现了框架设计的几个核心原则：

| 原则 | 体现 |
|------|------|
| **显式优于隐式** | 实现 `ITransfer` 才迁移，不实现就不迁移，行为明确可控 |
| **协议分离** | `ISerializeToEntity` 只管序列化协议，不耦合具体序列化库 |
| **统一扫描锚点** | `BaseAttribute` 让反射扫描逻辑只需一处，降低维护成本 |
| **零运行时开销** | 标记接口在运行时只做 `is` 检查，无虚方法调用开销 |
| **可组合** | 一个 Component 可同时实现 `ISerializeToEntity` 和 `ITransfer`，接口正交 |

在游戏框架的元数据体系中，这类"基础设施接口"的质量直接影响框架的可扩展性和开发者体验——它们是框架与业务代码之间最重要的契约。
