# 战斗系统架构深析与重构设计

> 本文不是简单的"代码注释文档"，而是一次真正的架构审视：现有系统建在哪些地基上？如果重来，怎么设计才更好？如何基于现有框架高效扩展新需求？

## 一、当前战斗系统的"技术栈全景"

先回答最根本的问题：**这套战斗系统依赖了哪些第三方/自研框架？**

### 1.1 框架依赖地图

```
战斗系统技术栈
```

## 二、核心框架深度解析

### 2.1 ET框架 —— 战斗的"骨架"

ET是整套战斗的**实体管理基座**，源自开源项目 [ET框架](https://github.com/egametang/ET)，项目对其进行了魔改适配。

**核心概念：**

```csharp
// 战斗中的每一个"对象"都是Entity
public class FSMComponent : Entity, IAwake, IFixedUpdate, IUpdate, IDestroy, IReset
public class SkillComponent : Entity, IAwake, ILateUpdate, IFixedUpdate, IDestroy, IReset
public class BTComponent : Entity, IAwake, IFixedUpdate, IUpdate, IDestroy, IUniAgent
```

**ET的生命周期接口：**

| 接口 | 触发时机 | 战斗用途 |
|------|---------|---------|
| `IAwake` | 组件挂载时 | 初始化状态、注册事件 |
| `IFixedUpdate` | 固定帧（15fps逻辑帧） | 帧同步逻辑、碰撞、FSM驱动 |
| `IUpdate` | 每帧（60fps渲染帧） | 表现同步、插值 |
| `ILateUpdate` | Update之后 | 技能后处理 |
| `ILateFixedUpdate` | FixedUpdate之后 | 碰撞结算（ColliderComponent用） |
| `IDestroy` | 组件销毁时 | 资源释放 |
| `IReset` | 对象池回收时 | 状态重置（战斗单位复用） |

**关键设计：Unit = 各种Component的挂载宿主**

```
Unit（战斗实体）
├── FSMComponent     状态机
├── SkillComponent   技能管理
├── BTComponent      AI行为树
├── ColliderComponent 碰撞体
├── HurtComponent    受伤参数
├── NumericComponent 数值（HP/ATK等）
└── BuffCacheComponent Buff缓存
```

**掌握要点：**
- 理解 FixedUpdate（逻辑帧）与 Update（渲染帧）的分工
- `IReset` 是对象池复用的关键，新增组件必须在这里清理状态
- Entity的Parent/Child关系决定了组件的生命周期归属

### 2.2 TrueSync —— 帧同步的"确定性保证"

帧同步最大的敌人是**浮点数不确定性**（不同CPU/平台对float计算结果不同）。TrueSync用**定点数（Fixed Point）**彻底解决这个问题。

```csharp
// 禁止：float elapsedTime = Time.fixedDeltaTime;
// 正确：FP类型，所有平台计算结果完全一致
public FP elapsedTime { get => Game.FixedTime; }

// 禁止：Vector3 pos = new Vector3(1.5f, 0, 2.3f);  
// 正确：TSVector，定点向量
public TSVector CurPredictPos { get; set; }
```

**核心规则（在战斗逻辑中必须遵守）：**

```
🚫 帧同步逻辑禁止区
├── float/double（用FP替代）
├── Vector3/Quaternion（用TSVector/TSQuaternion替代）
├── System.Random（用确定性随机替代）
├── Time.deltaTime（用Game.FixedTime替代）
├── Physics.Raycast（用TrueSync碰撞替代）
└── 任何依赖执行顺序的并发代码
```

**定点数的精度：**
```csharp
// FP内部是long，精度约为小数点后6位
// 项目中坐标缩放因子：
public const int VectorScaleFactor = 10000; // 1单位 = 10000内部精度
```

**掌握要点：**
- **新增任何帧同步逻辑，必须用FP/TSVector，绝不能混入float**
- `FRAME_INTERVAL_DEFAULT = 4` → 客户端60fps，逻辑帧15fps（每4帧逻辑1帧）
- 碰撞检测走`ColliderComponent`的自定义AABB，不走Unity物理引擎

### 2.3 UniFSM —— 状态机驱动战斗流程

战斗单位的状态流转（Idle→攻击→受击→倒地→起身）由`UniFSM`管理。

```csharp
// 状态枚举（EFSMState常量池）
public static class EFSMState
{
    public const string Idle = "Idle";
    public const string Hurt = "Hurt";
    public const string FloatState = "FloatState";   // 浮空
    public const string LieState = "LieState";       // 倒地
    public const string GetUpState = "GetUpState";   // 起身
    public const string Parry = "ParryState";        // 格挡
    public const string Doge = "DogeState";          // 闪避
    public const string BreakDefense = "BreakDefense"; // 破防
    public const string Dead = "Dead";
    // ...
}
```

**状态转换通过EInputKey事件触发：**

```csharp
// 触发状态转换
fsmComp.TryManualCondition(EInputKey.Hurt_BreakDefense);

// 清除事件
fsmComp.ClearManualCondition(EInputKey.Host);

// 查询当前状态
fsmComp.HasManualCondition(EInputKey.Host);
```

**每个StateAction对应一种行为模式：**

| StateAction类 | 对应状态 | 作用 |
|---|---|---|
| `HurtAction` | Hurt | 普通受击 |
| `FloatStateAction` | FloatState | 浮空处理 |
| `LieStateAction` | LieState | 倒地持续 |
| `GetUpStateAction` | GetUpState | 起身保护帧 |
| `ParryStateAction` | Parry | 格挡判定 |
| `DogeStateAction` | DogeState | 闪避无敌帧 |
| `BreakDefenseStateAction` | BreakDefense | 破防硬直 |
| `DeadAction` | Dead | 死亡处理 |

**掌握要点：**
- 新增状态：在`EFSMState`添加常量 → 创建对应`StateAction` → 在`FSMComponent`配置转换条件
- 状态优先级通过配表（FSM可视化编辑器）配置，不在代码里硬编码
- `CantBreak`标记用于技能期间禁止被状态打断

### 2.4 UniScript —— 技能逻辑的"可视化编程"

这是整套系统**最复杂也最强大**的部分。技能的所有行为（伤害判定、Buff施加、位移、特效触发）都通过UniScript节点组合实现。

**三层架构：**

```
UniScript节点体系
├── Timeline层（时间轴）        技能每帧做什么
│   ├── CLIP_HurtCollider      挂载伤害碰撞体
│   ├── CLIP_AttackEnd         攻击结束清理
│   ├── CLIP_OnColliderHit     碰撞命中回调
│   ├── CLIP_RootMotion        根运动控制
│   ├── CLIP_ConditionalDerive 条件分支（技能衍生）
│   └── CLIP_OnKeyInput        输入检测（连招窗口）
│
├── Task层（行为节点）          "做什么"
│   ├── AT_AttackDamage        造成伤害
│   ├── AT_AddBuff             添加Buff
│   ├── AT_RemoveBuff          移除Buff
│   ├── AT_ModifyNumeric       修改数值
│   ├── AT_Move                位移
│   ├── AT_FireFsmEvent        触发FSM事件
│   ├── AT_InputEvent          发送输入事件
│   └── AT_SetStateTag         设置状态标签
│
└── Condition层（条件节点）     "什么时候做"
    ├── TS_GetNumeric          获取数值判断
    ├── TS_GetAttackInfo       获取攻击信息
    ├── SwitchStateTag         状态标签分支
    └── SwitchPerfromResult    表现结果分支
```

**技能执行链路（简化）：**

```
玩家输入 → VKeyProcessor分发 
    → SkillComponent选择技能图(SkillGraph)
    → UniScript Timeline开始播放
    → 各帧触发CLIP节点
    → 碰撞检测 → CLIP_OnColliderHit命中
    → AT_AttackDamage计算伤害
    → 目标FSM → TryManualCondition(Hurt)
    → 目标进入受击状态
```

**掌握要点：**
- **新增技能效果 = 新增AT_（Action Task）节点**，继承基类，实现`Execute()`
- **新增技能条件 = 新增TS_/Switch（Target/Switch节点）**，实现条件判断
- 技能伤害数据通过`AttackInfo`结构在节点间传递
- `CSubSkill`配置决定了技能的基础参数，`AT_ModifySubSkillParams`可动态修改

### 2.5 Luban —— 数据驱动的"配表引擎"

战斗数值、技能参数、Buff效果全部走Luban配表，代码只读数据不硬编码。

```csharp
// 访问技能配置
var skillConfig = CfgManager.GetTable<TbSkill>().GetById(skillId);
var subSkill = skillConfig.SubSkills[index];

// 克隆配置做动态修改（避免污染原始配置）
public CSubSkill SubSkillConfigClone { get; } = new(); // SkillComponent中
```

**配表驱动的好处：策划可以直接调整战斗参数，无需程序介入。**

## 三、如果重构，这套系统怎么设计？

读完代码，坦率评估：**现有架构整体扎实，但有几处值得改进。**

### 3.1 保留的好设计

**✅ ET的ECS组件化** — 战斗单位的能力完全由挂载的Component决定，扩展新单位类型只需增减组件，不需改继承树。这个设计值得完整保留。

**✅ 帧同步 + 本地预表现** — `PVP_LOCAL_PLAYER_SID = 99999999`说明有本地预表现机制，用户操作即时响应，逻辑帧追帧校正。这是PVP体验的关键，设计正确。

**✅ Luban配表驱动** — 策划友好，数值热更新，减少程序迭代。

**✅ UniScript可视化脚本** — 让技能逻辑可以不依赖程序员独立迭代，战斗策划可以自己搭技能，这是非常成熟的工业化设计。

### 3.2 如果重构，这些地方会改

#### 问题1：FSM状态常量用string，有运行时错误风险

**现状：**
```csharp
// 字符串状态名，写错了编译期发现不了
public const string Hurt = "Hurt";
fsmComp.TryManualCondition(EInputKey.Hurt);  // OK
// 但如果写成 "hurt" 或 "HURT" 就静默失败
```

**重构方向：** 状态用int枚举，FSM内部用枚举ID索引，错误在编译期暴露。

```csharp
// 重构后
public enum EBattleState 
{
    Idle = 0, Hurt = 1, FloatState = 2, Dead = 3, ...
}
```

---

#### 问题2：BattleComponent.CachedParams 是全局可变状态

**现状：**
```csharp
public class BattleComponent : Entity
{
    public CachedParams CP = new();  // 攻击者/防御者/技能参数都缓存在这里
}
```

这意味着一次攻击计算期间，如果触发了连锁反应（比如Buff触发二次攻击），CP会被覆盖，导致数据错误。

**重构方向：** 将攻击上下文做成值类型（struct）栈传递，每次攻击独立上下文，彻底消除全局状态污染。

```csharp
// 重构后：每次伤害计算传入独立上下文
public readonly struct AttackContext 
{
    public readonly Unit Attacker;
    public readonly Unit Defender;
    public readonly CSubSkill SubSkill;
    // ...
}
```

---

#### 问题3：ColliderComponent的碰撞类型枚举过于扁平

**现状：**
```csharp
public enum EColliderCheckType { Attack, Parry, Doge, Perform, FakeAttack }
// 5种碰撞类型全在一个枚举，扩展时只能往后加
```

**重构方向：** 引入碰撞层（Layer）概念，用位掩码，支持组合查询：

```csharp
[Flags]
public enum EColliderLayer 
{
    None     = 0,
    Attack   = 1 << 0,  // 0001
    Hurt     = 1 << 1,  // 0010
    Parry    = 1 << 2,  // 0100
    Doge     = 1 << 3,  // 1000
    Perform  = 1 << 4,
}
// 支持：if ((layer & EColliderLayer.Attack) != 0)
```

---

#### 问题4：UniScript节点命名规范可以更统一

**现状：**
- Action Task：`AT_*`  ✅
- Target Select：`TS_*`  ✅
- Switch：`Switch*`（没有前缀）⚠️
- Global Script：`GS_*`  ✅
- Trigger：`TG_*`  ✅
- CLIP：`CLIP_*`  ✅

**重构方向：** Switch统一改为 `SW_*`，所有节点都有前缀，方便搜索和分类。

---

#### 问题5：被动技能（PassiveSkill）与Buff的边界模糊

从代码看，`SimpleBuff.ownerSkill` 是 `PassiveSkillInstance`，被动技能通过Buff的生命周期来管理。但`PassiveSkill`目录下的节点（`GS_AddPassiveSkill`、`TG_PSkillEvent`）说明存在单独的被动技能系统。

**重构方向：** 明确划分：
- **Buff** = 有持续时间的数值/状态修改
- **PassiveSkill** = 永久条件触发的效果（无持续时间概念）
- 两者通过统一的 `EffectInstance` 接口管理，而不是被动依附于Buff

## 四、如何系统掌握现有框架，应对新增战斗需求

### 4.1 新需求的分类与对应路径

```
新需求 → 分类 → 对应框架 → 具体操作
```

| 需求类型 | 对应框架 | 操作路径 |
|---------|---------|---------|
| 新增技能效果（伤害/位移/变形） | UniScript | 新建AT_节点 → 注册到UniScript |
| 新增技能触发条件 | UniScript | 新建TS_或Switch节点 |
| 新增战斗状态（如霸体/无敌） | UniFSM + ET | 在EFSMState加常量 → 新StateAction → 配置转换 |
| 新增Buff类型 | Luban + SimpleBuff | 配表添加 → 实现IHasHandler的处理逻辑 |
| 新增AI行为 | UniScript/UniBT | 在行为树里加新节点 |
| 新增数值属性 | Luban + NumericComponent | 配表加ENumericId → 业务逻辑使用 |
| 新增PVP同步逻辑 | VKeyProcessor | 新增VKey常量 → 实现IVKeyHandler |

### 4.2 掌握UniScript的最快路径

UniScript是这套系统最需要深入掌握的部分，建议这样学：

**第一步：读懂一个完整技能的节点图**

找一个最简单的普通攻击技能，在编辑器里打开它的Timeline，逐个节点弄清楚：
- 哪个CLIP负责开启攻击碰撞体？（`CLIP_SetAttackCollider`）
- 哪个CLIP监听命中？（`CLIP_OnColliderHit`）
- 命中后调用哪个AT节点造伤害？（`AT_AttackDamage`）

**第二步：跟踪一次伤害的完整调用链**

从`AT_AttackDamage.Execute()`开始，打断点（或加日志），追踪：
```
AT_AttackDamage.Execute()
  → BattleComponent.OnAttack()
    → AttackInfo填充
      → DefenderHurtComponent.OnHurt()
        → NumericComponent.ModifyHP()
          → FSMComponent.TryManualCondition(Hurt)
            → HurtAction.OnEnter()
```

**第三步：仿写一个最简单的AT节点**

比如"给目标加一个持续3秒的减速Buff"：
1. 复制`AT_AddBuff.cs`
2. 修改参数：buffId、duration
3. 在UniScript编辑器里注册新节点
4. 在测试技能里调用它

**完成这三步，就能独立完成80%的战斗新需求。**

### 4.3 帧同步代码的安全守则

新增帧同步逻辑时，用这个checklist自检：

```
帧同步代码安全清单 ✓
□ 所有位置/速度/距离 → TSVector / FP，不含float/Vector3
□ 随机数 → 使用帧同步专用随机（种子来自服务器）
□ 条件判断 → 基于FP比较，不用 Mathf.Approximately
□ 时间 → Game.FixedTime，不用 Time.time
□ 物理 → ColliderComponent自定义碰撞，不用Unity Physics
□ 外部调用 → FixedUpdate里执行，不在Update/Coroutine里修改逻辑状态
□ 对象创建 → 考虑是否走对象池（IReset），避免GC Alloc
```

### 4.4 建议建立的知识地图

```
战斗系统知识树
├── 必须深入
│   ├── ET组件生命周期（IAwake/IFixedUpdate等接口）
│   ├── UniScript节点开发（AT/TS/Switch/CLIP）
│   └── TrueSync定点数（FP/TSVector规范）
│
├── 需要了解
│   ├── UniFSM状态机配置方式
│   ├── Luban配表结构（CSubSkill/CBuff）
│   └── VKeyProcessor帧同步输入分发
│
└── 遇到再学
    ├── Animancer动画控制
    ├── KH.Network.FSP网络层
    └── YIUIFramework战斗UI绑定
```

## 五、总结

| 维度 | 评价 |
|------|-----|
| 架构整体 | **扎实**。ECS + 帧同步 + 可视化脚本是成熟的工业组合 |
| 扩展性 | **良好**。新技能、新Buff、新状态都有明确的扩展路径 |
| 主要风险 | `CachedParams`全局状态 + string状态名 是BUG温床 |
| 学习优先级 | UniScript节点开发 > ET生命周期 > TrueSync规范 |
| 最值得借鉴 | 可视化技能脚本系统，让策划有能力独立迭代技能 |

这套系统的核心理念是：**代码提供机制，配置决定行为**。理解这一点，就掌握了扩展它的正确姿势。
