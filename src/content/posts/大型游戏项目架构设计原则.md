## 什么是"好架构"

在讨论具体技术之前，必须先对齐"好架构"的定义。

**错误的定义**：
- 用了最新的技术框架
- 代码量最少
- 性能最高

**正确的定义**：
> 好架构是在当前约束条件下，让软件**最容易被修改**的设计。

游戏开发的现实是：需求永远在变，策划永远有新想法，运营需求随时插入。一个"好架构"不是不需要改，而是**改变的代价最小**。

## 一、架构的第一性原理

### 1.1 软件架构的本质矛盾

```
需求变化速度  >>  代码修改速度
```

解决这个矛盾有两种策略：

**策略一：预测变化**（难度高，收益高）
提前识别哪些部分会变，在这些地方设计扩展点。

**策略二：降低修改代价**（更现实，更可靠）
通过良好的代码组织，让每次修改的影响范围最小化。

### 1.2 架构的核心目标：管理依赖

所有架构问题的根源都是**依赖**：A 依赖 B，修改 B 就可能影响 A。

好架构的核心就是**管理依赖方向**：
- 高层策略不应依赖低层细节
- 依赖关系应指向更稳定的方向

```
❌ 糟糕的依赖关系（循环依赖）
UIManager → BattleManager → UIManager（循环！）
当修改 BattleManager 时，UIManager 也必须修改，反之亦然。

✅ 良好的依赖关系（单向依赖）
UIManager → Events（稳定的接口）← BattleManager
通过事件解耦，两者互不知晓对方的存在。
```

## 二、分层架构设计

### 2.1 标准分层模型

```
┌─────────────────────────────────────┐
│           Presentation Layer         │  UI、特效表现
├─────────────────────────────────────┤
│            Game Logic Layer          │  游戏逻辑、AI、技能
├─────────────────────────────────────┤
│             Service Layer            │  资源管理、网络、存储
├─────────────────────────────────────┤
│           Infrastructure Layer       │  对象池、事件系统、配置
└─────────────────────────────────────┘

规则：上层可以依赖下层，下层不能依赖上层
```

### 2.2 实际项目的模块划分

以一个中型 MMORPG 为例：

```
Assets/
├── Scripts/
│   ├── Core/               # 基础设施（最稳定，最底层）
│   │   ├── EventSystem/    # 事件总线
│   │   ├── ObjectPool/     # 对象池
│   │   ├── Timer/          # 定时器
│   │   └── Singleton/      # 单例基类
│   │
│   ├── Framework/          # 框架层（较稳定）
│   │   ├── UI/             # UI 框架
│   │   ├── Resource/       # 资源框架
│   │   ├── Network/        # 网络框架
│   │   └── Config/         # 配置框架
│   │
│   ├── GameLogic/          # 游戏逻辑（经常变）
│   │   ├── Battle/         # 战斗系统
│   │   ├── Character/      # 角色系统
│   │   ├── Skill/          # 技能系统
│   │   ├── Item/           # 道具系统
│   │   └── Quest/          # 任务系统
│   │
│   └── UI/                 # 界面层（最不稳定）
│       ├── Battle/         # 战斗界面
│       ├── Lobby/          # 大厅界面
│       └── Common/         # 公共组件
```

### 2.3 模块边界的设计

每个模块对外只暴露接口，隐藏实现细节：

```csharp

namespace Game.Battle

    // 对外接口（public）
    public interface IBattleSystem
    {
        void StartBattle(BattleConfig config);
        void EndBattle(BattleResult result);

    }
    
    // 内部实现（internal 或 private）
    internal class BattleSystem : IBattleSystem
    {
        private readonly SkillProcessor _skillProcessor;      // 内部细节
        private readonly DamageCalculator _damageCalculator;  // 内部细节
        // ...
    }

        _battle = battle;
```

## 三、核心设计模式的游戏实践

### 3.1 事件总线（Event Bus）

解耦是架构设计的核心目标，事件总线是最常用的解耦工具：

```csharp
/// <summary>
/// 类型安全的事件总线，支持零 GC 调用
/// </summary>
public static class EventBus
{
    private static readonly Dictionary<Type, List<Delegate>> _handlers = new();

    public static void Subscribe<T>(Action<T> handler) where T : struct
    {
        var type = typeof(T);
        if (!_handlers.ContainsKey(type))
            _handlers[type] = new List<Delegate>();
        _handlers[type].Add(handler);
    }

    public static void Unsubscribe<T>(Action<T> handler) where T : struct
    {
        if (_handlers.TryGetValue(typeof(T), out var list))
            list.Remove(handler);
    }

    public static void Publish<T>(T evt) where T : struct
    {
        if (!_handlers.TryGetValue(typeof(T), out var list)) return;
        
        // 快照避免回调中修改订阅列表
        for (int i = list.Count - 1; i >= 0; i--)
            ((Action<T>)list[i]).Invoke(evt);
    }
}

// 定义事件（值类型，零 GC）
public struct PlayerDiedEvent 
{ 
    public int PlayerId; 
    public Vector3 DeathPosition; 
}

public struct BossSpawnedEvent 
{ 
    public int BossId; 
    public Vector3 SpawnPosition; 
}

// 发布方（战斗系统）
public class BattleSystem
{
    void PlayerDied(int playerId, Vector3 pos)
    {
        EventBus.Publish(new PlayerDiedEvent 
        { 
            PlayerId = playerId, 
            DeathPosition = pos 
        });
        // 不需要知道谁关心这个事件
    }
}

// 订阅方（UI 系统）
public class BattleHUD : MonoBehaviour
{
    void OnEnable()
    {
        EventBus.Subscribe<PlayerDiedEvent>(OnPlayerDied);
        EventBus.Subscribe<BossSpawnedEvent>(OnBossSpawned);
    }
    
    void OnDisable()
    {
        EventBus.Unsubscribe<PlayerDiedEvent>(OnPlayerDied);
        EventBus.Unsubscribe<BossSpawnedEvent>(OnBossSpawned);
    }
    
    void OnPlayerDied(PlayerDiedEvent evt) { /* 显示死亡界面 */ }
    void OnBossSpawned(BossSpawnedEvent evt) { /* 显示 Boss 血条 */ }
}
```

### 3.2 命令模式：可撤销操作与录像回放

```csharp
// 命令接口
public interface ICommand
{
    void Execute();
    void Undo();
}

// 具体命令
public class MoveCommand : ICommand
{
    private readonly Character _character;
    private readonly Vector3 _targetPos;
    private Vector3 _previousPos;
    
    public MoveCommand(Character character, Vector3 targetPos)
    {
        _character = character;
        _targetPos = targetPos;
    }
    
    public void Execute()
    {
        _previousPos = _character.Position;
        _character.MoveTo(_targetPos);
    }
    
    public void Undo()
    {
        _character.MoveTo(_previousPos);
    }
}

// 命令管理器
public class CommandHistory
{
    private readonly Stack<ICommand> _undoStack = new();
    private readonly Stack<ICommand> _redoStack = new();
    
    public void Execute(ICommand command)
    {
        command.Execute();
        _undoStack.Push(command);
        _redoStack.Clear(); // 执行新命令后清空重做栈
    }
    
    public void Undo()
    {
        if (_undoStack.Count == 0) return;
        var command = _undoStack.Pop();
        command.Undo();
        _redoStack.Push(command);
    }
    
    public void Redo()
    {
        if (_redoStack.Count == 0) return;
        var command = _redoStack.Pop();
        command.Execute();
        _undoStack.Push(command);
    }
}

// 应用场景：帧同步录像（将所有输入记录为命令序列）
public class InputRecorder
{
    private readonly List<(int Frame, ICommand Command)> _recording = new();
    
    public void Record(int frame, ICommand command)
    {
        _recording.Add((frame, command));
        command.Execute();
    }
    
    public void Replay()
    {
        foreach (var (frame, command) in _recording)
        {
            // 回放到指定帧执行对应命令
            WaitForFrame(frame);
            command.Execute();
        }
    }
}
```

### 3.3 状态机：分层有限状态机（HFSM）

```csharp
// 状态机基础接口
public interface IState
{
    void OnEnter();
    void OnUpdate(float deltaTime);
    void OnExit();
}

// 层次状态机：子状态机嵌套
public class CharacterStateMachine
{
    private IState _currentState;
    private readonly Dictionary<Type, IState> _states = new();
    
    public void AddState<T>(T state) where T : IState
    {
        _states[typeof(T)] = state;
    }
    
    public void TransitionTo<T>() where T : IState
    {
        _currentState?.OnExit();
        _currentState = _states[typeof(T)];
        _currentState.OnEnter();
    }
    
    public void Update(float deltaTime) => _currentState?.OnUpdate(deltaTime);
}

// 战斗状态机示例
public class BattleState : IState
{
    // 战斗内部还有子状态机
    private readonly CharacterStateMachine _subMachine;
    
    public BattleState()
    {
        _subMachine = new CharacterStateMachine();
        _subMachine.AddState(new IdleState());
        _subMachine.AddState(new AttackState());
        _subMachine.AddState(new DodgeState());
        _subMachine.AddState(new StunnedState());
    }
    
    public void OnEnter()
    {
        _subMachine.TransitionTo<IdleState>();
    }
    
    public void OnUpdate(float deltaTime)
    {
        _subMachine.Update(deltaTime);
    }
    
    public void OnExit()
    {
        // 清理战斗状态
    }
}
```

### 3.4 观察者模式的正确打开方式

```csharp
// ❌ 常见错误：忘记取消订阅导致内存泄漏
public class Monster : MonoBehaviour
{
    void Start()
    {
        GameManager.Instance.OnLevelEnd += HandleLevelEnd; // 订阅
        // 忘记在 OnDestroy 中取消订阅！
    }
    
    void HandleLevelEnd() { /* ... */ }
    // 即使 Monster 被销毁，GameManager 仍持有其引用 → 内存泄漏
}

// ✅ 使用 IDisposable 确保取消订阅
public class SafeObserver<T> : IDisposable
{
    private readonly Action<T> _handler;
    private bool _isDisposed;
    
    public SafeObserver(Action<T> handler)
    {
        _handler = handler;
        EventBus.Subscribe(handler);
    }
    
    public void Dispose()
    {
        if (_isDisposed) return;
        _isDisposed = true;
        EventBus.Unsubscribe(_handler);
    }
}

// 在 MonoBehaviour 中配合 CompositeDisposable
public class Monster : MonoBehaviour
{
    private readonly List<IDisposable> _subscriptions = new();
    
    void Start()
    {
        _subscriptions.Add(new SafeObserver<PlayerDiedEvent>(OnPlayerDied));
        _subscriptions.Add(new SafeObserver<BossSpawnedEvent>(OnBossSpawned));
    }
    
    void OnDestroy()
    {
        foreach (var sub in _subscriptions) sub.Dispose();
        _subscriptions.Clear();
    }
}
```

## 四、依赖注入与服务定位器

### 4.1 服务定位器（Service Locator）

```csharp
// 服务定位器：集中管理服务实例
public static class ServiceLocator
{
    private static readonly Dictionary<Type, object> _services = new();
    
    public static void Register<T>(T service)
    {
        _services[typeof(T)] = service;
    }
    
    public static T Get<T>()
    {
        if (_services.TryGetValue(typeof(T), out var service))
            return (T)service;
        throw new Exception($"Service {typeof(T).Name} not registered");
    }
    
    // 支持懒加载
    public static T GetOrCreate<T>() where T : new()
    {
        if (!_services.ContainsKey(typeof(T)))
            Register(new T());
        return Get<T>();
    }
}

// 在游戏启动时注册所有服务
public class GameBootstrap : MonoBehaviour
{
    void Awake()
    {
        ServiceLocator.Register<IResourceManager>(new AddressablesResourceManager());
        ServiceLocator.Register<INetworkManager>(new PhotonNetworkManager());
        ServiceLocator.Register<IAudioManager>(new FMODAudioManager());
        ServiceLocator.Register<IBattleSystem>(new BattleSystem());
    }
}

// 在各系统中使用
public class SkillSystem
{
    // 在需要时才获取，避免启动顺序问题
    private IResourceManager ResourceManager => ServiceLocator.Get<IResourceManager>();
    
    public async UniTask<SkillEffect> LoadEffect(int effectId)
    {
        return await ResourceManager.LoadAsync<SkillEffect>($"Effects/Skill_{effectId}");
    }
}
```

### 4.2 构造函数注入（单元测试友好）

```csharp
// ✅ 构造函数注入：依赖明确，易于测试
public class DamageCalculator
{
    private readonly IBuffSystem _buffSystem;
    private readonly IEquipmentSystem _equipmentSystem;
    
    // 依赖通过构造函数传入
    public DamageCalculator(IBuffSystem buffSystem, IEquipmentSystem equipmentSystem)
    {
        _buffSystem = buffSystem;
        _equipmentSystem = equipmentSystem;
    }
    
    public int Calculate(int baseAttack, int targetDefense)
    {
        float attackMultiplier = _buffSystem.GetAttackMultiplier();
        int bonusAttack = _equipmentSystem.GetWeaponBonus();
        
        int finalAttack = (int)((baseAttack + bonusAttack) * attackMultiplier);
        return Math.Max(1, finalAttack - targetDefense);
    }
}

// 单元测试
[Test]
public void DamageCalculator_WithAttackBuff_DoesMoreDamage()
{
    // Arrange
    var mockBuff = new MockBuffSystem(attackMultiplier: 1.5f);
    var mockEquip = new MockEquipmentSystem(weaponBonus: 0);
    var calculator = new DamageCalculator(mockBuff, mockEquip);
    
    // Act
    int damage = calculator.Calculate(100, 20);
    
    // Assert：(100 * 1.5) - 20 = 130
    Assert.AreEqual(130, damage);
}
```

## 五、技术负责人如何做架构决策

### 5.1 架构决策的评估维度

面对一个架构选择，可以从以下维度评估：

```
1. 可维护性（最重要）
   - 新人多快能理解这个设计？
   - 需求变更时修改多少代码？
   - 出问题时多快能定位？

2. 性能
   - 是否满足性能目标？
   - 是否有明确的性能瓶颈？
   - 过度优化的代价是否值得？

3. 可测试性
   - 能否单独测试这个模块？
   - 能否在没有 Unity 的情况下运行逻辑测试？

4. 团队适配
   - 团队现有技术栈能否支撑？
   - 学习成本是多少？
   - 是否有足够的文档和社区支持？
```

### 5.2 技术债务的管理

```
技术债务矩阵：
                    高影响                低影响
高利息（很难改）  ┃ 立即还             ┃ 计划还
──────────────────╋────────────────────╋──────────────────
低利息（容易改）  ┃ 下次需求时顺手改   ┃ 暂时搁置
```

**技术负责人的职责**：
1. 让技术债务"可见"（维护技术债务清单）
2. 在每个迭代中预留 20% 时间还债
3. 防止新债的产生（代码审查）
4. 对高优先级债务制定专项计划

### 6.2 贫血模型（Anemic Domain Model）

```csharp
// ❌ 贫血模型：数据和行为分离
public class Character
{
    public int HP;
    public int MaxHP;
    public int Attack;
    // 只有数据，没有行为
}

// 行为堆积在 Service 类里
public class CharacterService
{
    public void TakeDamage(Character c, int damage) => c.HP -= damage;
    public void Heal(Character c, int amount) => c.HP = Math.Min(c.MaxHP, c.HP + amount);
    public bool IsDead(Character c) => c.HP <= 0;
}

// ✅ 充血模型：数据和行为在一起
public class Character
{
    public int HP { get; private set; }
    public int MaxHP { get; private set; }
    public int Attack { get; private set; }
    public bool IsDead => HP <= 0;
    
    public void TakeDamage(int damage)
    {
        HP = Math.Max(0, HP - damage);
        if (IsDead) OnDeath();
    }
    
    public void Heal(int amount)
    {
        HP = Math.Min(MaxHP, HP + amount);
    }
    
    private void OnDeath()
    {
        EventBus.Publish(new CharacterDiedEvent { Character = this });
    }
}
```

## 总结

大型游戏项目的架构设计，核心是回答三个问题：

1. **职责边界**：每个模块该做什么，不该做什么？
2. **依赖方向**：A 依赖 B 还是 B 依赖 A？通过接口还是具体实现？
3. **扩展点设计**：在哪里设计扩展点，让未来的需求可以低代价接入？

**技术负责人的价值**，不在于写出最复杂的代码，而在于**让整个系统保持简单**。

> 简单是最难实现的目标，也是最有价值的追求。
