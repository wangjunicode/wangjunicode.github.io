# 全局剧情变量组件——配置表默认值与服务端持久化同步机制

养成类剧情游戏有大量"记忆"——玩家是否看过某段剧情、某个角色的好感度是否达到了阈值、某个选项有没有选过……这些状态需要跨局持久化（存档并同步到服务端）。

VGame项目的`StoryVariableComponent`管理了这些全局剧情变量，本文分析它的初始化流程和实时更新机制。

## 一、双类型变量设计

```csharp
// 从TbGlobalStoryVarConfig配置表读取所有变量定义
foreach (var config in CfgManager.tables.TbGlobalStoryVarConfig.DataList)
{
    self.StoryVariables.Add(config.GlobalStoryVarName, new StoryVariable()
    {
        Value = config.DefaultValue,   // 配置表默认值
        VariableType = config.VariableType == cfg.dialogue.EVariableType.GLOBAL
            ? EVariableType.GLOBAL
            : EVariableType.LOCAL,
    });
}
```

变量分为两种类型：

- **GLOBAL（全局）**：在所有角色之间共享，比如"玩家的养成进度总天数"
- **LOCAL（局部）**：和特定剧情/角色绑定，比如"玩家在某段对话里选了哪个选项"

配置表定义了所有可能存在的变量，包括默认值（首次游玩时的初始状态）。

## 二、初始化流程：配置表→服务端数据覆盖

```csharp
public async static ETTask<bool> InitVariableFromServer(this StoryVariableComponent self)
{
    var eventDispatcher = self.ClientScene().GetComponent<EventDispatcherComponent>();
    
    // 防止重复注册：先注销，再注册
    eventDispatcher.UnRegisterEvent<Evt_StoryVariable>(self.UpdateVariableCall);
    eventDispatcher.RegisterEvent<Evt_StoryVariable>(self.UpdateVariableCall);
    
    // 防止二次初始化数据错乱：清空现有数据
    if (self.StoryVariables.Count > 0)
        self.StoryVariables.Clear();
    
    // 步骤1：用配置表默认值初始化所有变量
    foreach (var config in CfgManager.tables.TbGlobalStoryVarConfig.DataList)
    {
        self.StoryVariables.Add(config.GlobalStoryVarName, new StoryVariable()
        {
            Value = config.DefaultValue,
            VariableType = ...
        });
    }
    
    // 步骤2：向服务端请求已保存的变量值
    var req = new ZoneCultivateStoryRecordFieldInfoReq()
    {
        FieldTypeList = { 
            (int)DialogueEVariableType.Global, 
            (int)DialogueEVariableType.Local 
        }
    };
    
    var nr = await self.ClientScene().GetComponent<NetworkComponent>()
        .SendAsync<ZoneCultivateStoryRecordFieldInfoResp>(
            (uint)ZoneClientCmd.ZoneCsCultivateStoryFieldInfo, req);
    
    if (nr.IsSuccess)
    {
        // 步骤3：用服务端数据覆盖配置表默认值
        foreach (var list in nr.Data.FieldInfoList)
        {
            for (int i = 0; i < list.FieldList.Count; i++)
            {
                foreach (var tab in self.StoryVariables)
                {
                    // 通过 intID 匹配（服务端用整数ID，客户端用字符串名称）
                    if (list.FieldList[i].Key == self.GetVariableIntID(tab.Key))
                    {
                        tab.Value.Value = list.FieldList[i].Value;
                        break;
                    }
                }
            }
        }
        return true;
    }
    return false;
}
```

**"配置表初始化→服务端覆盖"的设计思路**：

1. 先用配置表默认值初始化：确保即使服务端没有数据（新玩家、新变量），系统也有合理的初始状态
2. 服务端数据覆盖：把玩家已有的存档值覆盖到对应变量

这类似数据库的"upsert"操作——存在则更新，不存在则保持默认。

**为什么要清空再初始化？**

注释解释了：`先清空，避免重复添加导致异常`。如果游戏在线热更新后需要重新初始化（配置表可能增加了新变量），如果不清空直接Add，会因为key已存在而抛异常。Clear+重新初始化确保了安全。

## 三、字符串ID到整数ID的映射

这是整个系统中最精妙的设计之一：

```csharp
public static int GetVariableIntID(this StoryVariableComponent self, string variableName)
{
    // 通过配置表找到 "字符串变量名" → "整数ID" 的映射
    var config = CfgManager.tables.TbGlobalStoryVarConfig.GetByName(variableName);
    return config?.IntID ?? 0;
}
```

**为什么客户端用字符串名称、服务端用整数ID？**

- **字符串（`"story_main_01_done"`）**：在代码和配置里易于阅读和维护，策划能直接理解变量的含义
- **整数（`10051`）**：传输效率高（4字节 vs 20+字节），在高频的网络同步消息中节省带宽

`TbGlobalStoryVarConfig`配置表是两者的映射桥梁，配置一条`GlobalStoryVarName: "story_main_01_done", IntID: 10051`的记录，客户端代码用名称操作，网络层自动转换为整数ID。

## 四、实时更新：服务端推送新值

```csharp
private static void UpdateVariableCall(this StoryVariableComponent self, Evt_StoryVariable evt)
{
    // 服务端推送了一个变量更新
    var variableName = self.GetVariableStringName(evt.VariableID); // 整数ID→字符串名称
    
    if (!string.IsNullOrEmpty(variableName))
    {
        if (self.StoryVariables.TryGetValue(variableName, out var variable))
        {
            variable.Value = evt.NewValue; // 更新本地缓存值
        }
    }
}
```

当玩家触发了某段剧情（服务端确认了），服务端会推送`Evt_StoryVariable`通知客户端更新本地变量缓存。这确保了客户端和服务端的变量状态始终一致。

**Evt_StoryVariable的数据流**：

```
玩家触发剧情 → 客户端发请求 → 服务端确认 → 服务端修改变量 
    → 服务端推送Evt_StoryVariable → UpdateVariableCall → 本地缓存更新
    → 触发器系统可以读取最新值 → 判断是否满足新的剧情触发条件
```

## 五、`UnRegister+Register`的防重复注册模式

```csharp
// 防止重复注册：先注销，再注册
eventDispatcher.UnRegisterEvent<Evt_StoryVariable>(self.UpdateVariableCall);
eventDispatcher.RegisterEvent<Evt_StoryVariable>(self.UpdateVariableCall);
```

这个模式在项目中多次出现，是防御性编程的体现。

如果`InitVariableFromServer`被调用两次（比如重新登录后重新初始化），直接RegisterEvent会导致同一个方法被注册两次，每次服务端推送都会触发两次更新（或更多）。

先UnRegister再Register，保证无论调用多少次，事件处理器只注册一次。

## 六、变量读写的接口设计

```csharp
// 读取变量值
public static int GetVariableValue(this StoryVariableComponent self, string name)
{
    if (self.StoryVariables.TryGetValue(name, out var variable))
        return variable.Value;
    
    Log.LogWarning("[StoryVariable] 变量 {0} 不存在", name);
    return 0; // 默认返回0，不抛异常
}

// 修改变量值（并上报服务端）
public static async ETTask SetVariableValue(this StoryVariableComponent self, string name, int value)
{
    if (!self.StoryVariables.TryGetValue(name, out var variable))
    {
        Log.LogError("[StoryVariable] 设置不存在的变量: {0}", name);
        return;
    }
    
    int oldValue = variable.Value;
    variable.Value = value;         // 先更新本地（即时生效）
    
    // 发布变量变化事件（触发器系统可以响应）
    EventSystem.Instance.Publish(self.ClientScene(), new Evt_StoryVariableChanged
    {
        VariableName = name,
        OldValue = oldValue,
        NewValue = value,
    });
    
    // 异步上报服务端（容错：失败了本地已经更新了）
    await UploadVariableToServer(self, name, value);
}
```

**"先更新本地，再上报服务端"的策略**：

优点：
- 即时响应，剧情逻辑不需要等待网络往返
- 即使网络慢，玩家也不会感知到延迟

风险：
- 如果服务端上报失败（网络断线），本地和服务端会不一致
- 通过服务端推送来同步，下次从服务端加载时会修正

对于剧情变量（玩家看没看过某段剧情），这种轻微的不一致通常影响不大（顶多是玩家重新登录后这段剧情又触发了一次）。

## 七、总结

全局剧情变量系统的设计展示了：

1. **配置表默认值兜底**：新玩家/新变量无需服务端数据，直接使用配置默认值
2. **两步初始化**：先配置表→再服务端覆盖，稳健可靠
3. **ID双映射**：字符串便于开发，整数便于网络传输，配置表是桥梁
4. **防重复注册**：UnRegister+Register的保险注册模式
5. **先更新本地**：即时响应，容忍网络失败

对新手来说，这套系统展示了一个重要的架构原则：**本地缓存+服务端同步**。游戏逻辑依赖本地缓存（速度快），服务端负责持久化（可靠），两者通过"初始化拉取+实时推送"保持一致。
