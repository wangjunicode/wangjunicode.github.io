# 帧同步战斗指令处理与网络同步系统

多人联机战斗的核心挑战是：如何让两个玩家看到的战斗结果完全一致？一方按了攻击键，对方需要立刻看到效果——但网络有延迟，处理器有差异。

帧同步（Lockstep）是解决这个问题的经典方案。本文深入解析这套帧同步战斗的网络同步系统。

---

## 帧同步的基本原理

帧同步的核心思想：

1. **所有玩家共用同一套初始状态**（角色属性、位置、随机种子）
2. **网络只传输玩家的"操作指令"**（按了哪个键），而不是游戏状态
3. **每个客户端在本地用相同的初始状态 + 相同的指令序列，重新计算战斗结果**
4. **由于计算过程完全确定性**（定点数+相同输入），所有客户端结果完全一致

关键约束：**计算必须完全确定性**。相同输入 → 相同输出，在任何平台、任何时间都成立。

## SyncFrame 和 SyncCmd：帧数据结构

```csharp
public class SyncFrame
{
    public int frameId;             // 帧号（全局递增）
    public List<SyncCmd> cmdList;   // 此帧所有玩家的操作指令
}

public class SyncCmd
{
    public int vkey;        // 虚拟按键（操作类型）
    public int playerId;    // 操作玩家 ID（或队伍 ID）
    public int[] args;      // 参数数组（如移动向量、技能 ID）
}
```

每一帧（如 1/30 秒），服务器收集所有玩家的操作指令，打包成 `SyncFrame`，广播给所有客户端。客户端接收到 `SyncFrame` 后，按照指令更新本地战斗状态。

## VKeyProcessorComponent：指令分发路由器

```csharp
[FriendOf(typeof(VKeyProcessorComponent))]
public static partial class VKeyProcessorComponentSystem
{
    [EntitySystem]
    private static void Awake(this VKeyProcessorComponent self)
    {
        // 扫描所有标记了 [VKeyProcess] 的处理器类，建立路由表
        var types = EventSystem.Instance.GetTypes(typeof(VKeyProcessAttribute));
        foreach (var type in types)
        {
            var attr = type.GetCustomAttribute<VKeyProcessAttribute>();
            self.handlers.Add(attr.vkey, Activator.CreateInstance(type) as IVKeyHandler);
        }
    }
    
    // 处理一个指令
    public static void Process(this VKeyProcessorComponent self, BattlePlayerComponent comp, SyncFrame frame, SyncCmd cmd)
    {
        var team = CommandHelper.GetCommandTeam(comp, cmd);  // 找到指令所属的队伍
        
        if (self.handlers.TryGetValue(cmd.vkey, out var handler))
        {
            handler.Handle(comp, frame, cmd, team);  // 路由到具体处理器
        }
        else
        {
            // 默认处理：直接注入输入到队伍输入组件
            if (team != null)
            {
                team.GetComponent<TeamInputComponent>().SetInputValue((EInputKey)cmd.vkey, true, true);
            }
        }
    }
}
```

这是一个典型的**命令模式（Command Pattern）+ 注解驱动路由**：

- 每种指令类型（`vkey`）有一个对应的处理器类（`IVKeyHandler`）
- 通过 `[VKeyProcess(vkey)]` 注解声明路由规则
- 系统启动时自动扫描所有处理器，建立路由表

这种设计的优点：新增指令类型只需要添加新的处理器类和注解，不需要修改路由器本身。

## 典型指令处理器

### 移动指令处理器

```csharp
[VKeyProcess((int)EInputKey.Move)]
public class MoveHandler : IVKeyHandler
{
    public void Handle(BattlePlayerComponent comp, SyncFrame frame, SyncCmd cmd, TeamEntity team)
    {
        if (team == null) return;
        
        // args[0,1,2] 是移动向量（×10000 精度，避免浮点数传输）
        var vector = new TSVector(cmd.args[0], cmd.args[1], cmd.args[2]);
        vector /= 10000;  // 还原真实向量值（定点数）
        
        team.GetComponent<TeamInputComponent>().SetInputValue((EInputKey)cmd.vkey, vector);
    }
}
```

**关键细节**：移动向量在发送前乘以 10000，接收后除以 10000。

这是因为网络传输通常使用整数，而定点数向量需要精度（如 0.7071 需要表示为 7071/10000）。通过整数传输，避免了浮点数在不同平台序列化可能产生的微小差异。

### 同步错误处理器

```csharp
[VKeyProcess(VKeyDef.PVP_SYNC_ERROR)]
public class SyncErrorHandler : IVKeyHandler
{
    public void Handle(BattlePlayerComponent comp, SyncFrame frame, SyncCmd cmd, TeamEntity team)
    {
        // 发布同步错误事件，触发自动保存日志等处理
        EventSystem.Instance.Publish(comp.DomainScene(), 
            new Evt_PVPSyncError() { frame = frame, cmd = cmd });
    }
}
```

当服务器检测到哈希不一致（Desync）时，发送 `PVP_SYNC_ERROR` 指令给所有客户端，客户端收到后可以触发日志收集、异常上报等处理。

### 战术调整指令处理器

```csharp
[VKeyProcess(VKeyDef.PVP_CHANGE_TACTIC_START)]
public class TacticStartHandler : IVKeyHandler
{
    public void Handle(BattlePlayerComponent comp, SyncFrame frame, SyncCmd cmd, TeamEntity team)
    {
        var tacticChangeIndex = cmd.args[2];
        
        // 防重放：只处理比上次超时索引更新的指令
        if (tacticChangeIndex <= comp.TacticTimeoutIndex) return;
        
        // 暂停战斗，等待战术调整
        comp.Domain.GetComponent<BattleComponent>().Pause();
        
        // 只有自己的队伍才弹出调整界面
        if (CommandHelper.IsMyTeam(comp, cmd.playerId))
        {
            // ... 弹出战术调整UI
        }
    }
}
```

**防重放机制**：`tacticChangeIndex <= comp.TacticTimeoutIndex` 这个检查防止了过期指令被重新执行。网络可能因为各种原因重传旧包，必须通过序列号来过滤。

## LocalBattleComponent：单机战斗驱动器

```csharp
[FriendOf(typeof(LocalBattleComponent))]
public static partial class LocalBattleComponentSystem
{
    [EntitySystem]
    private static void Awake(this LocalBattleComponent self)
    {
        // 监听战斗生命周期事件
        self.RootDispatcher().RegisterEvent<Evt_RealBattleStart>(self.OnPreBattleEnd);
        self.RootDispatcher().RegisterEvent<Evt_RealBattleEnd>(self.OnBattleEnd);
        self.RootDispatcher().RegisterEvent<Evt_RealBattleNeedPause>(self.OnBattleNeedPause);
        self.RootDispatcher().RegisterEvent<Evt_RealBattleCancelPause>(self.OnBattleCancelPause);
        
        // 监听来自服务器的控制指令
        self.battlePlayerComp.RegisterServerCommandCallback(
            VKeyDef.PVP_CONTROL_START, self.OnSyncControlStart);
        self.battlePlayerComp.RegisterServerCommandCallback(
            VKeyDef.PVP_ROUND_BEGIN, self.OnSyncRoundBegin2);
        self.battlePlayerComp.RegisterServerCommandCallback(
            VKeyDef.PVP_ROUND_END, self.OnSyncRoundEnd);
        self.battlePlayerComp.RegisterServerCommandCallback(
            VKeyDef.PVP_BATTLE_END, self.OnSyncBattleEnd);
        
        // 监听玩家输入（将输入打包发送给服务器）
        self.RootDispatcher().RegisterEvent<Evt_InputVKey>(self.OnInputVKey);
        
        // 开启帧录制
        self.battlePlayerComp.SetEnableRecordFrames(true);
    }
}
```

`LocalBattleComponent` 是单机模式的战斗驱动器，它：
1. 监听玩家输入 → 打包为 `SyncCmd` → 发送给服务器（即使是单机，也走同一套流程）
2. 接收服务器广播的 `SyncFrame` → 通过 `VKeyProcessorComponent` 路由处理
3. 驱动帧循环：收到帧数据 → 执行帧 → 推进战斗

## 帧缓冲区（FrameBuffer）

网络有抖动，不是每帧都能按时收到 `SyncFrame`。`PVPFrameBuffer` 处理这种情况：

```csharp
public class PVPFrameBuffer
{
    // 最多缓冲多少帧（超过会强制丢弃最旧的）
    public const int MAX_BUFFER_SIZE = 10;
    
    // 收到帧数据时存入缓冲区
    public void ReceiveFrame(SyncFrame frame) { ... }
    
    // 帧循环调用，查询这次应该执行几帧
    public int GetFrameCountOnEnterframe(int currentFrame)
    {
        // 正常情况：每次执行1帧
        // 如果落后太多（网络恢复后需要追帧）：执行多帧
        // 如果没有新帧（网络延迟）：返回0（等待）
    }
    
    // 取出并执行一帧
    public SyncFrame GetFrameAndTick(int currentFrame) { ... }
}
```

帧缓冲区的设计目标：
- **平滑播放**：不会因为单帧延迟而停顿
- **追帧能力**：网络恢复后可以快速追上落后的帧
- **防过快**：不允许比服务器超前执行（保证同步）

## 哈希验证：反作弊的最后防线

每 N 帧（可配置），客户端将关键状态的哈希发送给服务器：

```csharp
// LocalBattleComponent 中
private static void SyncHash(this LocalBattleComponent self)
{
    var statComp = self.DomainScene().GetComponent<BattleStatisticComponent>();
    var hash = statComp.GenerateBattleHash();
    
    // 打包成指令发送给服务器
    self.SendCommand(VKeyDef.PVP_SYNC_HASH, (int)hash, (int)(hash >> 32));
}
```

服务器收集所有客户端的哈希，如果发现不一致（Desync），发送 `PVP_SYNC_ERROR` 通知所有客户端，记录日志用于调试。

哈希不一致的常见原因：
1. **浮点数计算**：某处意外使用了 `float` 而不是定点数 `FP`
2. **随机数不同步**：两端的随机种子或调用次序不一致
3. **时间戳差异**：某处使用了实际时间而不是逻辑帧时间
4. **外部数据不一致**：两端加载了不同版本的配置文件

## 乐观执行与本地预测

在 PVP 模式中，等待服务器帧会有延迟（网络 RTT / 2）。为了让操作感觉即时，系统使用**本地预测**：

```csharp
// 玩家操作立刻在本地执行（乐观执行）
private static void OnInputVKey(this LocalBattleComponent self, Evt_InputVKey evt)
{
    // 立刻注入输入（不等服务器确认）
    var unit = ...; // 本地玩家的 Unit
    var inputComp = unit.GetComponent<VirtualInputComponent>();
    inputComp.OnTap((int)evt.vkey);
    
    // 同时打包发送给服务器
    self.SendCommand(evt.vkey, evt.args);
}
```

如果服务器确认与本地预测一致，万事大吉。如果不一致（罕见，如网络丢包重传了旧帧），需要**回滚**到服务器确认状态——这是帧同步中最复杂的部分（通常在高要求的格斗游戏中才实现完整的回滚机制）。

## 设计总结

| 组件 | 职责 |
|------|------|
| SyncFrame/SyncCmd | 帧数据结构，网络传输单元 |
| VKeyProcessorComponent | 指令路由器（Command 模式） |
| PVPFrameBuffer | 帧缓冲，处理网络抖动 |
| LocalBattleComponent | 单机/本地战斗驱动 |
| 哈希验证 | 检测帧同步不一致 |

帧同步系统的核心原则：
1. **网络只传指令，不传状态** → 网络流量极小
2. **计算完全确定性** → 定点数 + 固定帧率 + 相同随机种子
3. **哈希验证** → 及时发现同步错误
4. **帧录制** → 保存指令序列，支持回放和反作弊

理解帧同步，是理解多人动作游戏最核心的技术挑战。它要求系统的每一行代码都有"这个计算在所有平台上是否完全一致"的意识。这种思维训练，对写出高质量的游戏后端代码大有裨益。
