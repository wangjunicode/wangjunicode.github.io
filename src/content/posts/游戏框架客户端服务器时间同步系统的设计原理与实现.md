# 游戏框架客户端服务器时间同步系统的设计原理与实现

## 一、为什么游戏需要专属的时间系统

在游戏工程中，直接调用 `DateTime.Now` 或 `Time.time` 看似简单，但在以下场景下会暴露出严重问题：

- **帧内时间不一致**：同一帧的不同代码路径对 `DateTime.UtcNow` 的调用结果可能略有差异，导致定时器、技能 CD 等逻辑产生微小偏差。
- **客户端与服务器时钟漂移**：客户端本地时钟与服务器真实时间存在误差，不修正则排行榜、活动时间、抢购等强时序功能会出错。
- **时区混乱**：多地区服务器若用 `DateTime.Now`（本地时间），同一段逻辑在不同机器上行为不同。

ET 框架通过 `TimeInfo`（单例组件）和 `TimeHelper`（静态门面）两层设计，统一解决了以上问题。

## 二、核心数据结构

### 2.1 TimeInfo 类

```csharp
public class TimeInfo : Singleton<TimeInfo>, ISingletonFixedUpdate
{
    private int timeZone;
    public readonly DateTime dt1970;
    private DateTime dt;

    public long ServerMinusClientTime { private get; set; }
    public long FrameTime;
    ...
}
```

关键字段说明：

| 字段 | 类型 | 含义 |
|------|------|------|
| `dt1970` | `DateTime` | Unix 时间戳纪元（1970-01-01 UTC），用于 Ticks 差值计算，设为 `readonly`，构造后不可变 |
| `dt` | `DateTime` | 随 TimeZone 设置变化的基准时间，影响 `ToDateTime` 转换结果 |
| `ServerMinusClientTime` | `long` | 服务器时间 - 客户端时间（毫秒），由网络握手协商写入 |
| `FrameTime` | `long` | 当前帧起始时间戳，在 FixedUpdate 入口处冻结 |

### 2.2 时区处理

```csharp
public int TimeZone
{
    set
    {
        this.timeZone = value;
        dt = dt1970.AddHours(TimeZone);
    }
}
```

`dt` 会随 TimeZone 变化：默认 TimeZone=0 时 `dt` 等于 `dt1970`（UTC 起点）；设置 TimeZone=8 时 `dt` 偏移 8 小时，使 `Transition(d)` 和 `ToDateTime(ts)` 方法将所有时间戳对齐到北京时间。这样无论服务器部署在哪里，传入时区值后时间展示都是正确的。

## 三、时间计算方法详解

### 3.1 ClientNow() — 高精度 UTC 毫秒时间戳

```csharp
public long ClientNow()
{
    return ((DateTime.UtcNow).Ticks - this.dt1970.Ticks) / 10000;
}
```

- `DateTime.Ticks` 精度为 100 纳秒，除以 10000 得到毫秒级时间戳。
- 使用 `UtcNow` 而非 `Now`，彻底消除本地时区影响，保证跨平台一致性。
- 这是线程安全的操作（`DateTime.UtcNow` 为原子操作），可在多线程中调用。

**与 `DateTimeOffset.UtcNow.ToUnixTimeMilliseconds()` 的对比：**

```
// 两种方式结果完全等价，框架选择了更低层的 Ticks 运算
// 避免引入 DateTimeOffset 在早期 .NET 版本的潜在问题
```

### 3.2 ServerNow() — 服务器校正时间

```csharp
public long ServerNow()
{
    return ClientNow() + Instance.ServerMinusClientTime;
}
```

这是经典的 **NTP 时间校正** 思路：
1. 客户端与服务器握手时交换时间戳，计算出 `ServerMinusClientTime`（含网络延迟补偿）
2. 每次需要服务器时间时，用本地时间加上该偏移量即可，无需重复请求

典型握手计算：
```
ServerMinusClientTime = ServerTimestamp - (ClientSendTime + RTT/2)
```

### 3.3 FrameTime 与 ClientFrameTime() — 帧时间冻结

```csharp
public void BeforeFixedUpdate()
{
    Update();
}

public void Update()
{
    this.FrameTime = this.ClientNow();
}

public long ClientFrameTime()
{
    return this.FrameTime;
}
```

**为什么需要帧时间冻结？**

假设某帧处理了 100 个定时器回调，如果每个回调都调用 `ClientNow()` 获取"实时"时间，那么前 50 个和后 50 个的时间基准会略有差异（假设帧处理耗时 2ms）。这会导致：
- 同一帧触发的技能 CD 计时起点不同
- 帧同步时逻辑不确定

`FrameTime` 在 `BeforeFixedUpdate` 时冻结，整帧内所有逻辑共享同一时间基准，保证了帧内逻辑的确定性。

```
帧开始 → BeforeFixedUpdate → FrameTime 冻结
         ↓
         Update / FixedUpdate（全部使用 FrameTime）
         ↓
帧结束
```

### 3.4 ServerFrameTime() — 服务端帧时间

```csharp
public long ServerFrameTime()
{
    return this.FrameTime + Instance.ServerMinusClientTime;
}
```

帧时间 + 服务器偏移 = 服务器视角的当前帧时间。在需要对齐服务器逻辑（如战斗判定、排行榜时间窗口）时使用，比使用实时 `ServerNow()` 更稳定，因为它不会在帧内变化。

### 3.5 ToDateTime() 与 Transition() — 时间戳互转

```csharp
public DateTime ToDateTime(long timeStamp)
{
    return dt.AddTicks(timeStamp * 10000);
}

public long Transition(DateTime d)
{
    return (d.Ticks - dt.Ticks) / 10000;
}
```

两者互为逆运算：
- `ToDateTime(ts)`：毫秒时间戳 → DateTime（受 TimeZone 影响）
- `Transition(d)`：DateTime → 毫秒时间戳

注意：当 TimeZone 非零时，`dt` 已偏移，所以转换结果是本地时区的 DateTime，UI 展示时可直接使用。

## 四、TimeHelper — 静态门面层

```csharp
public static class TimeHelper
{
    public const long OneDay = 86400000;
    public const long Hour = 3600000;
    public const long Minute = 60000;

    public static long ClientNow()    => TimeInfo.Instance.ClientNow();
    public static long ServerNow()    => TimeInfo.Instance.ServerNow();
    public static long ClientFrameTime() => TimeInfo.Instance.ClientFrameTime();
    public static long ServerFrameTime() => TimeInfo.Instance.ServerFrameTime();
    public static long ClientNowSeconds() => ClientNow() / 1000;
    public static DateTime DateTimeNow() => DateTime.Now;
}
```

`TimeHelper` 是纯静态的薄封装，有以下设计考量：

1. **解耦调用方与 Singleton 实现**：调用方写 `TimeHelper.ClientNow()` 而非 `TimeInfo.Instance.ClientNow()`，当 `TimeInfo` 内部结构变化时，调用方不受影响。
2. **常量预置**：`OneDay / Hour / Minute` 以毫秒表示，方便计算时间差而无需 magic number。
3. **DateTimeNow 保留**：某些场景（如日志记录）需要本地时间，`DateTimeNow()` 直接返回 `DateTime.Now`，和框架时间体系分开。

## 五、ISingletonFixedUpdate 接口的作用

`TimeInfo` 实现了 `ISingletonFixedUpdate`，这意味着它受 `Game` 单例调度系统驱动：

```csharp
// ISingletonFixedUpdate 三段式回调
BeforeFixedUpdate()  // 帧前：冻结 FrameTime
FixedUpdate()        // 帧中：空实现（留扩展）
LateFixedUpdate()    // 帧后：空实现（留扩展）
```

这种三段式设计让 `FrameTime` 的冻结时机可控：**严格在本帧所有业务逻辑之前**完成，避免时间读取顺序问题。

## 六、工程实践：在项目中正确使用时间

### 6.1 定时器逻辑

```csharp
// ❌ 错误：不同调用时刻结果不同，帧内不稳定
long start = TimeHelper.ClientNow();

// ✅ 正确：帧内一致，所有逻辑共享同一基准
long start = TimeHelper.ClientFrameTime();
```

### 6.2 服务器时间展示

```csharp
// 获取服务器当前时间的 DateTime 表示（北京时间）
TimeInfo.Instance.TimeZone = 8;
DateTime serverTime = TimeInfo.Instance.ToDateTime(TimeHelper.ServerNow());
string display = serverTime.ToString("yyyy-MM-dd HH:mm:ss");
```

### 6.3 网络握手同步

```csharp
// 服务器返回其当前时间戳
void OnHandshakeResponse(long serverTimestamp)
{
    long clientNow = TimeHelper.ClientNow();
    // 简化版：忽略 RTT，实际应补偿 RTT/2
    TimeInfo.Instance.ServerMinusClientTime = serverTimestamp - clientNow;
}
```

### 6.4 跨天判断

```csharp
bool IsSameDay(long ts1, long ts2)
{
    // 利用 OneDay 常量计算所在天数
    return ts1 / TimeHelper.OneDay == ts2 / TimeHelper.OneDay;
}
```

### 6.5 CD 与冷却计时

```csharp
// 技能施放时记录帧时间
component.CastFrameTime = TimeHelper.ClientFrameTime();

// 后续帧检查 CD
bool IsCooldownFinished(long cooldownMs)
{
    return TimeHelper.ClientFrameTime() - component.CastFrameTime >= cooldownMs;
}
```

## 七、线程安全性分析

| 方法 | 线程安全 | 说明 |
|------|----------|------|
| `ClientNow()` | ✅ 安全 | `DateTime.UtcNow` 原子操作 |
| `ServerNow()` | ✅ 安全 | 读取 `ClientNow()` + `ServerMinusClientTime`，后者为简单 long 读取 |
| `ClientFrameTime()` | ⚠️ 注意 | 读取 `FrameTime` 字段，非 volatile，多线程下理论上可能读到旧值；但游戏逻辑通常单线程，实践中无问题 |
| `ServerMinusClientTime` setter | ⚠️ 注意 | 网络回调可能在子线程触发，应确保在主线程消费网络消息时再写入 |

## 八、设计亮点总结

1. **两层分离**：`TimeInfo`（状态持有）+ `TimeHelper`（无状态门面），符合单一职责原则
2. **帧时间冻结**：`BeforeFixedUpdate` 中一次性冻结 `FrameTime`，保证帧内逻辑确定性
3. **时区可配置**：`TimeZone` setter 统一修正基准时间，UI 展示无需手动加减时区
4. **服务器偏移透明**：`ServerNow` / `ServerFrameTime` 在调用方完全透明，不用关心如何校正
5. **常量预置**：`OneDay` / `Hour` / `Minute` 避免 magic number，提高可读性

这套时间系统虽然代码量不大，却是保证游戏逻辑正确性、可移植性和多端一致性的重要基础设施。
