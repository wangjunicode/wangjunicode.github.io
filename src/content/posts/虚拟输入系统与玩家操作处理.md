# 虚拟输入系统与玩家操作处理

动作游戏的手感来自于输入系统的精细设计。一个按键，在不同时机、不同角色状态下，可能产生完全不同的效果——普通状态按攻击是普通击，空中按攻击是下落斩，格挡状态按攻击是格挡反击。

更微妙的是**预输入（Pre-Input）**：玩家在动作结束前"提前"按键，系统应该记住这次输入并在动作结束后立即响应，而不是让玩家感觉"按了没反应"。

本文深入解析这套虚拟输入系统的设计。

---

## 第一性原理：为什么需要虚拟输入层？

最简单的输入处理：

```csharp
if (Input.GetKeyDown(KeyCode.J)) StartAttack();
```

这有几个问题：
1. **平台耦合**：PC 键盘和手机触屏的输入接口完全不同
2. **帧同步不兼容**：`Input.GetKeyDown` 依赖 Unity 的 Update，与帧同步的逻辑帧不对齐
3. **AI 无法使用**：AI 控制的角色也需要"输入"，不能依赖真实键盘
4. **录像回放无法重现**：真实输入无法被录制和重放

**虚拟输入层**解决了这些问题：所有输入（玩家、AI、回放）都转化为统一的"虚拟按键事件"，注入 `VirtualInputComponent`，由帧逻辑统一处理。

## 核心数据结构

```csharp
// 单次输入值
public struct SInputValue
{
    public EInputKey InputKey;    // 虚拟按键类型
    public TSVector Vector3Value; // 向量参数（如移动方向）
    public bool BoolValue;        // 布尔参数
    public bool Tap;              // 是否是单次点击（vs 持续按住）
}

// 预输入值（带时效性）
public struct SPreInputValue
{
    public SInputValue InputValue;
    public long RequestId;       // 操作唯一 ID（防重复处理）
    public FP TriggerTimestamp;  // 触发时间
    public FP AvailableTime;     // 有效期截止时间（超时丢弃）
}
```

`EInputKey` 是虚拟按键枚举，屏蔽了具体的物理按键：

```csharp
public enum EInputKey
{
    None = 0,
    Attack = 1,     // 攻击
    Jump = 2,       // 跳跃
    Dodge = 3,      // 闪避
    Skill = 4,      // 技能
    Guard = 5,      // 格挡
    SpecialAnimation = 100,  // 特殊动画（内部用）
    // ...
}
```

## 预输入缓冲队列（PreInputKeyQueue）

```csharp
public class PreInputKeyQueue : FixedLengthQueue<SPreInputValue>
{
    public FP CheckInterval = 0.25f;  // 检测时间间隔
    private FP _checkedTimestamp = 0;  // 下次检测的时间点

    public PreInputKeyQueue(int length) : base(length)
    {
        // 从配置表读取跳跃 CD（通常 0.25秒）
        CheckInterval = CfgManager.tables.TbParam.JumpCd;
        CheckInterval = (FP)0.25f;
    }

    // 重置下次检测时间
    public void ResetCheckTime(FP time)
    {
        if (time == 0)
        {
            _checkedTimestamp = Game.FixedTime + CheckInterval;
        }
        else
        {
            _checkedTimestamp = Game.FixedTime + time;
        }
    }

    // 是否到了检测时机
    public bool WaitForCheck => Game.FixedTime >= _checkedTimestamp;
}
```

`FixedLengthQueue` 是固定长度的循环队列：当队列满时，新输入会覆盖最老的输入（而不是丢弃新输入）。这保证了最新的操作意图被保留，而不是最早的。

预输入系统的工作原理：
1. 玩家按键（如攻击），产生一个 `SPreInputValue`，有效期 0.25 秒
2. 如果角色当前不能执行该动作（如正在受击），输入进入队列等待
3. 每次角色状态发生变化时（如受击结束），检查队列中是否有待处理的预输入
4. 如果预输入还在有效期内，立刻执行；如果已超时，丢弃

## 临时输入系统

```csharp
public struct TemporaryInputInfo
{
    public bool bTemplate;       // 是否是模板输入
    public int vkeyTemplateID;   // 模板 ID（用于复杂组合操作）
    public EInputKey key;        // 具体按键
    public bool bThroughAI;      // 是否通过 AI 触发
}

public List<TemporaryInputInfo> TemporaryInputList = new List<TemporaryInputInfo>();
```

临时输入（Temporary Input）用于技能系统向状态机传递输入的场景：

```csharp
// 技能节点在合适时机注入临时输入
public static void MarkTemporaryInput(this VirtualInputComponent self)
{
    // 将 TemporaryInputList 中的输入标记为已处理
    // 这些是被技能逻辑"注入"的输入，而非来自玩家
}
```

这让技能可以模拟玩家输入，触发状态机的特定转换，而不需要直接调用状态机的接口（保持了解耦）。

## VirtualInputComponent 的生命周期

```csharp
public class VirtualInputComponent : Entity, IAwake, ILateUpdate, ILateFixedUpdate, IDestroy, IReset
{
    public Dictionary<int, SInputValue> InputValues { get; } = new();
    // int key = (int)EInputKey，Dictionary 存储当前帧的输入状态
}
```

### 输入的时序流转

```
玩家点击 UI 按钮（或 AI 决策触发）
    ↓
VirtualInputComponent.OnTap(key)  / OnDown(key, vector)
    ↓
InputValues[key] = new SInputValue { InputKey = key, Tap = true }
    ↓
WorldComponent.FixedUpdate（逻辑帧）
    ↓
FSMComponent.CheckAnyState()  // 检查输入，决定状态转换
    ↓
World.LateFixedUpdate
    ↓
VirtualInputComponent.ClearInputValues()  // 清空本帧输入
```

`ClearInputValues` 在帧末清空，保证每帧的输入状态是独立的。如果不清空，上一帧的输入会"残留"到下一帧，导致角色不停重复触发动作。

## 输入转帧同步：如何保证多人一致性

在帧同步多人模式下，玩家输入通过网络发送给所有客户端：

```csharp
// 玩家本地按键
void OnAttackButtonClick()
{
    // 生成带帧号的输入数据
    var cmd = new SyncCmd
    {
        vkey = (int)EInputKey.Attack,
        playerId = LocalPlayerId,
        args = new int[] { /* 附加参数 */ }
    };
    
    // 上传到服务器/通过帧同步网络广播
    NetworkManager.SendPlayerInput(cmd);
}

// 收到帧数据（包含所有玩家的输入）后
void OnReceiveSyncFrame(SyncFrame frame)
{
    foreach (var cmd in frame.cmdList)
    {
        // 找到对应玩家的 Unit，注入虚拟输入
        var playerUnit = GetPlayerUnit(cmd.playerId);
        var inputComp = playerUnit.GetComponent<VirtualInputComponent>();
        inputComp.OnTap((EInputKey)cmd.vkey);
        
        // 驱动状态机立刻检查
        playerUnit.GetComponent<FSMComponent>().CheckAnyState();
    }
}
```

关键点：本地玩家的输入也通过这个流程，而不是直接执行。这保证了所有客户端在同一帧处理相同的输入，结果完全一致。

## AI 输入的统一接口

AI 决策结果也通过 `VirtualInputComponent` 注入，完全复用同一套输入处理逻辑：

```csharp
// AI 行为树节点：决定攻击
public class BT_AttackAction : ActionTask
{
    protected override void OnExecute()
    {
        var unit = agent as Unit;
        var inputComp = unit.GetComponent<VirtualInputComponent>();
        inputComp.OnTap((int)EInputKey.Attack);
        
        var fsmComp = unit.GetComponent<FSMComponent>();
        fsmComp.CheckAnyState();
    }
}
```

AI 控制的怪物和玩家控制的角色，在输入处理层面完全等同。这意味着：
- AI 可以在开发期间用玩家输入代替测试
- 玩家录像可以被 AI 学习（未来方向）
- 回放时，AI 的"输入"也可以被录制和重放

## 输入清空策略

```csharp
// WorldComponent.Tick 中
foreach (var unit in activeUnits)
{
    unit.GetComponent<FSMComponent>()?.EnterFrame(deltaTime);
    
    // 检查预输入
    unit.GetComponent<EventDispatcherComponent>()?.FireEvent(new Evt_OnCheckInput());
    
    // 清空本帧输入（在 FSM 处理之后才清空）
    unit.GetComponent<VirtualInputComponent>()?.ClearInputValues();
}
```

清空时机非常关键：
1. **FSM 先处理输入**（读取 InputValues）
2. **清空输入**（清空 InputValues）

如果顺序反了，FSM 读到的是空输入，所有操作都会失效。

## 设计总结

虚拟输入层是战斗系统中"玩家意图"到"游戏响应"的桥梁：

```
物理输入（键盘/触屏）
    ↓
平台适配层（InputSystem/Touch）
    ↓
VirtualInputComponent（统一虚拟按键）
    ↓
FSMComponent（状态机处理输入）
    ↓
技能/动作执行
```

| 特性 | 实现 |
|------|------|
| 平台无关 | EInputKey 枚举屏蔽硬件差异 |
| 帧同步兼容 | 通过帧数据网络广播，统一处理 |
| AI 复用 | AI 和玩家共享同一套输入接口 |
| 预输入 | FixedLengthQueue + 有效期管理 |
| 帧末清空 | LateFixedUpdate 清空，避免残留 |

这套设计让整个战斗系统对"输入来源"完全透明——不管是人还是 AI，不管是实时操作还是录像回放，输入进来都是一样的。这是优雅架构的典型体现。
