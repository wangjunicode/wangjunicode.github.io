# 大型游戏项目框架架构设计方法论

> 架构能力是区分高级工程师和技术负责人的核心分水岭。代码写得再好，如果架构腐化，项目迟早失控。

---

## 一、游戏架构的本质目标

### 1.1 好架构需要满足什么？

用第一性原理思考：游戏项目为什么需要好的架构？

```
游戏项目的痛点：
1. 需求变更快 → 代码耦合严重时，改一个功能牵连一片
2. 团队协作 → 多人同时修改同一文件导致冲突
3. 性能要求 → 每帧16ms，架构设计直接影响性能
4. 版本维护 → 老代码难以理解和修改
5. BUG定位 → 逻辑混乱导致问题难以复现和排查

好架构的核心目标：
✅ 低耦合：修改一个模块，不影响其他模块
✅ 高内聚：相关功能集中在一起，易于理解
✅ 可测试：关键逻辑可以单元测试
✅ 可扩展：新功能可以自然地融入现有架构
✅ 性能友好：架构不成为性能瓶颈
```

### 1.2 游戏架构与传统软件架构的差异

```
传统软件（Web/移动App）：
- 事件驱动为主
- 延迟容忍度高
- 状态较少变化

游戏：
- 每帧主循环（Game Loop）
- 16ms完成所有更新
- 海量状态持续变化
- 需要特殊考虑缓存局部性
```

## 二、游戏主框架设计

### 2.1 分层架构

```
┌─────────────────────────────────────────┐
│  表现层（View Layer）                    │
│  UI、特效、动画、音效                    │
├─────────────────────────────────────────┤
│  控制层（Controller Layer）              │
│  输入处理、相机控制、场景管理            │
├─────────────────────────────────────────┤
│  游戏逻辑层（Game Logic Layer）          │
│  战斗系统、任务系统、背包系统等          │
├─────────────────────────────────────────┤
│  数据层（Data Layer）                    │
│  配置数据、玩家数据、运行时状态          │
├─────────────────────────────────────────┤
│  基础服务层（Service Layer）             │
│  网络、资源管理、日志、事件系统          │
└─────────────────────────────────────────┘
```

**关键原则：上层依赖下层，下层不知道上层的存在。**

### 2.2 核心框架模块

```csharp
// 游戏框架入口：管理所有子系统的生命周期
public class GameFramework : MonoBehaviour
{
    // 单例（谨慎使用，只用于真正全局的东西）
    public static GameFramework Instance { get; private set; }
    
    // 子系统注册
    private readonly Dictionary<Type, IGameSystem> _systems = new();
    private readonly List<IUpdateable> _updatableSystems = new();
    
    void Awake()
    {
        Instance = this;
        DontDestroyOnLoad(gameObject);
        
        // 按依赖顺序初始化系统
        RegisterSystem(new EventSystem());           // 1. 事件系统（最基础）
        RegisterSystem(new ResourceSystem());        // 2. 资源系统
        RegisterSystem(new NetworkSystem());         // 3. 网络系统
        RegisterSystem(new DataSystem());            // 4. 数据系统
        RegisterSystem(new UISystem());              // 5. UI系统
        RegisterSystem(new AudioSystem());           // 6. 音频系统
        RegisterSystem(new SceneSystem());           // 7. 场景系统
        
        // 初始化所有系统
        foreach (var system in _systems.Values)
            system.Initialize();
    }
    
    void Update()
    {
        float deltaTime = Time.deltaTime;
        foreach (var system in _updatableSystems)
            system.Update(deltaTime);
    }
    
    public T GetSystem<T>() where T : IGameSystem
    {
        return (T)_systems[typeof(T)];
    }
    
    private void RegisterSystem<T>(T system) where T : IGameSystem
    {
        _systems[typeof(T)] = system;
        if (system is IUpdateable updateable)
            _updatableSystems.Add(updateable);
    }
}

// 子系统基础接口
public interface IGameSystem
{
    void Initialize();
    void Shutdown();
}

public interface IUpdateable
{
    void Update(float deltaTime);
}
```

## 三、事件系统——解耦的基石

### 3.1 事件系统设计

```csharp
// 类型安全的事件系统
public class EventSystem : IGameSystem
{
    private readonly Dictionary<Type, List<object>> _handlers = new();
    
    // 订阅事件
    public void Subscribe<T>(Action<T> handler) where T : struct
    {
        var type = typeof(T);
        if (!_handlers.ContainsKey(type))
            _handlers[type] = new List<object>();
        _handlers[type].Add(handler);
    }
    
    // 取消订阅
    public void Unsubscribe<T>(Action<T> handler) where T : struct
    {
        var type = typeof(T);
        if (_handlers.TryGetValue(type, out var handlers))
            handlers.Remove(handler);
    }
    
    // 发送事件
    public void Emit<T>(T eventData) where T : struct
    {
        var type = typeof(T);
        if (!_handlers.TryGetValue(type, out var handlers)) return;
        
        // 使用副本遍历，防止回调中修改列表
        var handlersCopy = handlers.ToArray();
        foreach (var handler in handlersCopy)
        {
            ((Action<T>)handler).Invoke(eventData);
        }
    }
    
    public void Initialize() { }
    public void Shutdown() => _handlers.Clear();
}

// 定义游戏事件（使用struct，避免GC）
public struct PlayerDiedEvent
{
    public int PlayerId;
    public int KillerId;
    public float3 DeathPosition;
}

public struct EnemyDamagedEvent
{
    public int EnemyId;
    public float Damage;
    public DamageType Type;
}

// 使用示例
public class UIHealthBar : MonoBehaviour
{
    private EventSystem _eventSystem;
    
    void Start()
    {
        _eventSystem = GameFramework.Instance.GetSystem<EventSystem>();
        _eventSystem.Subscribe<EnemyDamagedEvent>(OnEnemyDamaged);
    }
    
    void OnDestroy()
    {
        _eventSystem.Unsubscribe<EnemyDamagedEvent>(OnEnemyDamaged); // 务必取消订阅！
    }
    
    private void OnEnemyDamaged(EnemyDamagedEvent e)
    {
        UpdateHealthBar(e.EnemyId, e.Damage);
    }
}
```

## 四、UI框架设计

### 4.1 UI框架核心挑战

```
UI框架需要解决的问题：
1. 打开/关闭界面的管理（栈式导航）
2. 界面层级管理（正常UI、弹窗、系统提示）
3. UI与数据的绑定（数据变化→UI自动更新）
4. 界面资源的加载/卸载
5. 多界面同时存在时的交互（遮罩、点击穿透）
```

### 4.2 UI管理器实现

```csharp
public class UISystem : IGameSystem, IUpdateable
{
    // UI层级枚举
    public enum UILayer 
    { 
        Background = 0,  // 背景层（地图UI等）
        Normal = 100,    // 普通界面
        Popup = 200,     // 弹窗
        TopBar = 300,    // 顶部常驻UI（HP、金币）
        System = 400,    // 系统界面（加载、网络异常）
    }
    
    private readonly Dictionary<string, UIPanel> _activePanels = new();
    private readonly Stack<UIPanel> _panelStack = new(); // 导航历史
    private Canvas _rootCanvas;
    
    // 打开界面
    public async Task<T> OpenAsync<T>(UIOpenParams param = null) where T : UIPanel
    {
        string panelName = typeof(T).Name;
        
        // 防重复打开
        if (_activePanels.TryGetValue(panelName, out var existing))
        {
            existing.OnRefresh(param);
            return (T)existing;
        }
        
        // 加载UI预制体（异步）
        GameObject prefab = await GameFramework.Instance.GetSystem<ResourceSystem>()
            .LoadAsync<GameObject>($"UI/{panelName}");
        
        var go = Object.Instantiate(prefab, GetLayerTransform(GetLayer<T>()));
        var panel = go.GetComponent<T>();
        
        _activePanels[panelName] = panel;
        _panelStack.Push(panel);
        
        panel.OnOpen(param);
        return panel;
    }
    
    // 关闭界面
    public void Close<T>() where T : UIPanel
    {
        string panelName = typeof(T).Name;
        if (!_activePanels.TryGetValue(panelName, out var panel)) return;
        
        panel.OnClose();
        _activePanels.Remove(panelName);
        Object.Destroy(panel.gameObject);
    }
    
    // 返回（关闭栈顶界面）
    public void GoBack()
    {
        if (_panelStack.Count == 0) return;
        var top = _panelStack.Pop();
        Close(top.GetType());
    }
    
    public void Initialize() { /* 初始化Canvas等 */ }
    public void Shutdown() { /* 清理所有UI */ }
    public void Update(float deltaTime) { /* 更新逻辑 */ }
}

// UI基类
public abstract class UIPanel : MonoBehaviour
{
    public abstract void OnOpen(UIOpenParams param);
    public abstract void OnClose();
    public virtual void OnRefresh(UIOpenParams param) { }
}
```

### 4.3 数据绑定（MVVM简化版）

```csharp
// 可观察属性（驱动UI自动更新）
public class ObservableProperty<T>
{
    private T _value;
    public event Action<T, T> OnChanged; // (oldValue, newValue)
    
    public T Value
    {
        get => _value;
        set
        {
            if (EqualityComparer<T>.Default.Equals(_value, value)) return;
            var old = _value;
            _value = value;
            OnChanged?.Invoke(old, value);
        }
    }
}

// ViewModel层
public class PlayerViewModel
{
    public ObservableProperty<int> CurrentHP { get; } = new();
    public ObservableProperty<int> MaxHP { get; } = new();
    public ObservableProperty<float> HPPercent { get; } = new();
    
    public PlayerViewModel(PlayerData data)
    {
        CurrentHP.Value = data.hp;
        MaxHP.Value = data.maxHp;
        HPPercent.Value = (float)data.hp / data.maxHp;
        
        // 当HP变化时，自动更新百分比
        CurrentHP.OnChanged += (_, newHP) => 
            HPPercent.Value = (float)newHP / MaxHP.Value;
    }
}

// View层（UI组件绑定ViewModel）
public class PlayerHPBar : UIPanel
{
    [SerializeField] private Slider _hpSlider;
    [SerializeField] private Text _hpText;
    
    private PlayerViewModel _viewModel;
    
    public override void OnOpen(UIOpenParams param)
    {
        _viewModel = GameFramework.Instance.GetSystem<DataSystem>().PlayerViewModel;
        
        // 绑定：数据变化→UI自动更新
        _viewModel.HPPercent.OnChanged += OnHPPercentChanged;
        _viewModel.CurrentHP.OnChanged += OnCurrentHPChanged;
        
        // 初始化显示
        RefreshAll();
    }
    
    public override void OnClose()
    {
        _viewModel.HPPercent.OnChanged -= OnHPPercentChanged; // 取消绑定
        _viewModel.CurrentHP.OnChanged -= OnCurrentHPChanged;
    }
    
    private void OnHPPercentChanged(float _, float newPercent) 
        => _hpSlider.value = newPercent;
    
    private void OnCurrentHPChanged(int _, int newHP) 
        => _hpText.text = $"{newHP}/{_viewModel.MaxHP.Value}";
}
```

## 五、战斗系统架构设计

### 5.1 战斗系统核心需求

```
战斗系统的本质需求：
1. 数据驱动：策划可以通过配置表改变技能效果，无需改代码
2. 可扩展：新增技能类型不影响已有逻辑
3. 确定性：支持帧同步（如果项目需要）
4. 高性能：每帧可能执行大量伤害计算
```

### 5.2 技能系统——数据驱动设计

```csharp
// 技能配置数据（策划填写）
[Serializable]
public class SkillConfig
{
    public int Id;
    public string Name;
    public float Cooldown;
    public float CastRange;
    public List<SkillEffectConfig> Effects; // 技能由多个效果组成
}

[Serializable]
public class SkillEffectConfig
{
    public SkillEffectType Type; // 伤害/治疗/位移/控制
    public float Value;
    public float Radius; // 范围效果半径
    public TargetType TargetType; // 单体/范围/自身
    public float Duration; // 持续时间
}

// 效果处理器（每种类型一个处理器）
public interface ISkillEffectHandler
{
    SkillEffectType Type { get; }
    void Execute(SkillEffectConfig config, SkillContext context);
}

// 伤害处理器
public class DamageEffectHandler : ISkillEffectHandler
{
    public SkillEffectType Type => SkillEffectType.Damage;
    
    public void Execute(SkillEffectConfig config, SkillContext context)
    {
        float finalDamage = CalculateDamage(config.Value, context.Caster, context.Target);
        context.Target.TakeDamage(finalDamage, context.Caster);
    }
    
    private float CalculateDamage(float baseDamage, IUnit caster, IUnit target)
    {
        // 公式驱动的伤害计算
        float attackPower = caster.GetStat(StatType.AttackPower);
        float defense = target.GetStat(StatType.Defense);
        float damageReduction = defense / (defense + 100f); // 经典防御公式
        
        return baseDamage * attackPower * (1 - damageReduction);
    }
}

// 技能执行器
public class SkillExecutor
{
    private readonly Dictionary<SkillEffectType, ISkillEffectHandler> _handlers;
    
    public SkillExecutor()
    {
        _handlers = new Dictionary<SkillEffectType, ISkillEffectHandler>
        {
            { SkillEffectType.Damage, new DamageEffectHandler() },
            { SkillEffectType.Heal, new HealEffectHandler() },
            { SkillEffectType.Buff, new BuffEffectHandler() },
            { SkillEffectType.Move, new MoveEffectHandler() },
            // 新增类型只需添加新Handler，无需修改已有代码
        };
    }
    
    public void Execute(SkillConfig config, SkillContext context)
    {
        foreach (var effectConfig in config.Effects)
        {
            if (_handlers.TryGetValue(effectConfig.Type, out var handler))
            {
                handler.Execute(effectConfig, context);
            }
        }
    }
}
```

// Buff配置
public class BuffConfig
{
    public int Id;
    public string Name;
    public BuffStackType StackType; // 叠加方式：不叠/叠层/刷新/共存
    public int MaxStacks;
    public float Duration;
    public List<BuffModifier> Modifiers; // 属性修改
}

// 属性修改器
public class BuffModifier
{
    public StatType Stat;           // 修改哪个属性
    public ModifierType ModType;    // 加法/乘法
    public float Value;
}

// 属性计算（带Buff的完整公式）
public class StatCalculator
{
    public float GetFinalStat(IUnit unit, StatType statType)
    {
        float baseStat = unit.GetBaseStat(statType);
        
        // 收集所有Buff修改器
        float addBonus = 0;
        float multiBonus = 1;
        
        foreach (var buff in unit.ActiveBuffs)
        {
            foreach (var modifier in buff.Config.Modifiers)
            {
                if (modifier.Stat != statType) continue;
                
                if (modifier.ModType == ModifierType.Additive)
                    addBonus += modifier.Value * buff.Stacks;
                else if (modifier.ModType == ModifierType.Multiplicative)
                    multiBonus *= (1 + modifier.Value * buff.Stacks);
            }
        }
        
        // 最终公式：(基础值 + 加法) × 乘法
        return (baseStat + addBonus) * multiBonus;
    }
}

## 六、资源管理系统架构

### 6.1 资源管理的核心问题

```
资源管理需要解决：
1. 加载：同步/异步，优先级
2. 缓存：哪些资源常驻内存，哪些用完就卸载
3. 依赖：A依赖B，加载A前必须加载B
4. 引用计数：有人用就不卸载，没人用才卸载
5. 热更新集成：打补丁后资源路径可能变化
```

### 6.2 引用计数资源管理

```csharp
public class ResourceSystem : IGameSystem
{
    private readonly Dictionary<string, ResourceHandle> _cache = new();
    
    // 加载资源（带引用计数）
    public async Task<T> LoadAsync<T>(string path) where T : Object
    {
        if (_cache.TryGetValue(path, out var handle))
        {
            handle.AddRef(); // 增加引用计数
            return (T)handle.Asset;
        }
        
        // 从Addressables加载
        var asset = await Addressables.LoadAssetAsync<T>(path).Task;
        
        var newHandle = new ResourceHandle(asset);
        newHandle.AddRef();
        _cache[path] = newHandle;
        
        return asset;
    }
    
    // 释放资源引用
    public void Release(string path)
    {
        if (!_cache.TryGetValue(path, out var handle)) return;
        
        handle.RemoveRef();
        if (handle.RefCount <= 0)
        {
            // 引用归零，可以卸载
            Addressables.Release(handle.Asset);
            _cache.Remove(path);
        }
    }
    
    public void Initialize() { }
    public void Shutdown() 
    { 
        foreach (var handle in _cache.Values)
            Addressables.Release(handle.Asset);
        _cache.Clear();
    }
}

public class ResourceHandle
{
    public Object Asset { get; }
    public int RefCount { get; private set; }
    
    public ResourceHandle(Object asset) => Asset = asset;
    public void AddRef() => RefCount++;
    public void RemoveRef() => RefCount--;
}
```

## 七、大型项目架构演进

### 7.1 架构腐化的信号

```
架构开始腐化的典型症状：
❌ 加一个功能要改10个文件
❌ 不敢修改某段代码，怕出Bug
❌ 同一个逻辑在5个地方有副本
❌ 新人看代码不知道从哪开始
❌ 单元测试覆盖率持续下降
❌ 编译时间越来越长
```

### 7.2 渐进式重构策略

```
不要全面重写！采用绞杀者模式（Strangler Fig Pattern）：
1. 保持旧代码运行
2. 新功能用新架构实现
3. 逐步将旧代码迁移到新架构
4. 旧代码自然消亡

具体步骤：
Step 1: 建立新的基础设施（事件系统、资源系统）
Step 2: 新功能全部走新架构
Step 3: 高频修改的旧模块优先重构
Step 4: 测试覆盖保驾护航
```

## 八、面试常见架构题

### Q1：如何设计一个可扩展的技能系统？

**答：** 数据驱动 + 组合模式
- 技能配置化（JSON/Excel），策划可配置
- 技能由多个Effect组成（数据组合）
- 每种Effect对应一个Handler（代码扩展）
- 新增效果类型只需添加新Handler，O/C原则

### Q2：如何防止游戏中的循环引用导致内存泄漏？

**答：** 
1. UI关闭时取消所有事件订阅
2. 使用弱引用（WeakReference）处理缓存
3. 对象池回收时重置所有引用
4. 使用Profile工具定期检查内存增长

### Q3：如何设计热更新架构？

**答：** 
1. 资源热更：AssetBundle/Addressables增量包
2. 代码热更：HybridCLR（C#代码热更）或xLua（Lua逻辑层）
3. 配置热更：配置文件独立打包，可单独更新
4. 版本管理：MD5校验，断点续传

## 总结

架构设计没有银弹，但有原则：
- **SOLID原则**：每个模块职责单一，对扩展开放对修改关闭
- **依赖倒置**：高层模块依赖抽象，不依赖具体实现
- **数据驱动**：把变化的东西（游戏数值、行为）从代码中分离到数据
- **渐进演化**：不追求一开始就完美，随业务发展持续重构

**技术负责人的架构职责：** 不是设计完美架构，而是保证架构始终处于可维护状态，并随着项目规模扩大不断演进。
