## 为什么游戏需要热更新

移动游戏上线后面临的现实：
- iOS 审核周期 1~3 天，安卓各渠道审核 1~7 天
- 线上 Bug 需要立即修复，等不了重新过审
- 活动运营需要快速上线新内容
- 玩家不愿意为小更新下载 200MB 安装包

热更新的本质是：**在不重新发版的情况下，更新游戏内容和逻辑**。

---

## 一、热更新的两大类型

```
代码热更新
  在不重新打包安装的情况下，更新游戏逻辑代码
  实现方案：Lua (xLua/toLua)、HybridCLR、ILRuntime

资源热更新  
  更新游戏素材：图片、音频、模型、配置表等
  实现方案：AssetBundle、Addressables
  
通常两者配合使用
```

## 二、代码热更方案对比

### 2.1 Lua 方案（xLua / toLua）

```
原理：在 C# 中嵌入 Lua 虚拟机，将游戏逻辑写在 Lua 脚本中
     Lua 文件可以像资源一样更新，无需重新打包

优点：
  - 技术成熟，大量商业项目验证（王者荣耀、和平精英均有 Lua 热更）
  - Lua 轻量级，虚拟机小
  - 跨平台，iOS/Android/PC 行为一致

缺点：
  - 需要维护 C#/Lua 双语言体系
  - Lua 性能约为 C# 的 1/3 ~ 1/10（视计算类型）
  - 调试困难（需要配置 IDE 的 Lua 调试支持）
  - 类型不安全，运行时错误增多
  - 代码结构混乱（大型项目 Lua 代码量庞大难以管理）

适用场景：
  - 已有大量 Lua 代码的项目（维护成本高，不建议迁移）
  - 团队有丰富 Lua 经验
```

### 2.2 ILRuntime

```
原理：在 C# 中实现一个 .NET IL 解释器，将需要热更的逻辑编译成独立的 DLL，
     在运行时通过 ILRuntime 解释执行

优点：
  - 使用 C# 编写热更代码，无需学习新语言
  - 代码结构与主工程一致

缺点：
  - 开发维护趋于停滞（已被 HybridCLR 替代）
  - 解释执行，性能较差
  - 与 Unity 主工程的代码互操作复杂（适配器、CLR 绑定）

适用场景：
  - 不建议新项目采用，HybridCLR 是更好的选择
```

### 2.3 HybridCLR（推荐）

```
原理：对 Unity 的 IL2CPP 运行时进行扩展
     原 IL2CPP 将 C# 编译为 C++ 再编译为机器码（AOT 编译）
     HybridCLR 在此基础上增加了 IL 解释器
     部分程序集可以在运行时动态加载解释执行

特点：
  ✅ 纯 C# 开发，无需双语言
  ✅ 性能接近 AOT（热执行路径 JIT 优化）
  ✅ 对 Unity 各版本适配完整
  ✅ 商业项目大量使用（已有数百款游戏上线）
  ✅ 支持所有 C# 特性（反射、泛型等）
```

### 2.4 HybridCLR 工作原理深解

```
IL2CPP 正常流程：
C# 源码 → IL 字节码 → IL2CPP 工具 → C++ 代码 → 机器码

HybridCLR 扩展流程：
主要程序集（不热更）：正常走 AOT 路径 → 机器码（高性能）
热更程序集：保留为 IL 字节码 → 运行时动态加载 → HybridCLR 解释执行

关键技术：
1. 元数据补充（Supplementary Metadata）
   AOT 泛型的实例化在编译期确定，热更代码无法直接使用 AOT 泛型
   解决：加载原始 AOT 程序集的元数据，让解释器知道 AOT 类型信息

2. 解释执行优化
   频繁执行的代码段通过内联等优化提升性能
   性能约为纯 AOT 的 1/3 ~ 1/2（可接受）
```

    }

// 4. 热更代码加载
public class HotUpdateLoader : MonoBehaviour
{
    async UniTaskVoid Start()
    {
        // 加载补充元数据（解决 AOT 泛型限制）
        await LoadAOTMetadata();
        
        // 加载热更 DLL
        await LoadHotUpdateDll();
        
        // 进入游戏
        SceneManager.LoadScene("MainScene");
    }
    
    async UniTask LoadAOTMetadata()
    {
        // 需要补充元数据的 AOT 程序集列表
        var aotAssemblies = new[]
        {
            "mscorlib.dll",
            "System.dll", 
            "System.Core.dll",
            "UnityEngine.CoreModule.dll",
        };
        
        foreach (var assembly in aotAssemblies)
        {
            // 从 Addressables/AB 加载
            var handle = Addressables.LoadAssetAsync<TextAsset>($"AOT/{assembly}");
            await handle;
            
            if (handle.Status == AsyncOperationStatus.Succeeded)
            {
                // 加载补充元数据
                var err = RuntimeApi.LoadMetadataForAOTAssembly(
                    handle.Result.bytes, 
                    HomologousImageMode.SuperSet);
                    
                if (err != LoadImageErrorCode.OK)
                    Debug.LogError($"Failed to load AOT metadata: {assembly}, error: {err}");
            }
            
            Addressables.Release(handle);
        }
    }
    
    async UniTask LoadHotUpdateDll()
    {
        // 加载热更 DLL
        var handle = Addressables.LoadAssetAsync<TextAsset>("HotUpdate/GameLogic.dll.bytes");
        await handle;
        
        if (handle.Status == AsyncOperationStatus.Succeeded)
        {
            // 加载程序集
            Assembly hotUpdateAssembly = Assembly.Load(handle.Result.bytes);
            
            // 入口点调用
            Type entryType = hotUpdateAssembly.GetType("GameLogic.GameEntry");
            MethodInfo startMethod = entryType.GetMethod("Start");
            startMethod.Invoke(null, null);
        }
        
        Addressables.Release(handle);
    }
}

## 三、资源热更系统设计

### 3.1 Addressables 资源热更流程

```
资源热更的核心流程：

开发阶段：
  1. 将资源标记为 Addressable
  2. 分组（Group）：按更新频率分组
     - BuiltIn：随安装包（永远不热更）
     - Default：常规热更资源
     - Event：活动专用资源（活动结束可删除）

构建阶段：
  1. Build Addressables → 生成 AssetBundle
  2. 生成 catalog.json（记录所有资源的地址和 Hash）
  3. 上传到 CDN

运行时：
  1. 下载远端 catalog
  2. 对比本地 catalog（Hash 不同的需要更新）
  3. 下载差异资源
  4. 更新本地 catalog
  5. 加载资源（透明地从本地缓存或远端读取）
```

### 3.2 完整的资源热更管理器

```csharp
/// <summary>
/// 基于 Addressables 的资源热更管理器
/// </summary>
public class AssetUpdateManager
{
    public enum UpdateState
    {
        Idle,
        CheckingUpdate,
        Downloading,
        Complete,
        Error
    }
    
    public UpdateState State { get; private set; } = UpdateState.Idle;
    public float DownloadProgress { get; private set; }
    public long TotalDownloadSize { get; private set; }
    public long DownloadedSize { get; private set; }
    
    public event Action<UpdateState> OnStateChanged;
    public event Action<string> OnError;
    
    /// <summary>
    /// 完整的热更新流程
    /// </summary>
    public async UniTask<bool> CheckAndUpdateAsync(CancellationToken ct)
    {
        try
        {
            SetState(UpdateState.CheckingUpdate);
            
            // Step 1: 初始化 Addressables
            var initHandle = Addressables.InitializeAsync();
            await initHandle.WithCancellation(ct);
            
            // Step 2: 检查 Catalog 更新
            var checkHandle = Addressables.CheckForCatalogUpdates(false);
            await checkHandle.WithCancellation(ct);
            
            var catalogsToUpdate = checkHandle.Result;
            Addressables.Release(checkHandle);
            
            if (catalogsToUpdate == null || catalogsToUpdate.Count == 0)
            {
                Debug.Log("[HotUpdate] 没有需要更新的 Catalog");
                SetState(UpdateState.Complete);
                return true;
            }
            
            // Step 3: 更新 Catalog
            var updateCatalogHandle = Addressables.UpdateCatalogs(catalogsToUpdate, false);
            await updateCatalogHandle.WithCancellation(ct);
            
            var updatedLocators = updateCatalogHandle.Result;
            Addressables.Release(updateCatalogHandle);
            
            // Step 4: 获取下载大小
            var keys = GetAllUpdateKeys(updatedLocators);
            var sizeHandle = Addressables.GetDownloadSizeAsync(keys);
            await sizeHandle.WithCancellation(ct);
            
            long downloadSize = sizeHandle.Result;
            Addressables.Release(sizeHandle);
            
            if (downloadSize == 0)
            {
                Debug.Log("[HotUpdate] 资源已是最新");
                SetState(UpdateState.Complete);
                return true;
            }
            
            TotalDownloadSize = downloadSize;
            Debug.Log($"[HotUpdate] 需要下载: {FormatSize(downloadSize)}");
            
            SetState(UpdateState.Downloading);
            
            // Step 5: 下载资源
            var downloadHandle = Addressables.DownloadDependenciesAsync(keys, 
                Addressables.MergeMode.Union, false);
            
            while (!downloadHandle.IsDone)
            {
                DownloadProgress = downloadHandle.PercentComplete;
                DownloadedSize = (long)(downloadSize * DownloadProgress);
                await UniTask.NextFrame(ct);
            }
            
            if (downloadHandle.Status != AsyncOperationStatus.Succeeded)
            {
                Addressables.Release(downloadHandle);
                throw new Exception("资源下载失败");
            }
            
            Addressables.Release(downloadHandle);
            
            SetState(UpdateState.Complete);
            return true;
        }
        catch (OperationCanceledException)
        {
            Debug.Log("[HotUpdate] 热更被取消");
            SetState(UpdateState.Idle);
            return false;
        }
        catch (Exception e)
        {
            Debug.LogError($"[HotUpdate] 热更失败: {e.Message}");
            OnError?.Invoke(e.Message);
            SetState(UpdateState.Error);
            return false;
        }
    }
    
    private void SetState(UpdateState state)
    {
        State = state;
        OnStateChanged?.Invoke(state);
    }
    
    private List<object> GetAllUpdateKeys(List<IResourceLocator> locators)
    {
        var keys = new List<object>();
        foreach (var locator in locators)
            keys.AddRange(locator.Keys);
        return keys;
    }
    
    private string FormatSize(long bytes)
    {
        if (bytes < 1024) return $"{bytes}B";
        if (bytes < 1024 * 1024) return $"{bytes / 1024f:F1}KB";
        return $"{bytes / (1024f * 1024f):F1}MB";
    }
}
```

### 3.3 版本管理策略

```csharp
/// <summary>
/// 游戏版本管理：管理强更、热更、资源更新的关系
/// </summary>
[Serializable]
public class VersionInfo
{
    public string AppVersion;        // 安装包版本 "1.0.0"
    public string ResVersion;        // 资源版本 "1.0.0.100"
    public string MinRequiredApp;    // 最低兼容的安装包版本
    public string UpdateUrl;         // 下载更新的地址
    public string ChangeLog;         // 更新内容
    public bool ForceUpdate;         // 是否强制更新
}

public class VersionManager
{
    private const string VERSION_URL = "https://your-cdn.com/version.json";
    
    public async UniTask<UpdateDecision> CheckVersionAsync(CancellationToken ct)
    {
        // 从 CDN 获取最新版本信息
        var response = await UnityWebRequest
            .Get(VERSION_URL)
            .SendWebRequest()
            .WithCancellation(ct);
        
        var serverVersion = JsonUtility.FromJson<VersionInfo>(response.downloadHandler.text);
        var currentVersion = Application.version;
        
        // 判断是否需要强制更新安装包
        if (CompareVersion(currentVersion, serverVersion.MinRequiredApp) < 0)
        {
            return new UpdateDecision 
            { 
                Type = UpdateType.ForceApp,
                DownloadUrl = serverVersion.UpdateUrl,
                Message = "发现重要更新，请更新到最新版本以继续游戏"
            };
        }
        
        // 判断是否有新版本（可选更新）
        if (CompareVersion(currentVersion, serverVersion.AppVersion) < 0)
        {
            return new UpdateDecision 
            { 
                Type = UpdateType.OptionalApp,
                DownloadUrl = serverVersion.UpdateUrl,
                Message = $"发现新版本 {serverVersion.AppVersion}\n{serverVersion.ChangeLog}"
            };
        }
        
        // 只需要热更资源
        return new UpdateDecision { Type = UpdateType.HotUpdate };
    }
    
    // 版本比较 "1.2.3" vs "1.2.4"
    private int CompareVersion(string v1, string v2)
    {
        var parts1 = v1.Split('.').Select(int.Parse).ToArray();
        var parts2 = v2.Split('.').Select(int.Parse).ToArray();
        
        int len = Math.Max(parts1.Length, parts2.Length);
        for (int i = 0; i < len; i++)
        {
            int a = i < parts1.Length ? parts1[i] : 0;
            int b = i < parts2.Length ? parts2[i] : 0;
            if (a != b) return a.CompareTo(b);
        }
        return 0;
    }
}

public enum UpdateType { None, HotUpdate, OptionalApp, ForceApp }
public struct UpdateDecision { public UpdateType Type; public string DownloadUrl; public string Message; }
```

## 四、热更新的完整启动流程

### 4.1 启动流程设计

```
App 启动
  ↓
版本检查（VersionManager）
  ├── 需要强更安装包 → 跳转应用商店
  ├── 有可选更新 → 弹窗提示（可跳过）
  └── 只需热更 → 继续
  ↓
资源热更（AssetUpdateManager）
  ├── 检查 Catalog 更新
  ├── 计算下载量（告知玩家）
  ├── 下载并缓存（断点续传）
  └── 完成
  ↓
代码热更（HotUpdateLoader）
  ├── 加载 AOT 元数据补充
  ├── 加载热更 DLL
  └── 进入游戏
```

### 4.2 热更 UI 界面实现

```csharp
/// <summary>
/// 热更新加载界面控制器
/// </summary>
public class HotUpdateUI : MonoBehaviour
{
    [SerializeField] private Slider _progressBar;
    [SerializeField] private Text _statusText;
    [SerializeField] private Text _sizeText;
    [SerializeField] private GameObject _updatePanel;
    [SerializeField] private Button _updateButton;
    [SerializeField] private Button _skipButton;
    
    private CancellationTokenSource _cts;
    
    async void Start()
    {
        _cts = new CancellationTokenSource();
        
        // 检查版本
        var versionManager = new VersionManager();
        var decision = await versionManager.CheckVersionAsync(_cts.Token);
        
        switch (decision.Type)
        {
            case UpdateType.ForceApp:
                ShowForceUpdatePanel(decision);
                return; // 必须更新，不继续
                
            case UpdateType.OptionalApp:
                bool userWantsUpdate = await ShowOptionalUpdatePanel(decision);
                if (userWantsUpdate)
                {
                    Application.OpenURL(decision.DownloadUrl);
                    return;
                }
                break;
        }
        
        // 进行热更
        await DoHotUpdate();
    }
    
    private async UniTask DoHotUpdate()
    {
        var updateManager = new AssetUpdateManager();
        
        // 订阅进度
        updateManager.OnStateChanged += OnStateChanged;
        
        // 开始更新
        bool success = await updateManager.CheckAndUpdateAsync(_cts.Token);
        
        if (!success)
        {
            ShowRetryPanel();
            return;
        }
        
        // 加载热更 DLL
        _statusText.text = "正在加载游戏...";
        var loader = new HotUpdateLoader();
        await loader.LoadAsync(_cts.Token);
        
        // 进入主游戏
        SceneManager.LoadScene("MainMenu");
    }
    
    private void OnStateChanged(AssetUpdateManager.UpdateState state)
    {
        switch (state)
        {
            case AssetUpdateManager.UpdateState.CheckingUpdate:
                _statusText.text = "正在检查更新...";
                _progressBar.value = 0;
                break;
                
            case AssetUpdateManager.UpdateState.Downloading:
                _statusText.text = "正在更新资源...";
                break;
                
            case AssetUpdateManager.UpdateState.Complete:
                _statusText.text = "更新完成";
                _progressBar.value = 1;
                break;
        }
    }
    
    void OnDestroy() => _cts?.Cancel();
}
```

## 五、常见问题与解决方案

### 5.1 iOS 对热更代码的限制

```
Apple 审核规则：
  - 不允许下载可执行代码（机器码）
  - IL（字节码）被认为是"数据"，允许下载和解释执行

HybridCLR 的合规性：
  ✅ 热更的 DLL 是 IL 字节码，不是机器码
  ✅ 已有大量上架 App Store 的游戏使用
  ⚠️  注意：不要下载包含 unsafe/非托管代码的 DLL
  ⚠️  注意：AppStore 政策会变化，上线前再次确认
  
规避风险的额外措施：
  - 对 DLL 进行加密和混淆
  - 服务端做版本校验（防止绕过审核后强推高风险更新）
```

### 5.2 HybridCLR 的 AOT 泛型问题

```csharp
// 热更代码中使用了 AOT 中没有实例化的泛型 → 运行时报错
// 错误信息类似：ExecutionEngineException: Attempting to JIT compile method...

// 解决方案 1：在 AOT 代码中预先实例化
public class AOTHelper
{
    [Preserve] // 防止 IL2CPP 剔除
    static void AOTGenericInstances()
    {
        // 列出所有热更代码会用到的泛型实例化
        new Dictionary<int, GameConfig>();
        new List<BattleUnit>();
        new Queue<PlayerInput>();
        EventBus.Subscribe<DamageEvent>(null);
        // ...
    }
}

// 解决方案 2：使用 HybridCLR 的代码生成工具
// HybridCLR → Generate → AOT Generic Reference
// 工具会自动分析热更代码，生成所需的泛型实例化代码
```

### 5.3 热更分包策略

```
合理的分包策略（避免每次热更都下载大量资源）：

按更新频率分组：
  core/        - 核心资源，只在大版本更新
  ui/          - 界面资源，中频更新
  audio/       - 音效资源，中频更新
  event/       - 活动资源，高频更新（活动结束可清理）
  hotfix/      - Bug 修复资源，高频更新

按加载时机分组：
  startup/     - 启动时必须加载（等待下载完才能进入）
  optional/    - 可以在游戏内异步下载（进入游戏后后台更新）

好处：
  - 玩家只需等待 startup 组下载完即可进入游戏
  - optional 组在后台静默更新
  - 减少每次更新的等待时间
```

## 总结

热更新体系是大型移动游戏工程效能的核心组成部分：

1. **代码热更**：HybridCLR 是目前最优选择，用 C# 写热更代码，体验接近原生
2. **资源热更**：Addressables 提供完整的资源管理和热更框架
3. **版本管理**：设计清晰的版本号策略，区分强更/热更/资源更新
4. **分包策略**：合理分组，减少玩家等待时间

**技术负责人的职责**：建立热更基础设施，制定热更规范，确保热更不会引入新 Bug（热更代码的测试体系同样重要）。
