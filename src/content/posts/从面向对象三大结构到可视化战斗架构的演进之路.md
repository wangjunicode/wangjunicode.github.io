# 从面向对象三大结构到可视化战斗架构的演进之路

任何程序只由三种控制结构组成：**顺序、条件、循环**。这是计算机科学的基本定理。

那么，现代游戏战斗系统那复杂的 FSM + FlowGraph + Timeline 三层架构，究竟是如何从这三个最基础的积木演变而来的？

这篇文章试图还原这条演进路径——不是堆砌知识点，而是追问每一步"为什么要这样变"。

---

## 第一阶段：最朴素的写法

游戏战斗的本质需求就是：玩家按键 → 执行动作 → 判断结果。最直接的实现就是在 `Update()` 里用三种控制结构直接写：

```csharp
void Update() {
    // 条件：检测输入
    if (Input.GetKeyDown(KeyCode.Z)) {
        // 顺序：播动画 → 开判定框 → 算伤害
        PlayAnimation("attack");
        EnableHitbox();
        DealDamage(target, 100);
    }

    // 条件：死亡检测
    if (hp <= 0) Die();

    // 循环：每帧更新所有 buff
    foreach (var buff in buffs) buff.Update();
}
```

代码清晰，逻辑一目了然。这在原型阶段完全够用。

**问题在哪？**

技能一多，`Update()` 就变成几千行的意大利面。更关键的是：每次策划想改一个伤害值，要找程序修改代码、重新发版。**数值配置和逻辑代码混在一起**，没有边界。

---

## 第二阶段：面向对象封装 → Skill 类

自然的重构方向是把"一个技能"抽象成一个对象：

```csharp
class Skill {
    public float damage;    // 可配置
    public float cd;        // 可配置
    
    public void Execute(Unit caster, Unit target) {
        PlayAnimation(caster);
        DealDamage(target, damage);
    }
}
```

把 `damage`、`cd` 这些**数值**从代码里分离出来，写进 Excel 配置表。策划改数值不需要找程序，这是面向对象封装带来的第一个价值：**数据与逻辑分离**。

**新问题是什么？**

数值分离了，但**逻辑本身还是硬编码的**。

"第 8 帧开判定框、命中后根据目标状态决定是否触发破防、未命中则接续衍生技能"——这类需求让每个技能都变成一个特化的子类，覆写 `Execute()` 写满 if/else。技能一多，类爆炸，改一个技能逻辑还是要找程序。

问题的本质没有解决：**逻辑的条件分支和时序控制，还在代码里**。

---

## 第三阶段：状态机 FSM → 把"条件"从代码里解放出来

静下来想，角色行为的核心是什么？

是**状态转移**。角色任意时刻只处于一个状态（待机/攻击/受击/死亡），外部事件驱动状态之间的跳转。这本质上是把散落各处的 `if/else` 收归成一个**状态 × 事件的二维表**：

```csharp
void OnEvent(string evt) {
    switch (currentState) {
        case State.Idle:
            if (evt == "attack_pressed") TransitionTo(State.Attack);
            break;
        case State.Attack:
            if (evt == "hit_landed")  TransitionTo(State.Idle);
            if (evt == "took_hit")    TransitionTo(State.Hit);
            break;
        case State.Hit:
            if (evt == "recover_end") TransitionTo(State.Idle);
            break;
    }
}
```

**获得了什么？**

条件分支有了结构。"角色在攻击状态下被打中会进受击"——这条逻辑现在是一个清晰的表项，而不是深埋在某个 `if` 里。

更重要的是，这张表**可以成为数据**：用工具把状态图画出来，导出配置，策划就能配置"哪个状态响应哪个事件跳到哪里"，不需要改代码。

**还缺什么？**

状态内部是空的。"攻击状态"里，**第几帧做什么事**——这个时序控制无处安放，仍然只能写在代码里。

---

## 第四阶段：Timeline → 把"顺序"从代码里解放出来

程序员写时序逻辑，会用计时器和 coroutine：

```csharp
IEnumerator AttackCoroutine() {
    PlayAnimation("attack");
    yield return new WaitForSeconds(0.1f);  // 等第 8 帧
    EnableHitbox();
    yield return new WaitForSeconds(0.3f);  // 攻击窗口持续
    DisableHitbox();
}
```

这依然是代码。**Timeline 的本质是把"顺序执行 + 时间"显式建模成数据**：

```
Frame: 0────5────10────15────20────25────30
       │    │          │               │
     动画  开启判定框  检测命中事件    关闭判定框
     开始  
             │────────────────────│
             输入窗口（可接续按键）
                        │──────────────────
                        条件衍生检测开始
```

每个时间点挂的是一个 **Clip**（动作片段），Clip 本身是可序列化的数据对象。

**获得了什么？**

策划和 TA 可以直接在编辑器里拖动 Clip，精确到帧地配置技能时序——攻击判定框从第几帧开始、开多长、哪个时间点监听追击输入。**程序员不再需要介入每一个技能的时序调整**。

这是把"顺序"这种控制结构，从 Coroutine 代码变成了可视化数据。

---

## 第五阶段：FlowGraph + 可视化脚本 → 把"复杂条件分支"彻底解放

Timeline 解决了线性时序，但一个技能的执行逻辑往往不是线性的：

> 命中后：
> - 如果目标有"护盾" Buff 且我方有"破盾"状态 → 执行破盾流程
> - 否则 → 普通伤害
>
> 如果未命中 → 衍生技能

这是嵌套的 `if/else`。写代码没问题，但**策划看不懂、改不了**。

**FlowGraph（流图）** 把这个条件路由变成节点连线：

```
[CT_检测命中?]
    ├─ 是 → [CT_CheckBuff: 护盾?]
    │           ├─ 是 → [CT_CheckState: 破盾?]
    │           │           ├─ 是 → [AT_BreakShield]
    │           │           └─ 否 → [AT_NormalDamage]
    │           └─ 否 → [AT_NormalDamage]
    └─ 否 → [GS_SpawnDeriveSkill]
```

每个节点（`CT_` 是条件查询，`AT_` 是执行动作）都是程序封装好的**原子能力**，策划只负责**连线组合**。程序不需要为每种组合写特化代码。

这是把"条件分支"从代码变成了可视化数据。

## 现在的三层架构：演进的自然结果

把上面五个阶段叠在一起：

```
┌─────────────────────────────────────────────────────┐
│                    FSM（状态机层）                    │
│                                                     │
│   Idle ──[attack_pressed]──→ Attack                 │
│   Attack ──[hit_end]──→ Idle                        │
│   Attack ──[took_hit]──→ Hit                        │
│                                                     │
│   每个状态内含一个 FlowGraph                         │
└───────────────────────┬─────────────────────────────┘
                        │ 进入状态 → 启动流图
                        ▼
┌─────────────────────────────────────────────────────┐
│                 FlowGraph（流图层）                   │
│                                                     │
│   [Switch: 输入类型] → [GS_PlayTimeline]             │
│           ↓                                         │
│   [CT_检测命中] ──是→ [AT_伤害计算] → [发送结束事件]  │
│               └─否→ [GS_衍生技能]                   │
└───────────────────────┬─────────────────────────────┘
                        │ GS_PlayTimeline 节点触发
                        ▼
┌─────────────────────────────────────────────────────┐
│                  Timeline（时序层）                   │
│                                                     │
│  0───5───10───15───20───25───30───35───40           │
│      │         │              │                     │
│  开判定框   输入窗口开始    条件衍生检测              │
│      │─────────────│                                │
│           可接续输入                                 │
└─────────────────────────────────────────────────────┘
```

**三种控制结构，各有归宿：**

| 控制结构 | 被谁承载 | 谁来配置 |
|---|---|---|
| **条件**（宏观状态转移） | FSM 状态图 | 策划配置状态节点和跳转事件 |
| **顺序**（帧级时序控制） | Timeline Clip | TA / 策划拖拽 Clip |
| **条件**（技能内分支路由） | FlowGraph 节点连线 | 策划连线组合原子节点 |
| **循环**（每帧驱动） | 引擎 tick，完全透明 | 程序封装，策划无感 |

## 演进的内在逻辑

回头看，这条路径有一个贯穿始终的驱动力：

**每当某种控制结构"被锁在代码里"导致协作摩擦，就把它抽出来变成数据。**

- FSM 之前：条件（状态转移）散落在 `if/else` 里 → 策划改不了
- Timeline 之前：顺序（帧时序）散落在 Coroutine 里 → 策划改不了
- FlowGraph 之前：条件（技能内分支）散落在特化类里 → 策划改不了

每一层的诞生，都是把"程序才能改的逻辑"变成"策划能配置的数据"。

这就是为什么最终形态是这个形状：不是设计出来的，是**被协作需求逼出来的**。

---

## 给程序员的视角转换

理解了这条演进路径，再看这套架构就不会觉得"过于复杂"了：

- **FSM 不是为了好看**，是因为没有它，状态转移条件就散在代码各处，策划无法独立配置
- **Timeline 不是 Animator 的替代品**，是因为没有它，帧级时序只能写在 Coroutine 里，策划无法精确调整
- **FlowGraph 不是"蓝图给策划玩"**，是因为没有它，技能内的条件路由每次都要改代码

每一层都有其不可替代的职责，缺一不可。

---

## 结语

从 `if/else + for + 顺序语句`，到 FSM + FlowGraph + Timeline 三层架构，本质上是同一件事的三次重复：**把控制权从程序手里交给数据，再把数据的编辑权交给策划**。

当你下次看到某个战斗系统"不必要地复杂"时，不妨问一下：这一层是在解决哪种控制结构的协作问题？答案往往比架构图更清晰。
