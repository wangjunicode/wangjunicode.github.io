# 游戏引擎运行时配置与性能采样系统设计

每一个成熟的游戏框架都有一套"全局开关"机制——控制逻辑帧率、时间缩放、调试模式、平台差异等，并配套一个性能采样系统，让开发者可以在不入侵业务代码的前提下做精细化性能分析。本文解析 `EngineDefine`、`EngineRuntime`、`VProfiler` 三个核心组件的设计思路，带你理解这套"框架基础设施"的第一性原理。

---

## 一、EngineDefine：全局静态配置中心

```csharp
public class EngineDefine
{
    // 逻辑帧率
    public const int FPS = 60;
    // 逻辑 timeScale
    public static FP logicTimeScale = FP.One;
    // editor timeScale（仅在编辑器内调速）
    public static FP editorTimeScale = FP.One;
    // 综合 timeScale = 逻辑 × 编辑器
    public static FP timeScale => logicTimeScale * editorTimeScale;
    
    public static FP fixedDeltaTime => fixedDeltaTime_Orignal * timeScale;
    public static FP deltaTime => Game.deltaTime * timeScale;
    
    public static readonly FP fixedDeltaTime_Orignal = (FP)0.016667f;

    // IPv6 优先
    public static bool UseIPv6 = false;
    // UDP 下行包 IP 检测
    public static bool CheckUDPRemoteIP = true;
    
    public static bool HideJumpText = false;
    public static bool HideEditor = false;
    public static bool UseAnimStream = false;
    public static bool DisablePreload = false;
    public static bool DeepProfileMode = false;
    public static bool DisableHurtCollider = true;
    
#if !UNITY_EDITOR && !UNITY_STANDALONE
    public static bool isHybridCLR = true;
#else
    public static bool isHybridCLR = false;
#endif

#if UNITY_EDITOR
    public static bool EnableShelveCommit { ... }
    public const string AssetsMobile = "Assets_Mobile";
    public static bool EnableAssetsMobile = false;
#endif
}
```

### 1.1 为什么用 static class 而非 ScriptableObject

`EngineDefine` 选择纯静态类而非 Unity 的 `ScriptableObject` 配置资产，有几个明确的理由：

1. **零运行时开销**：静态字段直接内存访问，无需 `GetComponent` 或资产加载
2. **跨平台编译隔离**：大量 `#if` 指令在 ScriptableObject 中做不了，静态类天然支持条件编译
3. **热更友好**：HybridCLR 热更代码中可以直接引用，不依赖 Unity Asset System
4. **运行时动态修改**：`static` 字段可在运行中被调试工具、作弊菜单直接修改，做到真正的"运行时开关"

### 1.2 FP 类型：确定性浮点数

```csharp
public const int FPS = 60;
public static FP logicTimeScale = FP.One;
public static readonly FP fixedDeltaTime_Orignal = (FP)0.016667f;
```

`FP` 是 `TrueSync` 库的定点数类型（Fixed-Point），等价于 Q15.16 或类似精度的整数模拟浮点。

**为什么时间相关量都用 FP**：游戏采用帧同步（Lockstep）战斗，所有战斗逻辑运算必须跨平台确定性。浮点数在不同 CPU 架构（x86/ARM）上可能产生微小差异，累积后导致两端状态分歧（desync）。使用 FP 定点数保证所有端、所有平台计算结果完全一致。

### 1.3 双层 timeScale 设计

```csharp
public static FP logicTimeScale = FP.One;  // 战斗逻辑速度（慢动作、加速等）
public static FP editorTimeScale = FP.One; // 编辑器调速（不影响逻辑正确性）
public static FP timeScale => logicTimeScale * editorTimeScale;
```

两层分离的好处：
- `logicTimeScale`：战斗内慢动作演出、时间法术效果修改此值，会影响帧同步逻辑
- `editorTimeScale`：只在编辑器内使用，让策划可以加速/慢放查看逻辑，不影响联网战斗
- 组合计算：最终 `timeScale` = 两者相乘，系统不需要区分来源，统一取 `timeScale` 即可

### 1.4 关键标志位解析

| 标志 | 默认值 | 作用 |
|------|--------|------|
| `UseIPv6` | false | 优先尝试 IPv6 连接，支持 Happy Eyeballs 策略 |
| `CheckUDPRemoteIP` | true | UDP 包 IP 合法性校验，防地址欺骗 |
| `HideJumpText` | false | 关闭战斗跳字（性能测试时减少 UI 开销）|
| `DeepProfileMode` | false | 开启深度 Profile 采样，详见 VProfiler 节 |
| `DisableHurtCollider` | true | 禁用受击碰撞体（默认禁用，避免不必要物理开销）|
| `isHybridCLR` | 平台决定 | 是否运行在 HybridCLR 热更环境 |
| `DisablePreload` | false | 禁用资源预加载（测试/调试用）|

### 1.5 编辑器专属字段

```csharp
#if UNITY_EDITOR
    public static bool EnableShelveCommit { ... } // SVN搁置提交
    public const string AssetsMobile = "Assets_Mobile";
    public static bool EnableAssetsMobile = false; // 移动端资产目录
#endif
```

编辑器字段通过 `#if UNITY_EDITOR` 隔离，确保不进入打包产物。`EnableShelveCommit` 甚至持久化到 `PlayerPrefs`，跨编辑器会话保持开发者个人偏好。

## 二、EngineRuntime：战斗确定性随机运行时

```csharp
public class EngineRuntime
{
    public static bool Pause;
    public static int Seed;

    private TSRandom m_random;
    public TSRandom random => m_random;

    public EngineRuntime(int seed)
    {
        Seed = seed;
        m_random = TSRandom.New(seed);
    }

    public void SetRandomSeed(int seed)
    {
        Seed = seed;
        m_random = TSRandom.New(Seed);
    }

    public int GetRandomTime()
    {
        return m_random.CallTime;
    }
}
```

### 2.1 EngineRuntime 的核心职责

`EngineRuntime` 不是单例，而是按需创建、由战斗场景持有的运行时状态容器。它管理两样东西：

1. **暂停状态**：`Pause` 控制全局逻辑帧是否推进
2. **确定性随机数生成器**：`TSRandom` 是 TrueSync 的确定性随机，同 Seed 同调用序列必然产生相同序列

### 2.2 为什么需要确定性随机

帧同步战斗中，如果两端使用 `Random.Range()` 等 Unity 内置随机，由于系统随机状态不同，会产生不同结果，导致双端状态分歧（desync）。

```
端A: Random.Next() = 42   → 怪物移动方向：北
端B: Random.Next() = 71   → 怪物移动方向：东
→ 两端怪物位置分叉，战斗完全错误
```

`TSRandom` 的实现方式：使用相同 Seed 初始化，每次调用按固定算法步进，相同调用次序必然产生相同结果。

```
两端: TSRandom(seed=12345).Next() = 42（确定）→ 怪物移动方向：北（一致）
```

### 2.3 Seed 的管理流程

```
对战房间匹配完成
  → 服务器生成 Seed（或由双端哈希协商）
  → 战斗开始时通过网络下发 Seed
  → new EngineRuntime(seed)
  → 双端 TSRandom 状态对齐
  → 整场战斗的随机数序列完全相同
```

`GetRandomTime()` 返回 `m_random.CallTime`（已调用次数），用于调试时验证两端随机器调用次数是否同步——若不同步，必有逻辑分支差异（desync 排查关键指标）。

### 2.4 SetRandomSeed 的用途

```csharp
public void SetRandomSeed(int seed)
{
    Seed = seed;
    m_random = TSRandom.New(Seed);
}
```

重新设置 Seed 有两个主要用途：
1. **录像回放**：用同一个 Seed 重建 `EngineRuntime`，战斗录像完全可以本地重放
2. **回滚测试**：某些 Rollback Netcode 场景下，需要将随机器回滚到某一帧的状态

### 2.5 Pause 标志的作用

```csharp
public static bool Pause;
```

`Pause` 是 `static` 的——所有持有 `EngineRuntime` 的对象都共享同一个暂停状态，相当于全局广播。帧循环驱动层每帧开始时检查 `Pause`：

```csharp
void FixedUpdate()
{
    if (EngineRuntime.Pause) return;
    // 推进帧逻辑
}
```

这让暂停逻辑非常轻量，无需发布-订阅事件，不产生额外分配。

## 三、VProfiler：虚拟性能采样包装器

```csharp
public class VProfiler
{
    public static Action<string> HookBeginSample = null;
    public static Action HookEndSample = null;
    
    [Conditional("ENABLE_SAMPLE")]
    public static void BeginSample(string name)
    {
        if (HookBeginSample != null)
        {
            HookBeginSample(name);
            return;
        }
#if ONLY_CLIENT
        Profiler.BeginSample(name);
#endif
    }
    
    [Conditional("ENABLE_SAMPLE")]
    public static void EndSample()
    {
        if (HookEndSample != null)
        {
            HookEndSample();
        }
#if ONLY_CLIENT
        Profiler.EndSample();
#endif
    }
    
    [Conditional("ENABLE_SAMPLE")]
    public static void BeginDeepSample(string name)
    {
        if (EngineDefine.DeepProfileMode)
        {
            // 同 BeginSample，但只在 DeepProfileMode 下激活
        }
    }
    // ...
}
```

### 3.1 为什么不直接用 Profiler.BeginSample

Unity 的 `Profiler.BeginSample / EndSample` 有以下局限：

1. **只存在于 Unity 引擎层**：服务器端、纯 C# 测试环境中无法调用
2. **发布包仍有开销**：即使 Profiler 未连接，调用本身也有少量 CPU 开销
3. **无法扩展**：无法接入自定义性能分析后端（如自研性能监控平台、网络上报等）

`VProfiler` 解决了这三个问题：

### 3.2 `[Conditional("ENABLE_SAMPLE")]` 的原理

```csharp
[Conditional("ENABLE_SAMPLE")]
public static void BeginSample(string name)
```

`System.Diagnostics.ConditionalAttribute` 是 C# 的编译指令：**当调用侧编译环境没有定义 `ENABLE_SAMPLE` 宏时，所有对此方法的调用会在编译期被完整删除**（而非运行时判断跳过）。

```csharp
// 项目未定义 ENABLE_SAMPLE 时，编译后相当于这一行完全不存在
VProfiler.BeginSample("MyFunction");
```

这意味着发布包中完全没有 Profiler 相关代码，零性能开销，无需手动 `#if ENABLE_SAMPLE` 包裹每一处调用。

对比普通 `#if`：

```csharp
// 方式1：每处调用都要包 #if（繁琐，容易漏）
#if ENABLE_SAMPLE
Profiler.BeginSample("xxx");
#endif

// 方式2：VProfiler 封装（干净，集中控制）
VProfiler.BeginSample("xxx"); // 未定义 ENABLE_SAMPLE 时编译期消除
```

### 3.3 Hook 机制：可插拔后端

```csharp
public static Action<string> HookBeginSample = null;
public static Action HookEndSample = null;
```

两个静态 Action 允许任何代码注入自定义采样后端：

```csharp
// 接入自研性能监控
VProfiler.HookBeginSample = (name) => 
{
    MyPerfMonitor.BeginSpan(name);
};
VProfiler.HookEndSample = () =>
{
    MyPerfMonitor.EndSpan();
};

// 或在单元测试中注入 mock
VProfiler.HookBeginSample = (name) => testLog.Add($"BEGIN:{name}");
```

Hook 优先级高于 `Profiler.BeginSample`——一旦设置 Hook，Unity 原生 Profiler 不再调用，完全由自定义逻辑接管。

### 3.4 DeepSample：按需开启的深度采样

```csharp
[Conditional("ENABLE_SAMPLE")]
public static void BeginDeepSample(string name)
{
    if (EngineDefine.DeepProfileMode)
    {
        // 采样
    }
}
```

普通 `BeginSample` 在 `ENABLE_SAMPLE` 下始终采样。`BeginDeepSample` 在此基础上再加一层 `DeepProfileMode` 开关：

- **普通 Profile 模式**：只记录关键路径（BeginSample 处）
- **深度 Profile 模式**：额外记录所有 BeginDeepSample 处，包括热路径细节

这样设计是因为深度采样会产生大量数据，日常开发只开普通模式；当发现特定系统有问题时，临时开启 `DeepProfileMode` 放大该区域细节。

### 3.5 线程 Profile 支持

```csharp
[Conditional("ENABLE_SAMPLE")]
public static void BeginThreadProfiling(string threadname, string name)
{
#if ONLY_CLIENT
    Profiler.BeginThreadProfiling(threadname, name);
#endif
}
```

Unity Profiler 默认只追踪主线程。`BeginThreadProfiling` 将子线程的 CPU 耗时也纳入 Profiler 时间线，对分析 Job System、资源加载线程等至关重要。

## 四、三者的协作关系

```
游戏启动
  │
  ├─ EngineDefine.FPS = 60          → 帧循环驱动层读取，决定 fixedDeltaTime
  ├─ EngineDefine.DeepProfileMode   → VProfiler.BeginDeepSample 开关
  │
  ├─ new EngineRuntime(seed)        → 战斗随机器初始化
  │    └─ EngineRuntime.Pause       → 帧循环 FixedUpdate 是否推进
  │
  └─ VProfiler.BeginSample          → 性能采样（ENABLE_SAMPLE 宏控制）
       ├─ Hook 注入 → 自定义后端
       └─ Profiler.BeginSample → Unity Profiler
```

### 4.1 典型使用模式

```csharp
// 帧循环驱动（伪代码）
void FixedUpdate()
{
    if (EngineRuntime.Pause) return;
    
    VProfiler.BeginSample("FrameUpdate");
    
    // 所有系统逻辑 Update
    foreach (var system in updateSystems)
    {
        VProfiler.BeginDeepSample(system.Name);
        system.Update(EngineDefine.fixedDeltaTime);
        VProfiler.EndDeepSample();
    }
    
    VProfiler.EndSample();
}

// 随机数使用（战斗逻辑内）
void RollDamage(EngineRuntime runtime)
{
    int variance = runtime.random.Range(0, 10); // 确定性随机
    int damage = baseDamage + variance;
}
```

## 五、工程实践建议

### 5.1 EngineDefine 的扩展原则

- **只放框架级全局开关**，业务逻辑的开关放在各自的 Config/Manager 里
- **FP 类型贯穿战斗层**，渲染/UI 层仍可用 `float`（渲染不需要跨端确定性）
- 添加新开关时附上注释说明用途和默认值，避免"神秘标志"

### 5.2 确定性随机的陷阱

```csharp
// ❌ 错误：混用 Unity 随机和确定性随机
int dmg = Random.Range(10, 20);  // Unity 随机，不确定
int dodge = runtime.random.Range(0, 100); // 确定性

// ✅ 正确：战斗逻辑全部走确定性随机
int dmg = runtime.random.Range(10, 20);
int dodge = runtime.random.Range(0, 100);
```

**黄金规则**：所有影响战斗状态的逻辑，随机数必须来自 `EngineRuntime.random`，不得使用 `UnityEngine.Random`、`System.Random` 等非确定性来源。

### 5.3 VProfiler 接入指南

```csharp
// 开发阶段：接入 Unity Profiler（定义 ENABLE_SAMPLE + ONLY_CLIENT）
// 测试阶段：注入自研监控
VProfiler.HookBeginSample = CustomMonitor.Begin;
VProfiler.HookEndSample = CustomMonitor.End;

// 标注采样点的原则：
// - 主循环的每个 System 更新
// - 资源加载的关键阶段
// - 渲染提交前后
// - 不要在高频小函数内标注（采样本身有开销）
```

## 六、总结

| 组件 | 核心价值 |
|------|---------|
| **EngineDefine** | 全局静态配置，条件编译隔离平台差异，FP 定点数保证时间量确定性，双层 timeScale 分离逻辑与编辑器调速 |
| **EngineRuntime** | 战斗运行时状态容器，TSRandom 确定性随机保证帧同步双端一致，Seed 机制支持录像回放与回滚 |
| **VProfiler** | Conditional 编译期消除零开销，Hook 机制可插拔采样后端，DeepSample 按需放大细节，主/子线程均可采样 |

这三个组件的共同哲学是：**"在正确的地方抽象，在正确的地方关闭"**——EngineDefine 抽象全局开关、EngineRuntime 封装确定性状态、VProfiler 用编译期技巧让性能工具在发布包中完全消失。这正是成熟框架基础设施的设计之道。
