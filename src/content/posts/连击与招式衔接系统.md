# 11 连击与招式衔接系统

> 本文介绍战斗中的连击（Combo）机制设计，包括连击链的构建、招式衔接的条件判断、PoseDirection 姿势方向对连击的影响，以及连击系统与 FSM/技能的协作方式。

---

## 1. 系统概述

连击（Combo）系统让玩家可以按照一定规律连续发出多段攻击，形成流畅的招式衔接。本项目的连击系统具备以下特点：
- **多段子技能（SubSkill）**：一个技能可以包含多段连击动作
- **姿势方向（PoseDirection）**：当前招式结束后的角色姿态影响可接的下一招
- **输入打断（AbortInput）**：已被打断的输入不会在恢复后残留误触发
- **前摇预输入（PreCast）**：在当前招式后摇期间可以预先输入下一招，系统在合适时机自动执行
- **连击破坏（Combo Break）**：受击时连击重置，触发专属事件通知

### 1.1 连击的核心数据

```
连击相关核心字段（SkillComponent）
├── CurrentSubSkillIndex  ── 当前连击段落（第几段）
├── PoseDirection         ── 当前招式的姿势朝向
├── CurMoveSkillType      ── 当前的移动技能类型（影响衔接）
├── AbortInput            ── 被打断时记录的输入（防误触发）
├── AbortInputCnt         ── 各输入的打断计数
├── bInPreCast            ── 是否在前摇等待
└── RestPreCastDelay      ── 剩余前摇等待时间
```

## 2. 架构设计

### 2.1 子技能（SubSkill）链与 CurrentSubSkillIndex

技能（Skill）由多段子技能（SubSkill）组成，`CurrentSubSkillIndex` 追踪当前执行到哪段：

```
技能 ID 1001（普通连击）
├── SubSkill[0]（第一段）：左挥
│    ├── 条件：任意时刻可触发
│    ├── 时间窗口：[8~12帧] 可输入下一段
│    └── 结束姿势：PoseDirection.Right（向右的姿势）
│
├── SubSkill[1]（第二段）：右挥
│    ├── 条件：上一段姿势为 PoseDirection.Right
│    ├── 时间窗口：[7~11帧] 可输入下一段
│    └── 结束姿势：PoseDirection.Left
│
└── SubSkill[2]（第三段）：重击
     ├── 条件：上一段姿势为 PoseDirection.Left 或 PoseDirection.Right
     └── 结束姿势：PoseDirection.Forward（冲刺后前倾）
```

### 2.2 连击输入时间窗口

连击衔接不是随时都能触发的，而是有特定的**输入时间窗口**。在技能 Timeline 的后半段（攻击帧之后、技能完全结束之前），有一段可以接受连击输入的窗口：

```
技能 Timeline（20 帧总长）：
│
├─ [0~3帧]  前摇（不接受连击输入）
├─ [4~8帧]  攻击帧（攻击框激活）
├─ [8~14帧] 可接受连击输入窗口 ← 玩家在此期间输入下一招
└─ [14~20帧] 后摇（连击输入已确认，等待执行）
```

技术实现：通过 `CLIP_UniHurt` 的 `hurtAtkDelayFrames` 等字段控制窗口边界，由 Timeline Handler 动态设置 `AbortInput`。

### 2.3 PoseDirection —— 姿势方向衔接

`EPoseDirection` 决定当前招式结束后角色的动作姿态，影响下一段连击的动画选择：

```csharp
public enum EPoseDirection
{
    None,       // 无特定姿势
    Forward,    // 向前（冲刺后）
    Back,       // 向后（回撤后）
    Left,       // 向左（左挥后）
    Right,      // 向右（右挥后）
    Up,         // 向上（上挑后）
    Down,       // 向下（下砸后）
    Air,        // 空中
    Crouch,     // 蹲伏
}
```

**为什么需要姿势方向？**

考虑这种情况：剑客的普攻连击是"左挥→右挥→前刺"。如果不区分方向，玩家可以"左挥→左挥→左挥"无限连击，动画会不断重复同一个动作，看起来很奇怪且不真实。

通过 PoseDirection 约束：
- 第一段左挥结束 → 姿势变为 Right
- 第二段要求接收 Right 姿势 → 只有左挥后才能接到第二段
- 如果玩家在第二段完成后没有输入，姿势重置，连击链断开

### 2.4 连击破坏（Combo Break）

当连击期间角色受到攻击，连击链被打断：

```
连击进行中（SubSkillIndex = 1，第二段攻击帧）
    │
    └→ 被命中 → EHurtResult.Stiff 受击
         │
         └→ FSM 切换到 StiffState（僵直）
              │
              └→ SkillComponent 重置：
                   ├── CurrentSkill = null（技能停止）
                   ├── CurrentSubSkillIndex = 0（重置连击段数）
                   ├── PoseDirection = None
                   └── AbortInput 清空
              │
              └→ 触发 BPDef.ComboBreakEvtID = 1011（连击中断事件）
                   └→ Buff/被动技能响应（如"连击中断时触发特殊效果"）
```

### 2.5 AbortInput —— 打断输入管理

`AbortInput` 是一个精妙的设计，解决了"技能被打断后残留输入"的问题：

```
问题场景：
玩家正在连击第二段，同时按下了第三段的输入键
技能被打断（受击），进入受击状态
受击结束后，系统检测到"有第三段的输入"
→ 错误地触发了第三段！（玩家实际上没有这个意图）

解决方案（AbortInput）：
技能被打断时，记录当前未处理的输入到 AbortInput
被打断的输入在 AbortInput 中标记为"需要消耗"
下次检查连击条件时，先消耗 AbortInput 中的输入
消耗完之后，该输入不再有效触发连击
```

```csharp
// SkillComponent 中的 AbortInput 管理
public HashSet<EInputKey> AbortInput { get; } = new HashSet<EInputKey>();
public Dictionary<EInputKey, int> AbortInputCnt = new Dictionary<EInputKey, int>();

// 当技能被打断时：
// AbortInput.Add(EInputKey.SkillA);   // 记录被打断时的输入
// AbortInputCnt[EInputKey.SkillA]++;  // 记录打断次数
```

## 4. 连击流程详解

### 4.1 连击触发的完整流程

```
玩家输入 SkillA
    │
    └→ FSMComponent.TryManualCondition(EInputKey.NormalAtk)
         │
         └→ FSM 切换到 CustomSkill 状态
              │
              └→ GS_TryUseSkill(SkillId=1001)
                   │
                   └→ 检查 canUse（CD / AP / 距离）
                        │
                        └→ 满足 → 执行 onUse
                             │
                             └→ 根据 CurrentSubSkillIndex 选择子技能段
                                  │
                                  ├─ Index=0 → 播放 SubSkill[0] 的 Timeline
                                  ├─ Index=1 → 播放 SubSkill[1] 的 Timeline
                                  └─ Index=2 → 播放 SubSkill[2] 的 Timeline
                                  │
                                  └→ 进入连击时间窗口（CLIP_SetComboWindow 激活）
                                       │
                                       └→ 玩家再次输入 SkillA？
                                            ├─ 是 + 窗口内 → CurrentSubSkillIndex++
                                            │                  → 连击继续（下一段）
                                            └─ 否 / 窗口外 → 技能完成，Index 重置为 0
```

### 4.2 连击超时重置

```
每段技能都有连击等待超时：
GS_TryUseSkill 的 _successComplete 输出
    └→ GS_Timer（等待 N 帧 / N 秒）
         └→ 超时 → CurrentSubSkillIndex = 0（重置连击）
              + PoseDirection = None（重置姿势）
              + FSM 返回 Idle
```

### 4.3 换人技能对连击的影响

本项目有独特的**换人连击**机制（`EChangeSkillType.Atk/Def`）：

```
攻击换人（EChangeSkillType.Atk）：
    当前队员技能中途触发换人
    → 新队员接管，继续当前连击链
    → CurrentSubSkillIndex 保持（不重置）
    → 连击流畅延续到新角色

防守换人（EChangeSkillType.Def）：
    被攻击时触发换人（使用防守角色）
    → 旧队员退场，新队员进入受击状态
    → 连击链中断
    → CurrentSubSkillIndex = 0
```

## 5. 设计亮点

### 5.1 SubSkill 机制的极简实现
通过单个整数 `CurrentSubSkillIndex` 追踪连击段落，在技能图（SkillGraph）中通过 Switch 节点分流到不同的子技能 Timeline。逻辑简单但极具表达力，策划只需在技能图里增加分支即可新增连击段数。

### 5.2 PoseDirection 使连击动画更自然
姿势方向约束确保了连击动画的物理合理性——技能结束时的身体朝向决定下一段起始动画，避免了"右挥→右挥→右挥"这种视觉上不合理的循环，提升了战斗表现的质感。

### 5.3 AbortInput 的精确打断管理
AbortInput 机制解决了输入预缓冲和打断之间的冲突。玩家提前输入的操作（合理的操作习惯）被区分为"有效预输入"和"被打断后的残留输入"，前者被执行，后者被消耗丢弃，操作体验更加精准响应。

### 5.4 换人连击的战术深度
攻击换人保持 `CurrentSubSkillIndex` 的机制创造了独特的战术层次：玩家可以让一个角色打到连击中段，然后切换到爆发伤害更高的角色继续连击链，实现"一套连招换角色打"的高级操作技巧。

### 5.5 SubSkillConfigClone 的运行时修改
`SubSkillConfigClone` 允许 Buff/被动技能在运行时修改当前技能的参数（如增大攻击框范围、提高当前这段的伤害倍率），而不影响下一次使用该技能时的原始配置。这使得"本段连击增强"类的 Buff 效果可以精确控制作用范围。

## 6. 常见问题与最佳实践

### Q1：CurrentSubSkillIndex 会超出 SubSkill 列表的范围吗？
**A**：会做边界检查。在访问 SubSkill 列表时通过 `Min(index, list.Count-1)` 保护，或在 `canUse` 阶段检查索引有效性，超出则认为连击链已结束，技能执行失败，系统返回 Idle 状态。

### Q2：如何实现"后摇取消"（Cancel）？
**A**：在技能的后摇阶段（`CLIP_SetComboWindow` 的时间窗口内），接受特定的取消输入（如闪避键 EInputKey.Dodge），通过 `AbortInput` 管理当前技能输入，然后强制结束当前 Timeline，触发对应的取消状态（如闪避 DogeState）。连击段数根据取消类型决定是否保留或重置。

### Q3：连击系统如何处理 AI 单位？
**A**：AI 单位的连击决策由行为树（BTComponent）控制。行为树节点（如 `AT_UseAISkill`）在适当时机向 FSMComponent 注入对应段落的输入事件，模拟玩家在连击窗口内的输入。AI 的 `CurrentSubSkillIndex` 与玩家完全相同的逻辑路径处理。

### Q4：连击中断事件（ComboBreakEvtID）有什么用？
**A**：`BPDef.ComboBreakEvtID = 1011` 是给 Buff 和被动技能订阅的。例如：
- 特定装备：连击中断时获得一个护盾
- 被动技能：连击中断后下次攻击造成额外伤害
- 剧情事件：Boss 成功打断玩家连击时触发特殊对话

### Q5：PreCast（前摇预输入）和普通前摇有什么区别？
**A**：
- **普通前摇**：技能开始到攻击帧前的准备动作，无法被连击输入触发
- **PreCast（前摇预输入）**：在当前技能的后摇期，已经预先接受了下一招的输入，`bInPreCast = true`；等到合适时机（当前技能后摇结束或到达衔接窗口），自动执行下一招，不需要玩家再次按键，流畅度更高
