# 07_网络优化与延迟处理

> 低延迟是手游体验的核心指标之一。本文讲解本项目在网络延迟优化方面采用的技术方案和工程实践。

---

## 1. 系统概述

对于PVP对战游戏，延迟直接影响玩家体验：
- **延迟 < 80ms**：玩家感觉"丝滑"
- **延迟 80-150ms**：轻微感觉，可接受
- **延迟 150-300ms**：明显延迟感，影响操作
- **延迟 > 300ms**：体验极差，考虑断线重连

本项目的延迟优化策略包括：

1. **HttpDNS防域名劫持**：避免DNS解析异常导致连接错误节点
2. **UDP优先传输**：战斗帧数据用UDP而非TCP，减少握手和重传开销
3. **帧同步预测（SyncPredictTimeMS）**：提前拉帧，减少等待
4. **弱网参数自适应**：根据网络质量动态调整策略
5. **PING监控（VKeyDef.PVP_PING）**：实时测量并上报延迟
6. **Mesh网络**：P2P直连降低绕路延迟

## 2. 架构设计

### 2.1 延迟来源分析

```
玩家按下按钮
    │ 本地处理延迟（~1ms）
    ▼
Input序列化
    │ 序列化延迟（<1ms）
    ▼
网络发送
    │
    ├─ DNS解析（首次连接，可能100-500ms）→ HttpDNS优化
    ├─ TCP握手（首次，~100ms往返）→ 用UDP避免
    ├─ 传输延迟（取决于物理距离，20-100ms）→ CDN/就近接入
    ├─ 服务器处理（汇聚所有玩家输入，~0-5ms）
    └─ 广播回来
    │
    ▼
客户端收到该帧
    │ 帧缓冲处理（JitterBuffer）
    ▼
游戏逻辑执行
```

### 2.2 优化手段与效果

| 优化手段 | 目标延迟降低 | 实现位置 |
|---------|-------------|---------|
| HttpDNS | 首次连接减少100-400ms | GCloud HttpDns |
| UDP传输 | vs TCP减少20-50ms | LockSteper.Input(rawUdp=true) |
| 就近接入 | 减少50-200ms路由绕行 | GCloud自动调度 |
| 帧预测 | 减少1帧等待(~66ms) | SetCommonConfig SyncPredictTimeMS |
| 冗余发送 | 降低丢包率,减少重传 | LockStepInputFlag.DuplicateUpstream |
| 弱网降帧 | 弱网下减少卡顿 | 业务层动态调整帧率 |

### 2.3 PING实现原理

```
客户端发送 PVP_PING VKey (499)
    │ 记录发送时间 T1
    ▼
GCloud LockStep服务器
    │ 打上服务器时间戳
    ▼
广播到所有客户端
    │ 客户端收到，记录时间 T2
    ▼
RTT = T2 - T1
单向延迟 ≈ RTT / 2
```

```csharp
void ConnectToServer(string domain)
{
    // 先用HttpDNS解析IP，避免DNS劫持

    {

        ConnectByIP(result.IP, PORT);
    }
    else
    {
        Debug.LogWarning("[DNS] HttpDNS失败，使用系统DNS");
        ConnectByDomain(domain, PORT);
    }
}
```

### 3.2 UDP vs 可靠UDP的选择策略

```csharp
// 根据操作类型选择传输协议
public class BattleInputSender : MonoBehaviour
{
    // 普通移动操作：丢了无所谓（下一帧会更新位置）
    public void SendMoveInput(Vector2 direction)
    {
        byte[] data = SerializeMoveInput(direction);
        LockSteper.Instance.Input(data, rawUdp: true);  // 原始UDP，最快
    }

    // 技能释放：必须到达（否则玩家技能"没响应"）
    public void SendSkillInput(int skillId, Vector2 target)
    {
        byte[] data = SerializeSkillInput(skillId, target);
        LockSteper.Instance.Input(
            data,
            rawUdp: false,  // 可靠UDP，内置重传
            flag: LockStepInputFlag.Subscribe  // 要求服务器确认
        );
    }

    // 重要指令（对局结束确认等）：冗余发送降低丢包率
    public void SendCriticalInput(byte[] data)
    {
        LockSteper.Instance.Input(
            data,
            rawUdp: false,
            flag: LockStepInputFlag.DuplicateUpstream  // 上行冗余
        );
    }
}
```

### 3.3 帧预测配置（减少等待延迟）

```csharp
// 网络质量自适应配置管理器
public class NetworkQualityManager : MonoBehaviour
{
    private const int SAMPLE_COUNT = 30;       // 采样30帧
    private Queue<float> pingHistory = new Queue<float>();
    private Queue<float> lostRateHistory = new Queue<float>();

    private int lastTotalFrames = 0;
    private int lastLostFrames = 0;

    void Update()
    {
        UpdateNetworkStats();
    }

    private void UpdateNetworkStats()
    {
        int totalFrames = LockSteper.Instance.GetNetworkStat("totalFrameCount");
        int lostFrames = LockSteper.Instance.GetNetworkStat("totalLostFrameCount");

        // 计算最近的丢帧率
        int deltaFrames = totalFrames - lastTotalFrames;
        int deltaLost = lostFrames - lastLostFrames;
        
        if (deltaFrames > 0)
        {
            float currentLostRate = (float)deltaLost / deltaFrames;
            lostRateHistory.Enqueue(currentLostRate);
            if (lostRateHistory.Count > SAMPLE_COUNT)
                lostRateHistory.Dequeue();
        }
        
        lastTotalFrames = totalFrames;
        lastLostFrames = lostFrames;

        // 每5秒根据统计数据调整策略
        if (Time.frameCount % 300 == 0)
            AdaptNetworkConfig();
    }

    private void AdaptNetworkConfig()
    {
        if (lostRateHistory.Count == 0) return;
        
        float avgLostRate = lostRateHistory.Average();
        
        string config;
        string qualityLevel;

        if (avgLostRate < 0.02f)
        {
            // 优良网络：激进模式，缩短超时提升响应
            config = @"{""SyncTimeoutMS"": 300, ""SyncPredictTimeMS"": 40}";
            qualityLevel = "优良";
        }
        else if (avgLostRate < 0.05f)
        {
            // 一般网络：平衡模式
            config = @"{""SyncTimeoutMS"": 500, ""SyncPredictTimeMS"": 60}";
            qualityLevel = "一般";
        }
        else if (avgLostRate < 0.15f)
        {
            // 较差网络：保守模式，增大超时
            config = @"{""SyncTimeoutMS"": 800, ""SyncPredictTimeMS"": 100}";
            qualityLevel = "较差";
        }
        else
        {
            // 极差网络：最大容忍
            config = @"{""SyncTimeoutMS"": 1500, ""SyncPredictTimeMS"": 150}";
            qualityLevel = "极差";
        }
        
        LockSteper.Instance.SetCommonConfig(config);
        Debug.Log($"[NetOpt] 网络质量:{qualityLevel} 丢帧率:{avgLostRate:P1}");
    }
}
```

### 3.4 PING延迟上报（基于VKeyDef.PVP_PING）

```csharp
// PING延迟测量与上报
public class PingMonitor : MonoBehaviour
{
    private Dictionary<uint, long> pendingPings = new Dictionary<uint, long>();
    private uint pingSeqId = 0;
    private float lastPingSendTime = 0;
    private const float PING_INTERVAL = 3.0f;  // 每3秒发一次PING

    // 发送PING（在发送帧数据时附带）
    void Update()
    {
        if (Time.time - lastPingSendTime > PING_INTERVAL)
        {
            SendPing();
            lastPingSendTime = Time.time;
        }
    }

    private void SendPing()
    {
        pingSeqId++;
        long sendTime = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds();
        pendingPings[pingSeqId] = sendTime;
        
        // 构建PING消息
        byte[] pingData = BuildVKeyMessage(VKeyDef.PVP_PING, new int[] { (int)pingSeqId });
        LockSteper.Instance.Input(pingData, rawUdp: true);
    }

    // 当收到帧数据时，检查是否有PING的回包
    public void OnReceiveFrame(FrameInfo frame)
    {
        for (int i = 0; i < frame.ValidDataCount; i++)
        {
            var relayData = frame.DataCollection[i];
            if (relayData.PlayerId != LockSteper.Instance.PlayerID) continue;
            
            // 解析VKey，检查是否是PING回包
            var (vkey, args) = ParseVKeyMessage(relayData);
            if (vkey == VKeyDef.PVP_PING)
            {
                uint seqId = (uint)args[0];
                if (pendingPings.TryGetValue(seqId, out long sendTime))
                {
                    long rtt = DateTimeOffset.UtcNow.ToUnixTimeMilliseconds() - sendTime;
                    pendingPings.Remove(seqId);
                    
                    Debug.Log($"[PING] RTT={rtt}ms 单向延迟≈{rtt/2}ms");
                    UpdatePingUI((int)(rtt / 2));
                    
                    // 上报服务器（用于监控）
                    ReportPingToServer((int)(rtt / 2));
                }
            }
        }
    }
}
```

### 3.5 网络流量监控与带宽控制

```csharp
// 带宽使用监控
public class BandwidthMonitor : MonoBehaviour
{
    private uint prevIncomingBytes = 0;
    private uint prevOutgoingBytes = 0;
    private float lastSampleTime = 0;

    void Update()
    {
        if (Time.time - lastSampleTime < 1.0f) return;
        
        uint incoming = 0, outgoing = 0;
        LockSteper.Instance.GetFlowStat(ref incoming, ref outgoing);
        
        uint inDelta = incoming - prevIncomingBytes;
        uint outDelta = outgoing - prevOutgoingBytes;
        
        // 换算为KB/s
        float inKBps = inDelta / 1024f;
        float outKBps = outDelta / 1024f;
        
        Debug.Log($"[BW] 下行:{inKBps:F1}KB/s 上行:{outKBps:F1}KB/s");
        
        // 如果上行带宽过高，可以考虑压缩输入数据
        if (outKBps > 50f) // 超过50KB/s
        {
            Debug.LogWarning("[BW] 上行带宽异常，检查是否有冗余数据");
        }
        
        prevIncomingBytes = incoming;
        prevOutgoingBytes = outgoing;
        lastSampleTime = Time.time;
        
        // 同时获取UDP/TCP分别速度
        int udpSpeed = 0, tcpSpeed = 0;
        LockSteper.Instance.GetSpeed(ref udpSpeed, ref tcpSpeed);
    }
}
```

## 4. 设计亮点

### 4.1 三级延迟优化体系

本项目的延迟优化形成了三个层级：
1. **连接层**：HttpDNS + 就近接入（GCloud自动调度）
2. **传输层**：UDP优先 + 可靠UDP兜底 + 冗余发送
3. **应用层**：帧预测 + 弱网自适应 + 流量监控

### 4.2 SetCommonConfig支持运营期热调

`SetCommonConfig`接受JSON参数，可以通过服务器下发配置动态更新，不需要发版。遇到大规模网络问题时，可以临时给全服玩家推送弱网参数，降低投诉率。

### 4.3 多维度网络诊断API

GCloud提供了`GetNetworkStat`（帧统计）、`GetSpeed`（实时速度）、`GetFlowStat`（累计流量）三个诊断API，能从不同角度评估网络质量，是自适应优化的数据基础。

### 4.4 FrameProfile精细分析

`FrameProfile`记录了每一帧的`inputIntervalMS`（输入间隔）、`recvIntervalMS`（接收间隔）、`peekIntervalMS`（消费间隔）等精细指标，可用于离线分析卡顿原因。

## 5. 常见问题与最佳实践

### Q1：玩家反馈"技能没打出去"是什么原因？

可能原因：
1. 该操作用了`rawUdp=true`但发生丢包，客户端没收到确认，以为发送成功了
2. 解决：关键操作改用`rawUdp=false + LockStepInputFlag.Subscribe`

### Q2：如何判断延迟高是"网络问题"还是"服务器问题"？

- 如果`GetSpeed`显示UDP速度为0，说明UDP被拦截，走的是TCP，延迟偏高但可用
- 如果PING正常但帧同步延迟高，说明是服务器处理或帧率配置问题
- 如果PING也很高，说明是基础网络问题（玩家本地网络或运营商问题）

### Q3：`SyncPredictTimeMS`设置多大合适？

建议设置为`帧间隔时间的0.5-1倍`。如果帧率是15fps（约66ms/帧），设置40-80ms比较合适。设置太大会导致提前拉的帧数据不稳定；设置太小则没有提前效果。

### Q4：如何减少战斗中的数据量？

1. **压缩输入数据**：只发送变化的操作（差分压缩）
2. **合并输入**：多个短时间内的操作合并成一条
3. **减少冗余字段**：VKey消息只包含必要的参数
4. **Bit打包**：布尔值用bit位打包，减少字节数

### Q5：开发阶段如何模拟高延迟？

```csharp
// 在测试包中，通过GCloud的测试接口模拟延迟
// 或者使用网络限速工具（Charles Proxy）模拟弱网

// 代码层面可以用延迟队列模拟网络延迟
public class NetworkDelaySimulator : MonoBehaviour
{
    public float simulatedDelayMs = 200f; // 模拟200ms延迟
    private Queue<(float time, FrameInfo frame)> delayQueue 
        = new Queue<(float, FrameInfo)>();

    void Update()
    {
        while (true)
        {
            var rawFrame = LockSteper.Instance.PopFrame();
            if (rawFrame == null) break;
            delayQueue.Enqueue((Time.time + simulatedDelayMs/1000f, rawFrame));
        }
        
        while (delayQueue.Count > 0 && delayQueue.Peek().time <= Time.time)
        {
            var (_, frame) = delayQueue.Dequeue();
            ProcessFrame(frame);
        }
    }
}
```

## 6. 总结

网络延迟优化是一个系统工程，需要从DNS解析、传输协议选择、帧同步预测、弱网自适应等多个环节协同优化。本项目通过GCloud SDK提供的丰富API（`SetCommonConfig`、`GetNetworkStat`、`GetFlowStat`、`GetSpeed`），结合VKey协议的PING机制，实现了完整的延迟监控和自适应优化体系。

记住一个原则：**无法测量的延迟无法优化**。先把监控做好，才能针对性地优化。
