# 02_消息协议与序列化

> 本文讲解游戏中消息是如何被"打包"和"拆包"的。理解这部分内容，你才能看懂网络收发的二进制数据到底是什么意思。

---

## 1. 系统概述

网络通信的本质是双方按照约定的格式交换字节流。本项目使用了**两套序列化方案**：

1. **TDR（腾讯数据描述规范，Tencent Data Representation）**：用于GCloud底层与游戏服务器之间的结构化数据传输，这是腾讯自研的二进制序列化协议，高效紧凑。

2. **ApolloBuffer**：GCloud SDK的C#层序列化辅助类，封装了TDR的读写操作，提供更友好的API接口给业务层使用。

两者的关系是：**ApolloBuffer是TDR在C#层的封装**，你写代码时主要接触ApolloBuffer，底层传输用TDR二进制。

## 2. 架构设计

### 2.1 序列化层次图

```
业务数据结构（C# class）
      │
      ▼ WriteTo() / ReadFrom()
ApolloBufferWriter / ApolloBufferReader
      │
      ▼ 内部调用
TdrWriteBuf / TdrReadBuf（底层字节操作）
      │
      ▼
网络字节流（byte[]）→ 通过 GCloud Connector 发送
```

### 2.2 ApolloBuffer序列化框架

`ApolloBufferBase`是所有可序列化对象的基类，每个需要网络传输的消息类都继承它并实现`WriteTo`和`ReadFrom`：

```csharp
// GCloud SDK 提供的基类（伪代码表示接口约定）
public abstract class ApolloBufferBase
{
    // 将自身数据写入writer（序列化）
    public abstract void WriteTo(ApolloBufferWriter writer);
    
    // 从reader读取数据填充自身（反序列化）
    public abstract void ReadFrom(ApolloBufferReader reader);
    
    // 编码为byte[]
    public bool Encode(out byte[] data) { ... }
}
```

### 2.3 帧数据结构定义

战斗帧同步的消息结构定义在`LockSteper.cs`中，展示了ApolloBuffer序列化的完整实现：

```
FrameInfo（一帧数据）
  ├── FrameId: Int32          帧编号
  ├── RecvTickMS: Int32       接收时间戳（毫秒）
  ├── ValidDataCount: Int32   有效数据条数
  └── DataCollection: List<RelayData>  各玩家的操作数据
        └── RelayData（单个玩家的帧数据）
              ├── PlayerId: Int32      玩家ID
              ├── Flag: RelayDataFlag  状态标记（在线/离线）
              ├── DelayMS: Int32       延迟毫秒数
              ├── data: byte[]         玩家输入的原始数据
              └── SequenceId: Int32    序列号（用于去重）
```

### 2.4 VKey消息协议

除了帧同步的二进制数据，PVP系统还定义了一套**VKey（Virtual Key）消息协议**，用于标识特定的游戏事件。这些定义在`VKeyDef.cs`中：

```
VKey值范围与含义：
  100-106：策略卡调整相关操作
  402-407：对局生命周期事件
  441：PVE结算数据上报  
  492：客户端缓冲统计上报
  499：PING延迟上报
  502：哈希同步检测
  507：不同步错误通知
```

## 3. 核心代码展示

### 3.1 ApolloBufferWriter - 序列化写操作

来自 `ApolloBufferWriter.cs`（GCloud SDK核心文件）：

```csharp
// LockStepInitializeInfo 的序列化实现
// 来自 LockSteper.cs
public class LockStepInitializeInfo : ApolloBufferBase
{
    public int MaxBufferSize;       // 最大缓冲区大小
    public int StartFrameId = 0;    // 起始帧编号
    public LockStepManualUpdateOption ManualUpdateOption;
    public bool IsTcpCritical;      // 是否TCP优先
    public int MaxHistorySize;
    public bool EnableMesh;         // 是否开启Mesh网络
    public bool FastThreadMode;     // 是否快速线程模式
    public int HeartBeatInterval = 0; // 心跳间隔

    // 序列化：按顺序写入每个字段
    public override void WriteTo(ApolloBufferWriter writer)
    {
        writer.Write(MaxBufferSize);
        writer.Write(StartFrameId);
        writer.Write((int)ManualUpdateOption);
        writer.Write(IsTcpCritical);
        writer.Write(MaxHistorySize);
        writer.Write(EnableMesh);
        writer.Write(FastThreadMode);
        writer.Write(HeartBeatInterval);
    }

    // 反序列化：按同样顺序读取每个字段
    public override void ReadFrom(ApolloBufferReader reader)
    {
        reader.Read(ref MaxBufferSize);
        reader.Read(ref StartFrameId);
        int tmp = 0;
        reader.Read(ref tmp);
        ManualUpdateOption = (LockStepManualUpdateOption)tmp;
        reader.Read(ref IsTcpCritical);
        reader.Read(ref MaxHistorySize);
        reader.Read(ref EnableMesh);
        reader.Read(ref FastThreadMode);
        reader.Read(ref HeartBeatInterval);
    }
}
```

**关键规则**：`WriteTo`和`ReadFrom`的字段顺序必须完全一致，否则反序列化后数据会错乱。

### 3.2 RelayData 帧数据序列化

```csharp
// RelayData 序列化（来自 LockSteper.cs）
public class RelayData : ApolloBufferBase
{
    public Int32 PlayerId;
    public RelayDataFlag Flag = 0;
    public Int32 DelayMS = 0;
    byte[] data = new byte[1024];   // 玩家操作数据，最大1024字节
    private int dataLength;
    public Int32 SequenceId;

    public override void WriteTo(ApolloBufferWriter writer)
    {
        writer.Write(PlayerId);
        writer.Write((int)Flag);
        writer.Write(DelayMS);
        // 写入字节数组时需要传入实际长度
        if (dataLength == 0)
            writer.Write(null, 0);
        else
            writer.Write(data, dataLength);
        writer.Write(SequenceId);
    }

    public override void ReadFrom(ApolloBufferReader reader)
    {
        reader.Read(ref PlayerId);
        int temp = 0;
        reader.Read(ref temp);
        Flag = (RelayDataFlag)temp;
        reader.Read(ref DelayMS);
        // 读取字节数组，返回实际读取长度
        dataLength = reader.ReadArray(data);
        reader.Read(ref SequenceId);
    }
}
```

> **小贴士**：为什么用"网络字节序"（大端序）？因为不同CPU架构（x86是小端，部分ARM也是小端）存储多字节整数的字节顺序不同。网络上约定统一用大端序，接收方再转成自己的本地字节序，避免数据错乱。

### 3.5 VKeyArgPool - 零GC的参数池

为了减少战斗中的GC（垃圾回收），VKey参数使用了对象池：

```csharp
public class VKeyArgPool
{
    public static VKeyArgPool Instance = new VKeyArgPool();

    // 预创建不同长度的int数组池（1到6个参数）
    private List<int[]> m_args1Pool = new List<int[]>();
    private List<int[]> m_args2Pool = new List<int[]>();
    // ... 以此类推到6

    // 从池中取出一个数组，没有则新建
    public int[] GetArgs(int len)
    {
        var pool = m_arrCount2Pool[len - 1] as IList<int[]>;
        if (pool.Count > 0)
        {
            var args = pool[0];
            pool.RemoveAt(0);
            return args;
        }
        return new int[len];
    }

    // 用完后归还到池中（自动清零）
    public void ReturnArgs(int[] args)
    {
        for (int i = 0; i < args.Length; ++i)
            args[i] = 0;
        var pool = m_arrCount2Pool[args.Length - 1] as IList<int[]>;
        pool.Add(args);
    }
}
```

## 4. 设计亮点

### 4.1 TDR自动代码生成

项目中的`TdrBuf.cs`文件头部有注释`/* This file is generated by tdr. */`，说明这部分代码是由TDR工具链自动生成的。好处是：
- 服务端和客户端的协议结构定义保持一致（同一份TDR描述文件生成不同语言的代码）
- 协议变更时只需修改TDR描述，重新生成即可，不易出错

### 4.2 对象池减少GC

`VKeyArgPool`是一个典型的对象池模式。在帧同步战斗中，每一帧都会产生消息参数，如果每次都`new int[]`会产生大量垃圾对象，导致GC卡顿。通过池化复用，大幅减少内存分配。

### 4.3 结构化帧数据支持Clone和CopyTo

`FrameInfo`和`RelayData`都实现了`Clone()`和`CopyTo()`方法，这对于需要保存历史帧（用于回放、断线重连后的追帧）非常重要：

```csharp
// 深拷贝一帧数据
public FrameInfo Clone()
{
    FrameInfo clone = new FrameInfo();
    clone.FrameId = FrameId;
    clone.RecvTickMS = RecvTickMS;
    for (int i = 0; i < DataCollection.Count; i++)
    {
        RelayData data = DataCollection[i];
        if (data != null)
            clone.DataCollection.Add(data.Clone());
    }
    clone.ValidDataCount = ValidDataCount;
    return clone;
}
```

## 5. 常见问题与最佳实践

### Q1：为什么序列化字段顺序很重要？

ApolloBuffer是**无Schema**的二进制流，不像JSON那样有字段名。读取时完全按照写入顺序来解析字节。一旦`WriteTo`和`ReadFrom`顺序不一致，每个字段的数据都会错位，导致数据解析错误且往往没有明显报错，问题极难排查。

**最佳实践**：在类中使用注释明确标注字段顺序，并为每个可序列化类写单元测试：
```csharp
// 单元测试示例
var info = new LockStepInitializeInfo() { MaxBufferSize = 1024, StartFrameId = 5 };
byte[] data;
info.Encode(out data);
var decoded = new LockStepInitializeInfo();
decoded.Decode(data);
Assert.AreEqual(info.MaxBufferSize, decoded.MaxBufferSize);
```

### Q2：如何添加新的消息类型？

1. 继承`ApolloBufferBase`
2. 实现`WriteTo`和`ReadFrom`，字段顺序一致
3. 如需VKey标识，在`VKeyDef`中添加常量
4. 服务端同步更新协议（或通知服务端同学更新TDR描述）

### Q3：int[]的池化为什么用`RemoveAt(0)`而不是栈？

因为C#的`List<T>`在索引0移除后会移动后续元素，效率是O(n)。理论上用栈（`Stack<int[]>`）的`Push`/`Pop`效率更高。实际上由于战斗中参数数组数量不多（池的大小基本稳定），O(n)的开销可忽略。如果性能要求更高，可以改为使用`Stack`。

### Q4：`ValidDataCount`和`DataCollection.Count`有什么区别？

`DataCollection`是预分配的列表，可能包含已分配但无效的元素（如某个玩家本帧无操作）。`ValidDataCount`记录实际有数据的元素数量。遍历时应该用`ValidDataCount`而不是`DataCollection.Count`，避免处理空数据：

```csharp
// 正确写法
for (int i = 0; i < frame.ValidDataCount; i++)
{
    RelayData playerData = frame.DataCollection[i];
    // 处理玩家操作
}
```

### Q5：PVP_SYNC_ERROR（507）是怎么用的？

服务器检测到多个客户端在同一帧的哈希不一致时，会广播`PVP_SYNC_ERROR`（507）VKey，携带不同步的帧编号。客户端收到后记录日志并上报，供后续排查帧不同步问题（如浮点数精度差异、随机数种子不一致等）。

## 6. 总结

本项目的消息序列化以**ApolloBuffer**为核心，底层依托腾讯自研的**TDR**二进制协议，兼顾了高性能和跨平台兼容性。游戏事件通过**VKey**协议标识，战斗帧数据通过标准化的`FrameInfo`/`RelayData`结构传输。关键优化点在于对象池（`VKeyArgPool`）减少GC，以及克隆/拷贝方法支持帧历史管理。

理解了这套序列化体系，你就能读懂网络层收发的每一字节数据是什么含义。
