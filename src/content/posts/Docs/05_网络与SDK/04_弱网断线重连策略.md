# 04_弱网断线重连策略

> 网络不好是手游最常见的问题。本文讲解当玩家遇到网络中断时，系统是如何自动处理并尝试恢复的。

---

## 1. 系统概述

手机游戏面临比PC游戏复杂得多的网络环境：

- **信号弱区域**：地铁、电梯、地下室
- **网络切换**：WiFi切4G、4G切5G
- **运营商问题**：DNS劫持、QoS限速
- **NAT穿透失败**：某些路由器会拦截UDP包

本项目针对这些场景设计了**多层次的弱网与断线重连策略**：

1. **GCloud自动重连**：由GCloud LockStep SDK内置，对游戏业务层透明
2. **手动重连触发**：自动重连多次失败后，暴露`ReconnectManually()`接口给业务层
3. **网络状态监控**：通过`NetworkService`实时感知网络变化
4. **弱网参数自适应**：通过`SetCommonConfig`动态调整超时和预测参数

## 2. 架构设计

### 2.1 重连状态机

```
战斗进行中（Fighting）
       │
       │ 网络中断/超时
       ▼
  Reconnecting（重连中）
  ├── GCloud内置自动重连（透明进行）
  ├── 显示"网络不稳定"UI
  └── 超出自动重连次数/时间
       ├─── 成功 ──→ Reconnected（重连成功）→ 继续战斗
       └─── 失败 ──→ Error（重连失败）
                     │
                     ├── 提示玩家手动重连
                     │   调用 ReconnectManually()
                     └── 超时 → 退出战斗
```

### 2.2 网络状态变化触发流

```
设备网络变化（WiFi断开/切换）
       │
       ▼
TXNetwork（GCloud底层网络监听）
       │ TXNetworkChangedEvent
       ▼
NetworkService.onNetworkStateChanged()
       │ NetworkChangedEvent
       ▼
业务层订阅处理
       ├── 战斗中：触发LockStep重连
       ├── 大厅中：触发WebSocket重连
       └── 更新中：暂停更新，等待恢复
```

### 2.3 LockStep重连流程详解

```
检测到断线（StateChangedEvent = Reconnecting）
    │
    ├─ 暂停战斗逻辑（不再PopFrame）
    ├─ 显示"重连中..."UI
    ├─ 记录当前帧编号（用于重连后追帧）
    │
    ▼
GCloud自动尝试重连（最多N次，每次间隔递增）
    │
    ├─ 成功（StateChangedEvent = Reconnected）
    │   ├─ 服务器推送断线期间的帧数据
    │   ├─ 客户端追帧到最新帧
    │   └─ 恢复战斗逻辑
    │
    └─ 失败（StateChangedEvent = Error）
        ├─ 提示玩家点击"手动重连"
        │   LockSteper.Instance.ReconnectManually()
        └─ 超时（约60s）→ 结算为逃跑
```

## 3. 核心代码展示

### 3.1 LockStep断线重连接口

来自`LockSteper.cs`：

```csharp
// 核心重连API
public abstract class LockSteper : ApolloObject
{
    // 当发生状态变化时触发（包括断线、重连等）
    public abstract event StateChangedEventHandler StateChangedEvent;

    // 自动重连失败后，调用此方法手动触发重连
    public abstract bool ReconnectManually();

    // 通知SDK：游戏开始加载资源（暂停心跳计时，防止加载期间被判定超时）
    public abstract void NotifyLoadingBegin();

    // 通知SDK：资源加载完毕（恢复正常心跳）
    public abstract void NotifyLoadingEnd();
}

// 状态枚举
public enum LockStepState
{
    Fighting,       // 战斗正常进行
    Reconnecting,   // 正在重连中（GCloud自动处理）
    Reconnected,    // 重连成功
    StayInQueue,    // 在排队中（等待进入战斗）
    Error,          // 重连失败，发生严重错误
}

// 错误码
public enum LockStepErrorCode
{
    Success,
    ConnectFailed,       // 连接失败
    NetowrkException,    // 网络异常
    Timeout,             // 超时
    Unknown,             // 未知错误
    SessionStop,         // 对局已停止
    ServerFull,          // 服务器满员
    RoomNotFound,        // 房间不存在（可能已过期）
    AuthFailed,          // 鉴权失败
    LogoutNotFinished,   // 上次登出未完成
}
```

### 3.2 业务层的断线重连处理（推荐实现）

```csharp
// 战斗系统中的断线重连处理（示例代码）
public class BattleNetworkManager : MonoBehaviour
{
    private int reconnectAttempts = 0;
    private const int MAX_RECONNECT_ATTEMPTS = 3;
    private Coroutine reconnectCoroutine;

    void Start()
    {
        // 订阅状态变化事件
        LockSteper.Instance.StateChangedEvent += OnLockStepStateChanged;
        
        // 订阅系统网络变化
        NetworkService.Instance.NetworkChangedEvent += OnNetworkChanged;
    }

    private void OnLockStepStateChanged(LockStepState state, LockStepResult result)
    {
        switch (state)
        {
            case LockStepState.Reconnecting:
                // 暂停战斗逻辑，显示重连UI
                PauseBattleLogic();
                ShowReconnectingPanel(reconnectAttempts);
                Debug.Log($"[Network] 第{reconnectAttempts+1}次重连中...");
                break;

            case LockStepState.Reconnected:
                // 重连成功！恢复战斗
                reconnectAttempts = 0;
                HideReconnectingPanel();
                ResumeBattleLogic();
                Debug.Log("[Network] 重连成功，继续战斗");
                break;

            case LockStepState.Error:
                // 自动重连失败
                reconnectAttempts++;
                HandleReconnectFailure(result);
                break;
        }
    }

    private void HandleReconnectFailure(LockStepResult result)
    {
        Debug.LogWarning($"[Network] 自动重连失败: {result.Error}({result.ExtCode})");

        if (reconnectAttempts < MAX_RECONNECT_ATTEMPTS)
        {
            // 尝试手动重连
            bool canRetry = LockSteper.Instance.ReconnectManually();
            if (canRetry)
            {
                ShowManualReconnectPrompt(); // 显示"重连中，请稍候..."
            }
            else
            {
                // ReconnectManually返回false说明已无法重连
                ExitBattle(BattleExitReason.NetworkError);
            }
        }
        else
        {
            // 超出最大重连次数，强制退出
            ExitBattle(BattleExitReason.ReconnectExhausted);
        }
    }

    private void OnNetworkChanged(NetworkState state)
    {
        Debug.Log($"[Network] 网络状态变化: {state}");

        if (state == NetworkState.Disconnected)
        {
            // 网络完全断开，主动显示提示
            ShowNoNetworkToast("网络已断开，请检查网络连接");
        }
        else if (state == NetworkState.Connected)
        {
            // 网络恢复，GCloud会自动触发重连
            Debug.Log("[Network] 网络已恢复，等待GCloud自动重连...");
        }
    }
}
```

### 3.3 弱网自适应参数调整

```csharp
// 根据网络质量动态调整LockStep策略
public class NetworkQualityAdapter : MonoBehaviour
{
    private const int PING_CHECK_INTERVAL = 3000; // 每3秒检测一次
    private int lastPing = 0;

    void Update()
    {
        // 读取底层网络统计
        int totalFrames = LockSteper.Instance.GetNetworkStat("totalFrameCount");
        int lostFrames = LockSteper.Instance.GetNetworkStat("totalLostFrameCount");
        
        float lostRate = totalFrames > 0 ? (float)lostFrames / totalFrames : 0f;

        // 根据丢帧率动态调整策略
        if (lostRate > 0.1f) // 丢帧率超过10%
        {
            ApplyWeakNetworkConfig();
        }
        else if (lostRate < 0.02f) // 丢帧率低于2%
        {
            ApplyGoodNetworkConfig();
        }
    }

    private void ApplyWeakNetworkConfig()
    {
        // 弱网模式：增大超时时间，开启预测
        LockSteper.Instance.SetCommonConfig(
            @"{""SyncTimeoutMS"": 1000, ""SyncPredictTimeMS"": 120}"
        );
        Debug.Log("[Network] 切换到弱网模式");
    }

    private void ApplyGoodNetworkConfig()
    {
        // 良网模式：减小超时时间，提升响应
        LockSteper.Instance.SetCommonConfig(
            @"{""SyncTimeoutMS"": 300, ""SyncPredictTimeMS"": 40}"
        );
        Debug.Log("[Network] 切换到良网模式");
    }
}
```

### 3.4 NotifyLoadingBegin/End 防误判超时

```csharp
// 战斗开始前加载资源时，必须通知LockStep避免超时断线
public class BattleLoader : MonoBehaviour
{
    private IEnumerator LoadBattleAssets()
    {
        // 告诉LockStep："我正在加载资源，不要因为我没心跳就断开我"
        LockSteper.Instance.NotifyLoadingBegin();
        
        try
        {
            // 加载地图、角色模型等资源（可能需要3-10秒）
            yield return LoadMapAsync();
            yield return LoadCharacterModelsAsync();
            yield return LoadAudioAsync();
            
            Debug.Log("[Battle] 资源加载完毕");
        }
        finally
        {
            // 无论成功还是失败，都要通知LockStep加载结束
            LockSteper.Instance.NotifyLoadingEnd();
        }
    }
}
```

// 监控UDP/TCP实时带宽（来自 LockSteper.cs GetSpeed API）
public class NetworkSpeedMonitor : MonoBehaviour
{
    private int udpSpeedKBps = 0;
    private int tcpSpeedKBps = 0;

    {
        LockSteper.Instance.GetSpeed(ref udpSpeedKBps, ref tcpSpeedKBps);
        
        // 如果UDP速度为0但TCP有速度，可能是UDP被拦截
        if (udpSpeedKBps == 0 && tcpSpeedKBps > 0)

            Debug.LogWarning("[Network] UDP可能被防火墙拦截，仅TCP工作");
        }
        
        // 流量统计
        uint incomingBytes = 0, outgoingBytes = 0;
        LockSteper.Instance.GetFlowStat(ref incomingBytes, ref outgoingBytes);
    }
}

## 4. 设计亮点

### 4.1 分级重连策略

系统采用"先自动后手动"的分级重连策略：
- **第一级**：GCloud SDK内置的透明自动重连（玩家无感知）
- **第二级**：`ReconnectManually()`手动重连（给玩家一个继续游戏的机会）
- **第三级**：超出次数后退出战斗（避免玩家长时间处于挂起状态）

### 4.2 加载期间保活

`NotifyLoadingBegin()` / `NotifyLoadingEnd()` 这对API很关键。战斗开始前需要加载大量资源，期间客户端无法发送心跳，LockStep服务器会误判玩家离线。这两个接口告诉服务器"我在加载，别踢我"，避免了一加载完就重连的尴尬场景。

### 4.3 动态配置支持网络自适应

通过`SetCommonConfig`的JSON接口，可以在运行时（甚至通过热配置下发）调整：
- `SyncTimeoutMS`：拉帧超时时间，弱网时调大（等更久），良网时调小（响应更快）
- `SyncPredictTimeMS`：预测时间，提前拉帧，减少等待

### 4.4 多维度网络诊断

通过`GetNetworkStat`、`GetSpeed`、`GetFlowStat`三个API，可以全面了解当前网络质量，为弱网提示和重连决策提供数据支撑。

## 5. 常见问题与最佳实践

### Q1：重连成功后需要做什么？

重连成功（`LockStepState.Reconnected`）后，GCloud SDK会自动推送断线期间所有缺失的帧数据。业务层需要：
1. 关闭重连UI
2. 让战斗逻辑快速消费这些补帧（不显示动画，直接跳到当前帧）
3. 追帧完成后恢复正常战斗

```csharp
private void ResumeBattleLogic()
{
    isCatchingUp = true;
    while (LockSteper.Instance.GetCurrentMaxFrameId() > currentFrameId)
    {
        FrameInfo frame = LockSteper.Instance.PopFrame();
        if (frame == null) break;
        SimulateFrame(frame, fastForward: true); // 快速追帧，不渲染
        currentFrameId++;
    }
    isCatchingUp = false;
}
```

### Q2：`ReconnectManually()`返回false是什么意思？

返回false表示当前状态不允许重连（例如服务器已关闭该对局、或连接状态机不在Error状态）。此时应该直接退出战斗，不要再尝试重连。

### Q3：如何区分"网络断了"和"服务器关了"？

通过`LockStepResult.Error`错误码区分：
- `ConnectFailed` / `NetworkException` → 网络问题，可以重连
- `RoomNotFound` / `SessionStop` → 服务器已关闭对局，不需要重连，直接结算

### Q4：弱网时玩家操作有延迟感怎么优化？

1. **本地预测**：玩家操作立即在本地显示效果，等帧同步确认后对齐（可能有小幅回滚）
2. **增大SyncPredictTimeMS**：让SDK提前拉帧，减少等待时间
3. **操作压缩**：连续相同操作只发送一次，减少带宽占用

### Q5：如何测试弱网场景？

推荐使用Charles或Network Link Conditioner（iOS）进行弱网模拟：
- **丢包率20%**：测试重传机制
- **延迟500ms**：测试超时处理
- **断开1-10秒**：测试重连流程

在代码层面，可以通过`GCloudConnector`的测试接口注入模拟网络故障。

## 6. 总结

本项目的弱网断线重连策略以GCloud SDK为基础，通过`StateChangedEvent`感知网络状态变化，`ReconnectManually()`作为最后的手动重连手段，`NotifyLoadingBegin/End`防止加载期间被误踢，`SetCommonConfig`实现弱网参数自适应。整体设计做到了"对玩家透明，让系统自动处理大多数情况，仅在必要时才打扰玩家"。

良好的断线重连体验是手游品质的重要指标，理解这套机制对于处理线上玩家反馈的网络问题至关重要。
