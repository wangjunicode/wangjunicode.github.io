# 01_热更新架构总览

> 面向刚入行的毕业生 · 建议搭配源码阅读

---

## 1. 系统概述

热更新（Hot Update / Hot Patch）是移动游戏项目的核心能力之一。简单一句话描述：**不重新安装 App、只下载差量文件，就能让玩家跑到最新版本的代码和资源。**

对于一个商业手游项目来说，热更新的价值体现在以下几个维度：

| 维度 | 没有热更新 | 有热更新 |
|------|-----------|---------|
| 修复 Bug | 必须发版，玩家重新下载安装 | 推热更包，玩家下次登录自动修复 |
| 更新内容 | App Store/Google Play 审核周期 1-7 天 | 服务器一键推送，数分钟生效 |
| 包体大小 | 随版本叠加越来越大 | 只推差量，玩家感知极小 |
| 回滚 | 重新发旧版到渠道 | 服务端切回旧版本号，自动降级 |

本项目的热更新系统综合了业界成熟方案，整体分为**两大层次**：

1. **资源热更（Asset Hot Update）**：使用腾讯 GCloud Dolphin SDK + Unity Addressables 完成 AssetBundle 的版本控制与差量下载。
2. **代码热更（Code Hot Update）**：使用 HybridCLR（原 huatuo）方案，将热更 C# 程序集以 DLL bytes 形式打包进 AssetBundle，运行时通过 `Assembly.Load` 动态加载。

## 3. 核心代码展示

### 3.1 两段式更新入口（PatchManager）

```csharp
// PatchManager.cs - 核心启动入口
void GameUpdateAndBegin()
{
    hadInit = true;
    
    if (!isSkipPatch)
    {
        ClearOldPatch();   // 清除主版本不一致的旧补丁
        AppUpdate();       // 第一阶段：Dolphin 检测程序包更新
    }
    else
    {
        BeginAdressableCatalogUpdate(); // 编辑器直接跳过，刷新 AA Catalog
    }
}
```

**解读**：`isSkipPatch` 在 Android Release 包中默认为 `false`，在 Editor 和其他平台为 `true`。这保证了编辑器开发效率的同时，发布包必须走完整热更流程。

### 3.2 Addressables 路径重定向（热更核心机制）

```csharp
// 注册路径转换函数 —— 将 Addressables 的资源请求重定向到热更目录
aaPrefix = Addressables.RuntimePath + "/";
Addressables.InternalIdTransformFunc += TransformResourcePath;

private string TransformResourcePath(IResourceLocation location)
{
    string hotUpdatePath;
    if (!hotUpdatePathMap.TryGetValue(location.InternalId, out hotUpdatePath))
    {
        var key = location.InternalId.Replace(aaPrefix, "");
        // 热更新路径：Application.persistentDataPath/ResUpdate/<bundle路径>
        hotUpdatePath = Path.Combine(
            Application.persistentDataPath,
            HotUpdateRootDir,  // = "ResUpdate"
            key
        );
        hotUpdatePathMap[location.InternalId] = hotUpdatePath;
    }
    
    // 如果热更目录存在该文件，优先用热更版本
    if (File.Exists(hotUpdatePath))
        return hotUpdatePath;
    
    // 否则使用安装包内自带资源
    return location.InternalId;
}
```

**这是整个资源热更的关键所在**：Addressables 每次加载资源前都会调用这个转换函数，优先从热更目录 `ResUpdate/` 查找文件，从而实现"热更文件覆盖安装包文件"的效果。

### 3.3 版本号管理（VersionManager）

```csharp
// VersionManager.cs - 版本号读取
public string ResVersion
{
    get
    {
        if (!initialized)
        {
            initialized = true;
            LoadAppConfig(); // 从 P 目录 AppConfig.json 加载
        }
#if UNITY_ANDROID
        return AppConfig.AndroidSourceVersion;
#elif UNITY_IOS
        return AppConfig.iOSSourceVersion;
#else
        return Application.version + ".1";
#endif
    }
}
```

版本号格式为四位：`主.次.修.资源`，例如 `1.0.3.0`。Dolphin 以此判断是否需要推送更新。

## 4. 设计亮点

### 4.1 两阶段更新设计（Program + Source）

Dolphin 的更新被拆成了两个阶段：

- **AppUpdate（UpdateInitType_OnlyProgram）**：检测 APK/IPA 本身是否需要整包更新（大版本）
- **SourceUpdate（UpdateInitType_OnlySource）**：检测资源（AssetBundle/DLL）是否需要热更

这样的分离设计使得程序包更新和资源更新互不干扰，并且可以单独控制各阶段的灰度策略。

### 4.2 灰度更新支持

```csharp
#if APP_GRAY_UPDATE
info.grayUpdate = true;
#endif
```

通过编译宏 `APP_GRAY_UPDATE` 可以开启 Dolphin 的灰度发布能力，只向特定比例的用户推送更新，降低线上风险。

### 4.3 主版本号不一致时自动清空旧补丁

```csharp
// 比较主版本号
if (currentProgramVersion.Split(".")[0] != currentSourceVersion.Split(".")[0])
{
    // 删除旧的补丁文件目录 "ResUpdate"
    Directory.Delete(resUpdateDirectory, true);
}
```

防止大版本更新后，旧的热更文件与新安装包产生冲突。

### 4.4 路径转换缓存

`hotUpdatePathMap` 是一个 `Dictionary<string, string>`，将已经计算过的路径结果缓存下来，避免每次加载资源都做字符串拼接和文件存在判断，提升了性能。

## 5. 常见问题与最佳实践

### Q1：热更新文件存放在哪里？
**A**：热更文件统一放在 `Application.persistentDataPath/ResUpdate/` 目录下，子目录结构与 Addressables 的 RuntimePath 保持一致。

### Q2：编辑器里怎么测试热更新？
**A**：在 `PatchManager.cs` 中，`isSkipPatch` 字段在非 Android Release 环境下默认为 `true`，会跳过 Dolphin 检测直接进入游戏。如果要在编辑器测试热更逻辑，可以手动将 `isSkipPatch` 改为 `false`（但需要配置 GCloud 正确连接）。

### Q3：`isSkipPatch` 被误设为 `true` 导致线上不更新？
**A**：注意代码中的平台宏：
```csharp
#if !UNITY_EDITOR && UNITY_ANDROID
    public bool isSkipPatch = false;  // Android 正式包：不跳过
#else
    public bool isSkipPatch = true;   // Editor 及其他平台：跳过
#endif
```
iOS 正式包也需要确认是否需要将该字段初始化为 `false`（当前代码 iOS 走 else 分支，默认跳过，需关注）。

### Q4：如何判断当前在审核服还是正式服？
**A**：通过 `Resources.Load<TextAsset>("DeploymentEnvironment")` 加载一个文本资源，内容为 `"Review"` 则是审核服，使用 `DOPHIN_PRE_UPDATEURL` 通道；否则使用 `DOPHIN_UPDATEURL`。

### Q5：热更新后需要重启游戏吗？
**A**：代码热更（HybridCLR DLL）是在游戏启动时通过 `Assembly.Load` 一次性加载的，热更后的 DLL 在下次启动时生效，**不支持运行时代码替换**。资源热更（Addressables）由于使用了路径重定向机制，新下载的 AssetBundle 会在下次通过 Addressables 加载时自动使用新版本。

## 6. 总结

本项目的热更新架构是一个**经过商业验证的成熟方案**，核心思想是：

1. Dolphin 负责"检测和分发"（知道要下什么）
2. Addressables 负责"资源的运行时访问"（怎么找到文件）
3. HybridCLR 负责"代码的运行时执行"（怎么运行热更代码）
4. VersionManager + AppConfig 负责"版本状态的持久化"（记住当前版本是什么）
