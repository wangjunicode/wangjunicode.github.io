# 10_热更监控与回滚机制

> 面向刚入行的毕业生 · 建议搭配 PatchLog.cs 和 PatchManager.cs 阅读

---

## 1. 系统概述

### 1.1 为什么需要热更监控？

热更新的核心优势是"快速推送修复"，但热更本身也可能引入新问题：
- 新版本 DLL 有致命 Bug，导致更多用户崩溃
- 资源热更破坏了某个场景
- Catalog 损坏导致资源加载失败
- 热更包下载不完整导致游戏无法进入

没有监控，运营团队对线上热更的影响完全"盲飞"；没有回滚机制，一旦热更出现问题，就只能紧急发版，等待用户更新（可能需要1-2小时甚至更长）。

### 1.2 本项目的监控体系

本项目的监控体系由以下层次组成：

| 层次 | 工具 | 说明 |
|------|------|------|
| 热更阶段日志 | `PatchLog.cs` | 热更流程的本地日志，支持文件落盘 |
| 崩溃监控 | `App.LogHandler` + 腾讯 Bugly/鹰眼 | 捕获未处理异常，上报崩溃堆栈 |
| 更新成功率 | Dolphin SDK 上报 | 统计热更下载成功/失败比例 |
| 服务端回滚 | GCloud 控制台 | 切换版本号，触发客户端回滚 |
| 客户端自愈 | `ClearOldPatch()` + 降级 Fallback | 本地热更文件损坏时自动恢复 |

## 2. 架构设计

### 2.1 PatchLog 日志系统架构

```
PatchLog（静态工具类）
├─ LogLevel 控制（None/Error/Warning/Info）
│   └─ 生产环境：LogLevel.Error（只记录错误）
│   └─ 开发环境：LogLevel.Info（记录所有）
│
├─ 控制台输出（Debug.Log）
│   └─ 开发阶段使用，Profiler 可见
│
└─ 文件落盘（可选）
    ├─ 路径：persistentDataPath/PatchLogs/patch_log_YYYYMMDD.txt
    ├─ 按日期分文件，防止单文件过大
    └─ 异步写入（独立线程，不阻塞主线程）
```

### 2.2 热更监控数据流

```
客户端
├─ PatchLog → 本地文件
│   └─ QA/运维通过 adb pull 或远程文件管理拉取日志
│
├─ 崩溃上报（全局异常捕获）
│   └─ Index.Awake() 注册
│       AppDomain.CurrentDomain.UnhandledException += (sender, e) =>
│       {
│           Log.Error(e.ExceptionObject.ToString());
│       };
│   └─ App.LogHandler 拦截 Unity LogError/Exception
│
└─ 热更关键事件上报（扩展点）
    ├─ 热更开始（AppUpdate 调用时）
    ├─ 热更完成（OnNoticeChangeSourceVersion）
    ├─ 热更失败（OnUpdateMessageBoxInfo 错误回调）
    └─ DLL 加载失败（getAssembly 返回 null 时）

服务端（GCloud 控制台）
├─ 查看各版本的下载统计
├─ 监控更新成功率
└─ 一键回滚（切换版本号）
```

## 3. 核心代码展示

### 3.1 PatchLog 日志系统

```csharp
// PatchLog.cs - 热更专用日志系统
public static class PatchLog
{
    private static LogLevel _logLevel = LogLevel.Info;
    private static bool _writeToFile = false;
    private static string _logFilePath;
    private static StreamWriter _writer;
    private static readonly object _locker = new object(); // 线程锁

    // 初始化日志系统（在 PatchManager.Start() 中调用）
    public static void Initialize(LogLevel level = LogLevel.Info, bool enableFileLog = false)
    {
        _logLevel = level;
        _writeToFile = enableFileLog;

        if (_writeToFile)
        {
            // 日志目录：persistentDataPath/PatchLogs/
            string logDir = Path.Combine(Application.persistentDataPath, "PatchLogs");
            if (!Directory.Exists(logDir))
                Directory.CreateDirectory(logDir);
            
            // 按日期命名日志文件（如 patch_log_20241019.txt）
            string fileName = $"patch_log_{DateTime.Now:yyyyMMdd}.txt";
            _logFilePath = Path.Combine(logDir, fileName);
            
            // 使用独立线程打开文件，避免阻塞主线程
            new Thread(() =>
            {
                lock (_locker)
                {
                    _writer = new StreamWriter(_logFilePath, true, Encoding.UTF8);
                    _writer.AutoFlush = true; // 立即刷新，防止缓存丢失
                }
            }).Start();
        }
    }

    // 三个级别的日志接口
    public static void Info(object message)
    {
        if (_logLevel >= LogLevel.Info)
            WriteLog("[INFO] ", message);
    }

    public static void Warning(object message)
    {
        if (_logLevel >= LogLevel.Warning)
            WriteLog("[WARN] ", message);
    }

    public static void Error(object message)
    {
        if (_logLevel >= LogLevel.Error)
            WriteLog("[ERROR]", message);
    }

    private static void WriteLog(string prefix, object message)
    {
        string logMsg = $"{DateTime.Now:HH:mm:ss} {prefix} {message}";
        Debug.Log(logMsg); // 控制台输出

        if (_writeToFile && _writer != null)
        {
            lock (_locker) // 线程安全写文件
            {
                try
                {
                    string fileMsg = $"{DateTime.Now:yyyy-MM-dd HH:mm:ss} {prefix} {message}";
                    _writer.WriteLine(fileMsg);
                }
                catch (Exception e)
                {
                    Debug.LogError($"日志写入失败: {e.Message}");
                }
            }
        }
    }
    
    // 关闭日志（游戏退出时调用）
    public static void Shutdown()
    {
        lock (_locker)
        {
            if (_writer != null)
            {
                _writer.Flush();
                _writer.Close();
                _writer = null;
            }
        }
    }
}
```

// App.cs - 拦截 Unity LogError/Exception（可接 Bugly）
public void LogHandler(string logString, string stackTrace, LogType type)
{
    ET.Log.Info(logString);
    if (type == LogType.Error || type == LogType.Exception || type == LogType.Assert)
    {
        ET.Log.Info(stackTrace); // 记录堆栈信息
        // 扩展：Bugly.Report(logString, stackTrace); // 上报崩溃
    }
}

### 3.3 热更失败处理（错误监控与用户提示）

```csharp
// PatchManager.cs - Dolphin 错误回调
public void OnUpdateMessageBoxInfo(string msg, MessageBoxType msgBoxType, 
                                   bool isError, uint errorCode)
{
    PatchLog.Error($"OnUpdateMessageBoxInfo: {msg}, isError:{isError}, errorCode:{errorCode}");
    
    if (isError)
    {
        // 错误类型：网络中断、CDN 返回 4xx/5xx、文件校验失败等
        string message = msg + ", 错误码：" + errorCode + ", 是否重试？";
        
        // 上报错误事件（扩展点）
        // ReportEvent("hotupdate_error", new { error_code = errorCode, msg = msg });
        
        uiInstance.ShowTipMessage("提示", message,
            () => mgr.Continue(),   // 重试
            () => Application.Quit()); // 退出
    }
    else
    {
        // 非错误类消息（如进度提示），直接继续
        mgr.Continue();
    }
}
```

### 3.5 客户端自愈机制（热更文件损坏时）

```csharp
// PatchManager.cs - 主版本不一致时清除旧热更（防止版本污染）
private void ClearOldPatch()
{
    string currentProgramVersion = GetCurrentProgramVersion();  // "1.0.3"
    string currentSourceVersion = GetCurrentSourceVersion();    // "0.9.8.5"

    // 主版本号不一致（大版本跨越），旧热更文件可能与新安装包不兼容
    if (currentProgramVersion.Split(".")[0] != currentSourceVersion.Split(".")[0])
    {
        PatchLog.Info($"版本不一致：程序={currentProgramVersion}，资源={currentSourceVersion}，清除旧热更");
        
        string resUpdateDirectory = Path.Combine(Application.persistentDataPath, "ResUpdate");
        
        if (Directory.Exists(resUpdateDirectory))
        {
            try
            {
                Directory.Delete(resUpdateDirectory, true); // 删除整个热更目录
                PatchLog.Info("旧的补丁目录已删除，将从 CDN 重新下载");
            }
            catch (Exception ex)
            {
                PatchLog.Error($"删除失败: {ex.Message}");
                // 即使删除失败，也继续流程（Dolphin 会覆盖旧文件）
            }
        }
    }
}
```

### 3.6 Catalog 加载失败的降级 Fallback

```csharp
// PatchManager.cs
IEnumerator LoadCustomCatalog()
{
    string catalogPath = Path.Combine(
        Application.persistentDataPath, "ResUpdate", "catalog.json");
    
    if (File.Exists(catalogPath))
    {
        var handle = Addressables.LoadContentCatalogAsync(catalogPath);
        yield return handle;

        if (handle.Status == AsyncOperationStatus.Succeeded)
        {
            Addressables.AddResourceLocator(handle.Result);
            PatchLog.Info("自定义 Catalog 加载成功");
            // ...
        }
        else
        {
            // Catalog 损坏或格式错误：降级到安装包内的默认 Catalog
            PatchLog.Error("自定义 Catalog 加载失败，使用内置 Catalog");
            // 不做任何操作，Addressables 会自动使用安装包内的 Catalog
        }
    }
    
    // 无论 Catalog 是否加载成功，都继续进入游戏
    InitGame(false);
}
```

## 4. 回滚机制

### 4.1 服务端版本回滚（首选方案）

当热更出现问题时，最快的回滚方式是通过 GCloud 控制台操作：

```
GCloud 控制台操作步骤：
1. 登录 GCloud 控制台
2. 找到当前激活的版本（如 1.0.3.10）
3. 切换到上一个稳定版本（如 1.0.3.8）
4. 发布配置更改

效果：
- 新连接的客户端检测到"当前版本比服务端最新版本更高"
- Dolphin 会提示"无需更新"（isCurrentNewest = true）
- 已下载了 1.0.3.10 的客户端：
  - 如果是资源回滚：Addressables Catalog 会指向旧版本资源
  - 如果是 DLL 回滚：需要客户端下载旧版 DLL（Dolphin 下发旧版文件）
```

### 4.2 客户端本地热更清除（紧急回滚）

通过后台下发指令或在 GM 工具中触发，删除 `ResUpdate` 目录，让客户端回到安装包初始状态：

```csharp
// GM 工具接口（扩展实现）
public static void ClearHotUpdateCache()
{
    string resUpdateDir = Path.Combine(Application.persistentDataPath, "ResUpdate");
    if (Directory.Exists(resUpdateDir))
    {
        Directory.Delete(resUpdateDir, true);
        PatchLog.Info("[GM] 热更缓存已清除，重启后将重新下载");
    }
    
    // 清除版本记录
    PlayerPrefs.DeleteKey("GameLogicVersion");
    
    // 强制重启 App（让重新下载生效）
    Application.Quit();
}
```

### 4.3 版本回滚的数据一致性

**重要**：回滚代码（DLL）时需要同时考虑数据兼容性：
- 如果新版本修改了游戏存档格式，回滚后旧代码可能无法读取新格式的存档
- 建议在设计时保持存档的向前兼容（旧代码能读新格式，至少不崩溃）
- 热更 DLL 回滚时，同时回滚对应版本的资源 Bundle

## 5. 监控指标建议

### 5.1 关键监控指标

| 指标 | 采集时机 | 告警阈值（参考） |
|------|---------|----------------|
| 热更成功率 | OnNoticeUpdateSuccess | < 95% 触发告警 |
| 热更失败率 | OnUpdateMessageBoxInfo(isError=true) | > 1% 触发告警 |
| 平均下载耗时 | 从 StartUpdateService 到 OnNoticeUpdateSuccess | > 60s 触发告警 |
| DLL 加载失败 | getAssembly 返回 null | 任何出现触发告警 |
| Catalog 加载失败 | handle.Status != Succeeded | 任何出现触发告警 |
| 崩溃率 | AppDomain.UnhandledException | 相比热更前上升 > 0.5% |

### 5.2 热更日志关键节点

## 6. 设计亮点

### 6.1 线程安全的日志写入

```csharp
// PatchLog.cs - 使用独立线程 + 互斥锁保证线程安全
private static readonly object _locker = new object();

private static void WriteLog(string prefix, object message)
{
    // 主线程控制台输出
    Debug.Log(logMsg);
    
    // 文件写入：加锁保证线程安全
    if (_writeToFile && _writer != null)
    {
        lock (_locker)
        {
            _writer.WriteLine(fileMsg);
        }
    }
}
```

主线程的 `Debug.Log` 和文件写入线程互不干扰，保证了热更流程的稳定性。

### 6.2 AutoFlush 保证日志完整性

```csharp
_writer = new StreamWriter(_logFilePath, true, Encoding.UTF8);
_writer.AutoFlush = true; // 每次写入后立即刷新到磁盘
```

如果游戏崩溃，`AutoFlush = true` 确保最近的日志行已经写入磁盘，不会因缓存未刷新而丢失关键崩溃信息。

### 6.3 多层降级保障

整个热更流程设计了多层降级：

```
Dolphin 检测失败 → 跳过热更，直接进游戏
Catalog 加载失败 → 使用安装包内置 Catalog
DLL 加载失败     → 跳过该 DLL，继续加载其他 DLL
资源加载失败     → 打印错误，不崩溃（部分功能不可用）
```

这保证了即使热更流程出问题，玩家仍然能用安装包内的旧版本进入游戏，避免完全不可玩。

## 7. 常见问题与最佳实践

### Q1：如何拿到玩家手机上的热更日志？
**A**：
1. **Android ADB**：`adb pull /data/data/<包名>/files/PatchLogs/ ./logs/`
2. **iOS**：使用 Xcode 的 Devices & Simulators 工具下载 App Container
3. **远程文件管理**：使用 `HotUpdatePushWindow` 中的 RemoteFileExplorer 功能（需要设备在同一 WiFi 下连接到中转服务器）

### Q2：如何区分热更问题和代码 Bug 导致的崩溃？
**A**：查看崩溃堆栈：
- 如果崩溃发生在 `HybridCLR.Runtime` 相关的方法中，可能是热更 DLL 问题（如 AOT 泛型缺失）
- 如果崩溃发生在 `Addressables` 相关方法中，可能是资源加载问题
- 对比上次稳定版本的崩溃率，突增说明热更引入了新问题

### Q3：热更导致所有用户都崩溃，如何最快速恢复？
**A**：
1. **立即**：在 GCloud 控制台将版本回滚到上一个稳定版本（1-5 分钟内生效）
2. **同步**：通过游戏公告/社交媒体通知玩家"已发现问题，请重启游戏"
3. **排查**：从崩溃日志和 PatchLog 确认根本原因
4. **修复**：修复 Bug 后重新打包热更，在灰度环境验证后再全量发布

### Q4：如何预防热更引入崩溃问题？
**A**：
1. **灰度发布**：先对 5% 用户推送，观察崩溃率后再扩大
2. **热更前测试**：每次热更前在测试设备上完整跑一遍启动流程
3. **CI 自动化测试**：在 CI 流水线中运行自动化冒烟测试
4. **版本矩阵测试**：测试从不同历史版本升级到新版本的流程

### Q5：日志文件太大会影响性能吗？
**A**：`PatchLog` 使用独立线程写文件，主线程只做 `Debug.Log`（很快）。但长期运行下日志文件会累积：
- 建议定期清理（如每次启动时删除 7 天前的日志文件）
- 生产环境可以将 LogLevel 设置为 `LogLevel.Error`（只记录错误），大幅减少日志量
- 按日期分文件（`patch_log_YYYYMMDD.txt`）使得清理历史日志很方便

## 8. 总结

本项目的热更监控体系是一个**轻量但实用**的设计：

1. `PatchLog` 提供了开发期和生产期的日志记录能力
2. 全局异常捕获确保所有错误都有记录
3. 每个关键节点都有 Fallback 降级路径
4. `ClearOldPatch()` 提供了本地自愈能力
5. 服务端（GCloud 控制台）提供了快速回滚能力

对于初学者而言，理解这套体系的核心思想是：**监控是为了快速发现问题，回滚是为了快速缓解影响，降级是为了保证基本可用性**。在商业游戏项目中，这三者缺一不可。
