# 内存管理与对象池最佳实践

## 1. 系统概述

移动端游戏的内存管理直接影响帧率稳定性——GC（垃圾回收）是帧率抖动的主要原因之一。本项目通过多层次对象池策略，将战斗逻辑中的 GC 分配降至极低水平。

**主要优化手段：**
- `ObjectPool<T>`：通用对象池（ET 框架扩展版）
- `ListPool<T>`：容器对象池（避免频繁 new List<T>()）
- `ETTask` 对象池：异步任务对象复用
- `ZString`：零 GC 字符串格式化
- MemoryPack 二进制序列化（比 JSON 少 90% GC）

## 2. ObjectPool<T> 实现详解

### 2.1 接口定义

```csharp
// 位置：Scripts/Core/Pool/ObjectPool.cs
public interface IObjectPool<T> where T : class
{
    int CountInactive { get; }  // 池中空闲数量
    T Get();                    // 获取对象
    PooledObject<T> Get(out T v);  // RAII 模式获取
    void Release(T element);    // 归还对象
    void Clear();               // 清空池
}

// RAII 模式包装（using 语句自动归还，防止忘记 Release）
public struct PooledObject<T> : IDisposable where T : class
{
    private readonly T m_ToReturn;
    private readonly IObjectPool<T> m_Pool;
    
    void IDisposable.Dispose() => m_Pool.Release(m_ToReturn);
}
```

### 2.2 核心实现

```csharp
public class ObjectPool<T> : IDisposable, IObjectPool<T> where T : class
{
    // 共享池（全局单例，简单场景使用）
    public static ObjectPool<T> Shared = new ObjectPool<T>(null, null);
    
    private readonly Func<T> m_CreateFunc;         // 创建新对象的工厂函数
    private readonly Action<T> m_ActionOnGet;       // 获取时的初始化回调
    private readonly Action<T> m_ActionOnRelease;   // 归还时的清理回调
    private readonly Action<T> m_ActionOnDestroy;   // 池满时的销毁回调
    private readonly Stack<T> m_Stack = new();      // 空闲对象栈
    
    // 统计信息
    public int CountAll { get; private set; }       // 历史总创建数
    public int CountActive => CountAll - CountInactive; // 正在使用数
    public int CountInactive => m_Stack.Count;      // 空闲数
    
    // 构造器1：使用默认无参构造（T 必须有无参构造）
    public ObjectPool(Action<T> actionOnGet, Action<T> actionOnRelease)
    {
        m_CreateFunc = () => (T)Activator.CreateInstance(typeof(T));
        m_ActionOnGet = actionOnGet;
        m_ActionOnRelease = actionOnRelease;
    }
    
    // 构造器2：完整自定义
    public ObjectPool(Func<T> actionCreate, Action<T> actionOnGet, 
        Action<T> actionOnRelease, Action<T> actionOnDestroy = null)
    {
        m_CreateFunc = actionCreate;
        m_ActionOnGet = actionOnGet;
        m_ActionOnRelease = actionOnRelease;
        m_ActionOnDestroy = actionOnDestroy;
    }
    
    public T Get()
    {
        T element;
        lock (m_Stack)
        {
            if (m_Stack.Count == 0)
            {
                // 池空了，创建新对象
                element = m_CreateFunc.Invoke();
                CountAll++;
            }
            else
            {
                element = m_Stack.Pop();  // 从栈顶取出
            }
        }
        m_ActionOnGet?.Invoke(element);  // 初始化回调
        return element;
    }
    
    public void Release(T element)
    {
        m_ActionOnRelease?.Invoke(element);  // 清理回调
        lock (m_Stack)
        {
            m_Stack.Push(element);  // 归还到栈
        }
    }
}
```

## 3. RAII 模式使用示例

```csharp
// 方式一：手动 Get/Release（需要记住 Release，容易忘）
var list = ListPool<AttackInfo>.Get();
try
{
    // 使用 list...
}
finally
{
    ListPool<AttackInfo>.Release(list);  // 必须手动归还
}

// 方式二：RAII 模式（推荐）
using (var _ = pool.Get(out var list))
{
    // 使用 list...
}
// 离开 using 块时自动调用 Dispose → Release，零遗漏

// 在 YIUITweenComponentSystem 中的使用
var taskList = ListPool<Task>.Get();
try
{
    foreach (var tween in tweens)
    {
        taskList.Add(tween.OnShow());
    }
    await Task.WhenAll(taskList);
}
finally
{
    ListPool<Task>.Release(taskList);  // 确保归还
}
```

## 4. ETTask 对象池

```csharp
// 位置：Core/ETTask/ETTask.cs
// ETTask 是异步任务对象，每次创建会产生 GC
// 解决方案：通过 fromPool=true 参数从池中取出
public static ETTask Create(bool fromPool = false)
{
    if (!fromPool) return new ETTask();
    
    // 从池中取（池满时创建新对象）
    return ObjectPool<ETTask>.Shared.Get();
}

// 使用示例
public async ETTask DoSomethingAsync()
{
    var task = ETTask.Create(fromPool: true);  // 从池取
    try
    {
        // 异步操作...
        task.SetResult();
    }
    catch (Exception e)
    {
        task.SetException(e);
    }
    return task;
}

// ETTask 完成后，框架自动将其归还池中（在 GetResult 中调用 Return）
```

## 5. ZString 零 GC 字符串

```csharp
// 标准写法（有 GC）
string msg = string.Format("[战斗] 单位:{0} 受伤:{1}", unit.Id, damage);
Log.Info(msg);   // 创建了两个临时字符串

// ZString 写法（零 GC）
Log.Info(ZString.Format("[战斗] 单位:{0} 受伤:{1}", unit.Id, damage));

// StringBuilder 模式（多段拼接）
using (var sb = ZString.CreateStringBuilder())
{
    sb.Append("[技能] ");
    sb.Append(skillName);
    sb.Append(" 伤害:");
    sb.AppendFormat("{0}", damage);
    Log.Info(sb.ToString());  // 只在 ToString 时分配一次
}
```

## 6. MemoryPack 二进制序列化

项目使用 MemoryPack 替代 JSON 序列化，GC 减少 90%+：

```csharp
// JSON 序列化（有 GC，慢）
string json = JsonConvert.SerializeObject(data);  // 大量字符串分配
var restored = JsonConvert.DeserializeObject<Data>(json);

// MemoryPack（零 GC，快 10 倍）
byte[] bytes = MemoryPackSerializer.Serialize(data);  // 直接写入 byte[]
var restored = MemoryPackSerializer.Deserialize<Data>(bytes);

// 要使用 MemoryPack，数据类需要标注 [MemoryPackable]
[MemoryPackable]
public partial class RootMotionData
{
    public List<TSVector> rootMotionPositions;
    public List<TSQuaternion> rootMotionRotations;
}
```

## 8. 常见问题与最佳实践

**Q: 对象池大小多少合适？**  
A: 初始化时预热一定数量（如战斗开始时预创建 50 个 SkillGraph 实例），避免战斗中动态创建。最大容量根据峰值使用量 * 1.5 设置，超出部分直接销毁。

**Q: 对象归还后如何确保没有残留数据？**  
A: 归还时调用 `m_ActionOnRelease` 清理函数，重置所有字段。对于包含引用类型的对象，必须将引用置 null，防止 GC 根引用泄漏。

**Q: `lock(m_Stack)` 会不会有性能问题？**  
A: 战斗逻辑单线程执行，lock 无竞争，开销极小（JIT 会将无竞争锁优化为几纳秒的原子操作）。如确认单线程场景，可移除 lock 进一步优化。

**Q: MemoryPack 需要特殊处理 Unity 类型（Vector3 等）吗？**  
A: Unity 内置类型（Vector3、Quaternion 等）需要注册自定义 Formatter。项目中战斗系统使用的是 TrueSync 的 TSVector（定点数），已实现 MemoryPack 序列化接口。
