## 1. 系统概述

社交系统是提升玩家留存和粘性的重要基础设施，在本项目中主要体现在以下几个维度：

| 维度 | 实现形式 | 代码位置 |
|---|---|---|
| 邮件（Mail） | 系统通知、奖励发放 | `Hotfix/Function/Biz/Mail/` |
| 助战（AssistCard） | 借用其他玩家的角色卡 | `Player/AssistCard/` `Player/AssistanceCardItemData/` |
| 反思卡（ReflectionCard） | 玩家间互动道具 | `Player/ReflectionCard/` |
| PVP 竞技 | 间接社交竞争 | `GamePlay/PVP/` |
| 角色关系 | 剧情内角色间关系 | `Story/StoryLogic/StoryLogicCondition.cs` |

> 注：本项目当前代码中尚未见到独立的"好友系统"或"公会系统"目录，相关功能通过系统功能开关（`SystemEntry`）管理，底层接入腾讯 MSDK 平台能力（QQ/微信好友、部落等）。本文将重点讲解当前已有的社交相关实现，并阐述完整好友/公会系统的标准设计方案。

- 系统公告下发
- 奖励发放（背包满时转存邮件）
- 玩家间礼物（如助战回报）

### 2.2 助战卡系统（AssistCard）

助战系统允许玩家在 PVP 或养成战斗中借用好友/随机玩家的角色卡：

```

```

助战卡的使用在养成战斗准备阶段由 `DungeonComponentSystem` 处理。

### 2.3 PVP 竞技场（间接社交）

PVP 竞技（天平赛）是玩家间最直接的社交竞争，玩家双方不在线同时对战，而是"离线 AI 对战"：
- 玩家的阵容配置由服务端保存；
- 对战时服务端下发对手的阵容数据（`CellBoxInfoMsg`）；
- 客户端模拟战斗并上报结果。

### 2.4 角色关系系统（剧情内）

在剧本内，角色之间存在关系值（Relation），影响剧情走向和养成结果。相关判断逻辑：

```csharp
// StoryLogicCondition.cs
case condition.SL_RELATION_CHARACTER:
    return SL_RELATION_CHARACTER(comp, conditionParam);  // 检查角色间关系

case condition.SL_RELATION_POSITION:
    return SL_RELATION_POSITION(comp, conditionParam);   // 检查位置间关系
```

## 3. 标准好友/公会系统设计方案

### 3.1 好友系统架构

如需开发独立好友系统，标准设计如下：

```
FriendComponent（好友组件）
├── FriendList:    List<FriendData>    好友列表
├── BlackList:     List<FriendData>    黑名单
├── ApplyList:     List<FriendData>    申请列表
└── RecommendList: List<FriendData>    推荐列表

FriendData（好友数据）
├── GID:          long                  玩家 GID
├── Name:         string                昵称
├── Level:        int                   等级
├── ResidentIP:   IPCharacterEnum       常驻角色
├── OnlineStatus: EFriendOnlineStatus   在线状态
└── LastOnlineTime: long                最后在线时间
```

### 3.2 好友系统事件流

```
添加好友流程：
玩家点击"添加好友"
      │
      ▼
RequestFriendApply(targetGID)
      │  C/S
      ▼
服务端记录申请 → 推送给目标玩家
      │
      ▼
目标玩家收到 FriendApplyNotify
      │
      ▼
同意 → RequestFriendAccept → 双方 FriendList 更新
拒绝 → RequestFriendReject → 申请移除
```

### 3.3 公会系统架构

```
GuildComponent（公会组件）
├── GuildID:          int                公会 ID
├── GuildName:        string             公会名称
├── GuildLevel:       int                公会等级
├── MemberList:       List<GuildMember>  成员列表
├── GuildFunds:       long               公会资金
└── GuildTasks:       List<GuildTask>    公会任务

GuildMember（公会成员）
├── GID:        long
├── Name:       string
├── Position:   EGuildPosition          会长/副会长/成员
├── Contribution: long                  贡献值
└── JoinTime:   long
```

### 4.4 PVP 结算处理

```csharp
// X:\UnityProj\Assets\Scripts\Hotfix\Function\GamePlay\PVP\PVPSettlementComponentSystem.cs

/// <summary>
/// 处理 PVP 单局结算（间接社交：记录与其他玩家的胜负）
/// </summary>
public static void HandleRoundSettlement(
    this PVPSettlementComponent self,
    ZoneFightPvpGameEndNotify notify)
{
    // 更新双方得分
    self.Arena().PlayerScore   += notify.PlayerWin   ? 1 : 0;
    self.Arena().OpponentScore += notify.PlayerWin   ? 0 : 1;

    // 记录结算 ID（用于结算界面展示）
    self.Arena().RoundSettlementIds.Add(notify.SettlementId);

    // 触发 UI 更新
    EventSystem.Instance.Publish(self.ClientScene(),
        new Evt_PVPRoundSettlement { IsPlayerWin = notify.PlayerWin });
}
```

### 4.5 条件触发中的关系检查

```csharp
// X:\UnityProj\Assets\Scripts\Hotfix\Function\GamePlay\Story\StoryLogic\StoryLogicCondition.cs

/// <summary>
/// 检查指定角色与指定角色之间的关系值
/// 参数格式: "角色A_IP|角色B_IP|关系类型|比较符|目标值"
/// </summary>
private static bool SL_RELATION_CHARACTER(StoryComponent comp, string conditionParam)
{
    // 参数解析
    var parts = conditionParam.Split(SplitChar);
    var ipA         = (IPCharacterEnum)int.Parse(parts[0]);
    var ipB         = (IPCharacterEnum)int.Parse(parts[1]);
    var relationType = int.Parse(parts[2]);
    var compareOp   = parts[3];
    var targetValue = float.Parse(parts[4]);

    // 从养成组件获取角色关系值
    var cultivationComp = comp.GetCultivationComp();
    var charA = cultivationComp.GetCharacterByIP(ipA);
    var charB = cultivationComp.GetCharacterByIP(ipB);
    if (charA == null || charB == null) return false;

    var relationValue = cultivationComp.GetRelationValue(charA, charB, relationType);

    // 数值比较
    return CompareValue(relationValue, compareOp, targetValue);
}
```

## 5. 设计亮点

### 5.1 MSDK 平台能力的复用
通过接入腾讯 MSDK，游戏直接复用 QQ/微信的好友关系链，无需自建好友系统，大幅降低冷启动成本。在 `ClientEntryEvent.cs` 中已完成 `GCloud.MSDK.MSDK.Init()` 初始化，后续调用 MSDK 的好友相关 API 即可。

### 5.2 PVP 异步对战降低服务器压力
玩家不需要同时在线，通过存储阵容快照的方式实现"离线挑战"，服务器只需保存阵容数据和结算结果，无需维持长连接对战服务，大幅降低并发压力。

### 5.3 关系值作为剧情分支的触发条件
将角色间关系值纳入剧情条件判断体系（`StoryLogicCondition`），使得"社交关系"能直接影响剧情走向，提升了玩法的深度和可玩性。

### 5.4 三路奖励通知的社交闭环
助战奖励、PVP 胜利奖励等都通过 `AwardNotify` 三路（直发/邮件/展示）统一处理，助战好友会收到感谢邮件，形成社交正向反馈。

## 6. 常见问题与最佳实践

### Q1: 如何开发独立的好友列表 UI？
1. 创建 `FriendComponent` 组件，定义好友列表数据；
2. 在登录时请求好友列表（`RequestFriendList`）；
3. 注册好友状态变更通知处理器；
4. UI 层订阅好友数据变更事件刷新列表；
5. 参考背包系统的事件驱动模式即可。

### Q2: PVP 的对手阵容是如何获取的？
服务端在匹配成功后，通过 `ZoneFightPvpGameBeginNotify` 下发对手的 `CellBoxInfoMsg`（阵容数据），客户端调用 `PVPGetTeamSaveDataFromMsg` 转换为本地战斗数据格式。

### Q3: 公会任务和普通任务有什么区别？
公会任务是共享型任务，多个成员可以分别贡献进度，完成后所有成员都能领取奖励。普通任务是个人任务，进度独立。如需实现公会任务，需要在服务端维护共享进度，客户端通过实时通知更新显示。

### Q4: 如何防止玩家频繁发送社交请求（限流）？
在 `RequestXxx` 方法调用前，增加本地 CD 检查：
```csharp
private long _lastRequestTime;
public async ETTask<bool> RequestFriendApply(long targetGID)
{
    // 防刷：2秒内不能重复请求
    if (TimeHelper.ClientNow() - _lastRequestTime < 2000)
    {
        ShowTips("操作太频繁，请稍后再试");
        return false;
    }
    _lastRequestTime = TimeHelper.ClientNow();
    // ... 发送网络请求
}
```

### Q5: 社交数据（好友列表）需要做本地缓存吗？
建议做**轻量级本地缓存**（记录最近在线的好友列表），在网络请求完成前先展示缓存数据，请求完成后刷新，提升 UI 响应速度。但要注意缓存时效性，登录后必须强制刷新一次。
