# 02 角色系统与队伍管理

> 面向对象：刚入行的游戏客户端开发工程师

## 1. 系统概述

角色系统（Character System）负责管理游戏中所有 **IP 角色**（VTuber/偶像角色）的客户端数据，包括：
- 角色解锁列表的维护
- 常驻角色（陪伴角色）的设置与获取
- 角色好感度（Favor Level）的更新
- 角色星图序列（解锁顺序）的管理

队伍管理（Team Management）作为养成系统的子模块，负责：
- 养成剧本中队伍成员的组合
- 队伍属性（粉丝数、评分、赛点等）的实时计算
- 各角色在队伍内的位置、状态、关系的检查

## 2. 架构设计

### 2.1 数据结构概览

```
Scene (客户端场景)
  └─ CharacterComponent              角色组件
       ├── IPSequence: List<IPCharacterEnum>       角色星图解锁序列
       ├── ResidentCharacterIP: IPCharacterEnum    常驻角色 IP
       └── IPCharacters: List<IPCharacterData>     已解锁角色列表
              └── IPCharacterData
                    ├── IP: IPCharacterEnum
                    ├── FavorLevel: int           好感度等级
                    ├── FavorValue: int           好感度数值
                    └── ...（等级、解锁状态等）
```

### 2.2 CharacterComponent 与 CultivationComponent 的关系

- `CharacterComponent`：**持久化角色数据**，登录即同步，贯穿整个游戏生命周期。
- `CultivationComponent` 中的角色数据：**剧本内临时数据**，进入剧本后初始化，结束后销毁或结算。

两者通过 IP 枚举 `IPCharacterEnum` 关联，养成系统在需要展示角色基础信息时会向 `CharacterComponent` 查询。

### 2.3 队伍属性检查器（Checker）模式

队伍属性检查采用**策略模式**，每种队伍属性对应一个独立的 `Checker` 类：

```
ICultivationTeamPropertyChecker
  ├── CultivationFansChecker             粉丝数检查
  ├── CultivationPointsChecker           赛点检查
  ├── CultivationFansSupportRateChecker  粉丝支持率检查
  ├── CultivationRatingChecker           评分检查
  ├── CultivationMatchChecker            赛程检查
  ├── CultivationScheduleWatchCharacterChecker  日程观看角色检查
  ├── CultivationNightHealingCharacterChecker   夜间疗愈角色检查
  ├── CultivationBubbleStoryChecker      气泡剧情检查
  ├── CultivationActiveAssistCardChecker 助战卡检查
  ├── CultivationAssistCharacterChecker  助战角色检查
  └── CultivationInquiryChecker          询问状态检查
```

同样，角色属性也有对应的检查器：

```
ICultivationCharacterPropertyChecker
  ├── CultivationCharacterLeaderChecker       队长
  ├── CultivationCharacterPositionChecker     位置
  ├── CultivationCharacterFormChecker         阵型
  ├── CultivationCharacterPressureChecker     压力值
  ├── CultivationCharacterAttributeChecker    属性
  ├── CultivationCharacterSpecialityChecker   特长
  ├── CultivationCharacterRelationToPlayerChecker   与玩家的关系
  ├── CultivationCharacterRelationToCharacterChecker 角色间关系
  ├── CultivationCharacterStatusChecker       状态
  ├── CultivationCharacterScheduleChecker     日程安排
  └── CultivationCharacterScheduleLockChecker 日程锁定
```

这种设计的优势：**新增属性类型只需新增一个 Checker 类并在 `RefreshCharacterCheckers` 中注册即可，无需修改核心逻辑**。

## 3. 核心代码展示

### 3.1 从登录数据同步角色列表

```csharp
// X:\UnityProj\Assets\Scripts\Hotfix\Function\GamePlay\Player\Character\CharacterComponentSystem.cs

/// <summary>
/// 从登录同步角色列表
/// </summary>
public static void SyncCharacterListFromLogin(this CharacterComponent self, PlayerVtuberInfoMsg info)
{
    var ipCharacterList = info?.IpCharacterList;
    if (ipCharacterList == null)
    {
        Log.LogError("[Character][Network] 返回的 IP 角色列表为空");
        return;
    }

    var characters = new List<IPCharacterData>();
    foreach (var ipCharacter in ipCharacterList)
    {
        var characterCfg = CfgManager.tables.TbIPCharacter
            .GetOrDefault((IPCharacterEnum)ipCharacter.VtuberIpId);
        if (characterCfg == null)
            continue;

        // 只处理"可选择"类型的角色（过滤非展示角色）
        if (characterCfg.CharacterModuleType != ECharacterModuleType.Selection)
            continue;

        characters.Add(ServerDataConverter.GetIPCharacterData(ipCharacter));
    }

    self.IPCharacters = characters;
}

/// <summary>
/// 从登录同步常驻角色数据
/// </summary>
public static void SyncResidentCharacterFromLogin(this CharacterComponent self, PlayerVtuberInfoMsg info)
{
    var residentCharacter = info?.ResidentCharacterInfo;
    if (residentCharacter == null)
    {
        Log.LogError("[Character][Network] 返回的常驻角色数据为空");
        return;
    }
    self.ResidentCharacterIP = (IPCharacterEnum)residentCharacter.IpId;
}

/// <summary>
/// 从登录同步角色星图序列
/// </summary>
public static void SyncUnlockIPSequence(this CharacterComponent self, PlayerVtuberInfoMsg info)
{
    var sequence = info.UnlockCircle;
    self.IPSequence = ServerDataConverter.ToList(sequence, i => (IPCharacterEnum)i);
}
```

### 3.2 从剧本结算同步好感度

```csharp
/// <summary>
/// 从剧本结算同步角色好感度
/// </summary>
public static void SyncCharacterFavorFromCultivateScriptComplete(
    this CharacterComponent self,
    RepeatedField<CellBoxCharacterUpdateInfoMsg> characterInfos)
{
    if (characterInfos == null) return;

    foreach (var characterInfo in characterInfos)
    {
        var character = self.GetUnlockedIPCharacter((IPCharacterEnum)characterInfo.IpId);
        if (character == null) continue;

        // 通过好感值反查等级（等级与数值的换算在 CultivationUtil 中）
        character.FavorLevel = CultivationUtil.GetLevelByFavorValue(characterInfo.Favorability);
    }
}
```

    this CultivationComponent self, IPCharacterEnum characterIP)

## 4. 设计亮点

### 4.1 IP 枚举统一标识
所有角色都通过 `IPCharacterEnum` 枚举值唯一标识，既保证了类型安全，又便于配置表映射（表格中填数值，代码里用枚举，杜绝魔法数字）。

### 4.2 索引 Map 加速查找
`CharacterIPIndexMap`（`Dictionary<int, int>`）存储"IP → 数组索引"的映射，使得"按 IP 查角色"的时间复杂度从 O(n) 降为 O(1)，在帧率敏感的战斗逻辑中尤其重要。

### 4.3 Checker 策略模式的延迟初始化
```csharp
internal static Dictionary<int, ICultivationTeamPropertyChecker> GetTeamCheckers(
    this CultivationComponent self)
{
    // 懒加载：第一次调用时才初始化
    if (self.TeamCheckers == null)
        self.RefreshTeamCheckers();
    return self.TeamCheckers;
}
```
通过懒加载避免了 Component 初始化时的性能峰值，同时支持在运行时动态刷新检查器（如服务端下发新配置后重建）。

### 4.4 养成 vs 持久化数据隔离
`CharacterComponent` 的好感度等持久化数据只在**剧本结算时**通过明确的 `Sync*` 方法更新，日常 UI 展示读 `CharacterComponent`，剧本内逻辑读 `CultivationComponent`，两套数据清晰隔离，避免中间状态污染。

## 5. 常见问题与最佳实践

### Q1: 新增一个 IP 角色需要改哪些地方？
1. 策划在配置表 `TbIPCharacter` 中新增角色行（包括名字、属性、模块类型等）；
2. 在 `IPCharacterEnum` 枚举中添加新值（通常自动生成）；
3. 如果有特殊逻辑，在对应的 `CharacterChecker` 或扩展方法中处理；
4. **客户端代码层面通常不需要改动**，配置表驱动设计的优势所在。

### Q2: 为什么角色列表过滤了 `CharacterModuleType != Selection` 的角色？
游戏中有些角色是"非选择型"的（如仅用于剧情的 NPC 角色），这类角色不需要出现在玩家的角色列表中，所以在同步时就提前过滤，减少后续逻辑的复杂度。

### Q3: 如何添加一种新的队伍属性检查？
1. 定义新的 `ECultivationTeamPropertyType` 枚举值；
2. 创建实现 `ICultivationTeamPropertyChecker` 接口的新 Checker 类；
3. 在 `RefreshTeamCheckers` 方法中注册该 Checker；
4. 在需要检查的地方调用 `self.GetTeamChecker(propertyType).Check(...)` 即可。

### Q4: 好感度等级和好感度数值是什么关系？
好感度数值（FavorValue）是服务端存储的原始数值，好感度等级（FavorLevel）是客户端展示用的等级，通过 `CultivationUtil.GetLevelByFavorValue(value)` 换算。换算规则在配置表中定义，策划可以自由调整升级阈值而无需修改代码。

### Q5: 常驻角色（ResidentCharacter）是什么概念？
常驻角色是玩家在"主界面"选择的"陪伴角色"，会常驻展示在主界面的特定位置，并影响部分剧情和对话内容。通过 `CharacterComponent.ResidentCharacterIP` 标识，调用 `SetResidentIPCharacter` 方法修改并同步到服务端。
