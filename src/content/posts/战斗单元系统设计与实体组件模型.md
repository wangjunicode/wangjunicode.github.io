# 战斗单元系统设计与实体组件模型

在战斗系统中，"Unit"（战斗单元）是一切逻辑的载体。无论是玩家角色、怪物，还是投射物、特殊道具，都是 Unit 的不同表现形式。本文深入剖析 Unit 的数据模型和组件组装逻辑，帮助你理解"实体-组件（ECS 思想）"在商业项目中的实际应用。

---

## 第一性原理：为什么用 Entity-Component 模式？

传统面向对象设计会用继承来区分不同类型的角色：

```
Character
  ├── Player
  │     ├── Warrior
  │     └── Mage
  └── Monster
        ├── Boss
        └── Mob
```

这在类型少时可行，但随着游戏复杂度上升，继承树变得难以维护：
- 想给"Boss级别的怪物"加上"类玩家的技能系统"？
- 想让"特殊道具"拥有"移动能力"但不需要"HP"？

继承的组合爆炸让这些变得非常棘手。

**Entity-Component 模式的解决思路**：将"是什么"（类型）和"能做什么"（能力）分离。Entity 只是一个 ID，真正的能力由动态挂载的 Component 提供。

## Unit 的核心数据结构

```csharp
[MemoryPackable]
public partial class Unit : Entity, IAwake<int>, IDestroy, ITransform, IUniAgent
{
    public int LogicId { get; set; }    // 逻辑唯一ID
    public int ConfigId { get; set; }   // 配置表ID
    public EEntityType Type { get; set; } // 实体类型
    public int TeamCode { get; set; }   // 队伍编号
    
    // 坐标系（定点数，保证帧同步一致性）
    private TSVector position;
    private TSQuaternion rotation;
    public TSVector localScale = TSVector.one;
    
    // 父子关系
    public Unit RootUnit { get; set; }   // 创建者
    public Unit FollowUnit { get; set; } // 跟随对象
    
    // 运行时状态
    public bool Freeze => this.FreezeFrames > 0;
    public bool BattlePause => this.battlePause.Count > 0;
}
```

### 几个关键设计点

**① `[MemoryPackable]` 序列化支持**

Unit 被标记为可序列化，这是战斗录像/回放系统的基础——通过序列化 Unit 快照，可以在任意时间点还原战斗状态。

**② 定点数坐标**

```csharp
private TSVector position;   // TrueSync Vector，定点数
private TSQuaternion rotation; // TrueSync Quaternion，定点数
```

所有位置、旋转都使用定点数，这是帧同步战斗的铁律。

**③ 多层暂停机制**

```csharp
private HashSet<EPauseFlag> battlePause = new HashSet<EPauseFlag>();

public enum EPauseFlag
{
    common = 0,       // 普通暂停
    puppetShow = 1,   // 傀儡演出暂停
}
```

用 HashSet 而不是单一 bool 值来管理暂停，是因为多个系统可能同时希望暂停同一个 Unit。只有当所有暂停请求都撤销后，Unit 才真正恢复运行。这避免了"A 暂停了，B 强制恢复，导致 A 的暂停失效"的问题。

**④ 帧冻结（顿帧）**

```csharp
public bool Freeze => this.FreezeFrames > 0;
public bool PauseOrFreeze => Pause || Freeze;
```

帧冻结是动作游戏打击感的核心——在命中瞬间让目标停顿几帧，产生"硬直"感。通过计数器实现，每帧减1，自动恢复。

## Unit 类型与组件装配

不同类型的 Unit 需要不同的组件集合，`UnitFactory` 负责按类型装配：

```csharp
public enum EEntityType
{
    None,
    Player,      // 玩家角色
    Monster,     // 怪物
    BattleItem,  // 战斗道具（投射物等）
    Puppet,      // 傀儡（附属单位）
    Director,    // 导演（场景控制器）
}
```

### Player / Monster 的组件集合（最完整）

```csharp
if (unit.Type == EEntityType.Player || unit.Type == EEntityType.Monster)
{
    unit.AddComponent<PuppetOwnerComponent>();  // 傀儡拥有者
    unit.GetOrAddComponent<BuffComponent>();    // Buff 系统
    unit.AddComponent<PhysicsComponent>();      // 物理
    unit.AddComponent<SimpleMoveComponent>();   // 移动
    unit.AddComponent<FSMComponent>();          // 状态机
    unit.AddComponent<VirtualInputComponent>(); // 虚拟输入
    unit.AddComponent<AttackComponent>();       // 攻击
    unit.AddComponent<HurtComponent>();         // 受击
    unit.GetOrAddComponent<BlackboardComponent>(); // 黑板（AI数据）
    unit.AddComponent<ColliderComponent>();     // 碰撞
    var slotMgr = unit.GetOrAddComponent<PassiveSlotComponent>();
    slotMgr.AddOrGetSlot(EPSkillSlotType.Character); // 被动技能槽
}
```

可以看出，一个完整的战斗单元需要：
- 物理系统（位置计算）
- 状态机（行为控制）
- 输入处理（响应玩家/AI）
- 攻击/受击（战斗核心）
- 碰撞检测（判定是否命中）
- Buff 系统（状态效果）

### BattleItem（战斗道具）的精简组件

```csharp
else if (unit.Type == EEntityType.BattleItem)
{
    unit.AddComponent<SimpleMoveComponent>();
    unit.AddComponent<PhysicsComponent>();
    unit.AddComponent<FSMComponent>();
    var atkcomp = unit.AddComponent<AttackComponent>();
    atkcomp.Turn = unitInfo.CreateTurn;  // 记录创建的回合，避免自伤
    unit.AddComponent<ColliderComponent>();
}
```

投射物没有受击组件（不能被打到），没有 Buff 系统（不需要状态效果），没有玩家输入。这就是组件模式的威力——按需装配，没有冗余。

### Director（导演）的特殊组件

```csharp
else if (unit.Type == EEntityType.Director)
{
    var slotMgr = unit.GetOrAddComponent<PassiveSlotComponent>();
    slotMgr.AddOrGetSlot(EPSkillSlotType.Scene);  // 场景级被动技能
    unit.GetOrAddComponent<BuffComponent>();
}
```

Director 是一个特殊的不可见"单位"，用于控制整个战斗场景。它没有物理位置，没有碰撞，只有场景级的被动技能和 Buff，用于实现关卡特殊规则（如"每3秒所有单位获得1层buff"）。

## 坐标系操作

Unit 实现了 `ITransform` 接口，提供了完整的坐标变换操作：

```csharp
// 从世界空间转换到本地空间
public TSVector InverseTransformPoint(TSVector p)
{
    return TSMatrix4x4.Inverse(matrix4x4).MultiplyPoint3x4(p);
}

// 从本地空间转换到世界空间
public TSVector TransformPoint(TSVector p)
{
    return matrix4x4.MultiplyPoint3x4(p);
}

// 渲染帧插值移动（不立即更新渲染坐标）
public void LerpTransform(TSVector pos, TSQuaternion rot, TSVector scale)
{
    this.position = pos;
    if (LockX) position.x = 0;  // 可以锁定轴向
    if (LockZ) position.z = 0;
    this.rotation = rot;
    this.localScale = scale;
}

// 瞬移（触发位置同步事件）
public void TeleportTransform(TSVector pos, TSQuaternion rot)
{
    var beforePos = Position;
    EventSystem.Instance.Publish(this.DomainScene(), 
        new Evt_UnitWrap { Unit = this, BeforePosition = beforePos, AfterPosition = pos });
}
```

注意 `LerpTransform` 和 `TeleportTransform` 的区别：
- `LerpTransform` 只更新逻辑坐标，不立即通知视图更新，用于帧内的插值计算
- `TeleportTransform` 发布事件通知渲染层，用于真正的位置跳变（如传送、击飞落地）

## 实体关系系统

Unit 之间存在树形父子关系，系统提供了多种查询方式：

```csharp
// 按 Identifier 查找子 Unit
public static int SearchEntity(this Unit self, string identifier, List<Unit> results, bool includeChild = true)
{
    foreach (var child in self.Children.Values)
    {
        var unit = child as Unit;
        if (unit?.Identifier == identifier || string.IsNullOrEmpty(identifier))
        {
            results.Add(unit);
        }
        if (includeChild)
        {
            count += unit.SearchEntity(identifier, results, includeChild);
        }
    }
    return count;
}

// 按范围查找
public static int SearchEntityInRange(this Unit self, TSVector searchCenter, float range, List<Unit> results)
{
    foreach (var child in self.Children.Values)
    {
        var unit = child as Unit;
        if ((unit.Position - searchCenter).sqrMagnitude <= range * range)
        {
            results.Add(unit);
        }
    }
}
```

更重要的是"相对实体"查询系统：

```csharp
public static void GetRelativeEntity(this Entity entity, EOwnerType ownerType, bool isEnemy, List<Entity> lst)
{
    var team = GetRelativeEntity(entity, true, isEnemy) as TeamEntity;
    
    switch (ownerType)
    {
        case EOwnerType.Character:  // 当前角色
        case EOwnerType.Team:       // 所属队伍
        case EOwnerType.AllMember:  // 队伍所有成员
        case EOwnerType.OtherMember:// 除自己外的队友
        case EOwnerType.Main:       // 上场主角
    }
}
```

这套查询系统让技能/Buff 配置时只需要填写"目标类型"，不需要硬编码具体目标，极大提升了配置灵活性。

## 主机标志（bHost）

```csharp
public bool bHost
{
    get { return _bHost; }
    set
    {
        if (_bHost != value)
        {
            _bHost = value;
            var fsmComp = GetComponent<FSMComponent>();
            fsmComp.SetHostState(_bHost);  // 同步状态机模式
        }
    }
}
```

`bHost` 是多人战斗中区分"本地主机控制"和"远端同步"的标志。非主机的 Unit 只接收网络同步的状态数据，不执行本地预测逻辑，避免双重计算。

## 组件查找优化

在大量频繁调用的代码路径中，组件查找性能至关重要：

```csharp
// 带缓存的组件获取
public static BattleComponent GetBattleComponent(this Entity entity)
{
    return entity.CurrentScene().GetComponent<BattleComponent>();
}
```

每次调用 `GetComponent<T>()` 都涉及字典查找（O(1) 但仍有开销）。对于每帧都需要访问的核心组件，实际项目中会在 Awake 时缓存引用：

```csharp
// BattlePlayerComponentSystem.cs 中的做法
private static void Awake(this BattlePlayerComponent self, int frameInterval)
{
    self.battleStateComp = self.ClientScene().GetComponent<BattleStateComponent>();
    self.worldComp = self.Parent.GetComponent<WorldComponent>();
    // ↑ Awake 时缓存，避免每帧查找
}
```

## 生命周期管理

Unit 的创建和销毁都通过事件系统通知其他模块：

```csharp
// 创建时
EventSystem.Instance.Publish(unit.DomainScene(), new Evt_UnitCreate_View { Unit = unit });
unit.ClientScene().GetComponent<EventDispatcherComponent>()
    .FireEvent(new Evt_AfterUnitCreate { Unit = unit });

// 销毁时
public class UnitDestroySystem : DestroySystem<Unit>
{
    protected override void Destroy(Unit self)
    {
        EventSystem.Instance.Publish(self.DomainScene(), 
            new Evt_BeforeUnitDelete() { Unit = self });
    }
}
```

销毁时发布 `Evt_BeforeUnitDelete`，让各个系统有机会在单位真正消失前做清理工作（如取消技能、释放特效、更新UI）。

## 总结：为什么这样设计 Unit

| 设计选择 | 原因 |
|---------|------|
| Entity-Component 而非继承 | 灵活组合，避免类型爆炸 |
| 定点数坐标 | 帧同步多人一致性 |
| HashSet 管理暂停标志 | 支持多源暂停请求，避免竞争 |
| FreezeFrames 计数器 | 自动恢复的顿帧效果 |
| 工厂方法按类型装配 | 集中管理组件逻辑，便于维护 |
| 事件通知生命周期 | 解耦各子系统，便于扩展 |

`Unit` 的设计是整个战斗系统的地基。理解了 Unit 的数据模型，后续的技能、Buff、碰撞、AI 等系统才能真正理解其数据流向。
