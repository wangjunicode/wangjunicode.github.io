# 碰撞检测系统与有向包围盒算法

在动作游戏中，"打中没有"是最核心的体验问题之一。如果碰撞检测不准确，要么打空但扣血，要么明明打中却没效果，直接破坏战斗可信感。本文深入解析一套商业级战斗碰撞系统，从 OBB 数学原理到多种碰撞盒类型的工程设计，全面剖析。

---

## 第一性原理：为什么不用 Unity 的物理系统？

Unity 内置了 PhysX 物理引擎，有 `BoxCollider`、`SphereCollider` 等，为什么还要自己实现一套碰撞系统？

**原因一：帧同步一致性**

PhysX 是浮点运算，在不同 CPU 上可能产生微小差异。帧同步战斗要求所有客户端结果完全一致，必须使用定点数（TrueSync）实现的碰撞系统。

**原因二：精确控制**

战斗碰撞有精细的时序控制：
- 攻击盒只在动作特定帧开启
- 受击盒根据状态（格挡、闪避）接受不同类型的攻击
- 同一个攻击不能对同一个目标多次判定

Unity PhysX 的触发器/回调系统很难实现这种精确控制。

**原因三：多种碰撞类型**

格挡判定、闪避判定、正常攻击判定逻辑不同，用统一框架更容易管理。

## 碰撞盒类型体系

```csharp
public enum EColliderCheckType
{
    Attack,     // 正常攻击判定
    Parry,      // 格挡（弹反）判定
    Doge,       // 闪避判定
    Perform,    // 表演攻击（不扣血的演出用碰撞）
    FakeAttack, // 虚假攻击（某些特殊机制）
}
```

每种类型有独立的碰撞盒集合和判定逻辑：

```csharp
public class ColliderComponent : Component
{
    public List<ColliderInfo> AttackColliders = new();        // 攻击盒
    public List<ColliderInfo> HurtColliders = new();          // 受击盒
    public List<ColliderInfo> PerformAttackColliders = new(); // 表演攻击盒
    public List<ColliderInfo> PreParryAttackColliders = new();// 格挡预判攻击盒
    public List<ColliderInfo> PreDogeAttackColliders = new(); // 闪避预判攻击盒
    public List<ColliderInfo> FakeAttackColliders = new();    // 虚假攻击盒
    
    // 受击盒的默认盒（角色始终存在）
    public ColliderInfo tempHurtCollider;
    
    // 每帧结束时跳过标志
    public bool SkipThisFrame { get; set; }
    
    // 每种类型上一帧是否命中（用于逻辑决策）
    public Dictionary<EColliderCheckType, bool> bLastCollideHit = new();
}
```

受击盒支持细粒度的接受控制：

```csharp
public class ColliderInfo
{
    public EColliderType colliderType;  // Box（目前只支持Box）
    public TSVector Center;    // 中心偏移
    public TSVector Size;      // 尺寸
    public TSQuaternion Rotation; // 旋转（相对于Unit朝向）
    
    // 接受标志（可以细粒度控制哪些攻击类型能命中）
    public bool acceptAttack = true;   // 是否接受普通攻击
    public bool acceptParry = true;    // 是否接受格挡判定
    public bool acceptDoge = true;     // 是否接受闪避判定
    public bool acceptPerform = true;  // 是否接受表演攻击
    public bool acceptFakeAttack = true;
    
    public TSVector attackPoint; // 攻击点（用于多碰撞体时选择最近的）
}
```

这套设计非常灵活：
- 格挡状态下，将受击盒的 `acceptParry = true`，其他类型的攻击盒才能触发"被格挡"
- 无敌帧时，临时关闭所有 `accept*`，攻击穿透
- 特定身体部位的受击盒可以只接受特定类型

## OBB（有向包围盒）碰撞检测

```csharp
public static bool Intersects(Unit unit1, ColliderInfo collider1, Unit unit2, ColliderInfo collider2)
{
    if (collider1.Size == TSVector.zero || collider2.Size == TSVector.zero)
    {
        return false;  // 尺寸为零的盒不参与检测
    }
    
    if (collider1.colliderType == EColliderType.Box && collider2.colliderType == EColliderType.Box)
    {
        var scale = unit1.LocalScale.x;
        var otherScale = unit2.LocalScale.x;

        // 构建 OBB1：Unit1的位置 + Unit1朝向旋转的偏移 × 缩放
        var obb1 = OBB.Create(
            unit1.Position + unit1.Rotation * collider1.Center * scale,
            unit1.Rotation * collider1.Rotation * TSQuaternion.identity,
            collider1.Size * scale);
        
        // 构建 OBB2
        var obb2 = OBB.Create(
            unit2.Position + unit2.Rotation * collider2.Center * otherScale,
            unit2.Rotation * collider2.Rotation * TSQuaternion.identity,
            collider2.Size * otherScale);
        
        return obb1.Intersects(obb2);  // SAT 分离轴检测
    }

    return false;
}
```

### 为什么使用 OBB 而不是 AABB？

- **AABB（轴对齐包围盒）**：坐标轴对齐，计算简单，但当角色旋转时包围盒会"膨胀"，导致误判
- **OBB（有向包围盒）**：随角色朝向旋转，更贴合人物形状，判定更精准

动作游戏的角色随时可能朝着任意方向攻击，AABB 的误差不可接受。

### OBB 相交检测：分离轴定理（SAT）

OBB 相交检测通常使用**分离轴定理（Separating Axis Theorem）**：

> 如果两个凸多边形不相交，那么一定存在一个轴，使得两个多边形在该轴上的投影不重叠。

对于两个 OBB，需要检测 15 个潜在分离轴：
- OBB1 的 3 个轴（x, y, z）
- OBB2 的 3 个轴（x, y, z）  
- OBB1 和 OBB2 各轴的交叉积（3×3 = 9 个）

任意一个轴上投影不重叠 → 不相交；所有轴上投影都重叠 → 相交。

## 攻击判定的完整流程

### 正常攻击判定

```csharp
public static bool CanAttack(this ColliderComponent self, Unit target, 
    out ColliderInfo attackCollider, out ColliderInfo hitCollider)
{
    attackCollider = null;
    hitCollider = null;
    var targetCollider = target.GetComponent<ColliderComponent>();
    if (targetCollider == null) return false;
    
    foreach (var colliderInfo in self.AttackColliders)
    {
        using var hitColliders = ListComponent<ColliderInfo>.Create();
        foreach (var targetInfo in targetCollider.HurtColliders)
        {
            if (!targetInfo.acceptAttack) return false;  // 不接受攻击
            if (Intersects(self.GetParent<Unit>(), colliderInfo, target, targetInfo))
            {
                hitColliders.Add(targetInfo);
            }
        }
        
        // 多个受击盒命中时，选距离攻击点最近的
        if (hitColliders.Count > 0)
        {
            if (hitColliders.Count == 1)
            {
                attackCollider = colliderInfo;
                hitCollider = hitColliders[0];
                return true;
            }
            else
            {
                var attacker = self.GetParent<Unit>();
                // 计算攻击点世界坐标
                var atkPointWorldPos = attacker.Position 
                    + attacker.Rotation * colliderInfo.Center 
                    + attacker.Rotation * colliderInfo.Rotation * colliderInfo.attackPoint;
                
                TSVector nearestPoint = TSVector.zero;
                foreach (var collider in hitColliders)
                {
                    // 找到受击盒表面上距离攻击点最近的点
                    var p = OBBClosestPoint.ClosestPointOnSurface(
                        target.Position, target.Rotation,
                        collider.Center, collider.Rotation, collider.Size,
                        atkPointWorldPos, out var _);
                    
                    if (hitCollider == null || 
                        TSVector.Distance(p, atkPointWorldPos) < TSVector.Distance(nearestPoint, atkPointWorldPos))
                    {
                        nearestPoint = p;
                        hitCollider = collider;
                    }
                }
            }
        }
    }
    return hitCollider != null;
}
```

当攻击者的攻击盒与目标多个受击盒同时重叠时（如攻击刚好打在两个部位的边界），系统选择距离攻击点最近的受击盒，这保证了判定结果的确定性和合理性。

### 闪避判定的三状态返回

```csharp
// 返回值：0=闪避命中（触发闪避）, 1=未命中, 2=命中但不接受闪避判定
public static int CanPreDogeAttack(this ColliderComponent self, Unit target, out ColliderInfo dogeCollider)
{
    dogeCollider = null;
    var targetCollider = target.GetComponent<ColliderComponent>();
    if (targetCollider == null) return 1;

    foreach (var colliderInfo in self.PreDogeAttackColliders)
    {
        foreach (var targetInfo in targetCollider.HurtColliders)
        {
            if (Intersects(self.GetParent<Unit>(), colliderInfo, target, targetInfo))
            {
                if (!targetInfo.acceptDoge)
                {
                    return 2;  // 命中了，但目标当前状态不接受闪避判定
                }
                dogeCollider = colliderInfo;
                return 0;  // 触发闪避
            }
        }
    }
    return 1;  // 没有命中
}
```

三状态的设计很有必要：
- `0`：触发闪避效果（翻滚成功，无敌帧生效）
- `1`：攻击盒还未到达（时机不对，继续等待）
- `2`：攻击到了，但此时不能闪避（处于硬直中，闪避失效）

这三种情况对应不同的游戏逻辑响应，单纯的 `bool` 无法区分。

## AttackComponent：命中记录与反作弊

`AttackComponent` 配合 `ColliderComponent` 工作，负责记录"本次攻击已经命中过哪些目标"，防止一次攻击多次伤害同一目标：

```csharp
public static bool CanAttack(this AttackComponent self, ColliderComponent colliderComp, 
    Unit target, out ColliderInfo attackCollider, out ColliderInfo hitCollider)
{
    attackCollider = null;
    hitCollider = null;
    
    // 已经命中过这个目标，跳过
    if (self.AttackedEnemies.Contains(target)) return false;
    
    return colliderComp.CanAttack(target, out attackCollider, out hitCollider);
}
```

攻击开始时清空记录：

```csharp
public static void SetPrCorrection(this AttackComponent self, int timelineID)
{
    self.lastTimeline = timelineID;
    self.Clear(timelineID);  // 清理命中记录，开始新一轮判定
}

public static void Clear(this AttackComponent self, int timelineID)
{
    if (self.lastTimeline == timelineID)
    {
        self.AttackedEnemies.Clear();
        self.ParryAttackedEnemies.Clear();
        self.DogeAttackedEnemies.Clear();
        self.FakeAttackedEnemies.Clear();
        self.lastTimeline = -1;
    }
}
```

用 `timelineID` 关联，确保只有在正确的 Timeline（时间轴/动作）开始时才清理记录，防止跨技能的误清理。

## 碰撞盒的生命周期管理

碰撞盒不是一直存在的，需要精确的开关时机：

```csharp
// 帧结束后清理标志
public static void LateFixedUpdate(this ColliderComponent self)
{
    self.SkipThisFrame = false;
}

// 完全清理（技能结束/进入空闲）
public static void Clear(this ColliderComponent self)
{
    self.SetHurtColliders(self.tempHurtCollider);  // 只保留默认受击盒
    self.ClearAttack();  // 清除所有攻击类盒
}

// 只清攻击盒（受击盒保留）
public static void ClearAttack(this ColliderComponent self)
{
    self.ClearAttackColliders();
    self.ClearPerformAttackColliders();
    self.ClearPreDogeAttackColliders();
    self.ClearPreParryAttackColliders();
    self.ClearFakeAttackColliders();
    self.ClearToken();  // 清理关联的Token（特殊标记）
}
```

这套清理机制确保了：
- 技能结束时，攻击盒不会残留（否则角色站着不动也能造成伤害）
- 受击盒始终保留一个默认盒（角色始终可被攻击）
- Token（特殊标记，如"当前格挡中"）随碰撞盒一起管理

## 碰撞 Token 系统

```csharp
public static void ClearToken(this ColliderComponent self)
{
    var unit = self.GetParent<Unit>().RootUnit;
    var buffViewComp = unit.GetComponent<BuffViewComponent>();
    if (buffViewComp != null)
    {
        buffViewComp.ClearTokens(self.parryTokenInfo, EBoxType.Parry);  // 清理格挡Token
        self.parryTokenInfo.bUseToken = false;
        buffViewComp.ClearTokens(self.atkTokenInfo, EBoxType.Atk);      // 清理攻击Token
        self.atkTokenInfo.bUseToken = false;
        buffViewComp.ClearTokens(self.dodgeTokenInfo, EBoxType.Dodge);  // 清理闪避Token
        self.dodgeTokenInfo.bUseToken = false;
    }
}
```

Token 是与碰撞盒绑定的"状态标记"，用于 Buff/技能系统感知角色当前处于什么碰撞状态。例如：
- 格挡盒激活时，Token 告诉 Buff 系统"角色正在格挡"，可以触发"格挡时恢复HP"等效果
- 攻击盒激活时，Token 告诉系统"角色在攻击判定期间"

## 性能优化点

**1. SkipThisFrame 优化**

```csharp
public static void SkipThisFrame(this ColliderComponent self)
{
    self.SkipThisFrame = true;
}
```

某些情况下（如被打飞后落地瞬间），不需要做碰撞检测，直接跳过本帧，避免无效计算。

**2. 使用对象池管理 ColliderInfo 列表**

```csharp
using var hitColliders = ListComponent<ColliderInfo>.Create();
```

`ListComponent<T>.Create()` 是基于对象池的临时列表，用 `using` 确保用完后自动回收，避免每次检测都分配 GC。

**3. 提前返回优化**

```csharp
if (!targetInfo.acceptAttack) return false;
```

一旦发现目标不接受这种攻击，立刻返回，不继续迭代后续碰撞盒。

## 设计总结

| 设计选择 | 原因 |
|---------|------|
| 自实现碰撞系统 | 定点数保证帧同步；精确时序控制 |
| OBB 而非 AABB | 方向相关，更精准，减少误判 |
| 多种碰撞盒类型 | 格挡/闪避/表演有不同判定逻辑 |
| acceptXxx 标志 | 受击盒细粒度控制，无需修改代码 |
| AttackedEnemies 记录 | 防止一次攻击多次命中 |
| Token 系统 | 碰撞状态与 Buff 系统解耦通信 |
| SkipThisFrame | 精确控制某帧跳过检测 |

碰撞检测系统是"游戏感（Game Feel）"的核心组成部分。精准的判定让玩家感受到战斗的爽快，而底层的 OBB + 定点数 + 类型化碰撞盒，正是这种精准感的技术保障。
