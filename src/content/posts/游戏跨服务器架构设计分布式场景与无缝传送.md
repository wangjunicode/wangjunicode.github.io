## 一、分区服架构

```
区服架构（传统MMORPG）：

客户端 → 登录服 → 选择区服 → 区服（独立游戏世界）
                                    ↑
                                 每个区服包含：
                                 ├── 场景服务器（多个，按地图分）
                                 ├── 战斗服务器
                                 ├── 社交服务器（好友/公会）
                                 ├── 聊天服务器
                                 └── 本地数据库

跨服架构：
区服1 ──┐
区服2 ──┼──→ 跨服中心服 ──→ 跨服战场
区服3 ──┘              └──→ 跨服拍卖行
                        └──→ 跨服排行榜
```

## 二、无缝传送协议

```csharp
/// <summary>
/// 玩家在服务器间传送的协议
/// </summary>
public static class ServerTransferProtocol
{
    /*
    无缝地图传送流程：

    1. 玩家走近边界，当前服务器检测到需要切换
    2. 当前服务器（Server A）通知目标服务器（Server B）准备接收玩家
    3. Server A 将玩家完整状态序列化（位置/血量/背包/Buff等）
    4. Server B 接收后创建玩家Ghost对象，回复Ready
    5. Server A 告知客户端："切换到Server B，Token=xxx"
    6. 客户端断开Server A，连接Server B
    7. Server B 激活Ghost对象，玩家无缝进入新场景
    8. Server A 清理玩家数据

    关键：
    - 传送令牌（Token）有效期5秒，防止中间人攻击
    - Ghost对象预创建减少玩家感知到的延迟
    - 传送期间客户端显示简短加载（< 500ms）
    */

    [Serializable]
    public class TransferToken
    {
        public string PlayerId;
        public string TargetServerId;
        public string Token;           // 一次性令牌
        public long ExpireTime;        // 有效期（毫秒时间戳）
        public Vector3 EntryPosition;  // 在目标服务器的进入位置
    }

    [Serializable]
    public class PlayerTransferData
    {
        public string PlayerId;
        public string Token;
        
        // 玩家状态快照
        public float HP;
        public float MaxHP;
        public float MP;
        public int Level;
        public List<string> ActiveBuffIds;
        public List<InventoryItemData> Inventory;
        public Vector3 Position;
        public Quaternion Rotation;
        
        // 压缩与加密
        public byte[] CompressedData;
        public string Signature;       // HMAC签名，防篡改
    }
}
```

## 三、跨服匹配系统

```csharp
/// <summary>
/// 跨服匹配管理器（简化设计）
/// </summary>
public class CrossServerMatchmaker
{
    /*
    跨服匹配架构：
    
    各区服玩家 → 发起匹配请求 → 全局匹配服务
                                      ↓
                              等待池（按ELO分组）
                                      ↓
                            找到足够玩家后组成房间
                                      ↓
                       分配一个战场服（可能是临时创建）
                                      ↓
                   通知各区服玩家：传送Token + 战场服地址
    */
    
    public struct MatchRequest
    {
        public string PlayerId;
        public string OriginServerId;  // 来自哪个区服
        public int MMR;                // 段位分
        public string GameMode;        // pvp_3v3 / pvp_5v5 等
        public long RequestTime;
    }

    public struct MatchResult
    {
        public string MatchId;
        public string BattleServerId;  // 分配的战场服地址
        public string[] PlayerIds;     // 参与玩家
        public string[] TransferTokens; // 每人的传送令牌
    }
}
```

## 四、服务器弹性扩缩容

```yaml
# Kubernetes 配置示例（游戏场景服弹性扩缩）
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: game-scene-server
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: game-scene-server
  minReplicas: 2
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  # 自定义指标：在线玩家数
  - type: External
    external:
      metric:
        name: game_online_players_per_server
        selector:
          matchLabels:
            server_type: scene
      target:
        type: AverageValue
        averageValue: 100  # 每个实例目标100人
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30  # 30秒内不重复扩容
      policies:
      - type: Pods
        value: 4
        periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300  # 5分钟稳定窗口防抖
```

## 五、数据一致性分级

| 数据类型 | 一致性要求 | 方案 |
|----------|-----------|------|
| 货币/装备 | 强一致性 | 单服事务，跨服通过消息队列 |
| 玩家位置 | 最终一致 | 每帧同步，允许短暂误差 |
| 排行榜 | 准实时（5分钟）| 定期聚合，不需要实时 |
| 聊天消息 | 有序投递 | Kafka保证分区内有序 |
| 好友在线状态 | 准实时（30秒）| 心跳推送，可接受轻微延迟 |

**CAP定理在游戏中的取舍：**
- 战斗相关：牺牲分区容错（AP），保证一致性（战斗数值绝对正确）
- 社交相关：牺牲一致性（CP），保证可用性（聊天不因为网络分区停止服务）
