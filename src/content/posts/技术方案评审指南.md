## 为什么需要技术方案评审

```
没有技术评审的团队：
  开发3个月 → 发现架构方向错误 → 推倒重做
  两人独立实现了相同的模块
  代码合并时发现设计冲突
  新人不了解系统设计意图，乱改导致 Bug

有技术评审的团队：
  重要决策前充分讨论 → 问题暴露在纸面
  经验丰富的成员提前发现潜在风险
  所有人对架构有共识 → 执行效率更高
  留下决策记录 → 新人快速理解系统
```

技术评审不是流程负担，是**降低风险的投资**。

## 一、技术方案文档模板

### 1.1 高质量方案文档的结构

```markdown
# 技术方案：[功能/系统名称]

**作者**：[你的名字]
**状态**：草稿 / 评审中 / 已批准 / 已废弃
**评审截止日期**：YYYY-MM-DD
**预计工期**：X人日

---

## 1. 背景与问题（必读）

简洁说明：
- 现状是什么？
- 存在什么问题？
- 为什么现在需要解决？

示例：
  当前资源加载系统在关卡切换时存在1~3秒的卡顿，
  影响了约15%的玩家（数据来源：性能监控平台，2024-Q1），
  并收到了50+玩家的反馈投诉。

---

## 2. 目标（成功标准）

- 关卡切换卡顿 < 100ms（P95）
- 内存峰值不超过 1.2GB（当前 1.5GB）
- 对现有代码改动最小化（存量场景不需要修改）

---

## 3. 非目标（明确不做什么）

- 不优化游戏内资源加载（仅优化切换场景的加载）
- 不改变资源打包策略（下期再做）

---

## 4. 方案比较

### 方案A：预加载 + 异步加载
**描述**：在进入切换点前100米时开始预加载下一场景资源...
**优点**：无感知加载，玩家体验最好
**缺点**：内存峰值增加30%，需要精确控制预加载时机
**工期**：5人日

### 方案B：分步加载 + 加载动画
**描述**：保留加载界面，但将资源分批异步加载，优先加载关键资源...
**优点**：简单可控，内存安全
**缺点**：仍有加载界面（但时间缩短到 < 1s）
**工期**：3人日

### 方案C：...

---

## 5. 推荐方案

**推荐：方案B**

**理由**：
  当前内存预算已接近上限，方案A有OOM风险。
  方案B的加载时间可以满足P95 < 1s的目标，
  同时工期更短，风险更低。
```

## 二、评审会议的组织

### 2.1 评审前的准备

```
评审前至少2天发出：
  - 方案文档（让大家提前阅读）
  - 会议议程（什么时间讨论什么）
  - 需要决策的关键问题（评审要得出结论的点）

邀请正确的人：
  ✅ 方案执行者
  ✅ 上下游系统负责人
  ✅ 有相关经验的高级工程师
  ❌ 与决策无关的人（浪费时间）
  ❌ 人太多导致意见分散（最多5~7人）
```

### 2.2 评审会议流程

```
时间控制（45分钟为例）：

0~5分钟：背景同步
  - "我们解决什么问题？"（让没看文档的人快速入场）

5~15分钟：方案介绍
  - 作者介绍推荐方案和备选方案
  - 重点说明比较关键点，不念文档

15~35分钟：讨论质疑
  - 主持人（通常是 TL）引导讨论
  - 聚焦：技术风险、成本估算、遗漏场景
  
35~45分钟：决策
  - 明确：选择哪个方案？
  - 明确：需要修改文档的哪些部分？
  - 明确：行动项是什么？谁来做？什么时候完成？
```

### 2.3 有效的提问技巧

```
作为评审者，好的问题应该：

1. 具体，不空泛
  ❌ "这个方案有性能问题吗？"
  ✅ "方案B中分批加载，如果第一批资源还没加载完，
      玩家就触发了第一个交互，会发生什么？"

2. 聚焦风险
  ✅ "边界条件：如果玩家在加载途中强制关闭游戏，
      会不会破坏存档？"
      
3. 暴露假设
  ✅ "你的估算假设是每个批次50ms，
      但现有最大的批次有多大？实测过吗？"
      
4. 引导改进而非否定
  ❌ "这个设计不行"
  ✅ "这里有个我担心的点：[具体问题]。
      你有没有考虑过用 [X 方案] 来解决？"
```

## 三、架构决策记录（ADR）

### 3.1 ADR 的价值

```
问题场景：
  半年后新同学问："为什么这里用 A 方案而不是 B 方案？"
  如果没有记录，没人知道当初的原因
  
  更糟的情况：
  新同学不了解当初的约束，把 B 方案"优化"进去，
  结果触发了当时已知的问题

ADR（Architecture Decision Record）解决这个问题：
  记录：做了什么决策、为什么这样决策、有哪些替代方案
  让未来的人理解决策的历史背景
```

### 3.2 ADR 格式

```markdown
# ADR-0001：选择 UniTask 作为异步编程方案

**日期**：2024-03-15
**状态**：已采纳
**决策者**：技术负责人 张三

## 背景

项目中大量使用了 async/await 进行异步编程，
包括网络请求、资源加载、过场动画等场景。
我们需要选择一个统一的异步方案。

## 决策

使用 **UniTask**（Cysharp/UniTask）替代 C# 原生 Task。

## 原因

1. **零 GC 分配**：UniTask 基于值类型（struct），
   比 Task 少得多的堆内存分配，对游戏帧率友好
   
2. **PlayerLoop 集成**：支持 `UniTask.Yield(PlayerLoopTiming.Update)` 
   等在特定帧阶段执行，无法通过原生 Task 实现
   
3. **取消令牌**：与 MonoBehaviour/GameObject 生命周期集成，
   `destroyCancellationToken` 自动管理任务生命周期

## 备选方案

### 方案A：C# Task + SynchronizationContext
  问题：GC 压力大，且不支持 PlayerLoop 时序

### 方案B：Unity Coroutine
  问题：无返回值，错误处理复杂，无法跨脚本等待

## 风险

- UniTask 是第三方库，需要关注维护状态（目前活跃）
- 学习成本：新成员需要了解 UniTask 特有 API

## 结果

2024 Q2 迭代后，网络模块和资源加载模块均已迁移到 UniTask，
帧率稳定性提升约 2ms/frame（Profiler 数据）。
```

### 3.3 ADR 的管理方式

```
存放位置：
  /docs/decisions/
    ADR-0001-选择UniTask.md
    ADR-0002-资源管理系统设计.md
    ADR-0003-热更方案选型.md

每次重要技术决策后：
  1. 在 ADR 目录新增一条记录
  2. 提交到 Git（决策历史永久保存）
  3. 在 PR 描述中引用相关 ADR

搜索：
  当有人问"为什么用 X？"时，
  直接在 docs/decisions 中搜索，
  通常能找到完整的历史决策背景
```

## 四、技术负责人在评审中的角色

### 4.1 主持评审的技巧

```
TL 在评审中不是来"审判"方案的，而是：

1. 确保讨论聚焦
   把偏题的讨论拉回来：
   "这个话题很有价值，但不是今天要决策的，
   我们单独约时间讨论，先回到今天的核心问题"
   
2. 确保每个观点被听到
   "[安静的成员]，你对这个方案有什么看法？"
   
3. 推动决策
   "讨论了20分钟，我总结一下目前的共识和分歧...
   在 [某个具体问题] 上，我们有分歧，我来做个决定..."
   
4. 学会结束讨论
   有时候没有完美方案，需要在信息不完整的情况下做决策
   "我们知道风险是 X，缓解措施是 Y，先这样决定，
    遇到问题再调整"
```

### 4.2 如何处理意见分歧

```
技术方案评审中的意见分歧类型：

类型1：对事实的分歧（可以验证）
  "方案A真的更快吗？"
  → 做个原型验证，用数据说话
  
类型2：对价值取舍的分歧（需要决策）
  "应该优先开发速度还是性能？"
  → TL 需要结合项目阶段做决策
  
类型3：对未来的不确定性（接受模糊）
  "这个设计六个月后会不会成为瓶颈？"
  → 接受不确定性，做最小化赌注，预留扩展点

处理建议：
  - 快速区分是哪种分歧
  - 类型1：用事实解决
  - 类型2：TL 拍板，记录决策理由
  - 类型3：接受不确定性，设计时预留出路
```

## 总结

高质量技术评审的关键要素：

| 要素 | 差的实践 | 好的实践 |
|------|---------|---------|
| 文档 | 没有文档，口头讲方案 | 结构化文档，提前2天发出 |
| 参与者 | 人越多越好 | 5~7人，相关方 |
| 讨论 | 漫无边际 | 聚焦风险和关键决策点 |
| 结果 | 会议结束无定论 | 明确决策 + 行动项 |
| 记录 | 全靠记忆 | ADR 文档，提交到 Git |

**技术负责人的核心价值**：不是做出所有决策，而是**建立让团队做出好决策的流程**，并在重大决策节点提供判断力。
