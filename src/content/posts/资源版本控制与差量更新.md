# 04_资源版本控制与差量更新

> 面向刚入行的毕业生 · 建议搭配 VersionManager.cs 阅读

---

## 1. 系统概述

### 1.1 为什么需要版本控制？

在没有版本控制的情况下，每次热更新都需要玩家下载**全量资源**。对于一个几百MB的游戏来说，这意味着：
- 每次更新哪怕只改了1KB的代码，玩家也要下载几百MB
- 在弱网/移动网络下，用户体验极差
- CDN 费用高昂

版本控制 + 差量更新解决了这个问题：**只下载与当前版本有差异的文件**。

### 1.2 本项目的版本号体系

版本号格式：**`主版本.次版本.修订号.资源版本`**，例如 `1.0.3.8`

| 位段 | 示例 | 含义 |
|------|------|------|
| 主版本 | 1 | 大版本更新（一般对应重大功能迭代或引擎升级） |
| 次版本 | 0 | 中版本更新（功能模块迭代） |
| 修订号 | 3 | 小版本更新（Bug 修复、小功能） |
| 资源版本 | 8 | 热更资源版本（每次热更推送递增） |

**程序版本**（`Application.version`）：前三位，如 `1.0.3`，对应 App 安装包版本，在 App Store/Google Play 显示。

**资源版本**（`VersionManager.ResVersion`）：四位完整版本号，包含热更资源的版本信息，存储在 `AppConfig.json` 中。

## 2. 架构设计

### 2.1 版本信息存储架构

```
┌─────────────────────────────────────────────────┐
│             版本信息来源                          │
│                                                   │
│  Application.version                              │
│  （Unity 工程设置里设定的程序版本）                │
│  例：1.0.3                                        │
│                                                   │
│  AppConfig.json                                   │
│  （存储在 persistentDataPath/ResUpdate/ 下）       │
│  {                                                │
│    "AndroidSourceVersion": "1.0.3.8",             │
│    "iOSSourceVersion": "1.0.3.8",                 │
│    "UpdateSize": 314572800                        │
│  }                                                │
│                                                   │
│  AAVersionNumber.txt                              │
│  （存储在 ResUpdate/UpdateVersion/ 下）            │
│  内容：1.0.3.8 （当前 Addressables 资源版本）      │
└─────────────────────────────────────────────────┘
```

### 2.2 版本比较流程

```
GetCurrentSourceVersion()
        │
        ▼
VersionManager.Instance.ResVersion
        │
        ├─ [未初始化] → LoadAppConfig() 从文件加载
        │
        ▼
   检查 AppConfig.json 是否存在
        │
        ├─ 存在 → BinaryFormatter.Deserialize() 反序列化
        │         返回 AppConfig.AndroidSourceVersion / iOSSourceVersion
        │
        └─ 不存在 → 使用默认值（Application.version 派生）
                    GetDefaultResVersion():
                    appVer="1.0.3" → "1.0.4.0"（修订号+1，资源版本=0）
```

**为什么默认资源版本是修订号+1？** 这是一个容错设计：新安装的 App（`AppConfig.json` 不存在）默认认为自己"需要热更"，会触发 Dolphin 检查，从而获取服务端最新版本。

### 2.3 版本号比较算法

```csharp
// VersionManager.cs - 版本号字典序比较（逐段比较）
static private bool isHigherVersion(string oldString, string newString)
{
    string[] oldVersionNum = oldString.Split(".");
    string[] newVersionNum = newString.Split(".");

    bool bHigher = false;
    if (newVersionNum.Length == oldVersionNum.Length)
    {
        int oldNum = 0;
        int newNum = 0;
        for (int i = 0; i < oldVersionNum.Length; ++i)
        {
            int.TryParse(oldVersionNum[i], out oldNum);
            int.TryParse(newVersionNum[i], out newNum);
            if (oldNum > newNum)      // old 更高，new 不是更高版本
                break;
            else if (oldNum < newNum) // new 更高
            {
                bHigher = true;
                break;
            }
            // 相等继续比下一位
        }
    }
    return bHigher;
}
```

**示例**：
- `isHigherVersion("1.0.3.8", "1.0.3.9")` → `true`（9 > 8）
- `isHigherVersion("1.0.4.0", "1.0.3.9")` → `false`（4 > 3，new 不比 old 高）
- `isHigherVersion("1.0.3.8", "1.0.3.8")` → `false`（相等）

## 3. 核心代码展示

### 3.1 AppConfig 数据模型

```csharp
// VersionManager.cs
[System.Serializable]
public class AppConfig
{
    public string AndroidSourceVersion { get; set; }
    public string iOSSourceVersion { get; set; }
    public long UpdateSize { get; set; }  // 预期更新包大小（字节）
    
    public AppConfig()
    {
        var defaultRes = GetDefaultResVersion();
        AndroidSourceVersion = defaultRes;
        iOSSourceVersion = defaultRes;
        UpdateSize = 1024 * 1024 * 300; // 默认 300MB
    }
    
    // 默认版本号生成：在程序版本修订号上+1，资源版本=0
    public string GetDefaultResVersion()
    {
        try
        {
            var appVer = UnityEngine.Application.version; // 如 "1.0.3"
            string[] parts = appVer.Split('.');
            int num = int.TryParse(parts[2], out var n) ? n : 0;
            // 生成 "1.0.4.0"（修订号+1，资源版本=0）
            return string.Format("{0}.{1}.{2}.{3}", parts[0], parts[1], num + 1, "0");
        }
        catch
        {
            return "0.0.0.1";
        }
    }
}
```

### 3.2 AppConfig 持久化（二进制序列化）

```csharp
// 保存路径：persistentDataPath/ResUpdate/AppConfig.json
public string AppConfigPath => Application.persistentDataPath + "/ResUpdate/AppConfig.json";

// 保存（使用 BinaryFormatter 二进制序列化）
public void SaveAppConfig()
{
    try
    {
        using (FileStream fs = new FileStream(AppConfigPath, FileMode.Create))
        {
            BinaryFormatter formatter = new BinaryFormatter();
            formatter.Serialize(fs, AppConfig);
        }
    }
    catch (Exception e)
    {
        Debug.LogError("[AppConfigManager] Error saving AppConfig: " + e.Message);
    }
}

// 加载
public void LoadAppConfig()
{
    try
    {
        if (File.Exists(AppConfigPath))
        {
            using (FileStream fs = new FileStream(AppConfigPath, FileMode.Open))
            {
                BinaryFormatter formatter = new BinaryFormatter();
                AppConfig = (AppConfig)formatter.Deserialize(fs);
            }
        }
        else
        {
            AppConfig = new AppConfig(); // 文件不存在，使用默认配置
        }
    }
    catch (Exception e)
    {
        Debug.LogError("[AppConfigManager] Error loading AppConfig: " + e.Message);
        AppConfig = new AppConfig(); // 异常时也降级到默认配置
    }
}
```

### 3.3 热更成功后版本号更新

```csharp
// PatchManager.cs - 热更完成回调
public void OnNoticeChangeSourceVersion(string newVersionStr)
{
    PatchLog.Info($"[VGameDolphin] OnNoticeChangeSourceVersion: {newVersionStr}");
    
    exitUpdate = true;
    
    // 同时更新 Android 和 iOS 版本号（AppConfig 是共用的）
    VersionManager.AppConfig.AndroidSourceVersion = newVersionStr;
    VersionManager.AppConfig.iOSSourceVersion = newVersionStr;
    
    // 持久化到磁盘
    VersionManager.Instance.SaveAppConfig();
    
    // 继续 Addressables Catalog 更新
    BeginAdressableCatalogUpdate();
}
```

### 3.4 主版本不一致时清除旧热更

```csharp
// PatchManager.cs - 启动时的版本一致性检查
private void ClearOldPatch()
{
    string currentProgramVersion = GetCurrentProgramVersion();  // App 版本（如 "1.0.3"）
    string currentSourceVersion = GetCurrentSourceVersion();    // 资源版本（如 "0.9.8.5"）

    // 只比较主版本号（第一位）
    if (currentProgramVersion.Split(".")[0] != currentSourceVersion.Split(".")[0])
    {
        PatchLog.Info($"版本不一致：程序版本 {currentProgramVersion}，资源版本 {currentSourceVersion}");
        
        string resUpdateDirectory = Path.Combine(Application.persistentDataPath, "ResUpdate");
        
        if (Directory.Exists(resUpdateDirectory))
        {
            try
            {
                Directory.Delete(resUpdateDirectory, true); // 递归删除整个 ResUpdate 目录
                PatchLog.Info("旧的补丁目录 'ResUpdate' 已删除.");
            }
            catch (Exception ex)
            {
                PatchLog.Error($"删除 'ResUpdate' 目录失败: {ex.Message}");
            }
        }
    }
}
```

**重要性**：当玩家从旧大版本（如主版本 0）更新到新大版本（主版本 1）时，旧的热更文件与新安装包可能存在格式不兼容（Bundle 格式、Shader 变更等）。清除旧热更目录确保玩家使用安装包内的全新资源。

### 3.5 AAVersionNumber.txt 读取

```csharp
// VersionManager.cs - 从文件读取 AA 版本号
public static string ReadGameLogicVersionFromFile()
{
    string content = UnityEngine.Application.version; // 默认值
    string filePath = Application.persistentDataPath + "/ResUpdate/UpdateVersion/AAVersionNumber.txt";
    
    if (File.Exists(filePath))
    {
        string resContent = File.ReadAllText(filePath);
        
        // 只有文件中的版本比当前高时才采用
        if (!string.IsNullOrEmpty(resContent) && isHigherVersion(content, resContent))
        {
            // 持久化到 PlayerPrefs 供后续使用
            UnityEngine.PlayerPrefs.SetString(GAMELOGIC_VERSION_NAME, resContent);
            return resContent;
        }
    }
    
    return content;
}
```

## 4. 差量更新的实现原理

### 4.1 Dolphin 服务端差量计算

差量计算主要在服务端完成：

1. **构建时**：每次出包，将所有 AssetBundle 文件的 MD5 上传到 GCloud 控制台
2. **查询时**：客户端上报当前资源版本号，服务端计算出"需要下载的新/变更文件列表"
3. **下载时**：只下载列表中的差异文件

### 4.2 客户端对应机制

客户端通过两个机制配合差量更新：

**机制1：文件覆盖**
Dolphin 下载的文件直接覆盖 `ResUpdate/` 目录下的同名旧文件，未变更的文件不下载（保留旧文件）。

**机制2：Catalog 更新**
新的 `catalog.json` 包含所有资源的最新映射（包括未变更的资源映射旧路径，变更的资源映射新文件）。`AddResourceLocator(newCatalog)` 让 Addressables 使用新的映射关系。

### 4.3 版本升级矩阵

| 场景 | 程序版本变化 | 资源版本变化 | 用户感知 |
|------|-------------|-------------|---------|
| 纯资源热更 | 无（1.0.3） | 1.0.3.8 → 1.0.3.9 | 只下载变更 Bundle，无感 |
| 小包更新 | 1.0.3 → 1.0.4 | 1.0.3.x → 1.0.4.0 | 下载差量，重启App |
| 大版本更新 | 1.x → 2.x | 全量重新下载 | 去应用商店更新 |

## 5. 设计亮点

### 5.1 文件级差量（粒度精确）

Dolphin 的差量是以**文件为单位**的。Addressables 可以将单个资源打成独立的 Bundle（如每个重要 UI 打一个 Bundle），这样修改一个 UI 只需要下载那一个 Bundle，而不是整个 UI Bundle 包。这要求在打包策略上有合理的 Bundle 拆分。

### 5.2 版本号防回滚

`isHigherVersion` 的调用方向确保了版本号只能向前推进，不会因为网络异常收到旧版本号而回滚：

```csharp
// 只有新版本比当前版本更高时，才更新 PlayerPrefs
if (isHigherVersion(content, resContent))
{
    PlayerPrefs.SetString(GAMELOGIC_VERSION_NAME, resContent);
}
```

## 6. 常见问题与最佳实践

### Q1：`BinaryFormatter` 是否有跨平台风险？
**A**：有一定风险。`BinaryFormatter` 的序列化格式与平台、.NET 版本相关。如果 `AppConfig` 类的字段发生变化（增加/删除字段），旧的序列化文件可能无法反序列化，导致 `LoadAppConfig` 中的 catch 被触发，使用默认配置。**最佳实践**是在 `catch` 块中重建默认 AppConfig 并立即保存，防止下次启动继续报错。当前代码已经这样处理了。

### Q2：资源版本号和程序版本号不对齐怎么处理？
**A**：这是正常情况。程序版本（APK/IPA）不会频繁更新，资源版本（热更）可以每天多次更新。Dolphin 的两阶段更新设计就是为了处理这种不对齐：程序包检查和资源包检查是独立的两次 `InitUpdateMgr` 调用。

### Q3：如何查看当前设备上的版本信息？
**A**：在 PatchUI 的版本文字区域可以看到程序版本和资源版本：
```csharp
// PatchUI.cs
public void SetVersionText(string appVer, string resVer)
{
    verText.text = $"当前程序版本: {appVer}\n当前资源版本: {resVer}";
}
```
也可以直接读取 `AppConfigPath` 下的文件内容。

### Q4：热更完成但版本号没更新怎么处理？
**A**：确认 `OnNoticeChangeSourceVersion` 回调被触发，且 `SaveAppConfig()` 执行成功。常见问题：`ResUpdate` 目录写权限不足（特别是部分 Android 设备上）。

### Q5：多次热更后磁盘占用越来越大怎么办？
**A**：Dolphin 在热更时会用新文件**覆盖**旧文件（同名文件）。但如果某个 Bundle 被重命名（旧 Bundle 废弃，新 Bundle 用新名字），旧文件会残留。建议在 GCloud 控制台配置"清理旧文件"策略，或在客户端定期扫描 `ResUpdate` 目录中不在当前 Catalog 中的孤立文件并删除。
