## 为什么需要帧驱动定时器

游戏定时器通常有两种计时基准：

| 基准 | 代表 | 特点 |
|------|------|------|
| 毫秒/实时时间 | `TimerComponent`（服务端/客户端通用） | 精度高，与物理帧无关 |
| 帧计数 | `LogicTimerComponent` | 确定性，天然适配帧同步 |

在帧同步战斗中，所有客户端必须以相同的帧序列执行逻辑才能保证一致性。如果用毫秒时间触发"技能持续 2 秒"，不同设备的帧率差异会导致触发帧不同，破坏同步。**帧驱动定时器将"时间"转换为"帧数"，完全消除了这个不确定性。**

## 核心数据结构

```csharp
public class LogicTimerComponent : Singleton<LogicTimerComponent>, ISingletonFixedUpdate
{
    private readonly MultiMap<long, long> TimeId = new();          // 触发帧 → Timer ID 列表
    private readonly Queue<long> timeOutTime = new();              // 本轮超时的帧号
    private readonly Queue<long> timeOutTimerIds = new();          // 本轮超时的 Timer ID
    private readonly Dictionary<long, LogicTimerAction> timerActions = new(); // ID → TimerAction
    private long idGenerator;
    private long frameNow = 0;
    private long minTime = long.MaxValue;  // 最近一次触发帧，用于快速剪枝
}
```

### MultiMap 的选择

`TimeId` 用 `MultiMap<long, long>`（有序字典 → List）而非 `Dictionary<long, List<long>>` 的原因：

- **有序遍历**：每帧只需遍历 key ≤ frameNow 的条目，一旦碰到 key > frameNow 就可以 break，O(log N) 查找开销极低。
- **同帧多个定时器**：Value 是 `List<long>`，同一帧触发的多个定时器共享一个 key。

### minTime 优化

```csharp
private long minTime = long.MaxValue;

public void Update()
{
    if (this.TimeId.Count == 0) return;
    frameNow++;
    if (frameNow < this.minTime) return;  // 快速退出，无需遍历
    // ...
}
```

`minTime` 记录当前队列中最早的触发帧。每帧 Update 时，先与 `frameNow` 比较，如果还没到最早触发点，直接 return。这个优化在定时器稀疏时（大部分帧）可以让 Update 几乎为零开销。

## TimerAction 设计

```csharp
public class LogicTimerAction
{
    public long Id;
    public TimerClass TimerClass;    // OnceTimer / OnceWaitTimer / RepeatedTimer
    public object Object;            // 回调参数或 ETTask（等待型）
    public long Frame;               // 定时器间隔帧数
    public long StartFrame;          // 起始帧
    public int Type;                 // 触发事件类型（EventSystem 用）

    public static LogicTimerAction Create(long id, TimerClass timerClass, long startFrame,
                                          FP time, int type, object obj)
    {
        LogicTimerAction timerAction = ObjectPool.Instance.Fetch<LogicTimerAction>();
        // 关键转换：FP 秒 → 帧数
        timerAction.Frame = TSMath.RoundToInt(time / EngineDefine.fixedDeltaTime_Orignal);
        // ...
        return timerAction;
    }

    public void Recycle() { ObjectPool.Instance.Recycle(this); }
}
```

**时间 → 帧数转换**：`TSMath.RoundToInt(time / fixedDeltaTime)` 将定点数（FP）秒转换为帧数，全程使用定点数运算，杜绝浮点精度差异。

**对象池复用**：`TimerAction` 通过对象池管理，高频注册/取消定时器时避免 GC 压力。

## 三种触发模式

### 1. OnceTimer（一次性回调）

```csharp
public long NewOnceTimer(FP time, int type, object args)
{
    LogicTimerAction timer = LogicTimerAction.Create(GetId(), TimerClass.OnceTimer, frameNow, time, type, args);
    this.AddTimer(timer);
    return timer.Id;
}
```

触发时通过 `EventSystem.Instance.Invoke` 分发事件，适合"技能冷却结束"、"延迟出现的特效"等场景。调用者持有返回的 `timerId`，可在触发前 `Remove` 取消。

### 2. OnceWaitTimer（异步等待型）

```csharp
public async ETTask WaitAsync(FP time, ETCancellationToken cancellationToken = null)
{
    ETTask tcs = ETTask.Create(true);
    LogicTimerAction timer = LogicTimerAction.Create(GetId(), TimerClass.OnceWaitTimer, frameNow, time, 0, tcs);
    this.AddTimer(timer);
    long timerId = timer.Id;

    void CancelAction()
    {
        if (this.Remove(timerId)) tcs.SetResult();
    }

    try
    {
        cancellationToken?.Add(CancelAction);
        await tcs;
    }
    finally
    {
        cancellationToken?.Remove(CancelAction);
    }
}
```

这是最优雅的用法。**Object 字段存的是 `ETTask`**，定时器触发时调用 `tcs.SetResult()` 唤醒 async 流程。

用法：

```csharp
// 等待 0.5 秒（逻辑帧）
await LogicTimerComponent.Instance.WaitAsync(FP.FromFloat(0.5f));
// 这一行之后的代码在 0.5 秒后执行
DoSomethingAfterDelay();
```

支持 `ETCancellationToken`：技能被打断时取消等待，避免技能逻辑继续执行。

### 3. RepeatedTimer（循环定时器）

```csharp
private void Run(LogicTimerAction timerAction)
{
    case TimerClass.RepeatedTimer:
    {
        timerAction.StartFrame = frameNow;  // 更新起始帧
        this.AddTimer(timerAction);          // 重新注册（自身作为下一次触发）
        EventSystem.Instance.Invoke(timerAction.Type, ...);
        break;
    }
}
```

循环定时器触发后 **不回收自身**，而是用 `StartFrame = frameNow` 更新起始帧，重新加入 `TimeId`。这样每次触发间隔都精准保持 `Frame` 帧，不会累积误差。

## Update 主流程详解

```csharp
public void BeforeFixedUpdate()
{
    if (!EngineRuntime.Pause) Update();  // 游戏暂停时不推进帧计数
}

private void Update()
{
    if (this.TimeId.Count == 0) return;
    frameNow++;
    if (frameNow < this.minTime) return;

    // 第一遍：收集所有超时帧号
    foreach (KeyValuePair<long, List<long>> kv in this.TimeId)
    {
        if (kv.Key > frameNow)
        {
            this.minTime = kv.Key;  // 更新 minTime 为第一个未到期的帧
            break;
        }
        this.timeOutTime.Enqueue(kv.Key);
    }

    // 第二遍：收集超时的 Timer ID
    while (this.timeOutTime.Count > 0)
    {
        long time = this.timeOutTime.Dequeue();
        foreach (long timerId in this.TimeId[time])
            this.timeOutTimerIds.Enqueue(timerId);
        this.TimeId.Remove(time);
    }

    // 第三遍：执行回调
    while (this.timeOutTimerIds.Count > 0)
    {
        long timerId = this.timeOutTimerIds.Dequeue();
        if (!this.timerActions.Remove(timerId, out LogicTimerAction timerAction)) continue;
        this.Run(timerAction);
    }
}
```

**三阶段分离**的设计原因：在"收集阶段"不能边遍历 `TimeId` 边删除（会抛异常），也不能直接执行回调（回调内可能注册新的定时器，影响遍历）。将收集和执行分开，彻底避免了并发修改问题。

## 帧同步下的暂停处理

```csharp
public void BeforeFixedUpdate()
{
    if (!EngineRuntime.Pause)
        Update();  // 暂停时不推进 frameNow
}
```

`EngineRuntime.Pause` 是框架级暂停标志（通常用于加载/过场）。暂停时 `frameNow` 不递增，所有定时器"冻结"，恢复后从暂停帧继续计时，完美保持帧同步一致性。

## 与 TimerComponent 的对比

| 特性 | LogicTimerComponent | TimerComponent |
|------|--------------------|--------------------|
| 计时基准 | 逻辑帧（FP定点数） | 毫秒（long） |
| 确定性 | 完全确定 | 依赖系统时钟 |
| 暂停支持 | 内置 | 需要外部控制 |
| 适用场景 | 帧同步战斗 | 服务端/非战斗逻辑 |
| 精度 | 帧精度（约16.67ms/帧） | 毫秒精度 |

## 实际应用示例

### 技能持续时间

```csharp
public async ETTask ApplyBuff(BuffComponent buff, FP duration)
{
    buff.IsActive = true;
    var cancel = new ETCancellationToken();

    // 等待持续时间（可被外部取消）
    await LogicTimerComponent.Instance.WaitAsync(duration, cancel);

    if (!buff.IsDisposed)
        buff.Remove();
}
```

### 循环伤害（持续灼烧）

```csharp
// 每 0.5 秒触发一次灼烧伤害
long timerId = LogicTimerComponent.Instance.NewRepeatedTimer(
    FP.FromFloat(0.5f),
    TimerType.BurnTick,
    new BurnArgs { TargetId = target.Id, Damage = 100 }
);

// 效果结束时移除
LogicTimerComponent.Instance.Remove(ref timerId);
```

## 设计总结

`LogicTimerComponent` 是专为帧同步战斗设计的定时器，其核心优势在于：

1. **完全确定性**：帧计数 + 定点数，消除浮点误差和平台差异
2. **暂停感知**：`BeforeFixedUpdate` 钩子配合 `EngineRuntime.Pause`，游戏暂停时定时器自动冻结
3. **三段式执行**：收集超时帧 → 收集 Timer ID → 执行回调，安全处理回调内注册新定时器的情况
4. **minTime 剪枝**：99% 的帧无需遍历 MultiMap，几乎零开销
5. **async/await 友好**：`WaitAsync` 让战斗逻辑以线性方式书写，比回调式代码可读性高一个数量级
