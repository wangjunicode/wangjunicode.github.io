# 03_战斗帧同步网络同步方案

> 帧同步是多人实时对战游戏的核心技术。本文从"什么是帧同步"开始，带你彻底搞懂本项目的战斗同步方案。

---

## 1. 系统概述

### 1.1 为什么需要帧同步？

想象两个玩家在玩格斗游戏。玩家A按下了攻击键，这个操作需要同步给玩家B。如果简单地把"谁打了谁"的结果从服务器发给每个客户端，延迟波动会导致每个人看到的游戏状态不一致。

**帧同步（LockStep）**的核心思想是：
- **所有客户端运行相同的逻辑**
- **服务器只转发操作，不计算游戏逻辑**
- **只要所有人在同一帧执行相同操作，结果必然一致**

### 1.2 本项目帧同步方案

本项目使用腾讯GCloud提供的**LockStep SDK**实现帧同步。架构如下：

```
客户端A ──input──→ GCloud LockStep 服务器 ──broadcast──→ 客户端A
客户端B ──input──→ GCloud LockStep 服务器 ──broadcast──→ 客户端B  
客户端C ──input──→ GCloud LockStep 服务器 ──broadcast──→ 客户端C
```

每一"帧"（默认约66ms一帧，约15FPS），服务器将所有玩家在该帧的操作打包成`FrameInfo`广播给所有人。客户端收到帧后，所有人按相同逻辑执行，游戏状态必然一致。

## 2. 架构设计

### 2.1 LockStep状态机

GCloud LockStep有明确的状态机：

```
Uninitialized（未初始化）
    │ Initialize()
    ▼
  Idle（空闲）
    │ Login()
    ▼
 Logining（登录中）
    │ 登录成功
    ▼
 Logined（已登录）
    │ Ready() + 等其他人Ready
    ▼
  战斗进行中（LockStepState.Fighting）
    │ 网络异常
    ▼
Reconnecting（重连中）
    │ 重连成功/失败
    ▼
Reconnected / Error
```

### 2.2 帧同步关键组件

| 类/文件 | 职责 |
|---------|------|
| `LockSteper`（抽象类） | 帧同步系统的抽象接口，定义所有API |
| `LockStepImpl` | 实际实现，通过Native层调用C++底层SDK |
| `JitterBufferStrategy` | 抖动缓冲策略，平滑网络抖动对帧数据的影响 |
| `IBufferStrategy` | 缓冲策略接口，允许替换不同的缓冲算法 |
| `FrameInfo` | 单帧数据结构（含所有玩家的该帧操作） |
| `RelayData` | 单个玩家的单帧操作数据 |
| `FrameProfile` | 帧性能分析数据（输入间隔、接收间隔、丢帧统计等） |

### 2.3 帧数据流完整流程

```
【发送端】
游戏逻辑层
   │ 玩家按下技能按钮
   ▼
VKeyDef.PVP_XXXX（确定消息类型）
   │
   ▼
封装成byte[]（序列化）
   │
   ▼
LockSteper.Instance.Input(data, rawUdp=true)
   │  rawUdp=true → 使用UDP（低延迟）
   │  rawUdp=false → 使用可靠UDP（重要操作）
   ▼
GCloud Native层 → 发送到LockStep服务器

【接收端】
GCloud Native层接收到服务器广播
   │
   ▼
LockSteper.Instance.PopFrame() 或 ReadFrame(frame)
   │
   ▼
FrameInfo（含本帧所有玩家的操作）
   │
   ▼
游戏逻辑层按帧处理
```

### 2.4 Jitter Buffer（抖动缓冲）

网络不稳定时，帧数据不会按顺序、匀速到达。`JitterBufferStrategy`的作用是在缓冲区中保存若干帧，平滑地"喂给"游戏逻辑，避免忽快忽慢：

```
网络层（乱序、抖动）：帧5 → 帧3 → 帧7 → 帧4 → 帧6
                              ↓
                    JitterBuffer（排序+缓冲）
                              ↓
游戏逻辑层（平滑有序）：帧3 → 帧4 → 帧5 → 帧6 → 帧7
```

## 3. 核心代码展示

### 3.1 LockSteper 抽象接口（来自`LockSteper.cs`）

```csharp
public abstract class LockSteper : ApolloObject
{
    // 状态变化事件（登录成功、网络异常、重连等）
    public abstract event StateChangedEventHandler StateChangedEvent;

    // 广播事件（收到其他玩家发来的Broadcast数据）
    public abstract event BroadcastEventHandler BroadcastEvent;

    // 当前玩家ID
    public abstract UInt32 PlayerID { get; }

    // 是否已登录LockStep服务器
    public abstract bool HasLogined { get; }

    // 战斗是否已开始
    public abstract bool HasStarted { get; }

    // 初始化（传入配置信息）
    public abstract bool Initialize(LockStepInitializeInfo initInfo);

    // 登录（accessInfo由游戏服务器下发，相当于入场券）
    public abstract void Login(byte[] accessInfo, LoginEventHandler callback);

    // 准备就绪（告诉服务器"我准备好了"）
    public abstract void Ready(ReadyEventHandler callback);

    // 发送本帧操作
    // rawUdp=true：原始UDP（最快，可能丢包）
    // rawUdp=false：可靠UDP（稍慢，但不丢包）
    public abstract bool Input(byte[] data, bool rawUdp = true,
                               LockStepInputFlag flag = LockStepInputFlag.None);

    // 弹出下一帧数据（消费式读取）
    public abstract FrameInfo PopFrame();

    // 读取下一帧数据（非消费式）
    public abstract bool ReadFrame(FrameInfo frame);

    // 获取当前最大帧编号
    public abstract int GetCurrentMaxFrameId();

    // 设置拉帧策略配置（JSON格式）
    // 示例：{"SyncTimeoutMS": 500, "SyncPredictTimeMS": 60}
    public abstract void SetCommonConfig(string config);

    // 获取网络统计
    // key: "totalFrameCount", "totalLostFrameCount", "totalInputCount", "totalLostInputCount"
    public abstract int GetNetworkStat(string key);
}
```

### 3.2 LockStepImpl 底层实现（来自`LockStepImpl.cs`）

```csharp
public class LockStepImpl : LockSteper, ILockStep
{
    // --------- P/Invoke 调用C++底层 ---------
    [DllImport("__Internal")]
    private static extern bool gcloud_lockstep_has_logined();

    [DllImport("__Internal")]
    private static extern LockStepStatus gcloud_lockstep_get_status();

    // 初始化：将C#配置对象序列化后传给C++
    public override bool Initialize(LockStepInitializeInfo initInfo)
    {
        base.Initialize(false, true);
        return Initialize(initInfo, null);
    }

    public bool Initialize(LockStepInitializeInfo initInfo, IBufferStrategy bufferStrategy)
    {
        IGCloud gcloud = IGCloud.Instance;

        if (initInfo != null)
        {
            this.initInfo = initInfo;

            byte[] buffer;
            if (initInfo.Encode(out buffer) && buffer != null)
            {
                // 将序列化后的配置传给Native层
                return gcloud_lockstep_initialize(this.ObjectId, buffer, buffer.Length) == 0;
            }
        }
        return false;
    }
}
```

### 3.3 帧同步战斗中的业务调用（典型写法）

```csharp
// ---- 1. 初始化（进入战斗前）----
var initInfo = new LockStepInitializeInfo()
{
    MaxBufferSize = 1024 * 100,   // 100KB缓冲
    StartFrameId = 0,
    IsTcpCritical = false,        // 不强制TCP（UDP优先）
    EnableMesh = false,           // 不开启Mesh
    FastThreadMode = true,        // 启用快速线程模式
    HeartBeatInterval = 5000      // 5秒心跳
};
LockSteper.Instance.Initialize(initInfo);

// ---- 2. 登录（传入服务器下发的accessInfo）----
LockSteper.Instance.Login(accessInfo, (result) =>
{
    if (result.IsSuccess())
    {
        Debug.Log("LockStep登录成功，PlayerID: " + LockSteper.Instance.PlayerID);
        // 通知服务器"我准备好了"
        LockSteper.Instance.Ready((readyResult) =>
        {
            Debug.Log("所有玩家就绪，战斗开始！");
        });
    }
});

// ---- 3. 订阅状态变化 ----
LockSteper.Instance.StateChangedEvent += (state, result) =>
{
    switch (state)
    {
        case LockStepState.Reconnecting:
            ShowReconnectingUI();
            break;
        case LockStepState.Reconnected:
            HideReconnectingUI();
            break;
        case LockStepState.Error:
            HandleFatalError(result);
            break;
    }
};

// ---- 4. 每帧Update中拉取帧数据 ----
void Update()
{
    while (true)
    {
        FrameInfo frame = LockSteper.Instance.PopFrame();
        if (frame == null) break;

        // 处理该帧的所有玩家操作
        for (int i = 0; i < frame.ValidDataCount; i++)
        {
            RelayData playerData = frame.DataCollection[i];
            ProcessPlayerInput(playerData.PlayerId, playerData);
        }
    }
}

// ---- 5. 发送本帧操作 ----
void SendPlayerInput(byte[] inputData)
{
    // 普通操作用UDP
    LockSteper.Instance.Input(inputData, rawUdp: true);
}

void SendCriticalInput(byte[] inputData)
{
    // 重要操作用可靠UDP，设置Subscribe标记（服务器确认收到）
    LockSteper.Instance.Input(inputData, rawUdp: false,
                              flag: LockStepInputFlag.Subscribe);
}
```

### 3.4 JitterBuffer 缓冲策略

```csharp
// 来自 JitterBufferStrategy.cs
public class JitterBufferStrategy : IBufferStrategy
{
    public ILockStep Steper { get; set; }

    // 发送输入（代理给底层LockStep）
    public bool OnInput(byte[] data, int len, bool rawUdp, LockStepInputFlag flag)
    {
        if (Steper != null)
            return Steper.Write(data, len, rawUdp, flag);
        return false;
    }

    // 弹出帧（从底层LockStep读取并通过buffer平滑）
    public FrameInfo OnPopFrame()
    {
        if (Steper != null)
        {
            FrameInfo frame = new FrameInfo();
            bool ret = Steper.ReadFrame(frame);
            if (ret)
                return frame;
        }
        return null;
    }

    // 每帧更新（可在此实现平滑算法）
    public void Update() { }
}
```

## 4. 设计亮点

### 4.1 可靠UDP与原始UDP混用

LockStep SDK支持两种UDP传输：
- **rawUdp=true**：最低延迟，约1ms差异，但可能丢包（5-10%的操作可能丢失）
- **rawUdp=false（可靠UDP）**：内置重传机制，轻微增加延迟（1-3ms），但确保到达

实际战斗中，普通移动操作用rawUdp（丢了下帧补），技能释放等关键操作用可靠UDP。

### 4.2 SetCommonConfig 动态调参

通过JSON配置动态调整帧同步策略，无需重启：

```csharp
// 弱网环境下增大超时时间
LockSteper.Instance.SetCommonConfig(
    @"{""SyncTimeoutMS"": 1000, ""SyncPredictTimeMS"": 100}"
);

// 良好网络环境下减小超时时间，提升响应
LockSteper.Instance.SetCommonConfig(
    @"{""SyncTimeoutMS"": 300, ""SyncPredictTimeMS"": 40}"
);
```

### 4.3 帧统计API精准诊断

```csharp
// 获取关键统计指标
int totalFrames = LockSteper.Instance.GetNetworkStat("totalFrameCount");
int lostFrames = LockSteper.Instance.GetNetworkStat("totalLostFrameCount");
int lostInputs = LockSteper.Instance.GetNetworkStat("totalLostInputCount");

float lostFrameRate = (float)lostFrames / totalFrames * 100;
Debug.Log($"丢帧率: {lostFrameRate:F1}%");
```

### 4.4 LockStepInputFlag 精细控制

```csharp
[Flags]
public enum LockStepInputFlag
{
    None = 0,
    Subscribe = 0x01,        // 要求服务器确认（类似ACK）
    DuplicateUpstream = 0x10, // 上行数据冗余发送（降低丢包率）
}
```

### 4.5 FastThreadMode提升性能

`FastThreadMode=true`时，LockStep底层使用独立线程处理网络I/O，主线程只负责业务逻辑，不会因网络等待而卡帧。

## 5. 常见问题与最佳实践

### Q1：帧同步和状态同步有什么区别？

| 对比 | 帧同步（本项目） | 状态同步 |
|------|---------|---------|
| 服务器作用 | 仅转发操作 | 计算并下发结果 |
| 客户端计算 | 所有客户端自己算 | 服务器算好再拉取 |
| 延迟体感 | 低（操作立即响应） | 较高（等待服务器返回） |
| 反作弊难度 | 较难（客户端自主计算） | 较易（服务器权威） |
| 适用场景 | 实时格斗/MOBA | 大世界MMO |

### Q2：为什么会出现"帧不同步"（Desync）？

常见原因：
1. **浮点数运算**：不同CPU架构的浮点运算结果可能有精度差异，应全部改用整数或定点数运算
2. **随机数**：每个客户端必须使用相同种子的伪随机数生成器
3. **未确定性操作**：如`Dictionary`的遍历顺序在不同平台可能不同
4. **时序依赖**：不能依赖系统时钟做游戏逻辑，必须用帧计数

当检测到Desync时，游戏会收到`VKeyDef.PVP_SYNC_ERROR`（507）通知。

### Q3：LockStep登录需要什么参数？

`Login(byte[] accessInfo, callback)`中的`accessInfo`是**游戏服务器下发的入场凭证**，包含房间信息、玩家身份等。客户端不会直接连接LockStep服务器，必须通过游戏服务器获取这个凭证，避免玩家伪造连接。

### Q4：`PopFrame()`和`ReadFrame()`有什么区别？

- `PopFrame()`：消费式读取，读取后帧从缓冲区移除（不可重复读）
- `ReadFrame(FrameInfo frame)`：非消费式读取，帧仍保留在缓冲区

实际战斗循环中用`PopFrame()`；断线重连后需要追帧时可用`ReadFrame()`配合`SetStartFrame(frameId)`从指定帧开始重播。

### Q5：战斗结束后怎么清理？

## 6. 总结

本项目的帧同步方案基于**GCloud LockStep SDK**，通过UDP/可靠UDP混合传输实现低延迟的多人实时对战。核心设计是"服务器只转发，客户端自计算"，通过`FrameInfo`/`RelayData`结构统一封装帧数据，JitterBuffer平滑网络抖动，SetCommonConfig支持动态调参，FrameProfile支持精细诊断。

掌握了这套系统，你就能理解为什么玩家感觉"我的操作立刻有反应"——这正是帧同步低延迟的体现。
