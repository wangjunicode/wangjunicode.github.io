# 对象池驱动的可复用集合组件设计模式

## 前言

在游戏运行时，集合（List、Dictionary、HashSet 等）是使用频率最高的数据结构。它们通常出现在两种场景：

1. **长期持有**：组件级别的数据容器，随组件生命周期存在
2. **临时使用**：函数内部、事件处理中短暂使用，函数结束即可回收

第一种场景没什么问题，但第二种场景如果每次都 `new List<T>()`，则每次调用都会在堆上分配内存，给 GC 制造压力。在 60fps 或服务端高并发场景下，这些碎片化分配会累积成明显的 GC 停顿。

ET 框架的解法非常优雅：**让标准集合类继承自身，同时实现 `IDisposable`，通过工厂方法从对象池取出，用 `using` 语句自动归还**。本文从源码出发，完整解析这套设计的原理与用法。

## 一、设计基础：DisposeObject 与 IPool

在理解集合组件之前，先看框架的对象基类体系：

```csharp
// 最顶层抽象基类（纯标记）
public abstract class Object { }

// 可释放对象：实现 IDisposable + ISupportInitialize
public abstract class DisposeObject : Object, IDisposable, ISupportInitialize
{
    public virtual void Dispose() { }
    public virtual void BeginInit() { }
    public virtual void EndInit() { }
}

// 对象池归属标记接口
public interface IPool
{
    bool IsFromPool { get; set; }
}
```

`DisposeObject` 是框架中所有可被对象池管理对象的基类。它实现了 `ISupportInitialize`（来自 `System.ComponentModel`），这意味着它可以参与设计时序列化（如在 Inspector 编辑时触发 `BeginInit`/`EndInit`）。

`IPool` 接口的 `IsFromPool` 属性用于区分"这个对象是从对象池取出的"还是"直接 new 的"——对象归还时需要判断，防止把非池对象错误归还。

集合组件并没有继承 `DisposeObject`（因为 C# 不支持多继承，它们已经继承了 `List<T>` 等），而是直接实现 `IDisposable`，在 `Dispose` 时手动调用 `ObjectPool.Instance.Recycle(this)`。

## 二、ListComponent：最基础的可复用列表

```csharp
public class ListComponent<T> : List<T>, IDisposable
{
    public static ListComponent<T> Create()
    {
        if (ObjectPool.Instance == null) return new ListComponent<T>();
        return ObjectPool.Instance.Fetch(typeof(ListComponent<T>)) as ListComponent<T>;
    }

    public void Dispose()
    {
        if (ObjectPool.Instance == null) return;
        this.Clear();
        ObjectPool.Instance.Recycle(this);
    }
}
```

### 2.1 继承自 List<T> 的好处

`ListComponent<T>` 直接继承 `List<T>`，这意味着：

- **零拆箱开销**：不需要 `.Inner` 属性访问内部集合，`foreach`、`Count`、索引器直接可用
- **API 完整性**：`Sort`、`BinarySearch`、`AddRange` 等所有 `List<T>` 方法直接继承
- **类型兼容**：可以直接传给接受 `List<T>` 或 `IList<T>` 参数的方法

```csharp
// 直接当 List<T> 用
var list = ListComponent<Entity>.Create();
list.Add(entity1);
list.Sort((a, b) => a.Id.CompareTo(b.Id));
foreach (var e in list) { /* ... */ }
```

### 2.2 对象池 null 防御

注意 `Create` 和 `Dispose` 里都有 `ObjectPool.Instance == null` 检查：

```csharp
if (ObjectPool.Instance == null) return new ListComponent<T>();
```

这是为了支持**单元测试和非 Unity 环境**。在测试环境中，`ObjectPool` 可能还没初始化（或故意不初始化），这时候退化成普通 `new`，让测试代码正常运行而不抛 `NullReferenceException`。

### 2.3 标准使用范式：using 自动归还

```csharp
public async ETTask<List<Unit>> GetNearbyUnits(Vector3 pos, float radius)
{
    using var buffer = ListComponent<Unit>.Create();  // 从对象池取出

    foreach (var unit in this.Scene().GetComponent<UnitComponent>().GetAll())
    {
        if (Vector3.Distance(unit.Position, pos) <= radius)
        {
            buffer.Add(unit);
        }
    }

    // 这里处理 buffer 中的数据
    DoSomethingWith(buffer);

    return buffer.ToList(); // 需要持久化结果时，转为新 List
    // using 块结束时，Dispose() 自动调用，buffer 归还对象池
}
```

`using` 语句在 C# 8+ 版本可以写成 `using var`（不需要大括号），让代码更简洁。对象用完自动 `Dispose`，永远不会忘记归还。

### 2.4 陷阱：不要返回 Component 本身

```csharp
// ❌ 错误：返回了会被 Dispose 的对象
using var list = ListComponent<int>.Create();
list.Add(1);
return list; // 离开 using 块后 list 已被归还，调用方拿到的是"脏"对象

// ✅ 正确：在 using 内转换为普通集合再返回
using var list = ListComponent<int>.Create();
list.Add(1);
return list.ToList(); // 创建新 List<int>，安全返回
```

## 三、DictionaryComponent：支持初始容量的字典

```csharp
public class DictionaryComponent<T, K> : Dictionary<T, K>, IDisposable
{
    public static DictionaryComponent<T, K> Create(int capacity = 0)
    {
        DictionaryComponent<T, K> dict;

        if (ObjectPool.Instance == null)
        {
            dict = new DictionaryComponent<T, K>();
        }
        else
        {
            dict = ObjectPool.Instance.Fetch<DictionaryComponent<T, K>>();
        }

        if (capacity > 0)
        {
            dict.EnsureCapacity(capacity);  // .NET 6+ API
        }

        return dict;
    }

    public void Dispose()
    {
        this.Clear();
        ObjectPool.Instance.Recycle(this);
    }
}
```

### 3.1 capacity 参数：减少 rehash

字典在元素数量超过 `capacity * loadFactor`（默认 0.72）时会触发 rehash，重新分配更大的桶数组并迁移所有元素。如果已知大致规模，提前指定容量可以避免 rehash：

```csharp
// 知道大约有 50 个配置项时
using var configDict = DictionaryComponent<int, ConfigData>.Create(64); // 预分配容量

for (int i = 0; i < 50; i++)
{
    configDict[i] = LoadConfig(i);
}
```

`EnsureCapacity` 是 .NET 6 引入的 API（`Dictionary<TKey, TValue>.EnsureCapacity`），比构造函数传容量更灵活，可以在对象池复用时动态扩容。

### 3.2 泛型方法 Fetch vs 非泛型 Fetch

注意 `DictionaryComponent` 用的是泛型版本：

```csharp
dict = ObjectPool.Instance.Fetch<DictionaryComponent<T, K>>();
```

而 `ListComponent` 用的是非泛型版本：

```csharp
ObjectPool.Instance.Fetch(typeof(ListComponent<T>)) as ListComponent<T>
```

两种写法等价，只是代码风格不同。泛型版本更简洁，非泛型版本在某些反射场景下更灵活。

## 四、HashSetComponent：去重集合

```csharp
public class HashSetComponent<T> : HashSet<T>, IDisposable
{
    public static HashSetComponent<T> Create()
    {
        return ObjectPool.Instance.Fetch(typeof(HashSetComponent<T>)) as HashSetComponent<T>;
    }

    public void Dispose()
    {
        this.Clear();
        ObjectPool.Instance.Recycle(this);
    }
}
```

`HashSetComponent` 没有 null 防御（不像 `ListComponent` 和 `StackComponent`），因此在没有对象池的环境（单元测试）下调用 `Create()` 会抛 `NullReferenceException`。这是一处轻微的设计不一致，实际使用中要注意。

**典型场景：去重 Entity ID**

```csharp
// 收集需要广播的目标，自动去重
using var targetSet = HashSetComponent<long>.Create();
targetSet.Add(unit1.Id);
targetSet.Add(unit2.Id);
targetSet.Add(unit1.Id); // 重复添加，自动忽略

foreach (var id in targetSet)
{
    SendMessage(id, msg);
}
```

## 五、StackComponent：LIFO 栈

```csharp
public class StackComponent<T> : Stack<T>, IDisposable
{
    public static StackComponent<T> Create()
    {
        if (ObjectPool.Instance == null) return new StackComponent<T>();
        return ObjectPool.Instance.Fetch(typeof(StackComponent<T>)) as StackComponent<T>;
    }

    public void Dispose()
    {
        if (ObjectPool.Instance == null) return;
        this.Clear();
        ObjectPool.Instance.Recycle(this);
    }
}
```

`StackComponent` 继承 `Stack<T>`，适合需要 LIFO（后进先出）语义的临时操作。常见用途：

- **场景图遍历**：深度优先遍历 Entity 树时，用 Stack 代替递归
- **撤销/重做**：操作历史入栈，撤销时出栈

```csharp
// 非递归深度优先遍历 Entity 树
public static void VisitAll(Entity root, Action<Entity> visitor)
{
    using var stack = StackComponent<Entity>.Create();
    stack.Push(root);

    while (stack.Count > 0)
    {
        var current = stack.Pop();
        visitor(current);

        foreach (var child in current.Children.Values)
        {
            stack.Push(child);
        }
    }
}
```

## 六、EnumerableComponent：零分配过滤迭代器

`EnumerableComponent` 是这套组件中最精妙的一个，它实现了**过滤 + 转换**的惰性迭代，同时把迭代器本身放入对象池：

```csharp
public class EnumerableComponent<TSource, TResult>
    : IEnumerable<TResult>, IEnumerator<TResult>
    where TResult : class
    where TSource : class
{
    public delegate bool CheckHandler(TResult item);
    public delegate TResult ConvertHandler(TSource item);

    private IEnumerator<TSource> _sourceListEnumerator;
    private CheckHandler _func;
    private ConvertHandler _ConvertFunc;

    // 默认转换：直接 as 类型转换
    private static ConvertHandler defaultConvertFunc = defaultConvert;
    private static TResult defaultConvert(TSource source) => source as TResult;

    public static EnumerableComponent<TSource, TResult> Create(
        IEnumerable<TSource> sourceList,
        CheckHandler checkFunc,
        ConvertHandler convertFunc = null)
    {
        var item = ObjectPool.Instance.Fetch(typeof(EnumerableComponent<TSource, TResult>))
                   as EnumerableComponent<TSource, TResult>;
        item._sourceListEnumerator = sourceList.GetEnumerator();
        item._func = checkFunc;
        item._ConvertFunc = convertFunc != null ? convertFunc : defaultConvertFunc;
        return item;
    }

    public void Dispose()
    {
        Reset();
        ObjectPool.Instance.Recycle(this);
    }

    public bool MoveNext()
    {
        while (_sourceListEnumerator.MoveNext())
        {
            var current = _ConvertFunc(_sourceListEnumerator.Current);
            if (_func(current))
                return true;
        }
        return false;
    }

    public TResult Current => _ConvertFunc(_sourceListEnumerator.Current);
    // ... IEnumerable/IEnumerator 实现 ...
}
```

### 6.1 同时实现 IEnumerable 和 IEnumerator 的技巧

`EnumerableComponent` 同时实现了 `IEnumerable<TResult>` 和 `IEnumerator<TResult>`。这意味着**对象本身就是迭代器**，`GetEnumerator()` 直接返回 `this`：

```csharp
public IEnumerator<TResult> GetEnumerator() => this;
```

好处：
- 不需要再 `new` 一个迭代器对象，节省一次分配
- 配合对象池，整个迭代过程无 GC 分配

缺点：
- **不支持并发迭代**：同一个 `EnumerableComponent` 实例不能同时进行两次 `foreach`
- **不可重入**：迭代状态保存在对象上，不能在迭代中途保存引用后再次迭代

对于游戏逻辑中的单线程场景，这些缺点通常不是问题。

### 6.2 使用示例：过滤活着的玩家

```csharp
// TSource = Entity（原始类型）
// TResult = PlayerComponent（目标类型）
using var aliveEnumerable = EnumerableComponent<Entity, PlayerComponent>.Create(
    sourceList: unitComponent.GetAll(),
    checkFunc: p => p != null && !p.IsDisposed && p.HP > 0,
    convertFunc: e => e.GetComponent<PlayerComponent>()
);

foreach (var player in aliveEnumerable)
{
    // 只处理存活玩家
    player.TakeDamage(damage);
}
// using 结束，迭代器归还对象池
```

对比 LINQ 的写法：

```csharp
// LINQ 写法：每次 .Where().Select() 都会分配迭代器对象
foreach (var player in unitComponent.GetAll()
    .Select(e => e.GetComponent<PlayerComponent>())
    .Where(p => p != null && !p.IsDisposed && p.HP > 0))
{
    player.TakeDamage(damage);
}
```

在热路径（每帧或每次伤害计算都执行），`EnumerableComponent` 版本零 GC，LINQ 版本每次迭代都会分配。

## 七、对象池交互原理

所有集合组件都依赖同一个 `ObjectPool.Instance`。其核心操作：

```csharp
// 取出（如果池中有，直接返回；否则 new 一个）
T obj = ObjectPool.Instance.Fetch<T>();
// 或
T obj = ObjectPool.Instance.Fetch(typeof(T)) as T;

// 归还（Clear 后放回池，等待下次复用）
ObjectPool.Instance.Recycle(obj);
```

对象池按类型分槽（`Dictionary<Type, Queue<object>>`），`Fetch` 时 `Dequeue`，`Recycle` 时 `Enqueue`。集合在归还前必须 `Clear()`，否则下次取出时会携带上次的数据。

**为什么不在 Fetch 时 Clear？**

因为对象池不能预知取出的对象是什么状态——它可能是全新 new 出来的（是空的），也可能是之前归还的（理论上应该已 Clear）。如果双重 Clear，就浪费了一次遍历。框架约定：**归还时清空，取出时假设干净**。

## 八、与 FixedLengthQueue 的对比

框架中还有一个 `FixedLengthQueue<T>`（基于数组的固定长度循环队列），与本文的动态集合组件形成互补：

| | ListComponent | DictionaryComponent | FixedLengthQueue |
|---|---|---|---|
| 容量 | 动态扩容 | 动态扩容 | 固定（构造时指定） |
| 对象池 | ✅ | ✅ | ❌（直接 new） |
| 适用场景 | 临时列表 | 临时字典 | 环形缓冲/滑动窗口 |
| 访问方式 | 索引/迭代 | Key/迭代 | 入队/出队 |

## 九、EnumerableComponent 的静态委托优化

```csharp
private static ConvertHandler defaultConvertFunc = defaultConvert;
private static TResult defaultConvert(TSource source) => source as TResult;
```

`defaultConvertFunc` 是**静态字段**，不是每次 `Create` 时都 new 一个委托。这利用了 C# 委托的特性：方法组转换为委托时，如果目标方法是静态的，通常只分配一次（编译器缓存）。把它存为静态字段可以确保只有一个委托实例，进一步减少分配。

## 十、设计总结与最佳实践

### 核心设计模式

| 模式 | 实现方式 |
|---|---|
| **工厂方法** | `static Create()` 封装对象池取出逻辑 |
| **RAII（资源获取即初始化）** | `IDisposable` + `using` 自动归还 |
| **继承扩展** | 直接继承集合类，保留完整 API |
| **空对象防御** | `ObjectPool.Instance == null` 检查支持测试环境 |
| **惰性迭代** | `EnumerableComponent` 实现自身既是 Enumerable 又是 Enumerator |

### 使用原则

1. **临时集合用 Component，长期集合用普通 new**

   ```csharp
   // ✅ 临时处理用 using + Component
   using var temp = ListComponent<int>.Create();
   
   // ✅ 长期持有用普通集合
   public class PlayerComponent : Entity, IAwake
   {
       public Dictionary<long, Buff> Buffs = new();
   }
   ```

2. **using 结束前完成所有操作**：不要保存 Component 引用在 using 块外使用

3. **不要在 Dispose 后访问**：归还后数据已 Clear，访问会得到错误结果

4. **EnumerableComponent 不支持并发迭代**：嵌套迭代同一个实例会出错

5. **DictionaryComponent 预设容量**：已知大小时传入 capacity，避免 rehash

## 结语

`ListComponent`、`DictionaryComponent`、`HashSetComponent`、`StackComponent`、`EnumerableComponent` 构成了 ET 框架的**零 GC 临时集合工具箱**。它们遵循统一的 `Create + using + Dispose` 范式，把对象池集成到日常集合操作中，让开发者在享受标准集合 API 的同时，自动获得对象复用带来的性能红利。

理解这套设计，也就理解了 ET 框架对"分配=性能敌人"的一贯态度：不是不用集合，而是让集合的生命周期可控、可复用、可测量。
