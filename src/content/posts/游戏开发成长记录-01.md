> 这是「游戏开发成长记录」系列的第一篇。我想用这个系列，记录一个普通计算机专业学生，如何一步步成为游戏客户端工程师，再到带团队做项目的完整历程。不是成功学，是实话实说。

## 为什么是游戏开发

大三的时候，我面临一个所有 CS 学生都会面临的问题：方向选哪个？

前端、后端、算法、嵌入式、游戏开发……每个方向看起来都有前途，每个方向看起来又都很难。

选游戏开发，说来简单：**我喜欢玩游戏，我想做自己玩过的东西**。

但很快我发现，「喜欢玩游戏」和「能做游戏」之间，有一道不小的墙。

---

## 第一道墙：不知道学什么

游戏开发的技术栈比我想象的复杂得多。随便搜一下，能看到：

- 引擎：Unity、Unreal、Cocos
- 语言：C#、C++、Lua、Python
- 图形学：OpenGL、Vulkan、Shader
- 网络：TCP、UDP、帧同步、状态同步
- 还有：物理、AI、音频、工具开发……

每个方向展开都是无底洞。

后来我想明白了一件事：**不要试图先搞清楚所有方向，选一个点凿下去**。

我选了 Unity + C#，理由很简单：中文资料多、工作岗位多、可以快速出原型。

---

## C# 基础：不只是语法

学 C# 的第一个月，我以为自己在学语法。后来才明白，我学的是**面向对象的思维方式**。

几个当时觉得难但后来觉得很重要的概念：

### 委托与事件

```csharp
// 委托：把方法当变量传递
public delegate void OnDamageHandler(int damage);

// 事件：委托的安全封装，只能在声明类内部触发
public event OnDamageHandler OnDamage;

// 使用 Action<T> 简化委托声明（推荐）
public event Action<int> OnDamage;
```

当时觉得这个东西很绕，为什么不直接调方法？

后来写游戏 UI 的时候才真正理解：**事件让调用方和被调用方解耦**。血量变化时，UI、特效、音效可以各自订阅，不需要血量模块知道谁在关心它。

### 泛型

```csharp
// 没有泛型时：每种类型写一个容器
public class IntList { ... }
public class StringList { ... }

// 有了泛型：一个容器支持所有类型
public class MyList<T> {
    private T[] items;
    public void Add(T item) { ... }
    public T Get(int index) { ... }
}
```

游戏开发中泛型无处不在：对象池、事件系统、数据容器……理解泛型是写出可复用代码的前提。

### 装箱与拆箱的性能陷阱

```csharp
// 看起来无害，实际上每次 Add 都有装箱
ArrayList list = new ArrayList();
list.Add(1);  // int → object，装箱！

// 正确做法：用泛型集合
List<int> list = new List<int>();
list.Add(1);  // 无装箱
```

这是当时课上没人讲、但游戏开发中非常重要的知识点。每帧调用带装箱的代码，GC 就会频繁触发，游戏就会卡顿。

## 数据结构：游戏中的实际用途

学数据结构，不只是为了面试。

**平衡二叉树（BST）**：理解了它之后，才真正明白 `SortedDictionary` 和 `Dictionary` 的区别，知道什么时候选哪个。

**哈希表**：游戏中最常用的数据结构。技能配置、怪物配置、UI 页面……几乎所有「用 ID 查数据」的场景都是哈希表。

```csharp
// 游戏配置表：用字典存，O(1) 查找
Dictionary<int, SkillConfig> skillTable = new Dictionary<int, SkillConfig>();
SkillConfig config = skillTable[skillId]; // O(1)
```

**队列/优先队列**：A* 寻路的核心数据结构。后来真的在项目里用 A* 的时候，这段基础救了我很多时间。

---

## 网络同步：被一张图打开的世界

大三下学期，我第一次认真研究网络同步。起因是看到一篇讲帧同步的文章，里面一张图描述了服务器收集所有客户端输入、再统一下发的流程。

那张图改变了我对「多人游戏怎么运作」的认知：

```
客户端A ─── 操作输入 ──→
客户端B ─── 操作输入 ──→  服务器  ──→ 所有客户端收到同一帧的所有输入
客户端C ─── 操作输入 ──→           ──→ 各自运行相同的逻辑 → 结果一致
```

**帧同步的关键**：不同步结果，同步输入。所有客户端用相同的初始状态 + 相同的输入序列，运行相同的逻辑，得到相同的结果。

这个思路优雅到让我当时觉得有些震撼。

当然，这里有一个极其苛刻的前提：**逻辑必须是完全确定的**。

```csharp
// ❌ 帧同步中绝对禁止
float value = Random.value;           // 不同客户端随机数不同
float dist = Vector3.Distance(a, b);  // 浮点数跨平台结果可能不同
Physics.Raycast(...);                  // Unity 物理不确定

// ✅ 使用定点数替代浮点数
FixedPoint dist = FixedMath.Distance(a, b);
```

这些知识在大三时只是「了解」，真正用到是几年后参与实际项目。但正是这段时间打下的基础，让后来的技术讨论不会完全懵。

## 第一次写「完整」的小游戏

大三暑假，用 Unity 做了一个非常简陋的 2D 射击游戏：
- 玩家控制飞船移动、射击
- 敌人有简单的跟踪 AI
- 有血量、得分、死亡重开

代码质量惨不忍睹：一个 `GameManager` 挂载了所有逻辑，几百行的 `Update()` 函数，没有任何设计模式，全靠 `Find` 获取对象引用。

但它能跑，它是我做的。

现在回头看，这个「什么都不规范」的小项目其实很重要——它让我**感受到了游戏运行起来的感觉**，也让我开始真正思考「代码为什么要有结构」这个问题。

---

## 准备面试：第一次意识到差距

大四开始找工作。第一次面试游戏公司，被问到的问题我只会一半：

- C# 内存管理？讲了 GC，但不知道 GC 的具体算法
- 多态是怎么实现的？答出了虚函数表，但说不清楚 IL 层面
- 帧同步和状态同步的区别？答出来了，但说不出优劣取舍
- 设计模式用过哪些？只用过单例……

那次面试没过。但我知道自己差在哪了。

这种「知道自己哪里不足」的感觉，比面试通过更有价值。

---

## 这一阶段最重要的一件事

回头来看，大三这一年最重要的不是学了多少技术，而是**建立了「主动学习」的习惯**：

遇到不懂的，不绕开，去查、去写代码验证、记录下来。

这个习惯后来支撑了我在项目中快速上手新技术，解决陌生领域的问题。

游戏开发的技术栈太宽了，任何人都不可能全部精通。**能快速学习、快速上手**，比已经会什么更重要。
