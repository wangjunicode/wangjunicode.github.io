# 角色行为状态机设计与连招系统

在动作游戏中，角色的行为总是处于某种"状态"中：站着不动、奔跑、攻击、被打飞……从一个状态切换到另一个状态，有着严格的条件和规则。"什么时候可以取消当前动作"、"怎样连击更流畅"——这些都由状态机系统决定。本文深入解析一套商业级的 FSM（有限状态机）实现。

---

## 第一性原理：为什么需要状态机？

假设你用 `if-else` 管理角色行为：

```csharp
void Update()
{
    if (isAttacking && canCombo && Input.GetKey("attack"))
    {
        StartCombo();
    }
    else if (!isAttacking && isGrounded && Input.GetKey("attack"))
    {
        StartAttack();
    }
    else if (isHurt && !isInvincible)
    {
        TakeHurt();
    }
    // ...不断增加条件
}
```

随着状态增多，条件组合爆炸，代码越来越难维护。更致命的是：不同状态的逻辑相互交织，添加新状态几乎必然破坏已有状态。

**状态机的解决方案**：将每种状态封装为独立对象，每个状态只关心自己的逻辑和转换条件。系统只需要维护"当前状态"，调用其 Update 和处理转换即可。

## FSMComponent 核心结构

```csharp
public class FSMComponent : Entity
{
    public UniFSM Fsm;              // 当前运行的 FSM 实例
    public int curID;               // 当前 FSM 配置 ID
    public int DeadCnt;             // 死亡计数（防止重复死亡触发）
    public bool DoBreakDefense;     // 破防标志
    
    // 状态事件队列（VirtualInput -> FSM 的桥梁）
    public List<EInputKey> FsmStateEvents = new List<EInputKey>();
}
```

`UniFSM` 是基于节点图的 FSM 实现，每个节点代表一个状态，节点间的连线代表转换条件。策划在工具中连线，程序运行时执行。

## 状态层级：根状态与叶状态

战斗状态机采用**嵌套 FSM**（HFSM）设计：

```
根状态（Root State）
  ├── Idle（空闲）
  ├── Move（移动）
  ├── Attack（攻击）
  │     ├── NormalAttack（普通攻击）
  │     │     ├── Attack1（第1段）
  │     │     ├── Attack2（第2段）
  │     │     └── Attack3（第3段）
  │     └── SkillAttack（技能攻击）
  ├── Hurt（受击）
  │     ├── Stagger（硬直）
  │     ├── Float（浮空）
  │     └── KnockDown（倒地）
  └── Dead（死亡）
```

查询当前状态时：

```csharp
// 获取根状态标签（粗粒度）
public static EActionType CurrentRootStateTag(this FSMComponent self)
{
    if (self.Fsm == null || self.Fsm.currentState == null)
        return EActionType.None;
    return (EActionType)self.Fsm.currentState.tag;
}

// 获取叶状态标签（细粒度）
public static EActionType CurrentLeafStateTag(this FSMComponent self)
{
    var currentLeafState = self.CurrentLeafState();
    return currentLeafState == null ? EActionType.None : (EActionType)currentLeafState.tag;
}

// 递归获取叶状态
private static FSMState GetLeafState(FSM fsm)
{
    if (fsm == null || fsm.currentState == null) return null;
    
    // 如果当前状态是嵌套FSM，递归进入
    if (fsm.currentState is not NestedFSMState nestedFsmState)
    {
        return fsm.currentState;  // 已经是叶状态
    }
    return GetLeafState(nestedFsmState.subGraph);  // 递归
}
```

这种设计的优势：
- `CurrentRootStateTag` = `EActionType.Attack` 表示角色在攻击大类中
- `CurrentLeafStateTag` = `EActionType.Attack2` 表示在第2段攻击中
- 系统可以按需选择粒度来做判断

## 状态初始化：加载与缓存

```csharp
public static void ChangeToNormalFSM(this FSMComponent self)
{
    var unit = self.GetParent<Unit>();
    var entityType = unit.Type;
    
    if (entityType == EEntityType.Player || entityType == EEntityType.Monster)
    {
        // 从配置表获取玩家/怪物使用的 FSM ID
        self.StartFSMSync(CfgManager.tables.TbSkillConst.PlayerFsm);
    }
    else if (entityType == EEntityType.BattleItem)
    {
        self.StartFSMSync(CfgManager.tables.TbSkillConst.BattleItemFsm);
    }
    else if (entityType == EEntityType.Puppet)
    {
        // 傀儡的 FSM ID 由配置决定
        var conf = unit.GetConfig<CPuppet>();
        if (conf != null && conf.FsmId > 0)
        {
            self.StartFSMSync(conf.FsmId);
        }
    }
}

private static UniFSM LoadFSMInfoSync(this FSMComponent self, int stateId)
{
    var unitComp = self.Domain.GetComponent<UnitComponent>();
    var path = PathUtil.GetFSMPath(stateId);
    
    // 优先从缓存池取
    UniFSM info = null;
    if (unitComp.FSMCache.TryGetValue(stateId, out var fsmCache))
    {
        fsmCache.TryPop(out info);
    }
    
    // 缓存未命中，反序列化加载
    if (info == null)
    {
        info = SerializeHelper.Deserialize<UniFSM>(path);
    }
    
    // 注入 self 变量
    var v = info.localBlackboard.GetVariable<Unit>("self");
    if (v == null) v = info.localBlackboard.AddVariable<Unit>("self");
    v.value = self.GetParent<Unit>();
    v.isReadonly = true;
    
    return info;
}
```

FSM 和技能图一样使用对象池缓存，避免反复反序列化的开销。

## 每帧更新：双重推进模式

```csharp
public static void EnterFrame(this FSMComponent self, FP deltaTime)
{
    if (self.Fsm != null)
    {
        // 检查是否在 Host 状态（主机状态，等待服务器帧）
        bool bHost = self.Fsm.currentState.tag == (int)EActionType.Host;
        
        self.Fsm.UpdateGraph(deltaTime);  // 第一次更新
        
        // 如果从 Host 状态切出，立刻再更新一次
        // 避免 Host -> 正常状态 需要等到下一帧才开始执行的延迟
        if (bHost && self.Fsm.currentState.tag != (int)EActionType.Host)
        {
            self.Fsm.UpdateGraph(deltaTime);  // 第二次更新
        }
        
        self.FsmStateEvents.Clear();  // 清空本帧的状态事件
    }
}
```

`Host` 状态是一个特殊的等待状态：当战斗处于"等待服务器帧"时（如 PVP 模式等待对手帧数据），角色进入 Host 状态暂停所有行为。服务器帧来了，Host 状态退出后，**立刻**再运行一次 Update，让角色不因等待而损失一帧的响应。

## 手动触发条件（连招判定的核心）

```csharp
// 注册一个可被手动触发的转换条件
public static bool TryManualCondition(this FSMComponent self, EInputKey key)
{
    // 检查当前状态是否允许被此键中断
    var skillComp = self.Parent.GetComponent<SkillComponent>();
    if (!skillComp.AbortInput.Contains(key))
    {
        return false;  // 当前没有注册这个可中断的输入
    }
    
    self.FsmStateEvents.Add(key);  // 加入本帧的状态事件队列
    return true;
}
```

这是连招系统的关键：
- 攻击动画执行到"可衔接下一段"的时间窗口时，技能图节点调用 `AddAbortInput(key)` 注册
- 玩家按键触发 `TryManualCondition`，如果此时正好在可衔接窗口内，返回 true
- FSM 收到信号，切换到下一段攻击状态

## 状态感知：进入/退出回调

```csharp
public static void OnStateEnter(this FSMComponent self, IState state)
{
    KHDebug.LogTrack(0, (state as FSMState)?.tag ?? 0);  // 帧轨迹记录
}

public static void OnStateExit(this FSMComponent self, IState state)
{
    KHDebug.LogTrack(0, (state as FSMState)?.tag ?? 0);
    
    // 退出任何状态时，重置格挡命中次数
    var hurtComp = self.Parent.GetComponent<HurtComponent>();
    if (hurtComp != null)
    {
        hurtComp.AttackInfo.ParryHitTimes = 0;
    }
}
```

状态退出时自动重置格挡命中次数，保证了每次进入新状态时格挡计数从0开始，无需手动管理。

## 破防（BreakDefense）状态检查

```csharp
public static EBreakDefenseStatus GetBreakDefenseStatus(this FSMComponent self)
{
    var curStateTag = self.CurrentRootStateTag();
    
    // 正在播放破防动画
    if (curStateTag == EActionType.BreakDefense)
        return EBreakDefenseStatus.Breaking;
    
    // 护盾值归零（可以被破防）
    var numericComp = self.Parent.GetComponent<NumericComponent>();
    var guard = numericComp.GetFinalValue(ENumericId.RealGuard);
    if (guard <= 0)
        return EBreakDefenseStatus.Broken;
    
    return EBreakDefenseStatus.NotBroken;
}
```

`EBreakDefenseStatus` 的三种状态对应三种逻辑：
- `NotBroken`：护盾完好，攻击需要先消耗护盾
- `Broken`：护盾归零但未触发破防动画，下一次攻击可以触发破防
- `Breaking`：正在播放破防动画（被击中的短暂僵直状态）

## 异步等待状态：async/await 的战斗应用

FSM 系统提供了一套 async/await 接口，让技能图可以"等待某个状态的到来"：

```csharp
// 等待进入某个状态
public static async ETTask WaitState(this FSMComponent self, string stateName, ETTask cancelTask = null)
{
    while (true)
    {
        if (cancelTask != null && cancelTask.IsCompleted) return;
        
        if (self.Fsm?.currentStateName == stateName) return;  // 已在目标状态
        
        var waiting = await self.Fsm.WaitStateUpdate();
        if (!waiting) return;  // FSM 已停止
    }
}

// 等待某个状态结束
public static async ETTask WaitStateEnd(this FSMComponent self, string stateName, ETTask cancelTask = null)
{
    while (true)
    {
        if (cancelTask != null && cancelTask.IsCompleted) return;
        
        if (self.Fsm?.currentStateName != stateName) return;  // 已离开目标状态
        
        var waiting = await self.Fsm.WaitStateUpdate();
        if (!waiting) return;
    }
}
```

实际使用场景（特殊动画播放）：

```csharp
// 让角色进入特殊动画状态，等待播放完毕
self.AddVariable("Animation", "SpecialIdle")
    .AddVariable("AnimationLoop", "")
    .AddVariable("SpecialAnimationDuration", -1);

if (self.TryManualCondition(EInputKey.SpecialAnimation))
{
    await self.WaitState(EFSMState.SpecialAnimation);       // 等待进入状态
    await self.WaitStateEnd(EFSMState.SpecialAnimation);    // 等待状态结束
}
// 动画播放完毕，继续后续逻辑
```

这种写法极度清晰：先等进入状态，再等离开状态，比传统的回调链可读性高出一个数量级。

## 常见状态类型及其处理逻辑

| 状态类型 | EActionType | 说明 |
|---------|-------------|------|
| Idle | 空闲 | 可接受任何输入 |
| Move | 移动 | 可以被攻击打断 |
| Attack | 攻击中 | 根据连招窗口决定是否可打断 |
| Hurt | 受击硬直 | 通常不可被打断 |
| Float | 浮空受击 | 追打状态 |
| KnockDown | 倒地 | 等待起身 |
| GetUp | 起身 | 短暂无敌帧 |
| Parry | 格挡 | 可触发弹反 |
| Doge | 闪避 | 无敌帧期间 |
| BreakDefense | 破防演出 | 强制硬直 |
| Dead | 死亡 | 不再响应输入 |
| Host | 等待服务器帧 | PVP 特有 |

## 连招系统的完整流程

```
攻击1开始
  │
  ├─ SkillGraph 启动，播放攻击1动画
  │
  ├─ [特定帧] AddAbortInput(EInputKey.Attack) 注册可连招窗口
  │
  ├─ [玩家按攻击键] VirtualInputComponent.OnTap(Attack)
  │     │
  │     └─ FSMComponent.TryManualCondition(Attack)
  │           ├─ 检查 AbortInput.Contains(Attack) → 包含
  │           └─ 加入 FsmStateEvents
  │
  ├─ [窗口结束帧] RemoveAbortInput(Attack)
  │
  └─ FSM.UpdateGraph
        └─ 检测到 FsmStateEvents 中有 Attack
           └─ 触发状态转换 → 攻击2开始
```

关键在于"窗口期"：只有在 `AddAbortInput` 和 `RemoveAbortInput` 之间的这段时间内按键，才会触发连招。这让连招有了节奏感——不是随时都能连，而是需要在恰当的时机输入。

## 性能考量

**FSM 对象池**：

```csharp
if (unitComp.FSMCache.TryGetValue(stateId, out var fsmCache))
{
    fsmCache.TryPop(out info);
}
```

FSM 实例和 SkillGraph 一样使用对象池，避免频繁反序列化。

**手动更新模式**：

```csharp
info.StartGraph(self.Parent as IUniAgent, info.localBlackboard, Graph.UpdateMode.Manual);
```

FSM 使用手动更新模式，由 `WorldComponent` 的帧循环统一驱动，保证与其他系统的时序一致性，而不是由 FSM 内部计时器驱动（后者可能导致帧序混乱）。

## 总结

状态机是战斗系统的"行为大脑"，定义了角色什么时候能做什么。好的状态机设计：

| 特性 | 实现 |
|------|------|
| 嵌套状态 | HFSM，支持粗/细粒度查询 |
| 条件触发 | AbortInput + FsmStateEvents（连招系统） |
| 异步等待 | WaitState/WaitStateEnd |
| 对象池 | FSMCache 缓存 FSM 实例 |
| 手动更新 | UpdateMode.Manual，由帧循环统一驱动 |
| 双重更新 | Host 状态退出时立刻补一次 Update |

从一个初学者成长为动作游戏程序员，掌握状态机是最重要的一步。理解了"状态"的思维方式，你的代码会从混乱的 if-else 升华为优雅的有限自动机。
