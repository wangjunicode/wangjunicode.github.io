# 实体引用安全机制与场景根节点设计

在基于 ECS 的游戏框架中，实体（Entity）的生命周期由框架统一管理——可以随时被创建、销毁、复用。这带来一个隐患：持有某个实体引用的代码，可能在实体销毁之后仍然访问它，造成逻辑错误甚至崩溃。本文深入解析 `EntityRef<T>`、`EntityWeakRef<T>` 两套引用安全机制，并探讨 `EntitySceneFactory` 工厂方法和 `Root` 根节点单例如何共同构建出整个 ECS 世界的生命起点。

---

## 一、问题背景：ECS 中的悬空引用

传统面向对象编程里，持有一个对象的引用，只要该引用不被置为 null，GC 就会保证对象活着。但在 ECS 框架里，实体对象是被"逻辑销毁"的——`Dispose()` 之后，实体的 `InstanceId` 会被清零或改变，但 C# 对象本体可能还没被 GC 回收。

这意味着：

```csharp
Unit enemy = GetEnemy(); // 获取某个敌人实体
await SomeAsync();       // 等待异步操作
enemy.HP -= 10;          // enemy 可能已经 Dispose 了！
```

如果 `enemy` 在 `await` 等待期间被销毁，直接访问其属性会触发已销毁实体的业务逻辑，产生难以排查的 Bug。

### 1.1 InstanceId 的关键作用

框架的解决思路核心是 `InstanceId`——每个实体在创建时都会分配一个唯一的实例 ID，销毁时这个 ID 会被重置（归零或变更）。通过比对引用快照时的 InstanceId 与实体当前 InstanceId 是否一致，就能判断实体是否仍然有效。

## 二、EntityRef：带快照的强引用

```csharp
public struct EntityRef<T> where T: Entity
{
    private readonly long instanceId;
    private T entity;

    private EntityRef(T t)
    {
        if (t == null)
        {
            this.instanceId = 0;
            this.entity = null;
            return;
        }
        this.instanceId = t.InstanceId;
        this.entity = t;
    }
    
    private T UnWrap
    {
        get
        {
            if (this.entity == null)
            {
                return null;
            }
            if (this.entity.InstanceId != this.instanceId)
            {
                // instanceId 变化了，说明实体已被销毁/复用，解除强引用
                this.entity = null;
            }
            return this.entity;
        }
    }
    
    public static implicit operator EntityRef<T>(T t)
    {
        return new EntityRef<T>(t);
    }

    public static implicit operator T(EntityRef<T> v)
    {
        return v.UnWrap;
    }
}
```

### 2.1 设计要点

**结构体（struct）而非类（class）**：`EntityRef<T>` 是值类型，栈分配，零 GC 压力。多次创建/销毁引用包装不会带来堆内存分配。

**双字段存储**：
- `instanceId`：创建时快照，`readonly`，永不改变
- `entity`：实际持有的 C# 对象引用，可以被置 null

**`UnWrap` 的三步检查逻辑**：
1. `entity == null` → 快速返回 null
2. `entity.InstanceId != instanceId` → 实体已销毁，将 `entity` 置 null（让 GC 可以回收），返回 null
3. 以上都通过 → 返回有效实体

**隐式转换**：`EntityRef<T>` 支持与 `T` 之间的双向隐式转换，调用侧代码可以写成：

```csharp
EntityRef<Unit> unitRef = unit;          // T → EntityRef<T>
Unit u = unitRef;                         // EntityRef<T> → T（含安全检查）
if (u != null) { u.DoSomething(); }      // 安全使用
```

### 2.2 与普通引用的对比

| 特性 | 直接持有 T | EntityRef\<T> |
|------|-----------|--------------|
| 防止访问已销毁实体 | ❌ | ✅ |
| GC 阻止回收 | ✅ 阻止 | ✅ 阻止（entity 字段有效时） |
| 内存开销 | 8 bytes | 16 bytes（instanceId + 引用） |
| 适用场景 | 同帧内短暂使用 | 跨帧、跨异步持有 |

### 2.3 适用场景示例

跨异步操作是最典型的场景：

```csharp
private EntityRef<Player> _playerRef;

void OnPlayerCreated(Player player)
{
    _playerRef = player; // 快照 instanceId
}

async ETTask DelayedBuff()
{
    await TimerComponent.Instance.WaitAsync(3000);
    
    Player player = _playerRef; // UnWrap，自动检查有效性
    if (player == null) return;  // 3秒内玩家可能已离开
    
    player.AddBuff(BuffType.Haste);
}
```

## 三、EntityWeakRef：基于 WeakReference 的弱引用

```csharp
public struct EntityWeakRef<T> where T: Entity
{
    private long instanceId;
    private readonly WeakReference<T> weakRef;

    private EntityWeakRef(T t)
    {
        if (t == null)
        {
            this.instanceId = 0;
            this.weakRef = null;
            return;
        }
        this.instanceId = t.InstanceId;
        this.weakRef = new WeakReference<T>(t);
    }
    
    private T UnWrap
    {
        get
        {
            if (this.instanceId == 0) return null;

            if (!this.weakRef.TryGetTarget(out T entity))
            {
                this.instanceId = 0;
                return null;
            }

            if (entity.InstanceId != this.instanceId)
            {
                this.instanceId = 0;
                return null;
            }
            return entity;
        }
    }
    // 隐式转换同 EntityRef<T>
}
```

### 3.1 弱引用的核心价值

`WeakReference<T>` 允许 GC 在内存压力下回收对象，即使还有弱引用指向它。这解决了一个特殊问题：

**若用 `EntityRef<T>` 持有已 Dispose 的实体，`entity` 字段被置 null 之前，该对象无法被 GC 回收**（因为强引用还在）。而 `EntityWeakRef<T>` 通过 `WeakReference` 包装，允许 GC 在任何时机回收实体对象，不干扰垃圾回收节奏。

### 3.2 `UnWrap` 的三重验证

1. `instanceId == 0` → 从未设置或已明确失效，返回 null
2. `weakRef.TryGetTarget()` 失败 → GC 已回收对象，清零 instanceId，返回 null
3. `entity.InstanceId != instanceId` → 实体已重用为别的实例，清零 instanceId，返回 null

### 3.3 EntityRef vs EntityWeakRef

| 特性 | EntityRef\<T> | EntityWeakRef\<T> |
|------|--------------|-----------------|
| 阻止 GC 回收 | ✅ 是 | ❌ 否（弱引用） |
| 防悬空访问 | ✅ | ✅ |
| 内存压力场景 | 可能延迟回收 | GC 可自由回收 |
| WeakReference 额外开销 | 无 | 有（堆分配 WeakReference 对象） |
| 适用场景 | 常规跨帧/跨异步 | 缓存型弱观察者、观察者列表 |

> **注意**：源码注释提到 `WeakReference` 的性能尚未完整测试，因此实际项目中 `EntityWeakRef` 多用于非热路径场景（如编辑器工具、调试观察）。

## 四、EntityCustom：调试链路字符串

```csharp
public partial class Entity
{
    public virtual string GetDebugString()
    {
        return (parent == null || parent is Scene) 
            ? ViewName 
            : ZString.Concat(parent.GetDebugString(), " -> ", ViewName);
    }
}
```

`GetDebugString()` 递归构建实体的层级路径字符串，用于调试输出，例如：

```
GamePlay -> BattleScene -> Player -> SkillComponent
```

使用 `ZString.Concat`（Cysharp.Text）而非普通字符串拼接，避免 Debug 路径生成时产生大量 string 垃圾，对运行时性能友好。

## 五、EntitySceneFactory：场景工厂

```csharp
public static class EntitySceneFactory
{
    // 精确指定 id 和 instanceId（用于序列化恢复等场景）
    public static Scene CreateScene(long id, long instanceId, int zone, SceneType sceneType, string name, Entity parent = null)
    {
        Scene scene = new Scene(id, instanceId, zone, sceneType, name, parent);
        return scene;
    }

    // 自动生成 instanceId（常规创建）
    public static Scene CreateScene(int zone, SceneType sceneType, string name, Entity parent = null)
    {
        long instanceId = IdGenerater.Instance.GenerateInstanceId();
        Scene scene = new Scene(zone, instanceId, zone, sceneType, name, parent);
        return scene;
    }
}
```

### 5.1 工厂方法的意义

直接 `new Scene(...)` 需要手动传入所有参数，而且多处调用时容易遗漏参数校验。工厂方法：

1. **封装 instanceId 生成逻辑**：常规创建时自动调用 `IdGenerater` 分配唯一 ID，不需调用方关心
2. **两套重载满足不同需求**：精确重载用于存档恢复（需要与保存时相同的 id/instanceId），自动重载用于运行时动态创建
3. **为将来扩展提供钩子**：工厂内可以插入初始化注册、事件广播等逻辑，不污染 Scene 构造函数

### 5.2 SceneType 枚举的作用

`SceneType` 区分场景的用途（Process、Battle、Lobby 等），框架可基于类型执行不同的生命周期管理策略，例如 Battle 场景销毁时自动回收战斗相关资源。

## 六、Root：ECS 世界的起点

```csharp
public class Root: Singleton<Root>
{
    public Scene Scene { get; }

    public Root()
    {
        this.Scene = EntitySceneFactory.CreateScene(0, SceneType.Process, "Process");
    }

    public override void Dispose()
    {
        this.Scene.Dispose();
    }
}
```

### 6.1 Root 的职责

`Root` 是整个 ECS 实体树的根节点，继承自 `Singleton<Root>`，在游戏启动时创建，贯穿整个运行期。

**为什么需要 Root**：ECS 中所有 Scene、所有 Entity 都组织在树形结构里。没有根节点，就没有层次，也无法做统一的遍历、序列化、销毁级联。`Root` 持有 `Process` 类型的 Scene，这是"进程级别"的顶层 Scene，其下挂载所有其他子 Scene（大厅、战场、资源管理等）。

### 6.2 启动顺序

```
程序入口
  └─ Game（主循环调度单例）
       └─ Root（根节点单例）
            └─ EntitySceneFactory.CreateScene(0, SceneType.Process, "Process")
                 └─ Process Scene
                      ├─ LobbyScene（大厅场景）
                      └─ BattleScene（战斗场景）
                           └─ Unit（实体）
                                └─ SkillComponent（组件）
```

### 6.3 Dispose 级联

```csharp
public override void Dispose()
{
    this.Scene.Dispose(); // 触发 Process Scene 的 Dispose
}
```

`Root.Dispose()` 触发 `Process Scene.Dispose()`，后者会递归销毁所有子 Scene 和子 Entity，实现整个 ECS 世界的优雅关闭。这与 C++ 智能指针的 RAII 模式思想一致：析构/Dispose 自动传播。

### 6.4 使用示例

```csharp
// 游戏启动初始化
Game.AddSingleton<Root>();

// 在任意地方访问根场景
Scene processScene = Root.Instance.Scene;

// 在根场景下创建子场景
Scene battleScene = EntitySceneFactory.CreateScene(1, SceneType.Battle, "Battle", processScene);

// 关闭时
Root.Instance.Dispose(); // 整个 ECS 世界安全关闭
```

## 七、组合应用：异步安全的场景间通信

综合上述机制，来看一个完整的实际场景：战斗结算时，结算组件需要延迟通知大厅组件更新 UI：

```csharp
public class BattleSettleComponent: Entity
{
    // 用 EntityRef 而非直接持有 LobbyComponent
    private EntityRef<LobbyNotifyComponent> _lobbyRef;

    public void Init(LobbyNotifyComponent lobby)
    {
        _lobbyRef = lobby;
    }

    public async ETTask OnBattleEnd(BattleResult result)
    {
        // 广播战斗结果，等待动画播放完毕（3秒）
        await TimerComponent.Instance.WaitAsync(3000);

        // 解引用，自动检查 LobbyNotifyComponent 是否还存在
        LobbyNotifyComponent lobby = _lobbyRef;
        if (lobby == null)
        {
            Log.Warning("LobbyNotifyComponent already disposed, skip notify");
            return;
        }

        lobby.OnBattleResult(result);
    }
}
```

`EntityRef<LobbyNotifyComponent>` 在 3 秒等待期间，若大厅因玩家退出而销毁，`_lobbyRef` 解引用时自动返回 null，避免了空指针访问和已销毁实体的业务污染。

## 八、总结

| 机制 | 核心价值 |
|------|---------|
| **EntityRef\<T>** | 跨帧/跨异步持有实体引用时防悬空，InstanceId 快照验证 |
| **EntityWeakRef\<T>** | 弱持有，不阻止 GC 回收，适合缓存/观察者场景 |
| **EntityCustom.GetDebugString** | 零垃圾调试路径输出，ZString 拼接 |
| **EntitySceneFactory** | 工厂封装 instanceId 生成与场景构建，支持精确/自动两种模式 |
| **Root** | ECS 世界根节点，单例持有 Process Scene，Dispose 级联销毁整个实体树 |

这套机制的本质是"**以 InstanceId 为凭证的弱引用包装**"——不依赖 GC 判断对象活跃性，而是通过业务层的生命周期令牌（InstanceId 变更）来主动检测失效，在游戏框架高频实体创建销毁的场景下既安全又高效。
