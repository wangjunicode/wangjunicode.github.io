# 战斗世界帧循环驱动机制

在游戏战斗系统中，"帧循环"是整个战斗引擎跳动的心脏。每一帧发生的攻击判定、技能执行、物理模拟，都由一个统一的调度器有序驱动。本文将以技术负责人的视角，深入解析战斗世界（WorldComponent）如何构建这套核心循环，以及为什么要这样设计。

---

## 第一性原理：战斗循环到底是什么？

在回答"怎么做"之前，先思考"为什么需要"。

一个战斗场景里可能同时存在几十个 Unit（角色、怪物、投射物），每个 Unit 都需要在每一帧里更新自己的状态：执行技能、移动位置、检测碰撞、播放动画。如果每个 Unit 自己决定何时更新，系统就会变得混乱——甲可能在乙还没更新完时就读取了乙的位置，导致逻辑不一致。

**核心结论**：必须有一个全局协调器，在固定时间窗口内，按照严格的顺序，逐步更新所有 Unit 的状态。这就是 `WorldComponent` 的本质——一个**有序的全局帧调度器**。

---

## WorldComponent 的职责边界

```csharp
public class WorldComponent : Entity, IAwake, IFixedUpdate, IDestroy, IUpdate
{
    public FP _lastCustomInterpolationStartTime { get; set; } = -1f;
    public FP _lastCustomInterpolationDeltaTime { get; set; } = -1f;
    public FP lastTimeScale;
}
```

`WorldComponent` 实现了以下接口：

- `IAwake`：初始化时注册事件监听
- `IFixedUpdate`：固定帧率的逻辑更新（战斗核心逻辑在此）
- `IUpdate`：渲染帧更新（视觉插值在此）
- `IDestroy`：销毁时注销事件

注意，这里用的是 `FP`（Fixed-Point，定点数），而不是 `float`。这是帧同步战斗的关键技术决策——**定点数保证了在不同硬件、不同平台上，相同输入产生完全相同的计算结果**，这是多人联机战斗一致性的根基。

## 核心帧驱动：EnterFrame 详解

```csharp
public static void EnterFrame(this WorldComponent self)
{
    if (EngineRuntime.Pause)
    {
        return;
    }
    
    var unitComponent = self.Domain.GetComponent<UnitComponent>();
    if (unitComponent == null)
    {
        return;
    }
    
    VProfiler.BeginDeepSample("WorldComponent.EnterFrame");
    var activeUnits = unitComponent.GetActiveUnits();
    
    // 1. 处理帧冻结计数器
    foreach (var unit in activeUnits)
    {
        if (unit.FreezeFrames > 0) unit.FreezeFrames--;
    }
    
    // 2. 时间缩放更新（必须最先）
    self.Domain.GetComponent<TimeScaleComponent>()?.EnterFrame();
    self.ClientScene().GetComponent<EventDispatcherComponent>()
        ?.FireEvent(new Evt_OnFixedUpdateTimeScaleFresh());
    
    var deltaTime = EngineDefine.fixedDeltaTime;
    
    // 3. 队伍输入处理
    self.Domain.GetComponent<TeamComponent>().UpdateInput(deltaTime);
    
    // 4. 分组更新：普通Unit优先，Puppet(傀儡)后处理
    using var activeUnits1 = ListComponent<Unit>.Create();
    using var activeUnits2 = ListComponent<Unit>.Create();
    self.GetUnits(activeUnits, activeUnits1, activeUnits2);
    
    self.Tick(activeUnits1, deltaTime);
    self.Tick(activeUnits2, deltaTime);
    
    // 5. 战斗逻辑的多阶段更新
    self.Domain.GetComponent<BattleComponent>()?.EnterFrame2(deltaTime, activeUnits);
    self.Domain.GetComponent<BattleComponent>()?.LateEnterFrame(deltaTime, activeUnits);
    
    VProfiler.EndDeepSample();
}
```

### 关键设计决策逐一解读

**① 全局暂停检查放最前面**

```csharp
if (EngineRuntime.Pause) return;
```

这是游戏暂停功能的实现基础。一个布尔值的检查，阻断了整个战斗世界的推进。简洁而有效。

**② 时间缩放更新必须第一步**

注释写得很清楚："时间相关要最先执行，保证后续的逻辑和渲染的时间参数一致"。想象一下，如果时间缩放在中途才生效，那前半帧用的是旧的 deltaTime，后半帧用的是新的，逻辑就会出现分裂。

**③ 帧冻结（FreezeFrames）的实现**

```csharp
foreach (var unit in activeUnits)
{
    if (unit.FreezeFrames > 0) unit.FreezeFrames--;
}
```

帧冻结是动作游戏中实现"顿帧"效果的关键技术（如格斗游戏打中时的一瞬间停顿）。通过计数器减 1，而不是立刻让物理停止，保持了与帧循环的兼容性。

**④ Unit 分组处理顺序**

系统将 Unit 分为两类：
- `activeUnits1`：非 Puppet 类型（真实角色、怪物）
- `activeUnits2`：Puppet 类型（傀儡单位）

傀儡是跟随主角色的附属单位，必须等主角色更新完后再更新自己的位置和状态，否则傀儡永远滞后一帧。

## Tick 内部的更新顺序

```csharp
private static void Tick(this WorldComponent self, List<Unit> activeUnits, FP deltaTime)
{
    // 第一阶段：状态机更新（输入处理）
    foreach (var unit in activeUnits)
    {
        unit.GetComponent<FSMComponent>()?.EnterFrame(deltaTime);
        unit.GetComponent<EventDispatcherComponent>()?.FireEvent(new Evt_OnCheckInput());
        unit.GetComponent<VirtualInputComponent>()?.ClearInputValues();
    }
    
    // 第二阶段：动画和 RootMotion 预处理（帧冻结从这里生效）
    self.Domain.GetComponent<BattleComponent>()?.PreEnterFrame(deltaTime, activeUnits);

    // 第三阶段：物理和移动（冻结单位跳过）
    foreach (var unit in activeUnits)
    {
        if (unit.Freeze) continue;
        unit.GetComponent<PhysicsComponent>()?.EnterFrame(deltaTime);
        unit.GetComponent<SimpleMoveComponent>()?.EnterFrame(deltaTime);
        unit.GetComponent<VirtualInputComponent>()?.MarkTemporaryInput();
    }
    
    // 第四阶段：战斗逻辑（碰撞检测、伤害计算等）
    self.Domain.GetComponent<BattleComponent>()?.EnterFrame1(deltaTime, activeUnits);
}
```

这里有一个极其重要的设计：**状态机（FSM）不受帧冻结影响，但物理移动受影响**。

```csharp
// FSM 更新：即使冻结也执行
unit.GetComponent<FSMComponent>()?.EnterFrame(deltaTime);

// 物理更新：冻结时跳过
if (unit.Freeze) continue;
unit.GetComponent<PhysicsComponent>()?.EnterFrame(deltaTime);
```

为什么这样设计？在动作游戏中，"顿帧"是纯视觉和物理上的停顿，但技能逻辑（如何切换状态）不应该因此停止——否则玩家会感觉操作卡顿，无法响应输入。这种分离保证了体验的流畅性。

## 渲染帧插值：Update 与 PreLateUpdate

战斗逻辑运行在固定帧率（如 30fps），但渲染可以是 60fps 甚至更高。为了消除画面抖动，系统在渲染帧中做插值处理：

```csharp
public static void Update(this WorldComponent self, FP deltaTime)
{
    var activeUnits = unitComponent.GetActiveUnits();
    var units = ListComponent<Unit>.Create();
    foreach (var unit in activeUnits)
    {
        if (unit.Freeze) continue;  // 冻结单位不参与渲染插值
        units.Add(unit);
    }
    self.Domain.GetComponent<BattleComponent>()?.Update(deltaTime, units);
}
```

渲染帧的 Update 会将 Unit 的逻辑位置（定点数）插值到 GameObject 的渲染位置（浮点数），这就是为什么即使逻辑帧率低，画面看起来仍然丝滑。

## 完整的更新时序图

```
FixedUpdate（逻辑帧，固定30fps）
  │
  ├─ FreezeFrames 倒计时
  ├─ TimeScale 更新
  ├─ TeamInput 处理
  │
  ├─ Tick(普通Unit)
  │    ├─ FSM.EnterFrame（状态机，含输入）
  │    ├─ BattleComponent.PreEnterFrame（动画/RootMotion预处理）
  │    └─ Physics/SimpleMove.EnterFrame（物理，冻结跳过）
  │
  ├─ Tick(Puppet)
  │    └─ 同上，但晚于主Unit执行
  │
  ├─ BattleComponent.EnterFrame2（碰撞检测/伤害）
  └─ BattleComponent.LateEnterFrame（后处理）

Update（渲染帧，不固定）
  └─ BattleComponent.Update（位置插值）

PreLateUpdate（渲染后处理）
  └─ BattleComponent.LateUpdate（视图同步）
```

## 新手常见误区

**误区一：直接在 Update 里写战斗逻辑**

初学者往往把逻辑写在 Unity 的 `Update` 里，导致逻辑执行频率随帧率波动，在高帧率设备上角色移动"飞快"，低帧率设备上却"慢如蜗牛"。正确做法是将所有战斗逻辑放在 `FixedUpdate`（固定时间步长）中执行。

**误区二：忽视帧序依赖**

如果 B 的逻辑依赖 A 的结果，必须保证 A 先于 B 更新。`WorldComponent` 通过明确的分组（普通Unit先，Puppet后）和阶段（状态机先，物理后）解决了这个问题。不要随意打乱顺序。

**误区三：浮点数同步**

多人战斗中，千万不要用 `float` 做核心逻辑计算，不同 CPU 的浮点实现会导致结果不一致。必须使用定点数（如 `FP`/`TrueSync`）。

## 总结

`WorldComponent` 是战斗系统的"心脏调度器"，其核心价值在于：

1. **有序性**：严格规定每帧内各组件的更新顺序，消除依赖歧义
2. **分离性**：逻辑帧与渲染帧分离，保证游戏性与画面体验双优
3. **确定性**：定点数 + 固定帧率，实现帧同步多人一致性
4. **灵活性**：帧冻结与时间缩放互不干扰，支持丰富的表现效果

理解了这套驱动机制，你就掌握了整个战斗系统最底层的基础架构。后续的技能、Buff、碰撞检测，都是在这个框架上运转的齿轮。
