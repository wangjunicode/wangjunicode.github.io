# 08_分包与预加载策略

> 面向刚入行的毕业生 · 建议搭配 PatchManager.cs 和项目 Addressables 设置阅读

---

## 1. 系统概述

### 1.1 什么是分包？

分包（Sub-package / Bundle Split）是指将游戏资源按照逻辑功能、加载时机等维度拆分成多个独立的 Bundle 文件，而不是将所有资源打成一个大文件。

**不分包的问题**：
- 一个 Bundle 修改了1KB的内容，玩家需要下载整个 Bundle（可能几百 MB）
- 首次启动时需要下载全部资源，等待时间极长
- 内存中加载了大量不需要的资源

**合理分包的收益**：
- 差量更新精准：只下载真正变化的 Bundle
- 按需加载：玩家进入某个功能才下载对应资源
- 内存可控：不需要的 Bundle 可以卸载释放内存

### 1.2 本项目的分包思路

根据代码中的 Bundle 地址命名规范（如 `"Patch/PatchUI_New.prefab"`, `"Patch/SplashUI.prefab"`, `"Patch/VideoBackground.prefab"`），可以推断项目采用了**功能模块分组**的分包策略：

| Bundle 组 | 加载时机 | 说明 |
|-----------|---------|------|
| Patch 组 | App 启动时立即加载 | 热更 UI、启动视频等 |
| 核心资源组 | 进入游戏大厅前 | 基础 UI、角色模型 |
| 场景组 | 进入具体场景时 | 关卡资源、场景特效 |
| 活动组 | 进入活动时 | 限时活动资源 |

## 2. 架构设计

### 2.1 Addressables Bundle 分组策略

```
Addressables Groups（推荐结构）：
│
├─ Patch（Packed Separately，随包发布）
│   ├─ PatchUI_New.prefab
│   ├─ SplashUI.prefab
│   └─ VideoBackground.prefab
│
├─ Core（Packed Separately，随包发布）
│   ├─ 基础框架资源
│   └─ 常用 Shader
│
├─ UI_Common（Packed Together，热更）
│   └─ 公共 UI 元素
│
├─ UI_Login（Packed Separately，热更）
│   └─ 登录界面资源
│
├─ Scene_001（Packed Separately，热更）
│   └─ 第一个场景资源
│
└─ ...每个功能模块独立分组
```

### 2.2 预加载层次结构

```
启动预加载（必须完成，阻塞进入游戏）
├─ HybridCLR AOT 元数据
├─ 热更 DLL 程序集
├─ Addressables Catalog 初始化
└─ Patch 组资源（PatchUI 等）

进入大厅前预加载（可并行，用进度条展示）
├─ 公共 UI 资源（按钮、字体、图标等）
├─ 核心角色模型（主角）
└─ 常驻音效

场景预加载（懒加载，进入场景触发）
├─ 场景地图数据
├─ 场景内 NPC/怪物模型
└─ 场景音乐

活动资源（后台预加载，不阻塞）
└─ 限时活动专属资源
```

## 3. 核心代码展示

### 3.1 Patch 阶段的资源加载与释放

```csharp
// PatchManager.cs - 热更阶段的资源管理
private List<AsyncOperationHandle> AAhandlerList = new List<AsyncOperationHandle>();

// 封装的加载方法：记录所有 Handle 便于统一释放
private AsyncOperationHandle<T> loadAA<T>(string path)
{
    var aa = Addressables.LoadAssetAsync<T>(path);
    AAhandlerList.Add(aa);
    return aa;
}

// Start() 中加载热更 UI
var handler = loadAA<GameObject>("Patch/PatchUI_New.prefab");
handler.WaitForCompletion();
GameObject ui = GameObject.Instantiate(handler.Result);

// 进入游戏后统一释放热更阶段的资源
private void releaseAllAssets()
{
    for (int i = 0; i < AAhandlerList.Count; i++)
    {
        Addressables.Release(AAhandlerList[i]);
    }
    Resources.UnloadUnusedAssets(); // 触发 GC，释放无引用资源
}
```

### 3.2 延迟释放设计（PatchUI 销毁时机）

```csharp
// ClientBridge.cs
private static PatchManager s_patchManager;

/// <summary>
/// 通知 MapLoadingPanel 已打开，供 Hotfix 层调用，销毁 Patch 更新界面
/// </summary>
public static void NotifyMapLoadingPanelOpened()
{
    if (s_patchManager != null)
    {
        s_patchManager.DestroyPatchUI(); // 此时游戏已有 UI，可以安全销毁 PatchUI
    }
}

// PatchManager.cs
public void DestroyPatchUI()
{
    PatchLog.Info("[PatchManager] DestroyPatchUI called");
    
    if (!isSkipPatch && uiInstance != null)
    {
        GameObject.Destroy(uiInstance.gameObject);
        uiInstance = null;
    }
    
    releaseAllAssets(); // 统一释放所有热更阶段的 Addressables 资源
    GameObject.Destroy(gameObject); // 销毁 PatchManager 自身
}
```

**设计意图**：不在热更完成后立即释放资源，而是等到游戏的第一个 UI（MapLoadingPanel）显示后才释放。这防止了"热更 UI 消失 → 游戏 UI 还没出来 → 黑屏一帧"的视觉闪烁问题。

## 4. 分包与预加载最佳实践

### 4.1 Bundle 粒度建议

```
太粗（不推荐）：
  all_resources.bundle  ← 所有资源一个包，任何改动都要下全量

太细（不推荐）：
  每个 prefab 一个 bundle ← Bundle 数量过多，加载 overhead 大

推荐：
  按"功能模块 + 更新频率"分组
  - 频繁更新的策划资源（配置表、UI 文字）→ 单独小 Bundle
  - 稳定的框架资源（物理材质、通用着色器）→ 打入安装包，不热更
  - 场景资源按地图分组 → 进入场景时按需加载
```

### 4.2 热更 Bundle 的命名规范

```
命名格式：<模块>/<子模块>_<版本>.bundle

示例：
  patch/patch_ui.bundle       ← 热更 UI（高频更新）
  scene/map_001.bundle        ← 第一章地图（低频更新）
  char/hero_001.bundle        ← 英雄甲（角色更新时才变）
  activity/halloween_2024.bundle ← 万圣节活动（活动结束下架）
```

### 4.3 依赖 Bundle 的处理

Addressables 会自动处理 Bundle 依赖（共享资源会被打入 Common Bundle）：

```
hero_001.bundle 依赖 common_materials.bundle
↓
加载 hero_001 时，Addressables 自动先加载 common_materials
```

注意事项：不要让高频热更的 Bundle 依赖低频更新的 Bundle，否则修改 common_materials 时需要同时更新所有依赖它的 Bundle。

## 5. 预加载优先级策略

### 5.1 关键路径优化

```
优先级 1（阻塞）：游戏必须有这些资源才能启动
  - PatchUI（热更界面）
  - 登录界面资源
  - 主角基础模型

优先级 2（低优先级后台加载）：加载中不影响功能，但影响视觉品质
  - 高清贴图
  - 粒子特效

优先级 3（懒加载）：玩家进入对应功能时才加载
  - 副本/战斗资源
  - 活动专属资源
```

### 5.2 Shader 预热与分包配合

```csharp
// 利用热更下载等待时间预热 Shader
// 这是一个时间复用的好例子：
// 玩家在等待下载时，后台已经在预热 Shader
// 进入游戏后第一帧的卡顿大幅减少
ShaderPreloader_OnStart.Instance.PreloadCommonShader();
```

**推荐做法**：将常见的 Shader Variant 列表预先收集（通过 Unity 的 ShaderVariantCollection），在热更下载期间触发预热，避免玩家进入游戏后因 Shader 编译导致的卡顿。

## 6. 内存管理策略

### 6.1 Addressables 引用计数

```csharp
// 正确的加载/释放方式
var handle = Addressables.LoadAssetAsync<Sprite>("ui/icon_gold.png");
await handle.Task;
Sprite icon = handle.Result;
// ... 使用 icon ...
Addressables.Release(handle); // 用完必须释放！
```

**Addressables 内部使用引用计数**：同一个 Bundle 被加载多次，只存在一份内存，引用计数为 N。只有引用计数归零时，Bundle 才会被真正卸载。

### 6.2 热更阶段的内存占用

```
热更阶段存活的资源：
- PatchUI.prefab（热更进度条 UI）约 1-2MB
- SplashUI.prefab（闪屏 UI）约 5-20MB（视视频大小）
- VideoBackground.prefab（背景视频）约 1MB + 视频流

进入游戏后这些资源必须释放，否则白白占用内存。
PatchManager.DestroyPatchUI() 确保了这一点。
```

## 7. 常见问题与最佳实践

### Q1：如何决定一个资源该不该热更？
**A**：
- **必须热更**：游戏逻辑代码、策划配置、频繁更新的资源
- **建议热更**：UI 预制体、角色模型（需要随活动更换）
- **不建议热更**：底层框架代码（已在底包）、Unity 引擎 DLL、物理碰撞网格

### Q2：Bundle 越小越好吗？
**A**：不是。Bundle 文件头本身有开销，Bundle 数量过多时：
- 文件 IO 次数增加（每个 Bundle 都要打开/关闭文件）
- Catalog 文件变大（映射关系增多）
- HTTP 请求次数增加（每个 Bundle 一次网络请求）

经验值：单个 Bundle 建议在 1-20MB 之间，总 Bundle 数量控制在几百个以内。

### Q3：热更后旧的 Bundle 文件如何清理？
**A**：旧 Bundle 会一直留在 `ResUpdate/` 目录直到手动清理。建议：
1. 在大版本更新时删除整个 `ResUpdate` 目录（`ClearOldPatch()` 已实现主版本不一致时的清理）
2. 维护一个"活跃 Bundle 列表"，定期扫描并删除不在列表中的孤立文件

### Q4：如何监控每个 Bundle 的下载大小和次数？
**A**：通过 Dolphin 的下载统计接口，以及客户端本地记录 `ResUpdate/` 目录的文件增量可以统计。建议接入数据上报系统，监控：
- 每个版本的平均更新包大小
- 更新成功率
- 分包命中率（哪些 Bundle 被下载最多）

### Q5：预加载失败怎么处理？
**A**：关键资源（如 PatchUI）加载失败时需要有 Fallback 处理：
```csharp
if (prefab == null)
{
    // PatchUI 加载失败，使用 fallback UI 或直接跳过热更
    Log.LogWarning("[Index] SplashUI.prefab load failed, skip splash");
    OnSplashComplete();
    return;
}
```
非关键资源（如视频背景）加载失败可以静默忽略，不影响游戏正常运行。
