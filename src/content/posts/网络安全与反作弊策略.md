# 10_网络安全与反作弊策略

> 网络安全和反作弊是保护公平游戏环境的最后一道防线。本文讲解本项目在网络安全方面的设计思路和工程实践。

---

## 1. 系统概述

手游面临的主要安全威胁：

| 威胁类型 | 说明 | 本项目应对措施 |
|---------|------|-------------|
| 数据篡改 | 修改内存中的战斗数据（无限血量等） | 服务端权威帧同步 |
| 消息伪造 | 发送伪造的网络包（外挂发假操作） | UDP身份验证（PVP_AUTH） |
| 不同步作弊 | 故意让自己游戏状态与其他人不同 | 哈希校验（PVP_SYNC_HASH） |
| 中间人攻击 | 拦截并修改网络数据 | HTTPS/TLS加密传输 |
| DNS劫持 | 把服务器域名解析到恶意IP | HttpDNS直接返回IP |
| Token泄露 | 伪造玩家身份 | 临时Token机制 |
| 逆向破解 | 反编译代码分析协议 | Native层加密、IL2CPP |
| 撞库攻击 | 用泄露的账号密码登录 | MSDK多因子验证 |

## 2. 架构设计

### 2.1 安全分层模型

```
┌─────────────────────────────────────────────────────┐
│              应用层安全（反作弊逻辑）                   │
│  哈希校验 / 不同步检测 / 行为异常检测                   │
├─────────────────────────────────────────────────────┤
│              传输层安全                               │
│  UDP身份验证(PVP_AUTH) / TLS加密 / HTTPS              │
├─────────────────────────────────────────────────────┤
│              协议层安全                               │
│  GCloud LockStep令牌 / MSDK Token验证                 │
├─────────────────────────────────────────────────────┤
│              基础设施安全                             │
│  HttpDNS / COS临时密钥 / IP白名单                    │
└─────────────────────────────────────────────────────┘
```

### 2.2 帧同步安全机制

帧同步场景的安全核心是：**服务器不信任任何单一客户端的结果，只信任所有客户端达成一致的结果**。

```
所有玩家的战斗操作
    │ 通过GCloud LockStep转发（服务器只中转，不计算逻辑）
    ▼
每个客户端独立计算战斗结果
    │
    ▼
每个关键帧，计算当前状态的哈希值
    │ VKeyDef.PVP_SYNC_HASH (502)
    ▼
GCloud服务器汇总所有客户端的哈希
    │
    ├─ 所有哈希一致 → 继续
    └─ 哈希不一致  → 广播 PVP_SYNC_ERROR (507)
                      携带：时间戳、BusID、GameID、不同步帧号
```

## 3. 核心安全机制详解

### 3.1 UDP身份验证（PVP_AUTH）

战斗使用UDP传输，UDP无连接状态，任何设备都可以向服务器发送数据包。为防止攻击者伪造战斗操作，系统引入了`PVP_AUTH`（405）验证机制：

```csharp
// VKeyDef.cs 中定义
/// <summary>
/// UDP身份字段（用于验证UDP包来源合法性）
/// </summary>
public const int PVP_AUTH = 405;
```

**工作原理**：
1. 玩家通过MSDK登录后，游戏服务器下发一个`accessInfo`（包含加密的身份令牌）
2. 客户端在加入LockStep时携带该`accessInfo`
3. 每个UDP帧数据中包含`PVP_AUTH`字段（GCloud SDK自动附加）
4. 服务器验证`PVP_AUTH`，非法来源的包直接丢弃

```csharp
// 登录时携带accessInfo（从游戏服务器获取的入场凭证）
LockSteper.Instance.Login(accessInfo, (result) =>
{
    if (result.IsSuccess())
    {
        // accessInfo已被GCloud验证，后续UDP包自动携带身份信息
        LockSteper.Instance.Ready(callback);
    }
});
```

### 3.2 哈希校验与不同步检测

```csharp
// VKeyDef.cs
public const int PVP_SYNC_HASH = 502;   // 上报当前帧哈希
public const int PVP_SYNC_ERROR = 507;  // 不同步通知

// PVP_SYNC_ERROR的4个字段：
// [0] 时间戳
// [1] BusID（转16进制，用于定位业务）
// [2] GameID
// [3] 不同步帧号
```

**业务层哈希校验实现**：

```csharp
// 战斗中每隔N帧进行一次哈希校验（不是每帧都做，避免性能开销）
public class SyncHashChecker : MonoBehaviour
{
    private const int HASH_CHECK_INTERVAL = 30;  // 每30帧校验一次
    private int currentFrame = 0;

    void OnFrameProcessed(FrameInfo frame)
    {
        currentFrame = frame.FrameId;
        
        if (currentFrame % HASH_CHECK_INTERVAL == 0)
        {
            int hash = ComputeBattleStateHash();
            SendHashReport(currentFrame, hash);
        }
    }

    // 计算战场状态哈希（必须是确定性的，不能包含浮点数！）
    private int ComputeBattleStateHash()
    {
        int hash = 17;
        
        // 遍历所有棋子（按固定顺序，不能用Dictionary）
        foreach (var chess in GetAllChessInOrder())
        {
            hash = hash * 31 + chess.UnitId;
            hash = hash * 31 + chess.Hp;      // 必须是整数HP
            hash = hash * 31 + chess.Position.x;  // 整数坐标
            hash = hash * 31 + chess.Position.y;
        }
        
        return hash;
    }

    private void SendHashReport(int frameId, int hash)
    {
        int[] args = VKeyArgPool.Instance.GetArgs(2);
        args[0] = frameId;
        args[1] = hash;
        
        byte[] data = BuildVKeyMessage(VKeyDef.PVP_SYNC_HASH, args);
        LockSteper.Instance.Input(data, rawUdp: false);  // 哈希用可靠UDP
        
        VKeyArgPool.Instance.ReturnArgs(args);
    }
}
```

### 3.5 MSDK Token安全机制

```csharp
// MSDKLogin.cs 提供的Token安全体系
public class TokenSecurityManager
{
    private const int TOKEN_REFRESH_THRESHOLD = 1800; // 30分钟内过期则刷新

    // 检查Token是否即将过期，提前刷新
    public void CheckAndRefreshToken()
    {
        MSDKLoginRet loginRet = MSDKLogin.GetLoginRet();
        if (loginRet == null) return;
        
        long now = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
        long timeToExpire = loginRet.TokenExpire - now;
        
        if (timeToExpire < TOKEN_REFRESH_THRESHOLD)
        {
            Debug.Log($"[Security] Token即将在{timeToExpire}秒后过期，尝试刷新");
            MSDKLogin.AutoLogin("", "", "");  // 触发Token刷新
        }
    }
    
    // 向游戏服务器验证Token（不在客户端自验）
    public async Task<bool> VerifyTokenWithServer(string openId, string token, string channel)
    {
        // 发到游戏服务器，由服务器调用MSDK验证接口
        // 客户端永远不知道验证逻辑，防止绕过
        var response = await HttpHelper.Post("/api/login/verify", new
        {
            open_id = openId,
            token   = token,
            channel = channel
        });
        
        return response.success;
    }
}
```

### 3.6 COS临时密钥（STS机制）

```csharp
// 不在客户端存储固定的SecretId/SecretKey
// 而是向游戏服务器请求临时密钥
public class COSSecureUploader
{
    // 临时密钥缓存
    private static StsCredential cachedCredential;
    private static DateTime credentialExpireTime;
    
    // 上传前获取临时密钥
    private async Task<CosXml> GetCosClientAsync()
    {
        // 检查缓存是否有效（留60秒余量）
        if (cachedCredential != null && 
            DateTime.UtcNow < credentialExpireTime - TimeSpan.FromSeconds(60))
        {
            return BuildCosClient(cachedCredential);
        }
        
        // 向游戏服务器请求新的临时密钥
        var response = await HttpHelper.Get("/api/sts/cosCredential");
        cachedCredential = response.credential;
        credentialExpireTime = DateTime.UtcNow.AddSeconds(response.expiredIn);
        
        return BuildCosClient(cachedCredential);
    }
    
    private CosXml BuildCosClient(StsCredential credential)
    {
        CosXmlConfig config = new CosXmlConfig.Builder()
            .SetRegion("ap-guangzhou")
            .IsHttps(true)
            .Build();
        
        // 使用临时密钥（有效期约10分钟，过期自动获取新密钥）
        var credProvider = new DefaultSessionQCloudCredentialProvider(
            credential.tmpSecretId,
            credential.tmpSecretKey,
            credential.expiredTime,
            credential.sessionToken
        );
        
        return new CosXmlServer(config, credProvider);
    }
}
```

## 4. 反作弊深度设计

### 4.1 服务器权威性原则

帧同步本质上是"客户端执行权威"——每个客户端自己计算游戏逻辑。但通过以下机制实现有效的反作弊：

```
真实的反作弊逻辑 = 哈希校验 + 行为分析 + 事后审计
```

1. **哈希校验（实时）**：快速发现不同步，记录异常帧号
2. **行为分析（实时）**：检测异常操作频率、不可能的状态转换
3. **录像审计（事后）**：保存完整的帧数据录像，对可疑对局进行回放审查

### 4.2 VKey协议的安全设计

VKey本质上是游戏的"操作指令集"。安全设计要点：

```csharp
// 1. 只有合法的VKey才应该被处理
static bool IsValidVKey(int vkey)
{
    // 在合法范围内的VKey才处理
    return VKeyDef.IS_PVP_LOGIC_VKEY(vkey) ||
           (vkey >= 100 && vkey <= 106);  // 策略卡操作
}

// 2. 频率限制（防止刷包）
public class VKeyRateLimiter
{
    private Dictionary<int, int> vkeyCountInFrame = new();
    
    public bool CanProcess(int vkey, int frameId)
    {
        // 每帧每种VKey最多处理N次
        int count = vkeyCountInFrame.GetValueOrDefault(vkey, 0);
        if (count > GetMaxCountPerFrame(vkey))
        {
            Debug.LogWarning($"[Security] VKey {vkey} 在帧 {frameId} 超频，丢弃");
            return false;
        }
        vkeyCountInFrame[vkey] = count + 1;
        return true;
    }
    
    private int GetMaxCountPerFrame(int vkey)
    {
        return vkey switch
        {
            VKeyDef.PVP_SYNC_HASH => 1,   // 每帧只上报一次哈希
            VKeyDef.PVP_PING => 1,         // 每帧只发一次PING
            _ => 10                         // 其他VKey默认每帧最多10次
        };
    }
}
```

## 5. 设计亮点

### 5.1 双层防护：传输层+应用层

- **传输层**：HTTPS/TLS加密所有HTTP请求，GCloud SDK的UDP包携带`PVP_AUTH`身份验证
- **应用层**：哈希校验确保所有客户端状态一致

### 5.2 零信任架构

- 游戏服务器不信任客户端提供的Token，必须向MSDK验证
- COS操作使用临时密钥，不在客户端存放固定凭证
- UDP包通过`PVP_AUTH`验证，服务器不接受未认证的UDP包

### 5.3 事后审计能力

通过保存帧数据（每帧的`FrameInfo`完整序列），可以对可疑对局进行事后回放，复现不同步现象，定位作弊帧。

```csharp
// 录像存储（对局中开启）
public class BattleRecorder

    }
```

## 6. 常见问题与最佳实践

### Q1：哈希校验不一致，是否一定是有人作弊？

**不一定！** 更多情况是代码bug：
1. **浮点数计算**：不同设备的浮点精度差异，必须全用整数或定点数
2. **Dictionary遍历**：.NET的`Dictionary`遍历顺序不保证，改用排序列表
3. **随机数种子**：必须使用确定性伪随机数（固定种子），不能用`System.Random()`
4. **时间依赖**：不能用`Time.deltaTime`或系统时间做游戏逻辑

**排查步骤**：
1. 查看`PVP_SYNC_ERROR`中的不同步帧号
2. 找到该帧的录像数据
3. 对比两个客户端的帧数据差异，找出分叉点

### Q2：Token被中间人截获怎么办？

本项目Token通过HTTPS传输，已经被TLS加密，中间人看到的是密文。即使Token泄漏：
1. Token有过期时间（通常2小时），过期后无效
2. 游戏服务器对同一Token的并发登录有检测（异地登录踢人）
3. 支付等高风险操作有额外的二次验证

### Q3：UDP包可以被重放攻击吗？

GCloud LockStep SDK在内部处理了重放防护：
- 每个UDP包有`SequenceId`，服务器拒绝过期或重复的SequenceId
- `PVP_AUTH`令牌有时效性，过期的令牌发出的包会被服务器拒绝

### Q4：如何处理外挂注入（IL Hooking）？

应对IL级别的Hook攻击：
1. **IL2CPP**：发布时启用IL2CPP，将C#代码编译为本机C++，大幅提高逆向难度
2. **代码混淆**：关键逻辑混淆变量名和函数名
3. **完整性校验**：启动时校验关键Assembly的Hash（`MSDKLogin.CheckAppIntegrity()`）
4. **关键逻辑在Native**：核心战斗计算在GCloud C++层执行，不暴露在C#层

### Q5：弱网情况下频繁重连是否会被反作弊系统误判？

不会。GCloud LockStep SDK的重连流程对服务端是透明的，重连成功后服务器会补发缺失帧数据，不会因为临时断线而认为玩家在作弊。但如果同一玩家多次在关键帧（如濒临失败时）断线，可以在服务端设置策略触发人工审核。

## 8. 总结

本项目的网络安全体系以**纵深防御**为原则，从基础设施层（HttpDNS、HTTPS）到传输层（UDP身份验证）再到应用层（哈希校验、行为分析），形成多层次的安全防护网络。

最关键的设计理念是：**不信任客户端，但也不对所有异常用户进行惩罚**——先记录、后分析、再处置，避免误判影响正常玩家体验。对于新同学，记住一条原则：**所有安全关键验证在服务端执行，客户端只是数据的展示层**。
