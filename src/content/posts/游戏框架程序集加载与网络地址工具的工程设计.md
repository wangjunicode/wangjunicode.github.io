# 游戏框架程序集加载与网络地址工具的工程设计

在大型 Unity 游戏框架（尤其是 ET 框架 + HybridCLR 热更新方案）的日常开发中，**程序集（Assembly）管理**和**网络地址处理**是两个看似基础却频繁踩坑的领域。本文以真实项目中的 `AssemblyHelper.cs` 和 `NetworkHelper.cs` 源码为蓝本，深度剖析其设计思路、工程价值与背后的 .NET 底层机制。

---

## 一、为什么游戏框架需要专门管理程序集？

### 1.1 ET + HybridCLR 的双程序集结构

传统 Unity 项目只有一个程序集 `Assembly-CSharp.dll`，所有代码编译进同一个包。而 ET 框架配合 HybridCLR 热更新后，形成了**双程序集**乃至**多程序集**结构：

| 程序集 | 内容 | 是否可热更 |
|---|---|---|
| `Assembly-CSharp.dll`（Model层） | ECS 框架骨架、底层 Core | ❌ AOT 编译 |
| `Assembly-CSharp-Hotfix.dll`（Hotfix层） | 游戏逻辑、系统、Handler | ✅ HybridCLR 解释执行 |
| `ThirdParty.dll` 等 | 第三方插件 | 取决于配置 |

这种结构下，**同一个系统（如 EventSystem）需要在多个程序集里扫描类型**，单靠 `Assembly.GetCallingAssembly()` 或 `typeof(X).Assembly` 无法满足需求。`AssemblyHelper` 正是为此而生。

### 1.2 反射与类型字典的性能权衡

.NET 的 `Type.GetType(string)` 在跨程序集场景下**默认只搜索当前程序集和 mscorlib**，无法感知热更程序集。游戏框架必须自己维护一个"全局类型注册表"，用空间换时间：

```
启动时：扫描所有已加载程序集 → 构建 Dictionary<string, Type>
运行时：O(1) 按 FullName 查类型，无需再次反射扫描
```

## 二、AssemblyHelper 源码精读

```csharp
public static class AssemblyHelper
{
    public static Dictionary<string, Type> GetAssemblyTypes(params Assembly[] args)
    {
        Dictionary<string, Type> types = new Dictionary<string, Type>();
        foreach (Assembly ass in args)
        {
            foreach (Type type in ass.GetTypes())
            {
                types[type.FullName] = type;
            }
        }
        return types;
    }
}
```

代码极简，但每一个设计决策都值得深究。

### 2.1 `params Assembly[]` ——可变参数的工程意义

方法签名使用 `params` 关键字，调用方既可以传一个程序集，也可以一次性传入多个：

```csharp
// 单程序集模式（开发阶段，无热更）
var types = AssemblyHelper.GetAssemblyTypes(typeof(Game).Assembly);

// 双程序集模式（HybridCLR 热更载入后）
var hotfixAssembly = LoadHotfixAssembly(); // 从 AB 包加载
var types = AssemblyHelper.GetAssemblyTypes(
    typeof(Game).Assembly,    // Model 层
    hotfixAssembly            // Hotfix 层
);
```

这种设计让框架在开发期和生产期使用同一个方法签名，不需要重载。

### 2.2 `type.FullName` 作为键的陷阱

`FullName` 是 `命名空间.类名` 的完整形式，例如 `ET.EventSystem`。用它做键有以下考量：

**优势：**
- 全局唯一性高（相同 FullName 在同一进程中通常只存在一个，除非程序集版本冲突）
- 与 `Type.GetType(string)` 的参数格式一致，便于序列化/反序列化还原类型

**潜在问题：**

1. **泛型类型**：泛型类的 `FullName` 是 `ET.SomeClass\`1`，含反引号，需要注意字符串匹配
2. **嵌套类**：嵌套类型的 FullName 使用 `+` 分隔符（`OuterClass+InnerClass`），与命名空间 `.` 不同
3. **热更新覆盖**：若 Hotfix 程序集中有与 Model 层同名类型（正常不应该出现），后者会覆盖前者。当前代码用 `=` 赋值而非 `TryAdd`，即热更层覆盖基础层——这是一种有意为之的"热更覆盖"策略

### 2.3 在 ET 框架中的调用链

`AssemblyHelper.GetAssemblyTypes` 通常在框架启动时由 `EventSystem` 调用：

```csharp
// EventSystem.cs（伪代码示意）
public void Add(Dictionary<string, Type> addTypes)
{
    foreach (var (fullName, type) in addTypes)
    {
        // 扫描所有继承了 ISystem 的类，注册到队列
        if (typeof(ISystem).IsAssignableFrom(type))
        {
            RegisterSystem(type);
        }
        // 扫描所有带 [Event] 特性的类，注册事件监听
        var eventAttr = type.GetCustomAttribute<EventAttribute>();
        if (eventAttr != null)
        {
            RegisterEvent(type, eventAttr);
        }
    }
}
```

整个流程：
1. `Game.Start()` → 调用 `AssemblyHelper.GetAssemblyTypes`，传入所有程序集
2. 返回 `Dictionary<string, Type>`，交给 `EventSystem.Add()`
3. `EventSystem` 遍历字典，按特性/接口自动注册所有系统和事件处理器
4. HybridCLR 热更新完成后，**重新执行步骤 1-3**，新逻辑立即生效

## 三、NetworkHelper 源码精读

```csharp
public static class NetworkHelper
{
    public static string[] GetAddressIPs();
    public static IPAddress GetHostAddress(string hostName);
    public static IPEndPoint ToIPEndPoint(string host, int port);
    public static IPEndPoint ToIPEndPoint(string address);
    public static void SetSioUdpConnReset(Socket socket);
}
```

五个方法覆盖了游戏网络开发中最常用的地址处理场景。

### 3.1 GetAddressIPs —— 获取本机以太网 IP

```csharp
public static string[] GetAddressIPs()
{
    List<string> list = new List<string>();
    foreach (NetworkInterface networkInterface in NetworkInterface.GetAllNetworkInterfaces())
    {
        if (networkInterface.NetworkInterfaceType != NetworkInterfaceType.Ethernet)
        {
            continue;
        }
        foreach (UnicastIPAddressInformation add in 
            networkInterface.GetIPProperties().UnicastAddresses)
        {
            list.Add(add.Address.ToString());
        }
    }
    return list.ToArray();
}
```

**关键设计点：只保留 Ethernet 类型的网卡**

`NetworkInterfaceType` 枚举包含 30+ 种类型：`Loopback`（127.0.0.1）、`Wireless80211`（WiFi）、`Tunnel`（VPN/虚拟网卡）、`Ethernet` 等。游戏服务器配置中通常只关心物理以太网口，过滤掉回环地址、VPN 虚拟网卡、蓝牙 PAN 等干扰项。

**潜在局限性：**
- 现代游戏开发环境中开发机可能用 WiFi，此时此方法返回空数组，需按实际情况扩展过滤条件
- 在 Docker/K8s 容器中，网卡类型可能被标记为 `Unknown`，需增加兼容处理

### 3.2 GetHostAddress —— IPv4 优先的 DNS 解析

```csharp
public static IPAddress GetHostAddress(string hostName)
{
    IPAddress[] ipAddresses = Dns.GetHostAddresses(hostName);
    IPAddress returnIpAddress = null;
    foreach (IPAddress ipAddress in ipAddresses)
    {
        returnIpAddress = ipAddress;
        if (ipAddress.AddressFamily == AddressFamily.InterNetwork)
        {
            return ipAddress;  // 找到 IPv4 立即返回
        }
    }
    return returnIpAddress;  // 兜底返回最后一个地址（可能是 IPv6）
}
```

**为什么要 IPv4 优先？**

`Dns.GetHostAddresses` 在支持 IPv6 的系统上会同时返回 `::1`（IPv6 回环）和 `127.0.0.1`（IPv4 回环）。游戏服务器的 Socket 绑定逻辑通常不同时处理 IPv4/IPv6 双栈，明确指定 `AddressFamily.InterNetwork`（即 IPv4）可避免"本地测试 OK，服务器 IPv6 地址导致连接失败"的奇怪 Bug。

**AddressFamily 枚举对照：**

| 枚举值 | 含义 |
|---|---|
| `InterNetwork` | IPv4 |
| `InterNetworkV6` | IPv6 |
| `Unix` | Unix Domain Socket |

### 3.3 ToIPEndPoint —— 字符串到终端点的两种重载

```csharp
// 重载一：host + port 分开传
public static IPEndPoint ToIPEndPoint(string host, int port)
{
    return new IPEndPoint(IPAddress.Parse(host), port);
}

// 重载二：解析 "127.0.0.1:7777" 格式字符串
public static IPEndPoint ToIPEndPoint(string address)
{
    int index = address.LastIndexOf(':');
    string host = address.Substring(0, index);
    string p = address.Substring(index + 1);
    int port = int.Parse(p);
    return ToIPEndPoint(host, port);
}
```

重载二使用 `LastIndexOf(':')` 而非 `IndexOf(':')`，这是处理 **IPv6 地址**的正确姿势。IPv6 地址中含有多个冒号（如 `::1`），只有最后一个冒号才是端口分隔符，例如：

```
[::1]:7777     → host=[::1], port=7777
192.168.1.1:7777 → host=192.168.1.1, port=7777
```

不过此实现对带中括号的 IPv6+端口格式（`[::1]:7777`）未做特殊处理，实际项目中若有 IPv6 需求需额外兼容。

### 3.4 SetSioUdpConnReset —— 屏蔽 Windows UDP CONNRESET

```csharp
public static void SetSioUdpConnReset(Socket socket)
{
    if (!RuntimeInformation.IsOSPlatform(OSPlatform.Windows))
    {
        return;
    }

    const uint IOC_IN  = 0x80000000;
    const uint IOC_VENDOR = 0x18000000;
    const int SIO_UDP_CONNRESET = unchecked((int)(IOC_IN | IOC_VENDOR | 12));

    socket.IOControl(SIO_UDP_CONNRESET, new[] { Convert.ToByte(false) }, null);
}
```

这是一个只在 Windows 平台生效的 WinSock 特性修复，解决了一个经典的 UDP 开发 Bug。

**问题根源：**

Windows 下，如果 UDP 发包目标端口不存在或对端已关闭，操作系统会向发送方回复一个 ICMP Port Unreachable 报文。WinSock 收到后会将 UDP Socket 标记为"连接重置"状态，下一次 `ReceiveFrom` 调用会抛出 `SocketException: Connection reset by peer`（错误码 10054）。

这个行为**不符合 UDP 协议语义**——UDP 本就是无连接协议，发包到不存在的端口是完全合法的，不应导致 Socket 报错。Linux/macOS 正确地忽略了这个 ICMP 报文，但 Windows 没有。

**解决方案：**

通过 `IOControl` 设置 `SIO_UDP_CONNRESET` 控制码，禁用这个行为：

```csharp
socket.IOControl(SIO_UDP_CONNRESET, new[] { (byte)0 }, null);
// (byte)0 = false，即禁用 CONNRESET 报告
```

**工程影响：**

游戏的 UDP 服务端（无论是帧同步服务器还是 Relay 中继服）启动时必须调用此方法，否则在 Windows 开发机上调试时，任意一个客户端断开连接，服务端的 `ReceiveFrom` 就会抛异常，导致整个接收循环崩溃。

```csharp
// 标准的 UDP Server 初始化流程
Socket socket = new Socket(AddressFamily.InterNetwork, SocketType.Dgram, ProtocolType.Udp);
NetworkHelper.SetSioUdpConnReset(socket); // ← 关键！
socket.Bind(new IPEndPoint(IPAddress.Any, 7777));
```

## 四、两个工具类的设计哲学对比

| 维度 | AssemblyHelper | NetworkHelper |
|---|---|---|
| **职责** | 纯粹数据聚合：将多程序集类型合并为字典 | 跨平台差异抹平：统一处理 IP 解析与 Socket 配置 |
| **状态** | 无状态（纯静态方法，返回新对象） | 无状态（操作外部传入的 Socket） |
| **扩展点** | 方法本身即扩展点（传入不同 Assembly 组合） | 平台检测（`RuntimeInformation.IsOSPlatform`）|
| **错误处理** | 无（`GetTypes()` 可能抛 `ReflectionTypeLoadException`，未处理） | 无（`IPAddress.Parse` 若格式错误会抛异常，未处理） |
| **性能关键路径** | 启动时一次调用，可接受较高耗时 | 连接时调用，需尽量快速 |

### 4.1 AssemblyHelper 的防御性增强建议

生产项目中 `GetTypes()` 有时会因程序集中的某个类型依赖缺失而抛出 `ReflectionTypeLoadException`。更健壮的实现应该：

```csharp
public static Dictionary<string, Type> GetAssemblyTypes(params Assembly[] args)
{
    var types = new Dictionary<string, Type>();
    foreach (var ass in args)
    {
        Type[] assTypes;
        try
        {
            assTypes = ass.GetTypes();
        }
        catch (ReflectionTypeLoadException ex)
        {
            // 部分类型加载失败时，取已成功加载的类型
            assTypes = ex.Types.Where(t => t != null).ToArray();
            Log.Warning($"Assembly {ass.FullName} partial load: {ex.LoaderExceptions[0].Message}");
        }
        foreach (var type in assTypes)
        {
            types[type.FullName] = type;
        }
    }
    return types;
}
```

### 4.2 NetworkHelper 在帧同步框架中的位置

本项目的帧同步服务器（`KH.Network.FSP`）在初始化流程中会集中调用 `NetworkHelper`：

```
FSPServer.Start()
    ├─ NetworkHelper.GetAddressIPs()         → 获取绑定地址
    ├─ NetworkHelper.ToIPEndPoint(config)    → 解析配置文件中的地址
    └─ NetworkHelper.SetSioUdpConnReset(s)   → 屏蔽 Windows CONNRESET
```

这三步缺一不可，任何遗漏都可能导致 Windows 开发机与 Linux 生产服务器行为不一致的"环境 Bug"。

## 五、HybridCLR 热更新下的程序集生命周期

理解 `AssemblyHelper` 的真正价值，需要了解 HybridCLR 热更新的程序集生命周期：

```
游戏启动
    │
    ├─ [AOT 编译期] 加载 Assembly-CSharp.dll（Model 层）
    │     └─ AssemblyHelper.GetAssemblyTypes(modelAssembly) → EventSystem.Add()
    │
    ├─ 进入登录界面，触发热更新检测
    │     └─ 下载最新 Hotfix.dll
    │
    ├─ HybridCLR 解释执行 Hotfix.dll
    │     └─ AssemblyHelper.GetAssemblyTypes(modelAssembly, hotfixAssembly) → EventSystem.Add()
    │     └─ ← 关键：此处会重新注册所有系统，包括热更新后的新逻辑
    │
    └─ 游戏正式运行，所有系统均来自热更新程序集
```

这个机制允许游戏在不重启 App 的情况下完成逻辑替换。`AssemblyHelper` 作为程序集扫描的入口，在整个热更新流程中被调用两次，保证新旧程序集的过渡是无缝的。

## 六、实战问题集锦

### Q1：热更新后新增了一个 IAwakeSystem，为什么没有生效？

**原因**：热更新完成后，没有重新调用 `AssemblyHelper.GetAssemblyTypes` + `EventSystem.Add()`，导致新系统未被注册。

**解决**：在热更新完成回调中，显式重新初始化 EventSystem：
```csharp
// HotfixLoadComplete 回调
EventSystem.Instance.Add(AssemblyHelper.GetAssemblyTypes(
    typeof(Game).Assembly,
    hotfixAssembly
));
```

### Q2：Windows 本地调试 UDP 服务，一个客户端断开就崩溃

**原因**：未调用 `SetSioUdpConnReset`，Windows CONNRESET 问题。

**解决**：在 Socket 绑定前调用 `NetworkHelper.SetSioUdpConnReset(socket)`。

### Q3：`GetHostAddress` 在 DNS 返回 IPv6 地址时建立连接失败

**原因**：服务端绑定的是 IPv4 地址，但客户端 `GetHostAddress` 返回了 IPv6 地址。

**解决**：当前实现已有 IPv4 优先逻辑，检查 DNS 配置是否返回了纯 IPv6 记录，或考虑强制传入 IP 而非域名。

## 七、总结

`AssemblyHelper` 和 `NetworkHelper` 是 ET 框架 Core 层中最"小"但最具工程价值的两个工具类：

- **AssemblyHelper** 解决了多程序集 + 热更新场景下的类型发现问题，是 EventSystem 自动化注册的基础设施，也是 HybridCLR 热更新无缝过渡的关键环节
- **NetworkHelper** 封装了游戏网络开发中高频出现的 IP 处理逻辑，并通过 `SetSioUdpConnReset` 修复了 Windows 平台的 UDP 行为差异，避免了开发期的诡异 Bug

两个类都遵循"**单一职责、无状态、纯静态**"的设计原则，测试友好，易于理解。对于刚入行的游戏客户端开发者来说，读懂这两个类不仅仅是理解几十行代码，更是理解 ET 框架如何在工程层面处理**热更新程序集管理**和**跨平台网络兼容性**这两大核心挑战。
