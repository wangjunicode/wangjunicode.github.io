## 前言：为什么要自己实现异步框架？

C# 自带 `Task` / `async-await`，为什么还要造轮子？

第一性原理的追问：**`Task` 解决了什么问题，带来了什么代价？**

| 维度 | 原生 Task | 自研 ETTask |
|------|----------|------------|
| GC 压力 | 每次 await 都可能分配堆内存 | 对象池复用，零 GC |
| 上下文传递 | SynchronizationContext（Unity 有坑） | 自定义 Context 链 |
| 热更兼容 | 反射支持差 | 完全受控，支持热更 |
| 异常处理 | 未观察的异常静默丢失 | 全局 ExceptionHandler |

结论：**游戏帧循环 + 热更 + 极低 GC** 这三个约束叠加，原生 Task 无法满足。

---

## 一、`async/await` 的本质：编译器状态机

理解自研框架之前，必须先搞清楚编译器对 `async` 方法做了什么。

写这段代码：

```csharp
public async ETTask DoSomething()
{
    await SomeAsyncOp();
    Debug.Log("done");
}
```

编译器会把它展开成一个**状态机类**：

```csharp
// 伪代码，编译器实际生成
private struct DoSomethingStateMachine : IAsyncStateMachine
{
    public ETAsyncTaskMethodBuilder builder;
    public int state = -1;

    public void MoveNext()
    {
        switch (state)
        {
            case -1:
                var awaiter = SomeAsyncOp().GetAwaiter();
                if (!awaiter.IsCompleted)
                {
                    state = 0;
                    builder.AwaitOnCompleted(ref awaiter, ref this);
                    return;
                }
                goto case 0;
            case 0:
                Debug.Log("done");
                builder.SetResult();
                break;
        }
    }
}
```

**关键理解**：`async` 方法不是"暂停线程"，而是"保存状态 + 注册回调"。每次 `await` 就是一次 `MoveNext()` 调用。

## 二、`AsyncMethodBuilder`：接管异步机制的关键

`ETTask` 类头上有这个标记：

```csharp
[AsyncMethodBuilder(typeof(ETAsyncTaskMethodBuilder))]
public class ETTask : ICriticalNotifyCompletion, IETTask
```

`[AsyncMethodBuilder]` 告诉编译器：**当你遇到 `async ETTask` 方法，用我指定的 Builder 类来生成状态机，而不是用默认的 `AsyncTaskMethodBuilder`。**

这就是接管整个异步机制的入口。

### Builder 的职责

```csharp
public struct ETAsyncTaskMethodBuilder
{
    private IStateMachineWrap iStateMachineWrap;
    private ETTask tcs;

    // 1. 创建 Builder（每个 async 方法调用时执行一次）
    public static ETAsyncTaskMethodBuilder Create()
    {
        return new() { tcs = ETTask.Create(true) }; // fromPool=true
    }

    // 2. 暴露 Task 给调用方
    public ETTask Task => this.tcs;

    // 3. await 某个未完成的 awaiter 时
    public void AwaitOnCompleted<TAwaiter, TStateMachine>(
        ref TAwaiter awaiter, ref TStateMachine stateMachine)
    {
        // 包装状态机到对象池的 Wrap 中
        this.iStateMachineWrap ??= StateMachineWrap<TStateMachine>.Fetch(ref stateMachine);
        awaiter.OnCompleted(this.iStateMachineWrap.MoveNext);
    }

    // 4. 方法完成时
    public void SetResult()
    {
        this.iStateMachineWrap?.Recycle(); // 状态机回池
        this.tcs.SetResult();
    }
}
```

**核心设计**：状态机本身也被放入对象池（`StateMachineWrap`），彻底消除闭包分配。

## 三、ETTask 对象池：最危险也最重要的设计

```csharp
[StaticField]
private static readonly Queue<ETTask> queue = new();

public static ETTask Create(bool fromPool = false)
{
    if (!fromPool) return new ETTask();

    lock (queue)
    {
        if (!queue.TryDequeue(out var task))
            return new ETTask() { fromPool = true };
        return task;
    }
}

private void Recycle()
{
    if (!this.fromPool) return;

    this.state = AwaiterStatus.Pending;
    this.callback = null;
    this.Context = null;
    this.TaskType = TaskType.Common;

    lock (queue)
    {
        if (queue.Count > 1000) return; // 防止无限膨胀
        queue.Enqueue(this);
    }
}
```

源码注释直接写明了使用规则：

> 请不要随便使用 ETTask 的对象池，除非你完全搞懂了 ETTask！  
> 假如开启了池，**await 之后不能再操作 ETTask**，否则可能操作到再次从池中分配出来的 ETTask，产生灾难性的后果。

**为什么这么危险？**

```csharp
// ❌ 错误用法
var task = ETTask.Create(true); // fromPool=true
await task;
task.SetResult(); // 危险！task 可能已经被别人 Create 出来复用了
```

```csharp
// ✅ 正确用法
ETTask task = ETTask.Create(true);
var localRef = task; // 保存引用
// ... 把 localRef 传出去
await task;
// await 完成后，不再使用 task 引用
```

**第一性原理结论**：对象池本质是"时间维度的内存复用"，代价是"所有权语义失效"——你不能再假设某个引用代表"你创建的那个对象"。

## 四、Context 链：跨 await 传递上下文

```csharp
public interface IETTask
{
    public TaskType TaskType { get; set; }
    public object Context { get; set; }
}

public enum TaskType : byte
{
    Common,      // 普通任务
    WithContext, // 携带上下文
    ContextTask, // 专门用于传递 context 的任务
}
```

为什么需要 Context 机制？

在游戏框架中，很多异步操作需要知道"当前是在哪个 Scene / Entity 上下文中执行的"。原生 Task 的 `SynchronizationContext` 在 Unity 中有兼容性问题，且无法做到细粒度的 Entity 级别隔离。

```csharp
// 用法示例
await someTask.WithContext(currentScene); // 传入当前 Scene 作为上下文

// 在异步链的深处取回
var scene = await ETTask.GetContext<Scene>();
```

`SetContext` 的实现是一个链式遍历，沿着 await 链把 context "注入"进去：

```csharp
internal static void SetContext(this IETTask task, object context)
{
    while (true)
    {
        if (task.TaskType == TaskType.ContextTask)
        {
            ((ETTask<object>)task).SetResult(context); // 找到接收点，set 进去
            break;
        }
        task.TaskType = TaskType.WithContext;
        object child = task.Context;
        task.Context = context;
        task = child as IETTask; // 继续往下传
        if (task == null) break;
        if (task.TaskType == TaskType.WithContext) break; // 有人设置了新 context，停止
    }
}
```

## 五、WaitFrameFinish：帧完成的优雅等待

```csharp
// Game.cs
private static readonly Queue<ETTask> frameFinishTask = new Queue<ETTask>();

public static async ETTask WaitFrameFinish()
{
    ETTask task = ETTask.Create(true);
    frameFinishTask.Enqueue(task);
    await task;
}

public static void FrameFinishUpdate()
{
    while (frameFinishTask.Count > 0)
    {
        ETTask task = frameFinishTask.Dequeue();
        task.SetResult(); // 每帧末尾统一唤醒
    }
}
```

这是一个典型的"延迟到帧末执行"模式，用途：
- 等所有 System 的当帧逻辑执行完再做某些操作
- 确保数据一致性（避免帧中途读到半更新状态）

**快速开项目模板**：

```csharp
// 需要帧末才能读到最新状态的逻辑
await Game.WaitFrameFinish();
// 现在所有 Update 都跑完了
```

## 六、全局异常处理：不让异常静默消失

```csharp
[StaticField]
public static Action<Exception> ExceptionHandler;
```

项目启动时注册：

```csharp
ETTask.ExceptionHandler = e => Log.Error(e);
```

原生 Task 中，如果一个异步方法抛异常但没有被 `await`，异常会被"观察"到后才触发 `UnobservedTaskException`——而且默认不崩溃。游戏里这等于静默 Bug。

自研框架在 `SetException` 时统一调用 `ExceptionHandler`，保证所有异步异常都被记录。

## 七、快速开项目检查清单

### 必做

- [ ] 项目启动时设置 `ETTask.ExceptionHandler`
- [ ] 理解对象池 ETTask 的所有权语义，`await` 后不再持有引用
- [ ] 所有 `async` 返回类型用 `ETTask`，不要混用 `Task`

### 性能

- [ ] 热路径（每帧调用）用 `fromPool=true`
- [ ] 冷路径（初始化/一次性）用默认 `new ETTask()`，不用担心 GC
- [ ] 状态机 Wrap 对象池上限 1000，超出会走 GC，注意监控

### 调试

- [ ] `[DebuggerHidden]` 标记让调试器跳过框架内部，直接停在业务代码
- [ ] 异步调用栈默认会被折叠，调试时注意 `StackTrace` 可能不完整

## 总结

自研异步框架的核心价值链：

```
[AsyncMethodBuilder] 标记
    → 编译器使用自定义 Builder
    → Builder 从对象池拿 ETTask
    → 状态机包装进 StateMachineWrap（也入池）
    → await 完成后统一 Recycle
    → 整个异步链零 GC
```

这是"在语言层面合法地劫持编译器行为"——不是黑魔法，是 C# 的正式扩展点，只是大多数人不知道它存在。
