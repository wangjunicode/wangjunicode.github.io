# 主线程同步上下文：跨线程安全调度的生产者消费者模式深度解析

## 问题背景：Unity 的线程限制

Unity 引擎的绝大多数 API——Transform、GameObject、Component——都不是线程安全的，只能在主线程（渲染线程）中调用。但现代游戏框架大量使用异步任务（async/await）、ThreadPool、后台 IO 线程，不可避免地需要把回调"切回主线程"执行。

这正是 `SynchronizationContext` 的用武之地。

## SynchronizationContext 继承体系

.NET 标准库提供了抽象基类：

```csharp
public abstract class SynchronizationContext
{
    public virtual void Post(SendOrPostCallback d, object state);  // 异步派发
    public virtual void Send(SendOrPostCallback d, object state);  // 同步派发
    public static SynchronizationContext Current { get; }          // 当前线程的上下文
    public static void SetSynchronizationContext(SynchronizationContext sc);
}
```

框架实现自定义 `MainThreadSynchronizationContext`，继承并覆写 `Post` 方法，将回调排队到主线程执行队列，在主循环中统一 `Flush`。

## 核心数据结构：双队列交换

```csharp
public sealed class MainThreadSynchronizationContext : SynchronizationContext
{
    // 两个队列交替角色：一个接收生产者写入，另一个供主线程消费
    private Queue<Action> queue1 = new Queue<Action>();
    private Queue<Action> queue2 = new Queue<Action>();
    private Queue<Action> currentQueue;  // 指向"生产者队列"

    public MainThreadSynchronizationContext()
    {
        currentQueue = queue1;
    }
}
```

**为什么要双队列？**

单队列方案存在竞态问题：主线程迭代队列时，后台线程同时往里插入，导致迭代器失效或漏项。双队列通过原子指针交换实现零锁消费：
- `queue1`（生产者）：后台线程调用 `Post`，持续往这里写
- 主线程帧更新时：原子地把指针切换为 `queue2`（空队列）接管生产，自己拿 `queue1` 去消费
- 消费完后 `queue1` 清空待用，下次再交换

## Post 方法：后台线程的入口

```csharp
public override void Post(SendOrPostCallback d, object state)
{
    lock (currentQueue)
    {
        currentQueue.Enqueue(() => d(state));
    }
}
```

关键点：
1. `Post` 是异步的（非阻塞），适合在 async/await 的 `AwaitUnsafeOnCompleted` 中调用
2. `lock(currentQueue)` 保护写入，但锁粒度很小（仅 Enqueue 操作）
3. `() => d(state)` 闭包捕获参数，Action 化便于统一调度

## Flush 方法：主线程的消费逻辑

```csharp
public void Update()
{
    // 原子交换：将 currentQueue 换成空队列，拿走旧队列消费
    Queue<Action> q;
    lock (queue2)  // 保证 queue2 清空完成
    {
        queue2.Clear();
    }
    
    // Interlocked 方式交换（实际用 lock 或 Interlocked.Exchange）
    Queue<Action> tmp;
    lock (this)
    {
        tmp = currentQueue;
        currentQueue = (currentQueue == queue1) ? queue2 : queue1;
    }
    
    // 消费阶段：无竞争，不需要锁
    while (tmp.Count > 0)
    {
        Action action = tmp.Dequeue();
        try
        {
            action();
        }
        catch (Exception e)
        {
            Log.Error(e);
        }
    }
}
```

**双队列交换的关键洞察：**
- 交换完成后，`tmp` 是老队列，只有主线程访问（无竞争）
- `currentQueue` 已指向新空队列，后台线程继续往新队列写
- 两者完全解耦，消费阶段零锁开销

## 与 ETTask 的集成点

在 `AsyncETTaskMethodBuilder` 中，当 continuation 需要切回主线程时：

```csharp
public void AwaitUnsafeOnCompleted<TAwaiter, TStateMachine>(
    ref TAwaiter awaiter, ref TStateMachine stateMachine)
    where TAwaiter : ICriticalNotifyCompletion
    where TStateMachine : IAsyncStateMachine
{
    var wrap = StateMachineWrap<TStateMachine>.Create(stateMachine);
    
    // 如果当前线程不是主线程，Post 到主线程同步上下文
    var ctx = SynchronizationContext.Current;
    if (ctx != null && ctx is MainThreadSynchronizationContext)
    {
        awaiter.UnsafeOnCompleted(wrap.MoveNext);
    }
    else
    {
        // 跨线程场景：通过 Post 切回主线程
        var captured = SynchronizationContext.Current ?? 
                       Game.MainThreadSynchronizationContext;
        awaiter.UnsafeOnCompleted(() => 
            captured.Post(_ => wrap.MoveNext(), null));
    }
}
```

## 在 Game 单例中的注册

框架启动时，在主线程中把自定义上下文设置为当前线程的上下文：

```csharp
public static class Game
{
    public static MainThreadSynchronizationContext MainThreadSynchronizationContext 
        { get; private set; }
    
    public static void Init()
    {
        MainThreadSynchronizationContext = new MainThreadSynchronizationContext();
        SynchronizationContext.SetSynchronizationContext(MainThreadSynchronizationContext);
        // ...
    }
    
    public static void Update()
    {
        MainThreadSynchronizationContext.Update();  // 每帧 Flush 回调队列
        // ...
    }
}
```

之后任何线程通过 `SynchronizationContext.Current` 拿到的就是主线程上下文，`Post` 的回调最终在主线程 `Update` 中执行。

## 与 .NET 默认实现对比

| 特性 | .NET 默认（ThreadPoolSynchronizationContext） | 框架自实现 |
|------|---------------------------------------------|-----------|
| 执行线程 | ThreadPool 任意线程 | 严格主线程 |
| 帧对齐 | ❌ 随时执行 | ✅ 帧更新时批量执行 |
| Unity API 安全 | ❌ | ✅ |
| 锁竞争 | 低（ThreadPool 内部） | 极低（仅 Enqueue 时加锁） |
| 吞吐量控制 | 无 | 可在 Update 中限制每帧最大处理数 |

## 性能优化建议

**1. 批量限流**：帧更新时可设置每帧最大执行数量，避免大量回调集中在同一帧造成卡顿：

```csharp
public void Update(int maxPerFrame = 100)
{
    // ... 交换队列 ...
    int count = 0;
    while (tmp.Count > 0 && count++ < maxPerFrame)
    {
        tmp.Dequeue()();
    }
    // 剩余的留到下帧（需额外处理溢出逻辑）
}
```

**2. 避免深层闭包**：`Post` 中的 `() => d(state)` 会分配闭包对象，高频场景可用 `ObjectPool<ContextCallback>` 复用。

**3. 监控队列深度**：在开发期记录 `currentQueue.Count`，若持续增长说明主线程处理能力不足。

## 总结

`MainThreadSynchronizationContext` 是游戏框架多线程安全的核心基础设施：

- **双队列交换**：分离生产与消费，最小化锁竞争
- **帧对齐执行**：回调在主循环中批量处理，保证 Unity API 安全
- **透明集成**：通过 .NET `SynchronizationContext` 标准接口，async/await 无需额外适配
- **可观测性**：队列深度、帧处理数可作为性能监控指标

理解这个组件，是在 Unity 中正确使用 async/await 而不踩线程安全坑的关键。
