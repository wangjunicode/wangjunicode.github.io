# 战斗帧同步网络回放系统设计

战斗录像（回放）功能在游戏中越来越常见。用户可以回看精彩操作，游戏可以将录像上传用于反作弊分析。实现这个功能并不是简单地"录视频"，对于帧同步战斗而言，需要一套精密的帧数据存储和重播机制。本文深入解析这套系统的完整设计。

---

## 第一性原理：帧同步录像 vs 视频录像

**视频录像**：按固定帧率截图，存储每一帧的画面。缺点是体积大（每帧一张图），无法二次分析（只是图片序列）。

**帧同步录像**：只存储每帧的**输入指令**（玩家按了哪些键）。回放时，用相同的初始状态 + 相同的输入指令，重新运行一遍游戏逻辑，得到完全相同的结果。

帧同步录像的核心优势：
- **体积极小**：一场几分钟的战斗，录像文件可能只有几十KB（只有指令序列）
- **可分析**：可以任意时间点暂停，查看数值状态
- **防作弊**：服务器重放录像，可以验证客户端的战斗结果是否合法

## 核心数据结构

```csharp
public class DungeonRecord
{
    public int sceneId;         // 场景 ID
    public int battleID;        // 战斗 ID
    public int fightType;       // 战斗类型
    public List<FrameItem> framelst;   // 帧数据列表（核心）
    public List<TeamRecordData> teamDatas;  // 队伍初始数据
    public List<int> roundFeature;    // 回合特性
}

public class FrameItem
{
    public int frameId;          // 帧号
    public List<CmdItem> cmdlst; // 这一帧的指令列表
}

public class CmdItem
{
    public int vkey;        // 虚拟键（操作类型）
    public int playerId;    // 操作玩家 ID
    public List<int> args;  // 参数
}
```

录像本质上是：初始状态（队伍数据）+ 指令序列（framelst）。

## BattleRecordComponent：回放引擎

```csharp
[FriendOf(typeof(BattleRecordComponent))]
public static partial class BattleRecordComponentSystem
{
    [EntitySystem]
    private static void Awake(this BattleRecordComponent self)
    {
        // 注册事件监听
        self.RootDispatcher().RegisterEvent<Evt_FixedUpdate>(self.EnterFrame);
        self.battlePlayerComp = self.Parent.GetComponent<BattlePlayerComponent>();
        
        // 监听战斗控制指令（开始、结束）
        self.battlePlayerComp.RegisterServerCommandCallback(
            VKeyDef.PVP_CONTROL_START, self.OnSyncControllStart);
        self.battlePlayerComp.RegisterServerCommandCallback(
            VKeyDef.PVP_ROUND_BEGIN, self.OnSyncRoundBegin2);
        
        // 监听战斗结束事件（用于生成 Hash 验证）
        self.RootDispatcher().RegisterEvent<Evt_PointEvent>(self.OnPointEvent);
    }
```

### 启动回放

```csharp
public static void Start(this BattleRecordComponent self, DungeonRecord record)
{
    // 开启帧追踪（用于后期对账验证）
    FSPDebuger.EnableLogTrack = true;
    FSPDebuger.TrackBufferSize = 50000;
    FSPDebuger.BeginTrack();
    FSPDebuger.EnterTrackFrame(0);
    
    self.dungeonRecord = record;
    self.Running = true;

    // 处理第0帧的控制指令（战斗初始化）
    bool roundBegin = false;
    bool controlBegin = false;
    for (int i = 0; i < self.dungeonRecord.framelst.Count; ++i)
    {
        var item = self.dungeonRecord.framelst[i];
        if (item.frameId == 0 && item.cmdlst != null)
        {
            // 找到 ROUND_BEGIN 指令
            for (int icmd = 0; icmd < item.cmdlst.Count; ++icmd)
            {
                var cmd = item.cmdlst[icmd];
                if (cmd.vkey == VKeyDef.PVP_ROUND_BEGIN)
                {
                    self.OnSyncRoundBegin2(cmd.vkey, cmd.args.ToArray(), cmd.playerId);
                    roundBegin = true;
                    break;
                }
            }
            
            // 找到 CONTROL_START 指令
            for (int icmd = 0; icmd < item.cmdlst.Count; ++icmd)
            {
                var cmd = item.cmdlst[icmd];
                if (cmd.vkey == VKeyDef.PVP_CONTROL_START)
                {
                    self.OnSyncControllStart(cmd.vkey, cmd.args.ToArray(), cmd.playerId);
                    controlBegin = true;
                    break;
                }
            }
            
            if (roundBegin && controlBegin) break;
        }
    }
}
```

**第0帧的特殊处理**：帧号为0的指令是"控制指令"（如战斗开始、轮次开始），必须先处理这些，再开始正常的帧推进。

## 帧推进：EnterFrame 回放核心

```csharp
public static void EnterFrame(this BattleRecordComponent self)
{
    if (!self.BattlePlayerComp.IsRoundBegin)
        return;

    // 接收当前帧对应的历史帧数据
    self.battlePlayerComp.EnterFrame();

    if (self.m_RoundEnd == false)
    {
        self.ReceiveFrame();  // 从录像中"接收"帧数据
    }

    self.ExecFrame();  // 执行帧逻辑
}
```

### ReceiveFrame：从录像中喂帧数据

```csharp
private static void ReceiveFrame(this BattleRecordComponent self)
{
    int frameindex = self.battlePlayerComp.FrameIndex;
    if (frameindex <= 0) frameindex = 1;

    SyncFrame framedata = null;
    int curFrameIndex = self.m_lastRecordIndex;
    
    // 在录像列表中找到对应帧号的帧数据
    for (int len = self.dungeonRecord.framelst.Count; curFrameIndex < len; ++curFrameIndex)
    {
        var item = self.dungeonRecord.framelst[curFrameIndex];
        if (item.frameId == frameindex)
        {
            // 找到了，构建 SyncFrame 对象
            framedata = new SyncFrame();
            framedata.frameId = item.frameId;
            framedata.cmdList = new List<SyncCmd>();
            for (int j = 0; j < item.cmdlst.Count; j++)
            {
                var dcmd = item.cmdlst[j];
                var cmd = new SyncCmd();
                cmd.vkey = dcmd.vkey;
                cmd.playerId = dcmd.playerId;
                cmd.args = dcmd.args.ToArray();
                framedata.cmdList.Add(cmd);
            }
            break;
        }
        else if (item.frameId > frameindex)
        {
            break;  // 录像中没有这一帧的指令（该帧无输入）
        }
    }

    if (framedata != null)
    {
        self.m_lastRecordIndex = curFrameIndex + 1;  // 记录扫描位置（避免每次从头搜索）
        self.battlePlayerComp.InputFrameData(framedata);
    }
}
```

这里有个重要的性能优化：`m_lastRecordIndex` 保存了上次扫描到的位置。由于帧号是递增的，每次只需要从上次的位置继续搜索，而不是每次从头遍历整个录像列表。时间复杂度从 O(N*M) 降低到接近 O(N+M)。

### ExecFrame：执行帧逻辑

```csharp
public static int ExecFrame(this BattleRecordComponent self)
{
    if (!self.Running || !self.BattlePlayerComp.IsRoundBegin)
        return 0;

    // 询问帧缓冲区，这一次要推进多少帧
    int ticksNum = self.BattlePlayerComp.FrameBuffer
        .GetFrameCountOnEnterframe(self.battlePlayerComp.FrameIndex);
    
    if (ticksNum == 0)
    {
        // 没有新帧，推进一个空帧（维持时间流动）
        self.TickEngine();
    }

    // 执行所有待处理帧
    for (int i = 0; i < ticksNum; i++)
    {
        SyncFrame f = self.battlePlayerComp.FrameBuffer
            .GetFrameAndTick(self.battlePlayerComp.FrameIndex);
        
        if (f != null)
        {
            self.battlePlayerComp.ExecFrameCmd(f);  // 执行帧指令
        }
        
        if (f == null || f.frameId > 0)
        {
            self.TickEngine();  // 推进战斗引擎一帧
        }
    }

    return ticksNum;
}
```

`ticksNum` 的意义：回放时可能需要追帧（加速播放），也可能需要慢播（降速）。帧缓冲区根据当前帧索引和目标帧索引，决定这次 `EnterFrame` 需要推进几个逻辑帧。

## 战斗哈希：验证与反作弊

帧同步录像的一个核心特性是**可验证性**。服务器可以重放录像，检查最终结果是否和客户端上报的结果一致。

```csharp
public static ulong GenerateBattleHash(this BattleRecordComponent self)
{
    ulong hash = 0;
    FP fpHash = FP.Zero;
    
    var teamComp = self.DomainScene().GetComponent<TeamComponent>();
    int count = 0;
    
    // 遍历所有队伍成员，累加关键数值
    foreach (var team in teamComp.TeamList)
    {
        foreach (var member in team.TeamMember)
        {
            count++;
            self.HashActorAttribute(member, ref fpHash);
        }
    }

    // 以数值 Hash 为基础
    hash = (ulong)(fpHash * 10000).AsLong();
    hash += (uint)self._battleLimitFrame;  // 混入有效帧数
    
    // [8bit] 角色数量
    if (count >= 15) count = 15;
    hash <<= 8;
    hash += (ulong)count;
    
    // [16bit] 终止帧号
    hash <<= 16;
    hash += (uint)self.battlePlayerComp.FrameIndex;
    
    // [4bit] 胜利分组
    int wg = self.battlePlayerComp.winGroup;
    if (wg < 0) wg = 8 - wg;
    hash <<= 4;
    hash += (uint)wg;
    
    return hash;
}
```

哈希的组成（位布局）：

```
[高位]  关键数值加和  |  有效帧数  |  8bit角色数  |  16bit帧号  |  4bit胜利组  [低位]
```

参与哈希的关键属性：

```csharp
private static void HashActorAttribute(this BattleRecordComponent self, Unit pc, ref FP hash)
{
    var numericComp = pc.GetComponent<NumericComponent>();
    hash += pc.Group;                                          // 所属分组
    hash += numericComp.GetFinalValue(ENumericId.RealTeamHp); // 剩余血量
    hash += numericComp.GetPart(ENumericId.RealTeamHp, ENumericPart.Min); // 血量下限
    hash += numericComp.GetPart(ENumericId.RealTeamHp, ENumericPart.Max); // 血量上限
    hash += numericComp.GetFinalValue(ENumericId.ActionPoint); // 行动点
    hash += numericComp.GetFinalValue(ENumericId.BaseAtk);     // 基础攻击
    hash += numericComp.GetFinalValue(ENumericId.RealAtk);     // 实际攻击
    hash += numericComp.GetFinalValue(ENumericId.RealGuard);   // 护盾
    var pos = pc.Position;                                     // 位置（定点数）
    hash += pos.x;
    hash += pos.y;
    hash += pos.z;
}
```

通过将所有关键数值累加进哈希，任何一个数值的差异都会导致哈希不同。如果两端（客户端 vs 服务器）的战斗结果哈希一致，说明战斗过程完全相同，可信度极高。

## 帧追踪调试系统

`FSPDebuger` 是一个专门用于帧同步调试的工具：

```csharp
// 战斗开始时
FSPDebuger.EnableLogTrack = true;
FSPDebuger.TrackBufferSize = 50000;  // 预分配5万帧的缓冲区
FSPDebuger.BeginTrack();
FSPDebuger.EnterTrackFrame(0);

// 每帧进入时
FSPDebuger.EnterTrackFrame(self.frameIndex);

// 战斗结束时，可以保存轨迹数据
var bytesPath = FSPDebuger.SaveTrack(self.battlePlayerComp.FrameIndex);
```

这个系统的作用：当两个客户端的战斗结果出现偏差（即"帧同步不一致"，Desync），可以导出双方的帧追踪数据进行对比，精确定位到哪一帧、哪一个操作开始出现差异。

## 录像格式压缩：只记录有输入的帧

```csharp
// 录制时，只存储有指令的帧
if (frame.cmdList.Count > 0)
{
    var f = new SyncFrame();
    f.frameId = frame.frameId * self.frameBuffer.frameInterval;
    f.cmdList = new List<SyncCmd>(frame.cmdList.Count);
    // ... 复制指令
    self.listRecordFrames.Add(f);
}
```

大多数帧都是"空帧"（没有输入），不存储这些帧，录像大小可以减少 90% 以上。回放时，空帧就是让引擎正常推进一帧，不需要任何指令。

## 录像的应用场景

### 1. 玩家录像观看

存储最近 N 场战斗的录像，玩家可以回看自己的操作，分析失败原因。

### 2. 服务器验证

战斗结束后上传录像（帧数据），服务器重放并计算哈希，与客户端上报的结果对比：
- 哈希匹配 → 战斗合法，结算奖励
- 哈希不匹配 → 可能存在作弊，拒绝结算

### 3. 精彩片段分享

截取录像的特定时间段，生成分享链接，其他玩家下载后本地回放。

### 4. 开发调试

捕获线上 Bug 的录像，开发人员本地重放，在任意帧暂停、分析，精确复现问题。

## 设计总结

| 特性 | 实现方式 |
|------|---------|
| 极小体积 | 只存储输入指令，不存储状态快照 |
| 精确回放 | 定点数 + 帧同步，保证完全确定性 |
| 快速搜索 | m_lastRecordIndex 避免每次从头搜索 |
| 结果验证 | 多数值累加哈希，位域编码 |
| 速度控制 | FrameBuffer 控制每次推进的帧数 |
| 调试支持 | FSPDebuger 追踪和导出帧轨迹 |

帧同步录像系统是游戏可信度和玩家体验的双重保障。对于初学者，重要的是理解其核心：**只要初始状态相同 + 输入序列相同，任何平台上都能得到完全相同的结果**。这是帧同步战斗最强大也最精妙的特性。
