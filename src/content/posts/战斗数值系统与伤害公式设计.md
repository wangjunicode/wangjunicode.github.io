# 战斗数值系统与伤害公式设计

战斗数值系统是游戏平衡性的骨架。攻击力、防御力、会心率、闪避率……这些数值不只是简单的数字，背后有一套严密的分层架构和数学公式。本文从技术负责人视角，深入解析一个商业级战斗数值系统的完整设计。

---

## 第一性原理：数值系统需要解决什么问题？

想象最简单的情况：角色有一个"攻击力"属性，值为 100。

然后你遇到需求：
- 装备加攻击力 +50
- Buff 加攻击力 20%
- 特殊状态下再加攻击力 +30
- 另一个 Buff 加最终攻击力 10%
- 移除某个 Buff 后攻击力恢复

如果只是 `int atk += 50`，移除 Buff 时怎么知道减多少？多个 Buff 叠加怎么保证顺序？百分比和绝对值哪个先算？

**数值系统需要解决的核心问题**：
1. **多层叠加**：不同来源的加成如何叠加
2. **可撤销性**：加成可以精确移除，不影响其他加成
3. **公式联动**：某些数值依赖其他数值的公式计算
4. **一致性**：数值计算结果在任何时候都是可预测的

## 数值分层架构

系统将一个数值（如 HP）分解为多个"部分"（Part）：

```
/*
 * self.GetPart(ENumericId.Hp, ENumericPart.Base);    // 基础值
 * self.GetPart(ENumericId.Hp, ENumericPart.Add);     // 绝对值加成
 * self.GetPart(ENumericId.Hp, ENumericPart.Pct);     // 百分比加成
 * self.GetPart(ENumericId.Hp, ENumericPart.FinalAdd); // 二阶绝对值加成
 * self.GetPart(ENumericId.Hp, ENumericPart.FinalPct); // 二阶百分比加成
 * self.GetFinalValue(ENumericId.Hp);                  // 最终值
 */
```

最终值的计算公式通常是：

```
最终值 = (Base + Add) × (1 + Pct/100) + FinalAdd × (1 + FinalPct/100)
```

分两阶段的原因是：游戏设计上区分"战斗内加成"和"战斗外加成"，战斗外（装备、天赋）走一阶，战斗内（Buff、状态）走二阶，互不干扰，便于数值策划控制上限。

## NumericComponent 核心实现

### 初始化

```csharp
[EntitySystem]
private static void Awake(this NumericComponent self)
{
    int count = NumericComponent.NumericIdMax;
    if (count < 100) count = 1000;
    self.numericInfos = new NumericInfoArray(new NumericInfo[count]);
}
```

用数组（而非字典）存储 NumericInfo，通过 `ENumericId` 枚举值作为下标直接访问，O(1) 查找，缓存友好。这是高性能代码的典型做法。

### 获取和修改数值

```csharp
// 获取最终值（公式值优先）
public static FP GetFinalValue(this NumericComponent self, ENumericId key)
{
    return self.numericInfos[(int)key]?.Final ?? FP.Zero;
}

// 设置动态值（当前血量等实时变化的值）
public static void Set(this NumericComponent self, ENumericId nt, FP value)
{
    self.SetPart(nt, ENumericPart.Base, value);
}

// 增减动态值
public static void Modify(this NumericComponent self, ENumericId nid, FP delta)
{
    var value = self.GetPart(nid, ENumericPart.Base);
    value += delta;
    self.Set(nid, value);
}
```

### 修改器系统（可撤销加成的关键）

```csharp
// 添加加成，返回修改器句柄
public static ModifierHandle ModifyPart(this NumericComponent self, ENumericId nid, ENumericPart part, FP delta, object owner = null)
{
    var numericInfo = self.numericInfos[(int)nid];
    // 属性值：通过修改器堆叠，可以精确撤销
    var handle = numericInfo.AddModifier(part, delta, owner);
    return handle;  // 调用方保存句柄，用于之后撤销
}

// 撤销加成（通过句柄精确定位）
public static void RestorePart(this NumericComponent self, ENumericId nid, ENumericPart part, ModifierHandle op)
{
    var numericInfo = self.numericInfos[(int)nid];
    numericInfo.RemoveModifier(op);  // 精确移除这一次加成
}
```

使用示例（Buff 添加和移除攻击力加成）：

```csharp
// Buff 生效时
ModifierHandle handle = numeric.ModifyPart(ENumericId.Atk, ENumericPart.Add, 100, this);

// Buff 失效时
numeric.RestorePart(ENumericId.Atk, ENumericPart.Add, handle);
```

`ModifierHandle` 是一个轻量级的引用句柄，包含了足够信息精确定位到这次修改，移除时不影响其他来源的同类型加成。这是业界常见的"修改器（Modifier）"模式。

## 数值初始化流程

```csharp
public static void InitCharacterTemplate(this NumericComponent self, CCharacterNumeric template, BattleCharacterConf presetConf)
{
    self.BeginInitNumeric();  // 锁定，批量初始化期间不触发公式重算
    
    // 1. 初始化所有数值的默认值
    self.InitNumericDict(ENumericOwner.Character);
    
    // 2. 从配置模板设置基础值
    foreach (var kv in template.NumericMap)
    {
        self.SetPart(kv.Key, ENumericPart.Base, kv.Value);
    }
    
    // 3. 应用心得卡（外部装备）数值加成
    if (presetConf != null)
    {
        foreach (var numericChange in presetConf.NumericChanges)
        {
            self.ModifyPart(numericChange.Item1, numericChange.Item3, numericChange.Item2);
        }
    }
    
    // 4. 设置派生值（从基础值计算）
    var maxGuardValue = (FP)TSMath.RoundToInt(self.GetFinalValue(ENumericId.MaxGuard));
    self.SetPart(ENumericId.RealGuard, ENumericPart.Max, maxGuardValue);
    self.SetPart(ENumericId.RealGuard, ENumericPart.Base, maxGuardValue);
    
    self.EndInitNumeric();  // 解锁，触发一次全量公式重算
}
```

`BeginInitNumeric/EndInitNumeric` 是批量初始化的优化模式——在初始化期间锁定重算，等所有基础值设置完后再统一计算，避免每次 SetPart 都触发公式重算导致的性能开销和中间状态问题。

## 核心数值公式

### 防御减伤率

```csharp
public static FP 防御减伤率(this NumericComponent self, FP 穿透率)
{
    return 除法公式(self.最终防御力() * (1 - 穿透率), ConstA);
}

private const int ConstA = 200;

// 除法公式通常是 x / (x + C)，趋近于1但不会超过1
// 当防御力 = 200 时，减伤率 = 200/(200+200) = 50%
// 当防御力 = 600 时，减伤率 = 600/(600+200) = 75%
```

这是游戏界经典的"除法公式"（Diminishing Returns）。防御力越高，减伤率越高，但增速越来越慢，保证了防御力永远不会无限趋向100%。

### 概率类属性公式（会心率/闪避率等）

```csharp
private const int ConstB = 1600;

[NumericBinder(ENumericId.FocusProb)]  // 会心率绑定
public static FP 会心率_Base(this NumericComponent self)
{
    return 除法公式(self.最终会心值(), ConstB);
}

[NumericBinder(ENumericId.DogeProb)]   // 闪避率绑定
public static FP 闪避率_Base(this NumericComponent self)
{
    return 除法公式(self.最终闪避值(), ConstB);
}
```

`[NumericBinder]` 是一个自定义特性，将这个函数绑定为特定 `ENumericId` 的计算公式。当该数值被访问时，系统自动调用绑定的函数计算结果。

这是一种**函数式绑定**的设计：数值不只是静态的数字，可以是动态计算的表达式，且与数值 ID 解耦，便于扩展。

## 核心伤害公式

```csharp
[FormulaBinder(EExpression.AttackDamage)]
public static FP AttackDamage(NumericMap map)
{
    var self = map.self;    // 攻击方数值组件
    var other = map.other;  // 防御方数值组件

    // 1. 前台/后台增伤（队伍中第几号位有不同加成）
    var mainMulti = (self.获取主角色数值() == self 
        ? self.GetFinalValue(ENumericId.MainAtk) 
        : self.GetFinalValue(ENumericId.SupportAtk)) / 100;
    
    // 2. 百分比增减伤
    var atkMulti = (self.GetFinalValue(ENumericId.AtkMulti) 
        + self.队伍数值().GetFinalValue(ENumericId.AtkMulti) 
        - other.GetFinalValue(ENumericId.DamageReduce)) / 100;
    
    // 最终倍率（不低于0）
    var finalMulti = TSMath.Max(0, 1 + atkMulti + mainMulti);
    
    // 3. 单击基础伤害系数（技能配置的伤害倍率）
    var dmgPerhit = self.GetFinalValue(ENumericId.DmgPerHit) / 100;
    
    // 4. 防御力减伤
    var def = 1 - other.防御减伤率(self.最终穿防率());
    
    // 5. 会心/破招暴击增伤（×2）
    var atkResult = self.获取角色().GetAttackResult();
    var resDmg = (atkResult == EAttackResult.Critical || atkResult == EAttackResult.Focus) ? 2 : 1;
    
    // 6. 固定伤害（不受防御影响）
    var fixedDamage = self.GetFinalValue(ENumericId.FixedDamageIncrease) 
                    - other.GetFinalValue(ENumericId.FixedDamageReduce);

    // 最终伤害
    var finalDamage = self.最终攻击力() * dmgPerhit * finalMulti * def * resDmg + fixedDamage;
    
    return finalDamage;
}
```

### 伤害公式解读

**完整公式展开**：

```
最终伤害 = 攻击力 × 伤害系数 × 增减伤倍率 × (1 - 防御减伤率) × 暴击倍率 + 固定伤害
```

各项含义：
- `攻击力`：从数值组件获取，已包含所有加成
- `伤害系数（DmgPerHit）`：技能每段攻击的百分比，如"300%攻击力"就是3.0
- `增减伤倍率`：自身增伤 + 队伍增伤 - 对方减伤
- `防御减伤率`：考虑穿防率后的实际减伤比例
- `暴击倍率`：会心/破招时 ×2
- `固定伤害`：不受防御影响的固定数值（穿透类技能常用）

**`NumericMap` 的设计**：

```csharp
public struct NumericMap
{
    public NumericComponent self;           // 攻击方
    public NumericComponent other;          // 防御方
    public NumericComponent selfTeam;       // 攻击方队伍
    public NumericComponent otherTeam;      // 防御方队伍
    public NumericComponent selfTeamCurMain;   // 攻击方当前上场主角
    public NumericComponent otherTeamCurMain;  // 防御方当前上场主角
    
    public NumericMap(Unit owner)
    {
        TeamComponent teamSys = owner.Domain.GetComponent<TeamComponent>();
        self = owner.GetComponent<NumericComponent>();
        selfTeam = teamSys.GetTeamByUnit(owner).GetComponent<NumericComponent>();
        // ... 初始化所有引用
    }
}
```

`NumericMap` 将伤害公式所需的所有数值上下文打包成一个结构体，公式函数只需要接受这一个参数就能获取所有需要的数值。这避免了公式函数签名随需求变化而频繁修改。

## 概率系统：反作弊随机

普通的概率判定（如"30%会心率"）存在一个体验问题：理论上可能连续十几次都不触发，或连续多次都触发，造成体验极端。

系统实现了**伪随机分布（Pseudo-Random Distribution）**：

```csharp
public static bool GetRandResult(EExpression expression, Unit owner)
{
    FP randNum = CalcExpression(expression, owner);  // 面板概率（如0.3）
    FP prob = GetRandProb(expression, owner);         // 真实概率（随失败次数递增）
    
    FP result = battleSpt.Engine.random.NextFP();
    bool ret = result <= prob;
    
    SetExpressionResult(expression, owner, ret);  // 记录结果，更新失败计数
    return ret;
}
```

关键是 `GetRandRealP`，它将面板概率转换为"失败越多次，下次触发概率越高"的实际概率：

```csharp
public static FP GetRandRealP(FP randNum)
{
    // 二分法找到满足期望值的真实概率 p
    // 满足条件：期望触发间隔 = 1/面板概率
    // 但单次触发概率随失败次数线性递增：i * p
    
    FP low = 0, high = randNum, realP = 0;
    while (true)
    {
        realP = (high + low) / 2;
        
        // 计算这个 p 对应的期望触发次数
        FP expectation = 0;
        FP successP = 0;
        FP times = TSMath.CeilToInt(1 / realP);
        for (int i = 1; i <= times; i++)
        {
            FP currentP = (1 - successP) * TSMath.Min(1, realP * i);
            successP += currentP;
            expectation += currentP * i;
        }
        FP pr = 1 / expectation;  // 实际期望概率
        
        if (TSMath.Abs(pr - last) <= limit) break;
        if (pr > randNum) high = realP;
        else low = realP;
        last = pr;
    }
    return realP;
}
```

这套算法保证了：
- **期望概率不变**：长期来看触发率等于面板概率
- **方差降低**：不会出现"连续30次没触发"的情况
- **体验更公平**：玩家感受到"稳定的随机"

## 治疗系统

```csharp
public static void HealHp(this Entity entity, FP value, EValueAddType type = EValueAddType.Value, object owner = null)
{
    if (value <= 0) return;
    
    entity = entity.GetRelativeEntity(false, false);
    var team = entity.GetRelativeEntity(true, false) as TeamEntity;
    var numeric = team.GetComponent<NumericComponent>();
    var unitNumeric = entity.GetComponent<NumericComponent>();
    
    // 治疗量 × 治疗效果率
    var cureRate = unitNumeric.GetFinalValue(ENumericId.CureRate);
    var val = numeric.GetPercentValue(ENumericId.RealTeamHp, value, type) * cureRate;
    
    // 用临时修改器写入治疗量
    var op = numeric.ModifyPart(ENumericId.HpHeal, ENumericPart.Add, val, owner);
    
    // 触发前置事件（可被Buff拦截或修改）
    EventSystem.Instance.Publish(entity.DomainScene(), 
        new Evt_PointEvent() { Unit = entity as Unit, Team = team, EventID = BPDef.PreHealHp });
    
    // 实际加血
    numeric.ModifyPart(ENumericId.RealTeamHp, ENumericPart.Base, numeric.GetFinalValue(ENumericId.HpHeal));
    
    // 触发后置事件（可触发"回血后"特效）
    EventSystem.Instance.Publish(entity.DomainScene(), 
        new Evt_PointEvent() { Unit = entity as Unit, Team = team, EventID = BPDef.PostHealHp });
    
    // 清理临时修改器
    numeric.RestorePart(ENumericId.HpHeal, ENumericPart.Add, op);
}
```

治疗系统的"前置/后置事件"设计非常精妙：
- `PreHealHp`：可以被 Buff 拦截，阻止治疗或修改治疗量
- `PostHealHp`：治疗完成后，触发"治疗特效"、"治疗统计"等逻辑

这是"职责链"模式的应用，每个 Buff 都可以订阅这些事件，无需修改核心代码。

## 数值类型分类

```csharp
public enum ENumericType
{
    Base,    // 基础属性（攻击力、防御力等，多层叠加）
    Dynamic, // 动态属性（血量、怒气等，当前值+上下限）
    Formula, // 公式属性（会心率、闪避率，由函数计算）
}
```

- **Base 类型**：静态属性，通过修改器叠加，适合装备加成
- **Dynamic 类型**：有上下限的实时变化值，适合血量、能量条
- **Formula 类型**：依赖其他属性的计算结果，适合概率、比率类数值

## 设计总结

| 特性 | 实现方式 | 解决的问题 |
|------|---------|----------|
| 多层叠加 | Part 分层 + AddModifier | 来源清晰，不混淆 |
| 可撤销加成 | ModifierHandle 句柄 | Buff 移除时精确还原 |
| 公式数值 | NumericBinder 特性绑定 | 联动计算，自动更新 |
| 批量初始化 | BeginInit/EndInit 锁定 | 避免中间状态导致的错误计算 |
| 伪随机分布 | 失败累计概率递增算法 | 保证体验公平，降低极端情况 |
| 事件驱动治疗 | Pre/Post 事件钩子 | 可扩展，不修改核心代码 |

数值系统是战斗平衡的根基，也是最需要严格测试的模块。一个小小的公式错误可能导致某个角色无敌，或者某个 Buff 叠加 Bug 让数值爆炸。理解这套系统，是写出高质量战斗代码的必经之路。
