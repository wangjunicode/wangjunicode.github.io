## 1. 系统概述

本项目是一款以"IP 偶像养成"为核心玩法的手机游戏，整体采用 **ET 框架**（Entity-Component-System，简称 ECS）作为逻辑层基础，使用 **热更新（Hotfix）** 方案分离业务代码与引擎代码，保证上线后可快速迭代修复。

## 2. 架构设计

### 2.1 总体分层

```
┌──────────────────────────────────────────────┐
│             UI / 表现层（YIUI）               │
├──────────────────────────────────────────────┤
│          Hotfix / 游戏逻辑热更层              │
│   GamePlay  ─  Biz  ─  Framework             │
├──────────────────────────────────────────────┤
│    ET 框架核心（Entity / Component / Event）  │
├──────────────────────────────────────────────┤
│        Unity 引擎 + 资源管理（YooAsset）      │
└──────────────────────────────────────────────┘
```

### 2.2 ET 框架核心概念

**Entity（实体）**：游戏世界里的"对象"，本质是一个 ID，不包含任何逻辑。

**Component（组件）**：附着在 Entity 上的数据容器，如 `PlayerComponent`、`BackpackComponent`、`CultivationComponent`。

**System（系统）**：对 Component 进行操作的静态扩展方法集合，如 `PlayerComponentSystem`、`BackpackComponentSystem`。

**Event（事件）**：解耦模块通信的消息总线，如 `Evt_SwitchScene`、`Evt_BackpackDataUpdated`。

这种设计的核心理念：**数据与逻辑分离**。Component 只存数据，System 只写逻辑，通过 Event 跨系统通信。

### 2.3 SystemDomain（系统作用域）概念

游戏中不同"大玩法"（如主大厅、养成剧本、PVP 竞技场）之间的切换，统一由 `SystemDomain` 层管理：

```
ESystemDomainType
  ├── Unknown      (未初始化)
  ├── Common       (公共区域/登录)
  ├── Cultivation  (养成剧本玩法区)
  ├── PVPArena     (PVP 竞技场区)
  └── ...
```

每个 Domain 对应一个继承自 `SystemDomainHandler` 的处理类，负责该玩法区域的资源收集、Loading 显示、进入/退出钩子。

### 2.4 事件驱动流程

全局使用事件总线驱动状态机流转，典型流程如下：

```
玩家点击"进入养成"
      │
      ▼
Evt_SwitchSystemDomain { DomainType = Cultivation }
      │
      ▼
SwitchSystemDomainEvent.Handle()
  ├─ Prepare()           ─ 准备检查
  ├─ ExitPrevDomain()    ─ 退出当前作用域
  ├─ ShowLoading()       ─ 显示 Loading 界面
  ├─ LoadResources()     ─ 预加载资源包
  ├─ SwitchDomain()      ─ 切换到新作用域
  ├─ HideLoading()       ─ 隐藏 Loading
  └─ EnterNewDomain()    ─ 进入新作用域，打开 UI
```

### 2.5 网络层架构

所有 C/S 通信采用 **Protobuf + 自定义 RPC** 方案：

- 请求：`SendAsync<TResp>(cmdId, req)` 发送请求并等待回包
- 通知：`MessageHandler<Scene, TMsg>` 注册服务端推送消息处理器
- 数据转换：`ServerDataConverter` 统一负责协议 → 本地数据结构的转换

### 3.2 SystemDomain 基类（SystemDomainHandler.cs）

```csharp
// X:\UnityProj\Assets\Scripts\Hotfix\Function\GamePlay\SystemDomain\SystemDomainHandler.cs
public abstract class SystemDomainHandler : ISystemDomainHandler
{
    public ESystemDomainType Domain { get; set; }
    public SystemDomainComponent Owner { get; set; }

    // 收集该 Domain 需要预加载的资源
    public virtual void GatherResources(List<string> lstRes, List<string> lstOther) { }

    // 异步预加载额外资源
    public virtual async ETTask PreloadExtraResources()
    {
        await ETTask.CompletedTask;
    }

    // 进入该 Domain 时的前置处理
    public virtual async ETTask EnterPreprocess(SystemDomainSwitchContext context)
    {
        await ETTask.CompletedTask;
    }

    // 退出该 Domain 后的后置处理（默认关闭所有 UI 面板）
    public virtual async ETTask ExitPostprocess(SystemDomainSwitchContext context)
    {
        await ETTask.CompletedTask;
        EventSystem.Instance.Publish(Owner.ClientScene(),
            new Evt_CloseAllUIPanelExceptOne(context.LoadingPanelName));
    }
}
```

### 3.3 场景切换事件（SceneSwitchEvent.cs 节选）

```csharp
// 场景切换的完整生命周期管道
public override async ETTask Handle(Scene scene, Evt_SwitchScene a)
{
    var (success, context) = await Prepare(scene, a);
    if (!success)
    {
        SceneChangeHelper.BackToBootScene(scene).Coroutine();
        return;
    }

    // 严格有序的异步管道
    await ExitPrevSystemDomainPreprocess(context);
    var loadingOptions = await ShowLoading(scene, a);
    await ExitPrevSystemDomainPostprocess(context);
    await LoadAndEnterScene(scene, a, loadingOptions);
    await LoadDependentResources(scene, a);     // 多轮资源加载
    await LoadDependentResources2(scene, a);
    await LoadDependentResources3(scene, a);
    await LoadDependentResources4(scene, a);
    await LoadExtraResources(scene, a);
    await SwitchSystemDomain(context);
    await EnterNextSystemDomainPreprocess(context);
    await HideLoading(scene, a, loadingOptions);
    await EnterNextSystemDomainPostprocess(context);
    await Run(scene, a);
}
```

## 4. 设计亮点

### 4.1 热更新友好的 ECS 架构
代码逻辑全部在 Hotfix 层，System 类全部是**静态扩展方法**，通过 `this ComponentType self` 作为第一参数操作组件。热更新后无需重启游戏，新的业务逻辑立即生效。

### 4.2 Pipeline 模式的玩法切换
场景切换和系统域切换都采用**严格有序的异步管道**，每一步都是独立的 `await` 调用，便于在任意步骤插入新逻辑（如 AB 测试），且任何一步失败都能安全回滚。

### 4.3 事件总线解耦
模块间通过 `EventSystem.Instance.Publish` 发布事件，接收方用 `[Event]` 特性注册，发布者无需了解接收者的存在，实现了**零耦合的跨模块通信**。

### 4.4 配置表驱动
所有游戏数值（副本配置、养成日程配置、角色配置等）均从 `CfgManager.tables.Tb*` 读取，策划修改 Excel 后自动生成 C# 配置类，**零代码改动即可调整游戏数值**。

## 5. 常见问题与最佳实践

### Q1: 如何新增一个玩法大区域（Domain）？
1. 在 `ESystemDomainType` 枚举中添加新值；
2. 创建继承自 `SystemDomainHandler` 的处理类，打上 `[SystemDomainBinding(ESystemDomainType.YourDomain)]` 特性；
3. 在 `GatherResources` 中填写该区域需要预加载的资源；
4. 发布 `Evt_SwitchSystemDomain { DomainType = YourDomain }` 即可触发切换。

### Q2: ET 框架的 Component 什么时候需要用 [FriendOf]？
`[FriendOf(typeof(XxxComponent))]` 是 ET 的访问控制特性，表示当前 System 类可以访问该 Component 的私有/内部字段。如果你发现编译报"访问权限不足"，就需要在对应的 System 文件顶部加上这个特性。

### Q3: 异步方法 ETTask 和 async/await 是什么关系？
`ETTask` 是 ET 框架提供的轻量级异步任务，类似 UniTask，基于 ValueTask 优化内存分配。使用规则和标准 C# async/await 完全一致，只需把 `Task` 替换成 `ETTask` 即可。

### Q4: 收到服务端数据后为什么要走 ServerDataConverter？
服务端用 Protobuf 传输数据（`xxxMsg` 结构），客户端用自定义的本地数据结构（`xxxData`）。`ServerDataConverter` 是唯一的转换层，这样当协议变更时，只需修改 Converter，不影响其他业务代码。

### Q5: 如何调试事件流？
在任意 Event 类的 `Handle` 方法入口打断点，或在 `EventSystem.Instance.Publish` 调用处打断点。生产环境下可通过 `Log.LogInfo("[模块][网络] ...")` 风格的日志追踪调用链。

## 6. 小结

本项目的玩法系统以 **ET ECS + 热更新** 为基础，以 **SystemDomain 作用域管理** 为骨架，以 **事件总线** 为神经系统，构建了一套可扩展、可热更、可调试的游戏玩法框架。理解了这套总架构，就能快速定位和开发任意子系统。
