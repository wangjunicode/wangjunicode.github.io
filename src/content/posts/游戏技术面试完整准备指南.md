## 一、C# 高频面试题

### 1.1 值类型 vs 引用类型

```
必须掌握的区分：

值类型（struct, enum, int, float, bool 等）：
  存储在栈上（或包含它的对象内部）
  赋值是值拷贝
  
引用类型（class, interface, delegate 等）：
  对象本身存在堆上
  变量存储的是引用（堆上对象的地址）
  赋值是引用拷贝（两个变量指向同一对象）

面试中的陷阱题：
  struct A { public int X; }
  A a = new A { X = 1 };
  A b = a;          // 值拷贝！b 是独立的副本
  b.X = 2;
  Console.WriteLine(a.X); // 输出：1（不是 2！）
  
  class B { public int X; }
  B c = new B { X = 1 };
  B d = c;          // 引用拷贝！d 和 c 指向同一对象
  d.X = 2;
  Console.WriteLine(c.X); // 输出：2（同一对象！）
```

### 1.2 GC 与内存管理

```
Q：什么时候会触发 GC？
A：当 Gen 0 堆内存达到阈值时（通常几 MB）自动触发
   也可以手动调用 GC.Collect()（不推荐在游戏中）

Q：如何避免 GC 压力？
A：
  1. 避免在热路径（Update）中分配
  2. 使用对象池复用对象
  3. 使用 struct 替代小的 class
  4. 避免装箱（Boxing）
  5. 避免 LINQ（在热路径中）
  6. 使用 StringBuilder 替代字符串拼接

Q：什么是 LOH（Large Object Heap）？
A：超过 85KB 的对象直接分配到 LOH
   LOH 不会被整理（fragmentation 问题）
   只在完整 GC（Gen 2 GC）时被收集
```

### 1.3 async/await 原理

```
Q：async/await 底层原理是什么？
A：编译器将 async 方法转换为一个状态机（State Machine）
   每个 await 点是状态机的一个"挂起点"
   不会真正阻塞线程，而是注册回调，await 完成后继续执行
   
Q：await 之后在哪个线程执行？
A：取决于 SynchronizationContext
   在 Unity 主线程：await 之后会回到主线程（Unity 有自定义的 SynchronizationContext）
   在线程池：await 之后可能在任意线程
   
Q：UniTask 比 Task 好在哪里？
A：
  1. 基于 struct（值类型），零 GC 分配
  2. 与 Unity PlayerLoop 集成（Update/FixedUpdate 等阶段）
  3. 自动在 GameObject Destroy 时取消（destroyCancellationToken）
  4. 性能更好，适合游戏高频调用场景
```

## 二、Unity 高频面试题

### 2.1 生命周期

```
Q：Awake 和 Start 的区别？
A：
  Awake：GameObject 激活后立即调用，顺序不保证
         用于初始化自身（不依赖其他组件）
  Start：第一帧 Update 之前调用，Awake 都执行完后才执行
         用于引用其他组件的数据（此时其他组件已 Awake 完成）
         
  最佳实践：
  Awake 中：自身初始化，查找自身的组件，注册自己到管理器
  Start 中：跨组件访问，读取其他组件数据

Q：OnEnable 和 Awake 的执行顺序？
A：Awake → OnEnable → Start（首次激活时）
   再次 Enable：只执行 OnEnable，不执行 Awake/Start
```

### 2.2 渲染与性能

```
Q：什么是 DrawCall？如何减少？
A：DrawCall = CPU 向 GPU 发出的一条绘制指令
   减少方式：
   1. Static Batching（静态合批）
   2. Dynamic Batching（动态合批，网格 < 300 顶点）
   3. GPU Instancing（相同 Mesh + 材质的大量对象）
   4. Sprite Atlas（UI 图集，减少 UI DrawCall）

Q：什么是 Overdraw？
A：同一像素被多次绘制（多个半透明层叠加）
   特别影响移动端性能（带宽密集型）
   减少方式：
   1. 减少半透明物体层数
   2. 不透明物体从前到后排序（Early-Z 剔除）
   3. UI 尽量减少重叠的半透明元素

Q：TBDR 架构对移动端 Shader 优化有什么影响？
A：TBDR 的核心优化：
   1. 使用 Clear 而不是 Load（节省内存带宽）
   2. 合并多个 Pass（利用 On-chip Memory 传递中间结果）
   3. 减少 RT 切换（每次切换都会 Flush）
   4. 减少 discard/alpha-test（会破坏 HSR 优化）
```

### 2.3 架构设计

```
Q：Singleton 有什么问题？怎么解决？
A：问题：
   1. 全局状态，难以测试
   2. 依赖方向不清晰
   3. 多场景加载时 DontDestroyOnLoad 对象可能泄露
   
   解决方案：
   1. 依赖注入（DI）：通过接口注入依赖，而非直接访问 Singleton
   2. 服务定位器模式（Service Locator）：集中管理服务，比散落的 Singleton 好
   3. ScriptableObject 数据传递：用 SO 替代 Singleton 传递数据

Q：什么是 ECS？与 OOP 相比有什么优势？
A：ECS = Entity（实体）+ Component（纯数据）+ System（纯逻辑）
   
   OOP 问题：对象随机分布在内存 → Cache Miss 严重 → 性能差
   
   ECS 优势：
   1. 组件数据连续存储（SOA，Cache 友好）
   2. 便于并行化（Job System + Burst Compiler）
   3. 解耦：数据与逻辑完全分离
   
   实际使用：只在性能瓶颈处使用 ECS，不是所有场景都适合
```

## 三、算法与数据结构

### 3.1 游戏开发常考算法

```
A* 寻路（必考）：
  原理：f(n) = g(n) + h(n)
  g = 从起点到当前节点的实际代价
  h = 当前节点到终点的启发式估计
  
  关键实现：
  - 开放集合（优先队列，按 f 值排序）
  - 关闭集合（已探索节点）
  - 父节点记录（用于回溯路径）
  
  优化技巧：
  - 预计算的导航网格（NavMesh）代替格子地图
  - Theta*（允许任意角度移动，路径更优）
  - 分层寻路（大地图用多层寻路）

四叉树/八叉树：
  用于空间查询（"附近50米内的所有敌人？"）
  
  原理：递归把空间分成4/8份
  每个节点存储该区域内的对象
  
  查询：只遍历与查询范围相交的节点 → O(log n) ~ O(n)

BVH（包围体层次）：
  用于射线检测、碰撞检测
  Unity Physics 内部使用 BVH
```

### 3.2 数据结构选型

```
List<T>：
  随机访问 O(1)，末尾增删 O(1)（均摊）
  中间增删 O(n)
  游戏中最常用
  
Dictionary<K,V>：
  增删查 O(1)（均摊）
  注意：不保证顺序，foreach 顺序不确定
  
Queue<T>：
  FIFO，入队出队 O(1)
  适合：消息队列、BFS 算法
  
PriorityQueue<T>（C# 6.0+）：
  最优先元素 O(1)，增删 O(log n)
  适合：A* 的开放集合、技能优先级

SortedDictionary/SortedList：
  有序字典，增删 O(log n)
  适合：需要按 key 顺序访问的场景
  
HashSet<T>：
  快速判断包含关系 O(1)
  适合：去重、快速查找
```

## 四、图形渲染面试题

### 4.1 渲染管线

```
Q：请描述一个顶点从模型空间到屏幕的变换过程
A：
  1. 模型空间（Local Space）
     → Model Matrix（移动/旋转/缩放）
  2. 世界空间（World Space）
     → View Matrix（摄像机变换的逆）
  3. 观察空间（Camera/Eye Space）
     → Projection Matrix（正交或透视投影）
  4. 裁剪空间（Clip Space）
     → 齐次除法（除以 w 分量）
  5. NDC（Normalized Device Coordinates，[-1,1]）
     → Viewport 变换
  6. 屏幕空间（Screen Space）

Q：前向渲染（Forward Rendering）和延迟渲染（Deferred Rendering）区别？
A：
  前向渲染：
    每个对象渲染一次，同时计算所有光照
    n个对象 × m个光源 → n×m 次着色器调用
    优点：简单，支持半透明，MSAA 简单
    缺点：大量光源时性能差
    
  延迟渲染：
    Pass 1：将几何信息写入 G-Buffer（法线/位置/材质）
    Pass 2：对屏幕上每个像素计算光照
    n个光源 → n次全屏 Pass（与对象数无关）
    优点：大量光源性能好
    缺点：不支持半透明，MSAA 复杂，内存多
    
  URP 的特殊方案：Render Feature + 自定义 Pass
```

## 五、行为面试题（Behavioral）

### 5.1 STAR 法则

```
STAR = Situation（情境）+ Task（任务）+ Action（行动）+ Result（结果）

面试中关于经历的问题，用 STAR 回答

例：
Q：说一次你在项目中解决过的技术难题

A（STAR 结构）：
  S：我们的游戏在中低端设备上帧率只有 15fps，产品不可接受
  T：我负责找出原因并优化到 30fps
  A：用 Profiler 分析发现主要问题是渲染模块：
     - DrawCall 超过 400 个
     - 大量碎片化的小纹理没有打 Atlas
     - 血条是独立 Canvas，每帧 Rebuild
     
     具体措施：
     - 合并场景纹理到 Atlas（DrawCall 从 400 减少到 180）
     - 血条移到独立 Canvas（减少 80% 的 Rebuild）
     - 对静态场景物件开启 Static Batching
     
  R：帧率从 15fps 提升到 35fps，覆盖了目标机型的 90%
     这个优化也形成了内部的性能优化规范文档
```

## 六、反问环节

```
面试最后通常有"你有什么问题想问我的？"

好的问题：
  ✅ "您认为这个职位未来6个月最重要的挑战是什么？"
  ✅ "团队技术栈和架构上有什么最近在演进的方向？"
  ✅ "您觉得在这个团队成长最快的工程师有什么共同特质？"

避免的问题：
  ❌ "工资能高一点吗？"（待遇问题最后谈）
  ❌ "可以早退吗？"（把关注点放在贡献，不是限制）
  ❌ "下午可以不加班吗？"（同上）
```

## 总结

面试成功的三要素：

1. **技术基础扎实**：理解原理，不只背答案
2. **项目经验丰富**：有真实解决过的问题，能用 STAR 讲清楚
3. **表达清晰**：思考过程说出来，即使不会也要展示思路
