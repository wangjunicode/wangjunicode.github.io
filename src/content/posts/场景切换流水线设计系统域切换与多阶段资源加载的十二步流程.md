# 场景切换流水线设计——系统域切换与多阶段资源加载的十二步流程

游戏场景切换（从大厅进入战斗、从战斗返回大厅、进入养成玩法）是整个游戏最复杂的操作之一。它涉及资源加载、系统初始化/销毁、UI切换……每一步都可能失败，顺序必须严格正确。

xgame项目的`SceneSwitchEvent`用一个清晰的**十二步流水线**来管理这个过程。

## 一、十二步流水线

```csharp
public override async ETTask Handle(Scene scene, Evt_SwitchScene a)
{
    try
    {
        // 步骤0：初始化本次切换所需数据
        Init(scene, ref a);
        
        // 步骤1：Prepare - 检查切换条件，获取目标系统域上下文
        var (success, context) = await Prepare(scene, a);
        if (!success)
        {
            // 失败时返回登录场景（安全兜底）
            SceneChangeHelper.BackToBootScene(scene).Coroutine();
            return;
        }
        _systemSwitchContext = context;
        
        // 步骤2：退出当前系统域（前处理）
        await ExitPrevSystemDomainPreprocess(context);
        
        // 步骤3：显示Loading界面
        var loadingOptions = await ShowLoading(scene, a);
        context.LoadingPanelName = loadingOptions.UIPath;
        
        // 步骤4：退出当前系统域（后处理）—— Loading显示后才完成退出
        await ExitPrevSystemDomainPostprocess(context);
        
        // 步骤5：加载并进入新场景（Unity场景切换）
        await LoadAndEnterScene(scene, a, loadingOptions);
        
        // 步骤6-9：分四阶段加载依赖资源
        await LoadDependentResources(scene, a);
        await LoadDependentResources2(scene, a);
        await LoadDependentResources3(scene, a);
        await LoadDependentResources4(scene, a);
        
        // 步骤10：加载额外资源
        await LoadExtraResources(scene, a);
        
        // 步骤11：切换系统域
        await SwitchSystemDomain(context);
        
        // 步骤12：进入新系统域（前处理）
        await EnterNextSystemDomainPreprocess(context);
        
        // 步骤13：隐藏Loading界面
        await HideLoading(scene, a, loadingOptions);
        
        // 步骤14：进入新系统域（后处理）—— Loading隐藏后完成进入
        await EnterNextSystemDomainPostprocess(context);
        
        // 步骤15：执行切换后的自定义逻辑
        await Run(scene, a);
    }
    catch (Exception e)
    {
        Log.Error(e); // 任何步骤失败，记录错误（可以扩展为跳转错误场景）
    }
}
```

## 二、Prepare：前置条件检查

```csharp
private async ETTask<(bool success, SystemDomainSwitchContext context)> Prepare(
    Scene scene, Evt_SwitchScene a)
{
    // 1. 目标系统域是否存在处理器
    var systemDomainComp = scene.GetComponent<SystemDomainComponent>();
    if (!systemDomainComp.TryGetDomainHandler(a.NextSystemDomain, out var handler))
    {
        Log.LogError("[SceneSwitch] 没有找到系统域处理器: {0}", a.NextSystemDomain);
        return (false, null);
    }
    
    // 2. 让目标系统域处理器做额外的准备（可以拒绝切换）
    var context = new SystemDomainSwitchContext { NextDomainType = a.NextSystemDomain };
    var prepared = await handler.Prepare(context);
    if (!prepared)
    {
        Log.LogInfo("[SceneSwitch] 目标系统域拒绝切换: {0}", a.NextSystemDomain);
        return (false, null);
    }
    
    return (true, context);
}
```

**为什么需要Prepare？**

目标系统域（如战斗系统）可能有自己的前置条件：
- 网络连接是否正常？
- 必要的资源是否已就绪？
- 是否已经有一局战斗在进行中（防止重复进入）？

Prepare失败时，`BackToBootScene`是最安全的兜底——直接回到登录场景，清空所有状态，让玩家重新登录。这避免了"卡在中间状态"导致的各种奇怪BUG。

## 三、ExitPre vs ExitPost：Loading前后的差异

```csharp
// Loading显示前的退出处理（ExitPreprocess）
await ExitPrevSystemDomainPreprocess(context);

// 显示Loading界面
var loadingOptions = await ShowLoading(scene, a);

// Loading显示后的退出处理（ExitPostprocess）
await ExitPrevSystemDomainPostprocess(context);
```

**为什么要把"退出"分成Before和After Loading两个阶段？**

- **ExitPreprocess（Loading显示前）**：做一些玩家能看到的退出准备，比如播放退出动画、保存一些数据
- **ExitPostprocess（Loading显示后）**：做真正的重量级销毁工作，比如卸载战斗场景、释放大量内存。这些操作可能导致短暂卡顿，但Loading已经显示，玩家不会感知到

同理，EnterPreprocess（Loading隐藏前）做初始化，EnterPostprocess（Loading隐藏后）做"进入后"的视觉效果（如入场动画）。

## 四、分阶段资源加载

```csharp
await LoadDependentResources(scene, a);   // 第1阶段
await LoadDependentResources2(scene, a);  // 第2阶段
await LoadDependentResources3(scene, a);  // 第3阶段
await LoadDependentResources4(scene, a);  // 第4阶段
```

**为什么分4个阶段而不是一次性加载？**

有依赖关系的资源必须按顺序加载：
- 第1阶段：基础资源（角色模型、场景基础）
- 第2阶段：依赖第1阶段的资源（骨骼绑定、材质引用）
- 第3阶段：依赖第2阶段的资源（动画控制器）
- 第4阶段：最后的配置数据（需要角色实例才能初始化的数据）

如果一次性并行加载，会出现"A的材质用到了B的Shader，但B还没加载完"的依赖问题。

每个阶段内部的资源可以**并行加载**，阶段间**串行等待**，兼顾了效率和正确性。

## 五、Loading界面的三层选择逻辑

```csharp
protected override LoadingOptions GetLoadingOptions(Scene scene, Evt_SwitchScene evt)
{
    // 优先级1：调用方直接指定Loading面板（最高优先级）
    if (!string.IsNullOrEmpty(evt.LoadingPanelName))
        return new LoadingOptions { UIPath = evt.LoadingPanelName, ... };
    
    // 优先级2：场景的默认Loading面板
    var sceneComp = scene.GetComponent<SceneComponent>();
    var loadingPanelName = sceneComp.GetLoadingPanelName();
    
    // 优先级3：目标系统域的Loading可以覆盖场景的Loading
    var systemDomainComp = scene.GetComponent<SystemDomainComponent>();
    if (systemDomainComp.TryGetDomainHandler(nextDomainType, out var nextHandler))
    {
        var loadingConfig = nextHandler.GetLoadingName(_systemSwitchContext);
        if (loadingConfig.OverrideSceneLevel)
            loadingPanelName = loadingConfig.UIPath; // 系统域覆盖场景
    }
    
    // 兜底：通用Loading
    if (string.IsNullOrEmpty(loadingPanelName))
        loadingPanelName = PanelNameDefine.CommonLoadingPanel;
    
    return new LoadingOptions { UIPath = loadingPanelName, ... };
}
```

三层优先级：
1. 调用方明确指定（事件里带的参数）
2. 目标系统域覆盖（比如PVP战斗有专属的战斗Loading界面）
3. 场景默认（场景配置里的默认Loading）

这种"覆盖链"设计让代码可以在不同层次自定义Loading，而不需要所有调用方都指定Loading面板。

## 六、ResourcesGathering的分离设计

```csharp
protected override void GetDependentRes(Scene scene, Evt_SwitchScene evt, 
    List<string> lstRes, List<string> lstOther)
{
    // 资源列表收集
    var systemDomainComp = scene.GetComponent<SystemDomainComponent>();
    if (systemDomainComp.TryGetDomainHandler(nextDomainType, out var nextHandler))
        nextHandler.GatherResources(lstRes, lstOther);
}
```

**"收集资源"和"加载资源"分离**：

`GetDependentRes`只收集资源路径（同步方法），`LoadDependentResources`系列方法负责实际加载（异步）。

分离的好处：可以先收集所有需要加载的资源，然后对资源去重（同一资源被多个系统引用只加载一次），计算总资源量（用于进度条百分比），最后批量并行加载。

## 七、系统域（SystemDomain）的概念

`SystemDomainSwitchContext`和`SystemDomainComponent`贯穿整个流水线，"系统域"是什么？

系统域（SystemDomain）是游戏不同"大功能区"的抽象：
- `LobbyDomain`：大厅系统（主界面、商城、养成等）
- `BattleDomain`：战斗系统（帧同步战斗）
- `CultivationDomain`：养成玩法系统

每个系统域有独立的资源集合、初始化逻辑和生命周期。切换场景时，必须完整地退出当前系统域，进入新系统域。

这种分层设计让代码组织清晰：想知道"进入战斗需要加载什么资源"，看`BattleDomain.GatherResources()`就够了，不需要在场景切换的主流程里翻找。

## 八、总结

场景切换流水线的设计精华：

| 设计决策 | 解决的问题 |
|----------|---------|
| 十二步串行流水线 | 保证每步完成后再进行下一步 |
| Prepare失败→回登录 | 防止卡在中间状态 |
| Loading前后分两阶段 | Loading后再做重量级操作 |
| 四阶段资源加载 | 处理资源间的依赖关系 |
| 三层Loading选择 | 不同层次可以自定义Loading |
| GatherResources分离 | 资源去重和进度计算 |
| SystemDomain抽象 | 每个大功能区自管理资源和生命周期 |

对新手来说，这套流水线设计最重要的启示：**把复杂的流程拆分成有序的步骤，每步职责单一，异常处理清晰**。复杂不可怕，怕的是混乱——清晰的步骤让代码易于理解、维护和扩展。
