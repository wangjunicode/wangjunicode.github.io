# 12 战斗演出与表现系统

> 本文介绍战斗中的演出（Performance）与表现（Presentation）系统，讲解如何将逻辑层的战斗结果转化为玩家看到的动画、特效、摄像机表现，以及特殊大回合（SpChapter）等演出机制。

---

## 1. 系统概述

战斗演出与表现系统负责将**逻辑层的"发生了什么"**转化为**表现层的"看到了什么"**：
- 技能的攻击动画播放
- 命中时的特效、打击暂停（顿帧）
- Root Motion 驱动的位移
- 破防后的特殊大回合（SpChapter）演出
- 摄像机震动、锁定目标跟随
- 跳字（伤害数字）
- 单位暂停敌方（CLIP_PauseEnemy）制造戏剧性

### 1.1 表现层与逻辑层的解耦原则

**关键原则：逻辑层不直接调用表现层，通过事件通信。**

```
逻辑层（战斗计算）
    │
    └→ EventSystem.Publish(evt)   ← 发布事件
         │
         └→ 表现层监听者（View Handler）
               ├── 播放动画（Animator）
               ├── 生成特效（ParticleSystem）
               ├── 摄像机震动
               └── 跳字 UI 显示

优点：
- 逻辑层不引用任何 UnityEngine 的表现类
- 逻辑层可以在无渲染环境下运行（服务端验证、单元测试）
- 表现层可以独立迭代，不影响逻辑
```

### 1.2 演出的核心机制

| 机制 | 说明 |
|------|------|
| Root Motion | 动画位移驱动角色移动（技能前摇位移） |
| 顿帧（FreezeFrame） | 命中时短暂冻结逻辑更新，增强打击感 |
| CLIP_PerformRange | 标记技能的表演区间（影响受击响应方式） |
| CLIP_PauseEnemy | 暂停所有敌方单位（演出专用） |
| CLIP_SpChapter | 特殊大回合区间（破防追打演出） |
| CLIP_BattleEvent | 在 Timeline 中触发战斗点事件 |
| 换人演出（OnStageShowMode） | 角色上下场的特殊显示模式 |

## 2. 架构设计

### 2.1 表现层的事件监听架构

```
逻辑层事件（EventDef 命名空间）
├── Evt_UnitWrap          ← 单位移动（Teleport）
├── NumericChange         ← 数值变化（血量变化 → 血条更新）
├── Evt_UnitPauseStateChange ← 暂停状态变化
├── ChangeScale           ← 缩放变化
└── 自定义战斗事件（通过 BPDef 事件 ID）

表现层 Handler（监听上述事件）
├── UnitViewComponent     ← 单位视觉组件（动画/模型）
├── HPBarComponent        ← 血条 UI
├── JumpTextComponent     ← 跳字显示
├── CameraComponent       ← 摄像机控制
├── EffectComponent       ← 特效管理
└── AudioComponent        ← 音效（Wwise）
```

### 2.2 Root Motion 系统

Root Motion（根运动）让角色的移动由动画的骨骼位移数据驱动，而非外力设置位置。这使得技能动作中的位移更自然、更符合动画的"感觉"。

**确定性 Root Motion 的实现：**
```
动画制作时
    └→ 导出 RootMotionData（JSON格式）
         ├── 每帧的位移量 displacement[n]
         └── 每帧的旋转量 rotation[n]

逻辑层（CLIP_RootMotion）
    └→ GetSampleData(time, ref displacement, ref rotation)
         └→ 读取预烘焙的 RootMotionData
              └→ 应用位移（确定性定点数计算）
                   └→ Unit.LerpTransform(pos, rot, scale)
```

**Motion Warping（位移扭曲）：**
当技能需要让角色"追踪"到目标位置时：
```
motionWarping = true 时：
    在 [startTime, endTime] 时间窗口内
    原始 Root Motion 方向 → 重新映射到指向 CurPredictPos 的方向
    保持原始位移总量不变，只改变方向
```

### 2.3 顿帧（FreezeFrame）机制

顿帧是格斗游戏中增强打击感的经典技术：命中瞬间，角色的逻辑更新短暂停止，同时配合慢动作特效。

```csharp
// Unit 上的顿帧字段
public bool WillFreeze { get; set; }          // 下帧是否开始冻结
public int WillFreezeFrames { get; set; }     // 计划冻结的帧数
private int m_freezeFrames;
public int FreezeFrames { get => m_freezeFrames; set => m_freezeFrames = value; }

public bool Freeze => this.FreezeFrames > 0;

// 在 FixedUpdate 中：
// if (Freeze) { FreezeFrames--; return; }  // 冻结期间跳过逻辑更新
// 攻击方不冻结，受击方冻结
// → 攻击方继续流畅运动，受击方像"定格"了一瞬间
// → 视觉上产生强烈的打击感
```

### 2.5 特殊大回合（SpChapter）

`CLIP_SpChapter` 是破防成功后进入的特殊演出阶段：
```csharp
[Name("特殊大回合区间")]
[MemoryPackable]
public partial class CLIP_SpChapter : DurationActionClip
{
    // 特殊大回合 ID 列表（可同时关联多个特殊剧本）
    [fsSerializeAs("ConditionList")]
    [MemoryPackOrder(0)]
    public List<int> spIDs = new();
}
```

**SpChapter 的完整流程：**
```
攻击方破防成功
    └→ 目标进入 PerformHurtState（演出受击状态）
         └→ 攻击方技能 Timeline 进入 CLIP_SpChapter 区间
              └→ 系统执行特殊大回合逻辑：
                   ├── 摄像机切换到特殊角度
                   ├── 时间减慢（Time Scale 调整）
                   ├── 触发 BPDef.SpChapterStartEvtID 事件
                   ├── 攻击方执行连续追打技能
                   ├── 每次追打触发特殊受击动画
                   └── 结束后触发 BPDef.SpChapterEndEvtID
                        └→ 双方恢复正常状态
```

- 破防演出开始时暂停所有敌方（制造"时间静止"效果）

### 3.4 Unit 的暂停状态 —— EPauseFlag 与 演出联动

```csharp
// Unit 的多标志位暂停系统
public enum EPauseFlag
{
    common = 0,      // 战斗逻辑暂停（如全局暂停）
    puppetShow = 1,  // 演出暂停（表演期间暂停敌方逻辑）
}

// 设置演出暂停
public void SetBattlePause(bool value, EPauseFlag flag = EPauseFlag.common)
{
    bool oldValue = BattlePause;
    if (value)
        battlePause.Add(flag);
    else
        battlePause.Remove(flag);
    
    if (BattlePause != oldValue)
    {
        // 通知表现层：暂停状态改变（暂停时停止动画更新等）
        EventSystem.Instance.Publish(this.CurrentScene(),
            new Evt_UnitPauseStateChange() { unit = this });
    }
}
```

## 4. 表现层关键技术详解

### 4.1 逻辑帧与渲染帧的分离

```
逻辑层（FixedUpdate，30fps）
    │
    └→ 更新 Unit.position（定点数）
         └→ 不直接更新 Transform

渲染层（Update，60+fps）
    └→ 读取 Unit.position（上一个逻辑帧）和 Unit.nextPosition（下一个逻辑帧）
         └→ 插值（Lerp）得到当前渲染帧的位置
              └→ 更新 UnityEngine.Transform.position

效果：即使逻辑帧是 30fps，角色移动也能渲染出 60fps+ 的流畅效果
```

### 4.2 InForceTick 追帧模式

```csharp
// 断线重连时，追帧模式下跳过表现层更新
public bool InForceTick { get; set; } = false;

// 表现层 Handler 中的检查：
if (unit.InForceTick) return;  // 追帧时不播放动画/特效
                                // 只更新逻辑状态，快速追上当前帧
```

追帧时以最快速度（可能是正常速度的 10x）重放所有帧，期间禁用所有渲染相关的操作，追帧完成后 `InForceTick = false`，恢复正常表现。

### 4.3 条件性演出派生（CLIP_ConditionalDerive）

```csharp
// 根据条件派生不同的技能分支（如：普通命中 vs 破防命中走不同动画路径）
[Name("条件派生")]
[MemoryPackable]

    // 条件 Task 列表（每个条件对应一个派生分支）
    // 条件满足时 → 执行对应的派生技能分支
    // 条件不满足时 → 继续主 Timeline
}
```

### 4.4 摄像机演出设计

摄像机演出（推镜、特写、震动）通过以下机制实现：

```
战斗点事件触发
    └→ 表现层 CameraController 监听
         ├── SpChapterStartEvtID → 切换特殊大回合摄像机
         ├── OnCritical          → 短暂慢动作（TimeScale）
         └── CharacterEnterEvtID → 角色入场演出镜头

CLIP_OnPuppetEvent → 傀儡控制器（Puppet Show）
    └→ 镜头切换到预设的剧情机位
         └→ 播放完成后恢复战斗摄像机
```

## 5. 演出流程完整示例

### 5.1 破防演出完整时序

```
攻击方释放破防技能
    │
    ├─ [0~3帧] CLIP_RootMotion（冲向目标位移）
    │
    ├─ [3~6帧] CLIP_SetAttackCollider（攻击框激活）
    │           └→ 碰撞检测命中目标
    │                └→ DoBreakDefense = true
    │                └→ FSM 注入 Hurt_BreakDefense 事件
    │                     └→ 目标 FSM → BreakDefense → PerformHurtState
    │
    ├─ [6帧]   CLIP_PauseEnemy（暂停所有敌方 0.5 秒）
    │           └→ 逻辑层：目标 SetBattlePause(true, puppetShow)
    │           └→ 表现层：摄像机推近特写
    │
    ├─ [6~20帧] CLIP_SpChapter（特殊大回合区间）
    │            └→ BPDef.SpChapterStartEvtID 触发
    │                 └→ 攻击方获得额外出手权
    │                 └→ 执行追打连续技能
    │                 └→ 每次追打触发受击特效 + 跳字
    │
    └─ [20帧] 结束
              └→ CLIP_PauseEnemy 退出 → 恢复目标
              └→ BPDef.SpChapterEndEvtID 触发
              └→ 技能完成，FSM 返回 Idle
```

## 6. 设计亮点

### 6.1 表现层与逻辑层彻底隔离
所有表现类（Animator、ParticleSystem、Camera）都只存在于表现层，逻辑层代码不引用任何 `UnityEngine` 表现相关 API（部分文件有 `#if UNITY_EDITOR` 的编辑器辅助代码，但不在运行时路径上）。这让战斗逻辑可以在服务端（无 Unity 渲染环境）运行，用于服务端结果验证。

### 6.2 EPauseFlag 的多标志位暂停
演出暂停使用独立的 `EPauseFlag.puppetShow` 标志，与战斗逻辑暂停（`EPauseFlag.common`）完全分离。这样在演出结束后只清除演出标志，不影响可能同时存在的战斗逻辑暂停，避免了状态互相干扰的 Bug。

### 6.3 跳字的灵活配置
跳字不是全局统一样式，而是在每个 `ColliderInfo` 上单独配置，允许不同技能、不同攻击框有不同的跳字效果。例如：普通攻击小跳字，暴击大跳字+特殊颜色，背刺从背后方向出字，AOE 技能多个跳字同时出现。

### 6.4 Root Motion + Motion Warping 的精准控制
`CLIP_RootMotion` 支持 Motion Warping，让技能的位移自动追踪目标。这解决了格斗游戏常见的痛点：技能前摇动画中角色朝某个固定方向冲刺，如果目标移动了就会打空。Motion Warping 在时间窗口内动态调整位移方向，保持技能的命中率。

### 6.5 特殊大回合（SpChapter）的戏剧性
SpChapter 机制是这套战斗系统的核心特色之一。破防成功后，敌方进入特殊受击状态，配合 CLIP_PauseEnemy 的定格效果、摄像机的特写推镜、连续追打的视觉冲击，营造出电影化的战斗演出体验，与普通格斗游戏的攻击动画有本质区别。

## 7. 常见问题与最佳实践

### Q1：表现层的 Handler 如何知道逻辑事件发生了？
**A**：通过 `EventSystem.Instance.Subscribe<T>()` 订阅事件（如 `NumericChange`），逻辑层 `Publish` 事件后，所有订阅者的回调自动被调用。表现层 Component 在 `Awake` 时订阅，`Destroy` 时取消订阅。

### Q2：追帧（InForceTick）时为什么要跳过表现层？
**A**：追帧的目标是尽快让客户端的逻辑状态与服务端同步，可能需要在 1 秒内执行几百帧。如果每帧都播放动画/特效，一是性能不允许，二是表现上会非常混乱。跳过表现层后，只有逻辑状态在快速推进，追帧完成后表现层再从当前正确状态开始渲染。

### Q3：FreezeFrames（顿帧）是在逻辑层还是渲染层？
**A**：逻辑层。`FreezeFrames > 0` 时，`Unit.Freeze = true`，逻辑层的 FixedUpdate 会跳过该单位的更新（`if (unit.Freeze) { unit.FreezeFrames--; return; }`）。这是帧同步安全的，所有客户端的冻结帧数相同，渲染层通过事件知道单位被冻结，配合慢动作特效。

### Q4：CLIP_BattleEvent 和直接在代码里 Publish 事件有什么区别？
**A**：`CLIP_BattleEvent` 允许在技能 Timeline 的**任意帧**精确触发战斗事件，由策划在编辑器中控制触发时机。直接在代码里 Publish 则是固定在某个代码执行点触发。前者的优点是策划可以不改代码地调整事件触发时机（如"第 8 帧触发攻击开始事件"改为"第 6 帧"只需在 Timeline 上拖拽 Clip 位置）。

### Q5：换人演出如何保证逻辑和表现的同步？
**A**：换人是逻辑层先执行（单位进出场、属性变化），然后通过 `CharacterEnterEvtID`/`CharacterExitEvtID` 事件通知表现层。表现层响应事件播放入场/退场动画。逻辑和表现是异步的——逻辑已经换完了，表现还在播放入场动画，这是游戏设计上的有意为之（视觉流畅性优先）。

### Q6：跳字的位置是如何计算的？
**A**：跳字位置基于 `ColliderInfo.attackPoint`（攻击挂点坐标）和 `AttackInfo.DefenderHitPos`（受击点坐标）的插值，通常显示在被击者附近。`FollowTargetDuration` 配置跳字跟随目标移动的时长，让跳字看起来"贴在"被击者身上，而不是浮在固定世界坐标位置。
