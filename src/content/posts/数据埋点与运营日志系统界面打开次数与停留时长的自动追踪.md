# 数据埋点与运营日志系统——界面打开次数与停留时长的自动追踪

"玩家最爱去哪个界面？"、"角色详情页平均停留多久？"、"有多少玩家在看完教程后立刻退出了游戏？"……这些数据是游戏运营的核心指标，需要通过埋点系统来收集。

VGame项目的`DataAnalysisComponent`实现了一套轻量级的埋点系统，专注于界面行为数据的采集。

## 一、初始化时预定义追踪目标

```csharp
public class DataAnalysisComponentAwakeSystem : AwakeSystem<DataAnalysisComponent>
{
    protected override void Awake(DataAnalysisComponent self)
    {
        self.OpenPanelTime = 0;
        
        // 追踪"打开次数"的界面列表（初始值为0）
        self.OpenPanelDic = new Dictionary<string, int>()
        {
            { PanelNameDefine.ReflectionCardBagPanel, 0 },  // 心得卡背包
            { PanelNameDefine.CharacterUnlockPanel, 0 },    // 角色解锁界面
            { PanelNameDefine.BagPanel, 0 },                 // 背包
            { PanelNameDefine.AchievementPanel, 0 },         // 成就
            { PanelNameDefine.AssistCardPanel, 0 },          // 协助卡
            { PanelNameDefine.AssistCardDetailPanel, 0 },    // 协助卡详情
            { PanelNameDefine.MemoryCardSelectPanel, 0 },   // 记忆卡选择
            { PanelNameDefine.DormEntraPanel, 0 }            // 宿舍入口
        };
        
        // 追踪"从哪里跳转过来"的界面（每个界面记录来源分布）
        self.OriginToPanel = new Dictionary<string, Dictionary<string, int>>()
        {
            { PanelNameDefine.CharacterDetailsPanel, new Dictionary<string, int>() }, // 角色详情
            { PanelNameDefine.NewTeamPresetEditorPanel, new Dictionary<string, int>() }, // 芯盒
            { "Pnl_StyleDetail", new Dictionary<string, int>() },    // 技能详情
            { "UnLockCharacterTip", new Dictionary<string, int>() },  // 角色解锁提示
        };
        
        // 追踪"停留时长"的界面列表（时间戳，0表示未开始计时）
        self.PanelTimeDic = new Dictionary<string, long>()
        {
            { "PVPReflectionCardShopPanel", 0 },           // PVP心得卡商店
            { "PVEReflectionCardShopPanel", 0 },           // PVE心得卡商店
            { PanelNameDefine.NewTeamPresetEditorPanel, 0 }, // 芯盒
            { PanelNameDefine.CharacterDetailsPanel, 0 },   // 局外角色详情
            { PanelNameDefine.CharacterUnlockPanel, 0 },    // 角色解锁
            { PanelNameDefine.BattlePreparePanel, 0 },      // 战前准备
            { PanelNameDefine.DormEntraPanel, 0 }           // 宿舍
        };
    }
}
```

**预先定义追踪目标的设计意图**：

不是所有界面都需要追踪，只追踪"运营关心"的界面。在Awake里明确列出追踪列表，有几个优点：

1. **一目了然**：代码即文档，看Awake就知道追踪了哪些指标
2. **防止意外追踪**：只有在这里注册了的界面才会被追踪，不会因为代码bug把不该追踪的界面误计入
3. **初始值为0**：字典初始化时就赋值，之后无需检查key是否存在再创建

## 二、打开次数统计

```csharp
public static void OpenPanelNumberLog(this DataAnalysisComponent self, string panelName)
{
    if (self.OpenPanelDic == null || self.OpenPanelDic.Count <= 0) return;
    
    if (self.OpenPanelDic.TryGetValue(panelName, out var num))
    {
        num += 1;
        self.OpenPanelDic[panelName] = num;
        
        // 日志输出（开发阶段调试用）
        string logArgs = ZString.Format(
            "[DataAnalysis] 打开界面:{0},打开次数：{1}",
            panelName, self.OpenPanelDic[panelName]);
        Log.LogInfo(logArgs);
    }
}
```

**TryGetValue + 更新的模式**：

如果用`ContainsKey` + 访问两步操作，会有两次哈希查找。`TryGetValue`一次查找同时检查存在性并获取值，更高效。

这里有个小BUG：`num += 1`修改的是局部变量`num`，而不是字典里的值。`self.OpenPanelDic[panelName] = num`正确地回写了，但写法可以更简洁：

```csharp
// 更简洁的写法
self.OpenPanelDic[panelName]++;
```

不过两种写法效果相同，只是风格差异。

## 三、停留时长追踪

```csharp
// 界面打开时：记录开始时间
public static void OpenPanelTimeLog(this DataAnalysisComponent self, string panelName)
{
    if (!self.PanelTimeDic.TryGetValue(panelName, out _)) return; // 不在追踪列表则跳过
    
    self.PanelTimeDic[panelName] = TimeInfo.Instance.ClientNow(); // 记录打开时间戳（毫秒）
}

// 界面关闭时：计算停留时长并上报
public static void ClosePanelTimeLog(this DataAnalysisComponent self, string panelName)
{
    if (!self.PanelTimeDic.TryGetValue(panelName, out var openTime)) return;
    if (openTime <= 0) return; // 未记录过打开时间，跳过
    
    var closeTime = TimeInfo.Instance.ClientNow();
    var stayDuration = closeTime - openTime; // 停留时长（毫秒）
    
    self.PanelTimeDic[panelName] = 0; // 重置计时
    
    // 上报给运营系统
    string logArgs = ZString.Format("stayTime={0}ms", stayDuration);
    self.SubmitOssLog((int)EDataAnalysisLogType.PanelStayTime, 
        ZString.Format("{0}|{1}", panelName, logArgs)).Coroutine();
}
```

**`openTime <= 0`的防御**：如果玩家打开界面时来不及记录就关闭了（系统初始化时机问题），`PanelTimeDic`的值为0，跳过上报。

**`.Coroutine()`的非阻塞上报**：上报是网络请求，不需要等待结果再继续，用`.Coroutine()`启动后忘记。

## 四、来源追踪（Funnel分析）

```csharp
// 记录从哪个界面跳转来的
public static void OriginToPanelLog(this DataAnalysisComponent self, string fromPanel, string toPanel)
{
    if (!self.OriginToPanel.TryGetValue(toPanel, out var originDic)) return;
    
    if (!originDic.TryGetValue(fromPanel, out var count))
    {
        originDic[fromPanel] = 1; // 首次从这个来源来
    }
    else
    {
        originDic[fromPanel] = count + 1; // 再次从这个来源来
    }
}
```

`OriginToPanel`的数据结构是：`目标界面 → (来源界面 → 次数)`的嵌套字典。

**实际应用场景**：

假设`CharacterDetailsPanel`（角色详情）的来源字典积累了：
```
"CharacterListPanel" → 150次
"BattlePreparePanel" → 80次
"TaskRewardPanel" → 30次
```

这告诉运营：大多数用户是从角色列表直接进入角色详情（150次），其次是从战前准备界面进入（80次），从任务奖励界面进入最少（30次）。如果运营想增加角色详情的访问，优化角色列表入口效果最明显。

这就是**漏斗分析（Funnel Analysis）**的数据基础。

## 五、OSS日志上报

```csharp
public static async ETTask<bool> SubmitOssLog(
    this DataAnalysisComponent self, int logType, string logArgs)
{
    // 禁止上报空内容
    if (string.IsNullOrEmpty(logArgs)) return false;
    
    Log.LogInfo("[DataAnalysis][Network] 协议 {0}({1}) 请求开始", ...);
    
    var req = new ZoneCientOssLogReq
    {
        Type = logType,   // 日志类型（枚举值）
        LogArgs = logArgs // 日志内容（JSON或键值对字符串）
    };
    
    var nr = await self.ClientScene().GetComponent<NetworkComponent>()
        .SendAsync<ZoneCientOssLogResp>(
            (uint)ZoneClientCmd.ZoneCsClientOssLog, req, false); // false=非关键请求
    
    if (nr.IsCompleteSuccess)
    {
        Log.LogInfo("[DataAnalysis][Network] 协议 {0}({1}) 接收成功", ...);
        return true;
    }
    
    Log.LogError("[DataAnalysis][Network] 协议 {0}({1}) 接收失败", ...);
    return false;
}
```

**`false`参数（非关键请求）**：

`SendAsync`的第三个参数表示是否是"关键请求"（影响游戏逻辑的）。埋点上报是非关键的——即使上报失败，不影响游戏体验。非关键请求在网络繁忙时可以被降级处理（比如允许超时重试次数少）。

**logArgs的格式**：使用`"|"`分隔字段，例如：
```
"CharacterDetailsPanel|stayTime=12340ms"
"AssistCardPanel|openCount=5"
```

服务端解析这种格式，入库后供BI（商业智能）分析。

## 六、DataAnalysisEvtHandler：事件自动触发

```csharp
// 监听界面打开事件，自动记录
[Event(SceneType.Client)]
public class OnPanelOpenDataAnalysis : AEvent<Evt_OnPanelOpen>
{
    protected override void Run(Scene scene, Evt_OnPanelOpen args)
    {
        var dataAnalysis = scene.GetComponent<DataAnalysisComponent>();
        if (dataAnalysis == null) return;
        
        dataAnalysis.OpenPanelNumberLog(args.PanelName);  // 次数+1
        dataAnalysis.OpenPanelTimeLog(args.PanelName);    // 开始计时
        dataAnalysis.OriginToPanelLog(args.FromPanel, args.PanelName); // 来源记录
    }
}

// 监听界面关闭事件，自动计算停留时长
[Event(SceneType.Client)]
public class OnPanelCloseDataAnalysis : AEvent<Evt_OnPanelClose>
{
    protected override void Run(Scene scene, Evt_OnPanelClose args)
    {
        var dataAnalysis = scene.GetComponent<DataAnalysisComponent>();
        if (dataAnalysis == null) return;
        
        dataAnalysis.ClosePanelTimeLog(args.PanelName); // 计算时长并上报
    }
}
```

**业务代码零感知**：

UI框架在每次面板打开/关闭时自动发布事件，`DataAnalysisComponent`监听这些事件自动记录数据。业务代码（各个面板的逻辑）完全不需要知道埋点的存在，不需要主动调用任何埋点函数。

这是**横切关注点（Cross-Cutting Concern）**的优雅处理：埋点和业务逻辑正交，互不侵入。

## 七、总结

数据分析系统的设计展示了：

1. **预定义追踪目标**：白名单制，只追踪明确需要的界面
2. **三种追踪维度**：打开次数、停留时长、来源分布
3. **事件驱动零侵入**：UI事件自动触发埋点，业务代码不感知
4. **非关键上报**：网络优先级低，失败不影响玩家体验
5. **防御性处理**：上报前检查空值、未计时则跳过

对新手来说，"用事件监听实现横切关注点"是一种非常实用的设计技巧。日志、埋点、监控这些"非业务"功能，都可以通过监听业务事件来实现，而不需要在每个业务函数里手动调用。
