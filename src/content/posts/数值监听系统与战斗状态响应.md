# 数值监听系统与战斗状态响应

当队伍血量降为0时，需要触发死亡逻辑。当护盾降为0时，需要触发破防状态。当能量变化时，需要更新UI。

这些"数值变化→逻辑响应"的联动，如果都写在修改数值的地方，代码会变得非常耦合——改血量的代码里夹杂着死亡判断、UI更新、音效触发……

本文解析一套基于观察者模式的数值监听系统，理解如何用声明式方式将数值变化与业务响应解耦。

---

## 第一性原理：为什么要数值监听系统？

不好的写法：

```csharp
public static void TakeDamage(TeamEntity team, FP damage)
{
    var numeric = team.GetComponent<NumericComponent>();
    var oldHp = numeric.GetFinalValue(ENumericId.RealTeamHp);
    numeric.Modify(ENumericId.RealTeamHp, -damage);
    var newHp = numeric.GetFinalValue(ENumericId.RealTeamHp);
    
    // 在这里写死亡判断
    if (oldHp > 0 && newHp <= 0) team.TryDead();
    
    // 在这里通知UI
    UIManager.UpdateHpBar(team, newHp);
    
    // 在这里触发音效
    AudioManager.PlayHurtSound();
    
    // ...越来越多的耦合
}
```

问题很明显：`TakeDamage` 函数知道太多了，它需要了解 UI、音效、死亡系统……这违反了单一职责原则，也让单元测试变得困难。

**观察者模式的解法**：

```csharp
// TakeDamage 只关心数值修改本身
public static void TakeDamage(TeamEntity team, FP damage)
{
    team.GetComponent<NumericComponent>().Modify(ENumericId.RealTeamHp, -damage);
    // 就这一行，其他都由数值监听系统自动触发
}

// 各系统独立监听，各自处理
[NumericWatcher(SceneType.Current, (int)ENumericId.RealTeamHp)]
public class NumericWatcher_Hp : INumericWatcher
{
    public void Run(Entity entity, NumericChange args)
    {
        if (args.Old > 0 && args.New <= 0) team.TryDead();
        UIManager.UpdateHpBar(team, args.New);
    }
}
```

## NumericWatcher：监听器注册机制

```csharp
[AttributeUsage(AttributeTargets.Class)]
public class NumericWatcherAttribute : BaseAttribute
{
    public SceneType SceneType { get; }
    public int numericId { get; }
    
    public NumericWatcherAttribute(SceneType sceneType, int numericId)
    {
        this.SceneType = sceneType;
        this.numericId = numericId;
    }
}
```

通过特性（Attribute）声明监听关系，系统在启动时自动扫描所有标记了 `[NumericWatcher]` 的类，建立数值ID到监听器列表的映射表。

当任何数值发生变化时，系统自动查找对应的监听器列表并调用：

```csharp
// NumericComponent 中（伪代码）
private void OnValueChanged(ENumericId id, FP oldValue, FP newValue)
{
    var change = new NumericChange { Old = oldValue, New = newValue };
    
    // 查找所有监听这个 ID 的监听器
    var watchers = NumericWatcherComponent.GetWatchers((int)id);
    foreach (var watcher in watchers)
    {
        watcher.Run(this.Parent, change);
    }
}
```

## NumericWatcher_Hp：血量监听

```csharp
[NumericWatcher(SceneType.Current, (int)ENumericId.RealTeamHp)]
public class NumericWatcher_Hp : INumericWatcher
{
    public void Run(Entity entity, NumericChange args)
    {
        if (entity is TeamEntity)
        {
            var team = entity as TeamEntity;
            
            // 情况1：血量首次归零（触发死亡）
            if (args.Old > 0 && args.New <= 0)
            {
                team.TryDead();
            }
            // 情况2：血量持续为零（处理延迟死亡，如护盾还在时先不死）
            else if (args.Old <= 0 && args.New <= 0)
            {
                if (team.bDelayDead)
                {
                    team.TryDead();
                }
            }
            
            // 通知 UI 更新血条
            int max = entity.GetComponent<NumericComponent>()
                .GetPart(ENumericId.RealTeamHp, ENumericPart.Max).AsInt();
            entity.ClientScene().GetComponent<EventDispatcherComponent>()
                .FireEvent(new Evt_TeamHpChange() {
                    team = entity as TeamEntity,
                    OldHp = args.Old.AsInt(),
                    Hp = args.New.AsInt(),
                    Hp_Max = max
                });
        }
    }
}
```

### 细节：Old > 0 && New <= 0 的重要性

这个条件看似简单，实则关键：

- 只有"从有血量变为无血量"才触发 `TryDead`
- 如果血量已经是0，再次受到伤害扣减到更负的值，不应该再次触发死亡
- 如果是治疗，血量从负数（特殊情况）变为正数，也不触发死亡

`bDelayDead` 处理一个特殊场景：角色有护盾（Guard），血量为0但护盾不为0时，应该先触发"破防"而不是"死亡"。当护盾也归零后，`bDelayDead = true`，下次血量变化（即使还是0）时触发真正的死亡。

## NumericWatcher_Guard：护盾监听与破防触发

```csharp
[NumericWatcher(SceneType.Current, (int)ENumericId.RealGuard)]
public class NumericWatcher_Guard : INumericWatcher
{
    public void Run(Entity entity, NumericChange args)
    {
        // 护盾首次归零
        if (args.Old > 0 && args.New <= 0)
        {
            if (entity is Unit unit)
            {
                var team = teamSys.GetTeamByUnit(unit);
                var numericComp = team.GetComponent<NumericComponent>();
                
                // 只有队伍血量还有的情况下才触发破防
                // （已经死了就不需要破防演出）
                if (numericComp.GetFinalValue(ENumericId.RealTeamHp) > 0)
                {
                    skillComp.TryBreakDefense();  // 触发破防技能
                }
                
                // 标记"已破防"（影响血量监听的延迟死亡逻辑）
                var buffComp = unit.GetComponent<BuffComponent>();
                buffComp.bDead = true;
            }
        }
        // 护盾持续为零（处理延迟破防）
        else if (args.Old <= 0 && args.New <= 0)
        {
            if (entity is Unit unit)
            {
                if (skillComp.HasDelayBreakDefense())
                {
                    skillComp.TryBreakDefense();  // 延迟的破防触发
                }
            }
        }
        // 护盾从零恢复
        else if (args.Old <= 0 && args.New > 0)
        {
            if (entity is Unit unit)
            {
                var buffComp = unit.GetComponent<BuffComponent>();
                buffComp.bDead = false;  // 取消已破防标记
            }
        }
        
        // 通知 UI 更新护盾条
        int max = entity.GetComponent<NumericComponent>()
            .GetPart(ENumericId.RealGuard, ENumericPart.Max).AsInt();
        entity.ClientScene().GetComponent<EventDispatcherComponent>()
            .FireEvent(new Evt_UnitGuardChange() {
                unit = entity as Unit,
                OldGuard = args.Old.AsInt(),
                Guard = args.New.AsInt(),
                Guard_Max = max
            });
    }
}
```

护盾监听器需要处理三种状态转换：
1. `有 → 无`：触发破防
2. `无 → 无`：处理延迟破防（被技能阻止的情况）
3. `无 → 有`：恢复标记（护盾通过技能恢复时）

这三种情况对应了游戏中"护盾"机制的完整状态机。

## NumericWatcher_Mana：能量监听

```csharp
[NumericWatcher(SceneType.Current, (int)ENumericId.Mana)]
public class NumericWatcher_Mana : INumericWatcher
{
    public void Run(Entity entity, NumericChange args)
    {
        int max = entity.GetComponent<NumericComponent>()
            .GetPart(ENumericId.Mana, ENumericPart.Max).AsInt();
        
        entity.ClientScene().GetComponent<EventDispatcherComponent>()
            .FireEvent(new Evt_ManaChange() {
                entity = entity,
                OldValue = args.Old.AsInt(),
                value = args.New.AsInt(),
                value_Max = max
            });
    }
}
```

能量监听相对简单——主要是通知 UI 更新能量条。但重要的是，这个通知触发了一个事件（`Evt_ManaChange`），而不是直接调用 UI 接口。

这意味着：
- UI 系统订阅这个事件，更新能量条显示
- 成就系统可以订阅这个事件，检测"能量满载"成就
- 技能系统可以订阅这个事件，在能量充满时自动触发某些效果
- 任何系统都可以独立响应，不需要修改监听器代码

## NumericWatcher 与 EventDispatcher 的组合

数值监听系统和事件分发系统形成了一个两层的解耦架构：

```
数值修改
    ↓
NumericWatcher.Run（第一层：数值变化监听）
    ↓
EventDispatcher.FireEvent（第二层：事件广播）
    ↓
各业务系统的事件处理函数（最终响应方）
```

第一层（NumericWatcher）：处理与数值直接相关的核心逻辑（如死亡判断、破防触发）

第二层（EventDispatcher）：广播状态变化，让所有感兴趣的系统都能响应（UI、音效、特效、成就）

这种双层设计让核心逻辑与表现层完全解耦——NumericWatcher 不知道 UI 的存在，UI 也不知道 NumericWatcher 的存在，它们通过事件这个"公告板"间接通信。

## 防止数值变化的无限循环

一个潜在问题：如果监听器 A 响应数值 X 的变化，在响应中修改了数值 Y，而监听器 B 响应数值 Y 的变化时又修改了数值 X，就会无限循环。

解决方案：

```csharp
// NumericComponent 中的变化通知（带重入保护）
private bool _notifying = false;

private void OnValueChanged(ENumericId id, FP oldValue, FP newValue)
{
    if (_notifying) return;  // 防止重入
    _notifying = true;
    
    try
    {
        var watchers = GetWatchers((int)id);
        foreach (var watcher in watchers)
        {
            watcher.Run(this.Parent, new NumericChange { Old = oldValue, New = newValue });
        }
    }
    finally
    {
        _notifying = false;
    }
}
```

或者通过"先完成所有数值修改，再批量通知"的方式避免中间状态问题：

```csharp
// NumericComponent 批量修改模式
self.BeginInitNumeric();  // 锁定通知
// ... 批量修改数值
self.EndInitNumeric();    // 解锁并统一通知
```

## 设计总结

NumericWatcher 系统的核心价值：

| 特性 | 实现方式 |
|------|---------|
| 声明式监听 | [NumericWatcher] 特性，自动注册 |
| 解耦变化源与响应方 | 中间者模式，互不直接引用 |
| 多监听器支持 | 同一数值可以有多个监听器 |
| 状态机式响应 | Old/New 值判断不同状态转换 |
| 双层解耦 | Watcher → Event → UI/音效/成就 |

这套系统让战斗数值系统可以被独立测试：无需 UI、无需渲染，只需要数值计算和监听器，就可以验证血量归零时是否正确触发死亡逻辑。这是高质量战斗系统的工程素养体现。
