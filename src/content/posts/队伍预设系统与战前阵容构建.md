# 队伍预设系统与战前阵容构建

在大多数战斗类游戏中，玩家在战斗前需要配置阵容：选择角色、装备心得卡、设置上场顺序。这些配置数据需要在进入战斗时被精确地转化为战斗角色的数值和技能。

本文深入解析队伍预设（TeamPreset）系统，理解从阵容配置到战斗角色的完整数据流。

---

## 第一性原理：阵容配置的核心问题

阵容配置系统需要解决：

1. **数据序列化**：配置需要网络传输（PVP对战时双方互传阵容）
2. **多层数据合并**：角色基础数值 + 装备加成 + 心得卡被动技能
3. **状态持久化**：玩家关闭游戏后，阵容配置应该保存
4. **版本兼容**：随游戏更新，配置数据结构可能变化，需要兼容旧数据

---

## 核心数据模型

### PassiveSkillInfo：被动技能信息

```csharp
[MemoryPackable]
public partial struct PassiveSkillInfo
{
    [MemoryPackOrder(0)] public int skillID;     // 技能 ID
    [MemoryPackOrder(1)] public int level;        // 技能等级
    [MemoryPackOrder(2)] public int SourceID;     // 来源 ID（装备/羁绊/星盒的具体ID）
    [MemoryPackOrder(3)] public int Sourcelevel;  // 来源等级
    [MemoryPackOrder(4)] public ERarityType rarity;  // 稀有度
    [MemoryPackOrder(5)] public EPSkillSourceType type;  // 来源类型
}
```

`EPSkillSourceType` 枚举定义了被动技能的来源：

```csharp
public enum EPSkillSourceType
{
    Nan = 0,           // 无效
    Chess = 1,         // 棋子/心得卡（核心装备系统）
    Style = 2,         // 角色风格（类似专精/流派）
    CharacterrSkill = 3, // 角色自带技能
    Bond = 4,          // 羁绊（特定角色组合加成）
    Board = 5,         // 星盒（等级突破系统）
    DefaultRule = 6,   // 默认规则（所有队伍都有）
    ExtraState = 7,    // 额外状态（场景特殊效果）
    SceneFeature = 8,  // 场景特性（关卡BUFF）
    RoundFeature = 9,  // 爆点（轮次特殊规则）
    Tactic = 10,       // 策略卡（战斗中手动使用的卡牌）
}
```

这个枚举展示了游戏的深度层次——同一个"被动技能"可以来自十种不同的系统，每种来源都对游戏深度有独特贡献。

### BattleCharacterConf：角色战斗配置

```csharp
[MemoryPackable]
public partial class BattleCharacterConf
{
    [MemoryPackOrder(0)] public CharacterBuildData Info;   // 角色养成数据（等级、突破等）
    [MemoryPackOrder(2)] public int ChessID;               // 装备的心得卡 ID
    [MemoryPackOrder(3)] public int TeamCode;              // 所属队伍编号（0或1）
    [MemoryPackOrder(4)] public List<BattleElementConf> Feature;  // 特性列表（被动技能集合）
    
    // 数值修改列表（装备带来的数值加成）
    [MemoryPackOrder(5)] public List<(ENumericId, FP, ENumericPart)> NumericChanges;
}
```

`NumericChanges` 是三元组列表：`(数值ID, 数值, 部分)`。

例如：
```
(ENumericId.Atk, 50, ENumericPart.Add)  // 攻击力绝对值 +50
(ENumericId.Hp, 10, ENumericPart.Pct)   // HP 百分比 +10%
```

这些加成在角色初始化时统一应用到 `NumericComponent`：

```csharp
// NumericComponent 初始化时应用 NumericChanges
if (presetConf != null)
{
    foreach (var (nid, value, part) in presetConf.NumericChanges)
    {
        self.ModifyPart(nid, part, value);  // 添加修改器
    }
}
```

注意：这里使用修改器（ModifyPart），而不是直接 SetPart，所以这些加成是可撤销的（战斗结束或重置时可以精确移除）。

### BattleElementConf：战斗元素配置

```csharp
[MemoryPackable]
public partial class BattleElementConf
{
    [MemoryPackOrder(2)] public PassiveSkillInfo skill;      // 被动技能信息
    [MemoryPackOrder(3)] public BlackboardSource extraArgs;  // 额外参数（通过黑板传入技能）
    
    public BattleElementConf(IBlackboard extraArgs)
    {
        this.extraArgs = new BlackboardSource();
        // 只复制以 "v_" 前缀开头的变量（过滤掉内部变量）
        this.extraArgs.OverwriteFromWithHead(extraArgs, "v_");
    }
}
```

`BlackboardSource` 是黑板的数据源，可以在战斗开始时将参数注入到被动技能的黑板中。"v_" 前缀过滤确保只有标记为"对外可见变量"的黑板变量才会被传递。

### TeamExtraBuildData：队伍额外数据

```csharp
[MemoryPackable]
public partial class TeamExtraBuildData
{
    public Dictionary<int, CharacterExtraBuildData> data;  // 角色ID → 额外数据
    public int fans;        // 观众人气（影响某些游戏效果）
    public int winRate;     // 胜率统计
    public string teamName; // 队伍名称
    public string teamIcon; // 队伍图标
    
    // 查询角色上场状态
    public bool IsCharacterForceOnStage(int ipid)
    {
        if (data.ContainsKey(ipid))
            return data[ipid].OnStageState == EOnStageState.ForceOn;
        return false;
    }
    
    // 查询角色技能解锁状态
    public bool IsCharacterSkillUnlock(int ipid, int vTypeID, int skillID)
    {
        if (data.ContainsKey(ipid))
            return data[ipid].UnlockSkillList.Contains(new Vector2_Int(vTypeID, skillID));
        return false;
    }
}
```

`EOnStageState` 控制角色的上场规则：

```csharp
public enum EOnStageState
{
    Normal = 0,     // 正常（根据出场顺序决定）
    ForceOn = 1,    // 强制上场（某些活动中主角必须在场）
    ForceLeave = 2, // 强制不上场（某些惩罚机制）
}
```

## 数据序列化：MemoryPack

所有配置类都标注了 `[MemoryPackable]`，使用 `MemoryPack` 进行二进制序列化：

```csharp
// 序列化（客户端准备发送阵容）
var bytes = MemoryPackSerializer.Serialize(teamConf);

// 反序列化（收到对方阵容数据）
var teamConf = MemoryPackSerializer.Deserialize<TeamConf>(bytes);
```

为什么使用 MemoryPack 而不是 JSON？

| 特性 | JSON | MemoryPack |
|------|------|------------|
| 序列化速度 | 慢（文本处理）| 极快（直接二进制映射）|
| 数据大小 | 大（文字有冗余）| 小（纯二进制） |
| 可读性 | 人类可读 | 二进制不可读 |
| 适用场景 | 配置文件、调试 | 网络传输、高频数据 |

PVP 对战时，阵容数据需要频繁网络传输，MemoryPack 的极高性能是关键优势。

## 数据流：从配置到战斗

```
玩家配置阵容（UI层）
    ↓
TeamPresetComponent.Save() → 本地持久化
    ↓
战斗开始时
    ↓
TeamPresetComponent.BuildBattleConf()
    ↓
BattleCharacterConf（数值+特性+技能）
    ↓
战斗初始化
    ↓
NumericComponent.InitCharacterTemplate(conf)
    ├── 设置基础数值（角色等级对应的属性）
    └── 应用 NumericChanges（装备加成）
    ↓
PassiveSkillInstance.Invoke() for each Feature
    └── 激活所有被动技能（心得卡效果、羁绊效果等）
    ↓
战斗开始
```

## 棋盘系统（Board/Chess）

```csharp
[MemoryPackable]
public partial class ChessData
{
    public int ID;           // 棋子（心得卡）ID
    public int Level;        // 棋子等级
    public int ConfigID;     // 配置表 ID
    public int RarityType;   // 稀有度
}

[MemoryPackable]
public partial class DetailBoard
{
    public List<DetailChess> Chesses;   // 棋盘上的所有棋子
    public int BoardID;                 // 棋盘 ID
    public int BoardLevel;              // 棋盘等级
}
```

"棋盘"是游戏中的装备/养成系统的可视化表示（类似于星盘/符文/圣遗物）。玩家在棋盘上放置心得卡（棋子），获得对应的被动技能加成。

从棋盘数据构建被动技能：

```csharp
public static List<PassiveSkillInfo> GetPassiveSkillsFromBoard(DetailBoard board, int characterId)
{
    var result = new List<PassiveSkillInfo>();
    
    foreach (var chess in board.Chesses)
    {
        // 从心得卡配置表获取对应的被动技能
        var chessConf = CfgManager.tables.TbChess.Get(chess.ConfigID, chess.Level);
        if (chessConf == null) continue;
        
        // 检查是否有角色限定（某些心得卡只对特定角色有效）
        if (chessConf.CharacterId > 0 && chessConf.CharacterId != characterId) continue;
        
        result.Add(new PassiveSkillInfo
        {
            skillID = chessConf.PassiveSkillID,
            level = chess.Level,
            SourceID = chess.ID,
            type = EPSkillSourceType.Chess
        });
    }
    
    return result;
}
```

## 羁绊系统（Bond）

羁绊是"特定角色组合时生效的加成"：

```csharp
public static List<PassiveSkillInfo> GetBondSkills(List<int> teamCharacterIds)
{
    var result = new List<PassiveSkillInfo>();
    
    foreach (var bondConf in CfgManager.tables.TbBond.DataList)
    {
        // 检查羁绊的所有角色是否都在队伍中
        bool allInTeam = bondConf.RequiredCharacters.All(id => teamCharacterIds.Contains(id));
        if (allInTeam)
        {
            result.Add(new PassiveSkillInfo
            {
                skillID = bondConf.PassiveSkillID,
                level = bondConf.Level,
                type = EPSkillSourceType.Bond
            });
        }
    }
    
    return result;
}
```

当队伍中同时有 A 和 B 角色时，自动激活 A+B 的羁绊被动。这种设计鼓励玩家使用特定组合，创造了深层的阵容策略。

## 技能解锁系统

```csharp
public bool IsCharacterSkillUnlock(int ipid, int vTypeID, int skillID)
{
    if (data.ContainsKey(ipid))
    {
        return data[ipid].UnlockSkillList.Contains(new Vector2_Int(vTypeID, skillID));
    }
    return false;
}
```

技能解锁状态存储在 `UnlockSkillList` 中，战斗开始时，只有解锁的技能才会被加载。未解锁的技能（如高级技能需要达到特定好感度）直接跳过。

## 设计总结

队伍预设系统是"游戏外系统"与"战斗系统"的衔接桥梁：

| 层次 | 系统 | 作用 |
|------|------|------|
| 表现层 | UI配置界面 | 玩家交互 |
| 存储层 | TeamPresetComponent | 本地持久化 |
| 传输层 | MemoryPack 序列化 | 网络传输 |
| 初始化层 | BattleCharacterConf | 战斗数据构建 |
| 执行层 | NumericComponent + PassiveSkillInstance | 数值和技能生效 |

复杂度管理：
- 多来源被动技能通过 `EPSkillSourceType` 统一管理
- 数值加成通过修改器系统统一应用
- 序列化使用高效的 MemoryPack

对初学者来说，这套系统展示了"数据驱动"的极致：战斗系统几乎不知道具体是什么角色、什么装备，它只知道"有这些数值加成"和"有这些被动技能"，完全由配置数据决定战斗表现。
