## 游戏安全的现实

```
为什么游戏要做安全？

竞技游戏：作弊者一人破坏多人体验
  → 自瞄、透视外挂：玩家大量流失
  → 必须有效果显著的反作弊

数值游戏（RPG/MMO）：内存修改无限资源
  → 金币/道具外挂：破坏经济平衡
  → 付费玩家心态崩塌

所有游戏：提速外挂
  → 速度加倍刷资源：破坏游戏平衡

现实：客户端无法被完全信任，但可以提高作弊成本
核心原则：重要逻辑必须在服务端验证，客户端只负责表现
```

## 一、服务端权威架构

### 1.1 什么应该在服务端验证

```
服务端必须验证（客户端数据不可信）：
  ✅ 伤害计算（防止无限伤害）
  ✅ 移动合法性（防止瞬移、加速）
  ✅ 道具使用（防止无限使用）
  ✅ 交易金额（防止负数金币）
  ✅ 成就达成（防止直接解锁）

客户端可以计算（但服务端要对比）：
  ⚠️ 战斗结果（客户端计算，服务端复核）
  ⚠️ 路径（客户端选路，服务端验证距离）

客户端负责（无需服务端）：
  ✅ 特效、动画（纯表现）
  ✅ UI 显示（不影响数值）
```

### 1.2 服务端验证示例

```
以战斗伤害为例：

客户端：
  1. 玩家按下攻击键
  2. 播放攻击动画
  3. 发送消息到服务端：{"skill_id": 1001, "target_id": 9527}
  4. 等待服务端返回结果

服务端：
  1. 收到攻击请求
  2. 查询玩家当前状态（位置、技能 CD、Buff）
  3. 计算伤害（使用服务端的数值，不信任客户端）
  4. 验证合法性（目标是否在攻击范围内？技能是否 CD？）
  5. 返回结果：{"damage": 1500, "is_crit": true, "target_hp": 2000}

客户端：
  6. 收到服务端结果，播放受击动画
  7. 更新显示的血量

效果：即使黑客修改了客户端伤害值，服务端计算的结果才是准确值
```

## 二、客户端反作弊

### 2.1 内存数值保护

```csharp
/// <summary>
/// 数值加密存储：防止内存编辑器修改
/// 原理：不直接存储真实值，而是存储真实值 XOR 随机密钥
/// </summary>
public struct EncryptedInt
{
    private int _encryptedValue;
    private int _key;
    
    public static EncryptedInt Create(int value)
    {
        var ei = new EncryptedInt();
        ei._key = UnityEngine.Random.Range(int.MinValue, int.MaxValue);
        ei._encryptedValue = value ^ ei._key;
        return ei;
    }
    
    public int Value
    {
        get => _encryptedValue ^ _key;
        set => _encryptedValue = value ^ _key;
    }
    
    // 运算符重载（使用起来和普通 int 一样）
    public static implicit operator int(EncryptedInt e) => e.Value;
    public static implicit operator EncryptedInt(int value) => Create(value);
    
    // 篡改检测（进阶）
    // 可以维护一个"影子值"（另一种方式存储同一值），定期比对
}

// 使用：和普通 int 完全相同的写法
public class PlayerStats : MonoBehaviour
{
    private EncryptedInt _hp = 100;
    private EncryptedInt _gold = 0;
    
    public int HP => _hp;
    public int Gold => _gold;
    
    public void TakeDamage(int damage)
    {
        _hp = (int)_hp - damage; // 内存中存储的是加密后的值
    }
}
```

### 2.2 时间戳验证（防加速外挂）

```csharp
/// <summary>
/// 反加速外挂：检测游戏速度异常
/// </summary>
public class AntiSpeedHack : MonoBehaviour
{
    private float _lastRealTime;
    private float _lastGameTime;
    private const float TOLERANCE = 0.2f; // 20% 误差容忍
    private int _warningCount = 0;
    
    void Update()
    {
        float realDelta = Time.realtimeSinceStartup - _lastRealTime;
        float gameDelta = Time.time - _lastGameTime;
        
        _lastRealTime = Time.realtimeSinceStartup;
        _lastGameTime = Time.time;
        
        if (realDelta <= 0 || gameDelta <= 0) return;
        
        // 如果游戏时间流逝比真实时间快很多，说明可能被加速
        float ratio = gameDelta / realDelta;
        
        if (ratio > 1f + TOLERANCE || ratio < 1f - TOLERANCE)
        {
            _warningCount++;
            Debug.LogWarning($"[AntiCheat] Speed anomaly detected: ratio={ratio:F2}, count={_warningCount}");
            
            if (_warningCount > 10)
            {
                // 上报服务端
                ReportCheat("speed_hack", new Dictionary<string, float>
                {
                    ["ratio"] = ratio
                });
            }
        }
        else
        {
            _warningCount = Mathf.Max(0, _warningCount - 1); // 正常时降低计数
        }
    }
    
    private void ReportCheat(string cheatType, Dictionary<string, float> data)
    {
        // 上报到服务端，不立即踢出（避免误判），服务端决策

    }
}
```

### 2.3 完整性检测

```csharp
/// <summary>
/// 游戏文件完整性检测
/// 防止修改游戏资源文件（如修改角色属性配置）
/// </summary>
public static class IntegrityChecker
{
    // 关键文件的预期哈希（构建时计算并写入）
    private static readonly Dictionary<string, string> _expectedHashes = new()
    {
        ["config/player_config.json"] = "abc123...", // 预期哈希
        ["config/skill_config.json"] = "def456...",
    };
    
    public static async UniTask<bool> VerifyFilesAsync()
    {
        foreach (var (relativePath, expectedHash) in _expectedHashes)
        {
            string fullPath = Path.Combine(Application.streamingAssetsPath, relativePath);
            
            if (!File.Exists(fullPath))
            {
                Debug.LogError($"[Integrity] Missing file: {relativePath}");
                return false;
            }
            
            string actualHash = await ComputeFileHashAsync(fullPath);
            
            if (actualHash != expectedHash)
            {
                Debug.LogError($"[Integrity] File modified: {relativePath}");
                // 上报并阻止游戏进入
                return false;
            }
        }
        
        return true;
    }
    
    private static async UniTask<string> ComputeFileHashAsync(string path)
    {
        await UniTask.SwitchToThreadPool();
        
        using var sha = System.Security.Cryptography.SHA256.Create();
        using var stream = File.OpenRead(path);
        byte[] hash = sha.ComputeHash(stream);
        
        await UniTask.SwitchToMainThread();
        
        return BitConverter.ToString(hash).Replace("-", "").ToLower();
    }
}
```

### 3.1 HTTPS 与证书锁定

```csharp
/// <summary>
/// 证书锁定（Certificate Pinning）
/// 防止中间人攻击（使用代理工具分析/修改网络包）
/// </summary>
public class SecureHttpClient
{
    // 预期的服务器证书公钥哈希（Base64）
    private static readonly HashSet<string> _pinnedHashes = new()
    {
        "your-server-cert-public-key-hash-base64",
    };
    
    public static UnityWebRequest CreateSecureRequest(string url)
    {
        var request = new UnityWebRequest(url);
        
        // 注意：Unity 默认不支持证书锁定
        // 需要使用原生插件（Android NSS / iOS Security Framework）
        // 或者使用第三方库
        
        // 替代方案：自定义证书验证
        // ServicePointManager.ServerCertificateValidationCallback = ValidateCert;
        
        return request;
    }
    
    private static bool ValidateCert(object sender, 
        System.Security.Cryptography.X509Certificates.X509Certificate cert,
        System.Security.Cryptography.X509Certificates.X509Chain chain,
        System.Net.Security.SslPolicyErrors errors)
    {
        if (errors != System.Net.Security.SslPolicyErrors.None) return false;
        
        // 检查证书公钥哈希是否在允许列表中
        string certHash = cert.GetCertHashString();
        return _pinnedHashes.Contains(certHash);
    }
}
```

### 3.2 消息签名防篡改

```csharp
/// <summary>
/// 消息签名：防止客户端篡改请求参数
/// </summary>
public static class MessageSigning
{
    private static readonly string _secret = "game_api_secret"; // 与服务端约定
    
    /// <summary>
    /// 为请求添加签名
    /// </summary>
    public static Dictionary<string, string> SignRequest(Dictionary<string, string> parameters)
    {
        var sorted = new SortedDictionary<string, string>(parameters);
        
        // 按 key 排序后拼接
        var sb = new System.Text.StringBuilder();
        foreach (var kv in sorted)
        {
            sb.Append(kv.Key).Append('=').Append(kv.Value).Append('&');
        }
        
        // 添加时间戳（防重放攻击）
        long timestamp = DateTimeOffset.UtcNow.ToUnixTimeSeconds();
        sb.Append("timestamp=").Append(timestamp);
        sb.Append("&secret=").Append(_secret);
        
        // 计算 HMAC-SHA256 签名
        string sign = ComputeHMAC(sb.ToString(), _secret);
        
        var result = new Dictionary<string, string>(parameters)
        {
            ["timestamp"] = timestamp.ToString(),
            ["sign"] = sign
        };
        
        return result;
    }
    
    private static string ComputeHMAC(string message, string key)
    {
        using var hmac = new System.Security.Cryptography.HMACSHA256(
            System.Text.Encoding.UTF8.GetBytes(key));
        byte[] hash = hmac.ComputeHash(System.Text.Encoding.UTF8.GetBytes(message));
        return BitConverter.ToString(hash).Replace("-", "").ToLower();
    }
}
```

## 四、安全体系建设

### 4.1 分层防御策略

```
防御层次（从内到外）：

层1 - 数值加密（客户端内存保护）：
  目标：对抗 Cheat Engine 等内存修改工具
  方法：EncryptedInt/Float，变量混淆

层2 - 完整性检测（防文件篡改）：
  目标：防止修改游戏资源和配置
  方法：文件哈希校验，重要代码混淆

层3 - 网络安全（防中间人）：
  目标：防止封包分析和修改
  方法：HTTPS，消息签名，关键数据加密

层4 - 服务端验证（最重要的防线）：
  目标：即使客户端被破解，服务端结果正确
  方法：重要逻辑全在服务端，客户端只是表现层

层5 - 行为分析（智能反作弊）：
  目标：检测外挂行为模式
  方法：异常数据上报，AI 行为分析

投入产出比建议：
  优先 Layer 4（服务端验证）：一劳永逸，效果最好
  次之 Layer 5（行为分析）：持续对抗外挂更新
  之后 Layer 1/2/3：提高门槛，但攻防对抗无止境
```

### 4.2 安全事件响应

```markdown
## 安全事件响应 SOP

### P0（紧急，1小时内响应）
- 服务器被入侵
- 玩家数据泄露
- 批量作弊导致游戏瘫痪

处理步骤：
  1. 立即通知安全团队和项目负责人
  2. 评估影响范围
  3. 紧急修复/封堵漏洞
  4. 回滚受影响的数据（如果可能）
  5. 向玩家说明情况

### P1（高，4小时内响应）
- 外挂工具大量传播
- 单一作弊方式影响大量玩家

处理步骤：
  1. 收集外挂特征（抓包、逆向分析）
  2. 添加检测规则
  3. 批量封禁，通知受影响玩家
```

## 总结

游戏安全的核心认知：

1. **没有完美的客户端安全**：客户端代码最终都可以被逆向，不要依赖"秘密算法"
2. **服务端验证是根本**：重要数值和逻辑都要在服务端维护和验证
3. **提高作弊成本**：不是让作弊不可能，而是让作弊成本高于收益
4. **快速响应**：发现新外挂后快速更新检测规则，持续对抗

**技术负责人的任务**：
- 架构设计时就考虑"客户端不可信"原则
- 建立安全事件响应流程
- 定期进行安全审计（自己尝试破解自己的游戏）
- 投资服务端权威验证，而不是过度依赖客户端反作弊
