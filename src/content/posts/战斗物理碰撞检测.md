# 08 战斗物理碰撞检测

> 本文介绍战斗系统中的碰撞检测机制，包括攻击框/受击框的数据结构、多种碰撞体类型、帧精确的碰撞判定逻辑，以及 Token 消耗系统。

---

## 1. 系统概述

战斗碰撞检测是连接"技能执行"和"伤害计算"的桥梁：技能 Timeline 在特定帧激活攻击框，碰撞系统检测攻击框与敌方受击框的重叠，一旦检测到命中，就触发后续的伤害计算和受击表现流程。

### 1.1 碰撞检测的核心概念

| 概念 | 说明 |
|------|------|
| 攻击框（Attack Collider） | 技能发出的伤害区域，激活时检测是否覆盖到敌方受击框 |
| 受击框（Hurt Collider） | 角色身上的可被命中区域，通常覆盖角色的躯干 |
| 格挡框（Parry Collider） | 格挡状态时的特殊受击框，命中后触发格挡逻辑 |
| 闪避框（Doge Collider） | 闪避状态时的受击框，命中后触发闪避逻辑 |
| 表演框（Perform Collider） | 演出击中框，触发特殊表演动作 |
| 假攻击框（Fake Attack） | 不造成伤害，仅用于触发特效/反应的虚假攻击框 |

### 1.2 帧同步中的碰撞检测

与 Unity 物理引擎（`Physics.OverlapBox`）不同，本项目的碰撞检测**完全在逻辑层手动实现**：
- 使用定点数（`FP`/`TSVector`）计算，保证帧同步确定性
- 不依赖 Unity 物理系统（PhysX 不确定性）
- 每帧手动计算 OBB（有向包围盒）与 OBB 的重叠

### 2.3 ColliderInfo —— 碰撞体数据结构

每个碰撞体（攻击框或受击框）都由 `ColliderInfo` 描述：

```
ColliderInfo
├── 几何形状
│   ├── EColliderType colliderType  ← 目前只有 Box（矩形包围盒）
│   ├── TSVector center             ← 相对于 Unit 根节点的中心偏移
│   ├── TSQuaternion rotation       ← 碰撞体的旋转（支持斜向攻击框）
│   └── TSVector size               ← 半尺寸（x/y/z 各轴的半长）
│
├── 攻击框专用
│   └── TSVector attackPoint        ← 攻击挂点（用于受击特效位置计算）
│
├── 受击框专用
│   ├── bool acceptAttack           ← 是否接受普通攻击
│   ├── bool acceptDoge             ← 是否接受闪避框
│   ├── bool acceptParry            ← 是否接受格挡框
│   ├── bool acceptPerform          ← 是否接受表演框
│   ├── bool acceptFakeAttack       ← 是否接受假攻击框
│   └── bool forceHurtEffectBox     ← 是否作为受击特效定位框
│
└── 跳字配置
    ├── bool overrideJumpText       ← 是否覆盖默认跳字
    ├── FP jumpTextDuration         ← 跳字持续时间
    └── FP jumpTextScale            ← 跳字缩放
```

### 2.4 Token 消耗系统

`ColliderTokenInfo` 是一个特殊机制，用于控制同一攻击框在整段攻击期间最多命中几次：

```csharp
public struct ColliderTokenInfo
{
    public bool bUseToken;               // 是否开启 Token 消耗
    public bool bUseDefaultTokenTemplete; // 是否使用默认 Token 模板
    public int tokenTempleteID;          // 自定义 Token 模板 ID
    public bool bUsed { get; set; }      // 本轮是否已消耗（防重复命中）
}
```

**典型场景**：一段横扫攻击，攻击框持续激活 5 帧，但如果 Token 系统设置为"每段只能命中一次"，即使攻击框持续与目标重叠，也只在第一次检测到重叠时结算伤害，后续帧跳过。

### 2.5 Timeline 中的攻击框控制

攻击框的激活时机完全由 Timeline 上的 Clip 控制：

```
CLIP_SetAttackCollider  ← 激活攻击框（Clip 进入范围时设置 AttackColliders）
CLIP_ClearHurtCollider  ← 清空受击框（某些无敌帧）
CLIP_HurtCollider       ← 按帧配置受击框（支持逐帧改变受击框形状）
```

**CLIP_HurtCollider 的逐帧受击框配置**（特别设计）：
```
[0帧]: 受击框 A（正常站姿）
[2帧]: 受击框 B（下蹲动作，高度变矮）
[5帧]: 受击框 C（跳跃姿态，位置上移）
```
每帧根据动画姿态配置精确的受击框，而不是用一个固定的大框，这样既保证了碰撞精度，又避免了"明明看起来没打到却判定命中"的问题。

### 3.2 CLIP_HurtCollider.cs —— 帧精确受击框配置

```csharp
[Name("设置受击框")]
[Description("设置受击框")]
[Attachable(typeof(ActorActionTrack), typeof(ActorColliderTrack))]
[MemoryPackable]
public partial class CLIP_HurtCollider : ActionClip
{
    // 帧碰撞体列表：每帧可以有不同的受击框配置
    [fsSerializeAs("frameColliders")]
    [MemoryPackOrder(0)]
    public List<FrameCollider> frameColliders = new();

    // 获取特定帧的碰撞体列表（精确匹配）
    public List<ColliderInfo> GetCollidersByFrame(int frame)
    {
        for (int i = frameColliders.Count - 1; i >= 0; i--)
        {
            if (frameColliders[i].frame == frame)
                return frameColliders[i].colliders;
        }
        return null;
    }

    // 获取最近帧的碰撞体（帧之间插值使用）
    public List<ColliderInfo> GetNearestCollidersByFrame(int frame)
    {
        for (int i = frameColliders.Count - 1; i >= 0; i--)
        {
            if (frameColliders[i].frame <= frame)
                return frameColliders[i].colliders;
        }
        return null;
    }
}

// 帧碰撞体数据（某一帧的碰撞体配置）
[MemoryPackable]
public partial class FrameCollider
{
    [fsSerializeAs("f")]
    public int frame;                     // 帧号

    [fsSerializeAs("c")]
    public List<ColliderInfo> colliders;  // 该帧的碰撞体列表
}
```

## 4. 碰撞检测流程详解

### 4.1 每帧碰撞检测流程

```
ColliderComponent.LateFixedUpdate（在所有 FixedUpdate 之后执行）
    │
    ├─ 检查 SkipThisFrame → 是 → 跳过本帧
    │
    └─ 遍历攻击方 Unit（具有攻击框的）
         └─ 攻击方 AttackColliders 列表非空？
              └─ 是：遍历场景中所有敌方 Unit
                   └─ 目标 Unit 有受击框（HurtColliders）？
                        └─ 是：执行 OBB vs OBB 相交检测
                             ├─ 相交？→ 记录命中（AttackedEnemies.Add）
                             │         → 检查 TokenInfo（防重复命中）
                             │         → 触发命中事件（BPDef.AttackStartEvtID）
                             └─ 不相交 → 继续检测下一对
```

### 4.2 OBB 碰撞检测原理

OBB（Oriented Bounding Box）是有向包围盒，支持任意旋转方向的矩形碰撞体：

```
传统 AABB（轴对齐包围盒）
┌──────┐
│      │  只能是轴对齐的矩形
└──────┘

OBB（有向包围盒）
  ┌──────┐
 /        /
└──────┘  可以任意旋转（如斜向挥剑）
```

检测方法：**分离轴定理（SAT）**
- 如果两个 OBB 在任何一个轴上的投影不重叠，则两者不相交
- 检查 15 个候选分离轴（两个盒子各 3 条边轴 + 6×3=18种叉积轴，取其中 15 个）
- 全部轴都重叠 → 两个 OBB 相交（命中）

### 4.3 背击判定

```
IsBackHit 判定逻辑：
1. 计算从目标到攻击者的方向向量 V
2. 计算目标当前朝向向量 F
3. Dot(V, F) > 0.7（约 45°以内）→ 背击
```

背击通常触发额外伤害加成（如技能配置中的背击倍率系数）。

## 5. 设计亮点

### 5.1 帧精确的受击框配置
`CLIP_HurtCollider` 支持在 Timeline 上为每一帧单独配置受击框，配合角色的动画姿态，受击框可以精确匹配角色的实际形体。这解决了格斗游戏中常见的"幽灵碰撞"问题（角色明显避开但仍判定命中）。

### 5.2 多类型攻击框并行存在
`ColliderComponent` 同时维护多个列表（普通攻击、格挡预判、闪避预判、表演攻击、假攻击），允许角色在同一时刻处于多种攻击状态（如格挡准备阶段同时激活普通攻击框和格挡框），支持复杂的格斗博弈机制。

### 5.3 Token 消耗防止多段命中
一段持续 5 帧的攻击动作中，攻击框每帧都与目标重叠，若不加控制会命中 5 次。Token 机制（`ColliderTokenInfo`）允许设计者精确控制"每段攻击最多命中 N 次，或只命中一次"，是格斗游戏攻击判定的标准做法。

### 5.4 受击框的层级过滤
每个 `ColliderInfo` 上有 `acceptAttack`、`acceptDoge` 等开关，允许精细控制"哪种类型的攻击框可以命中该受击框"。例如：某种特殊受击框只响应表演框（演出专用），不响应普通攻击框，实现了精准的击中类型分类。

### 5.5 独立于 Unity 物理系统
所有碰撞计算完全在逻辑层（定点数运算）中完成，不调用 Unity 的 `Physics.OverlapBox` 等 API，确保了帧同步的确定性。表现层的特效位置等可以使用 Unity 物理，但不会影响战斗逻辑结果。

## 6. 常见问题与最佳实践

### Q1：攻击框和受击框的坐标空间是怎样的？
**A**：`ColliderInfo.center` 是相对于 Unit 根节点（`Unit.Position`/`Unit.Rotation`）的局部坐标。实际碰撞检测时，需要将局部坐标通过 Unit 的变换矩阵（`TSMatrix4x4.TRS`）转换为世界坐标后再进行相交检测。

### Q2：为什么 ColliderInfo 只支持 Box 类型，不支持 Sphere/Capsule？
**A**：Box（OBB）可以通过旋转和缩放模拟大多数攻击形状（细长的剑刺、宽扁的扫击等），且 OBB vs OBB 的 SAT 检测实现相对简单。Capsule/Sphere 支持可以后续扩展，在当前战斗设计中 Box 已满足需求。

### Q3：一个角色可以同时有多个攻击框吗？
**A**：可以。`AttackColliders` 是 `List<ColliderInfo>`，一次攻击可以有多个攻击框（如双剑攻击，左手剑和右手剑各有一个攻击框）。碰撞检测遍历所有攻击框，任意一个命中都视为命中。

### Q4：如何在 Timeline 编辑器中调试受击框？
**A**：`CLIP_HurtCollider` 在编辑器中重写了 `OnClipGUI`，会在 Timeline 上以颜色标记有碰撞体配置的帧，选中后在 Inspector 面板显示 `_currentFrameCollider` 的详细信息。同时，Scene View 中应有受击框的 Gizmo 可视化（通常在对应的表现层 Component 中实现）。

### Q5：AttackComponent 的 AttackedEnemies 什么时候清空？
**A**：每次新的攻击段开始时（通常在技能 Timeline 的 `CLIP_AttackEnd` 触发后或新的 `CLIP_SetAttackCollider` 进入时）清空。这确保每段连击独立记录命中对象，不同段之间的命中不会相互干扰。
