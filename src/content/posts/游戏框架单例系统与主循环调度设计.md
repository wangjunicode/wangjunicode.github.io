## 概述

在 ECS 架构游戏框架中，单例系统（Singleton System）承担了"全局唯一服务"的角色：日志、定时器、事件系统、渲染管线调度……这些系统都以单例形式存在。但与传统 `getInstance()` 不同，这套设计将所有单例的**生命周期**统一交给 `Game` 静态类管理，实现了：

- 统一注册 / 销毁，不会出现忘记清理的内存泄漏
- 生命周期回调（Awake / Update / FixedUpdate / LateUpdate）自动分发
- 支持热重载（静态字段用 `[StaticField]` 标注，HybridCLR 重载时可识别清理）

## 核心接口设计

### ISingleton

```csharp
public interface ISingleton : IDisposable
{
    void Register();
    void Destroy();
    bool IsDisposed();
}
```

这是所有单例的最小契约：注册、销毁、查询状态。`IDisposable` 让单例可以被 `using` 语句或 `try/finally` 块安全释放。

### Singleton\<T\> 基类

```csharp
public abstract class Singleton<T> : ISingleton where T : Singleton<T>, new()
{
    private bool isDisposed;
    [StaticField]
    private static T instance;

    public static T Instance => instance;

    public void Register()
    {
        if (instance != null)
            throw new Exception($"singleton register twice! {typeof(T).Name}");
        instance = (T)this;
    }

    public void Destroy()
    {
        if (this.isDisposed) return;
        this.isDisposed = true;
        T t = instance;
        instance = null;
        t.Dispose();
    }
    // ...
}
```

几个关键设计点：

1. **泛型约束 `where T : Singleton<T>, new()`**：CRTP（Curiously Recurring Template Pattern），保证 `Instance` 静态属性类型安全，子类无需重复声明。
2. **`[StaticField]` 标注**：这是配合 HybridCLR 热更新的约定，框架在热重载时会扫描并清空带此标注的静态字段，避免旧引用残留。
3. **`Register` 时检查双重注册**：在 `Game.AddSingleton` 调用前不会设置 `instance`，防止不通过 `Game` 直接 `new` 的误用。
4. **`Destroy` 先置 null 再 Dispose**：这样即使 `Dispose` 内部再次访问 `Instance`，也不会拿到已销毁的对象。

## 生命周期接口

框架定义了多个可选的生命周期接口，单例按需实现：

```csharp
public interface ISingletonAwake         { void Awake(); }
public interface ISingletonAwake<A>      { void Awake(A a); }
public interface ISingletonUpdate        { void Update(); }
public interface ISingletonFixedUpdate   { void BeforeFixedUpdate(); void FixedUpdate(); void LateFixedUpdate(); }
public interface ISingletonLateUpdate    { void LateUpdate(); }
```

这种"按需实现"模式的好处：

- **零成本抽象**：不需要 Update 的单例不实现 `ISingletonUpdate`，`Game` 的调度队列里就没有它，没有任何额外开销。
- **物理与逻辑分离**：`ISingletonFixedUpdate` 提供 `BeforeFixedUpdate` / `FixedUpdate` / `LateFixedUpdate` 三段，满足物理帧中"前置数据准备 → 物理模拟 → 后置同步"的需求。

## Game：全局调度中枢

`Game` 是一个纯静态类，不继承任何基类，本身不是单例，而是"单例的容器和调度器"。

### 内部数据结构

```csharp
[StaticField] private static readonly Dictionary<Type, ISingleton> singletonTypes = new();
[StaticField] private static readonly Stack<ISingleton> singletons = new();
[StaticField] private static readonly Queue<ISingleton> updates = new();
[StaticField] private static readonly Queue<ISingleton> fixedUpdates = new();
[StaticField] private static readonly Queue<ISingleton> lateUpdates = new();
[StaticField] private static readonly Queue<ETTask> frameFinishTask = new();
```

- **`singletonTypes`（Dictionary）**：Type → ISingleton 映射，用于防重注册检测。
- **`singletons`（Stack）**：记录注册顺序，`Close()` 时 **反序** Pop 销毁，确保依赖靠后注册的系统先销毁。
- **`updates / fixedUpdates / lateUpdates`（Queue）**：需要每帧调度的单例列表。

### 注册流程

```csharp
public static T AddSingleton<T>() where T : Singleton<T>, new()
{
    T singleton = new T();
    if (singleton is ISingletonAwake singletonAwake)
        singletonAwake.Awake();
    AddSingleton(singleton);
    return singleton;
}

public static void AddSingleton(ISingleton singleton)
{
    // 防重检测
    if (singletonTypes.ContainsKey(singletonType)) throw ...;
    singletonTypes.Add(singletonType, singleton);
    singletons.Push(singleton);
    singleton.Register();

    if (singleton is ISingletonUpdate)        updates.Enqueue(singleton);
    if (singleton is ISingletonFixedUpdate)   { beforeFixedUpdates.Enqueue(singleton); ... }
    if (singleton is ISingletonLateUpdate)    lateUpdates.Enqueue(singleton);
}
```

**分析**：注册完成后，根据接口自动入队，后续调度完全由队列驱动，无需手动调用。

### 每帧调度

Update 的实现是经典的"快照计数"模式：

```csharp
public static void Update()
{
    int count = updates.Count;  // 快照当前数量
    while (count-- > 0)
    {
        ISingleton singleton = updates.Dequeue();
        if (singleton.IsDisposed()) continue;
        updates.Enqueue(singleton);  // 重新入队，循环调度
        try { ((ISingletonUpdate)singleton).Update(); }
        catch (Exception e) { Log.Error(e); }
    }
}
```

这个模式的精妙之处：
- `count` 是进入 Update 前的快照，**本轮 Update 内新注册的单例不会被调用**，避免遍历时修改集合的竞态。
- 已销毁的单例（`IsDisposed()`）直接 continue 跳过，不重新入队，自然从调度列表中移除。
- `try/catch` 包裹每次调用，单个系统异常不会导致整帧崩溃。

### FixedUpdate 三段式

物理帧按 `BeforeFixedUpdate → FixedUpdate → LateFixedUpdate` 三阶段调度：

```csharp
public static void BeforeFixedUpdate()
{
    FixedFrames++;
    FixedTime = FixedFrames * EngineDefine.fixedDeltaTime_Orignal;
    // ... 同 Update 快照模式调度 BeforeFixedUpdate
}
```

- **`BeforeFixedUpdate`**：物理帧递增计数，更新逻辑帧时间（`FixedTime`）。适合在此准备输入数据、清空帧开始的临时状态。
- **`FixedUpdate`**：执行物理模拟、帧同步逻辑。
- **`LateFixedUpdate`**：同步物理结果到游戏逻辑层（如更新实体位置）。

### WaitFrameFinish 异步帧结束

```csharp
public static async ETTask WaitFrameFinish()
{
    ETTask task = ETTask.Create(true);
    frameFinishTask.Enqueue(task);
    await task;
}

public static void FrameFinishUpdate()
{
    while (frameFinishTask.Count > 0)
        frameFinishTask.Dequeue().SetResult();
}
```

这是一个极实用的"帧尾同步点"：任何系统都可以 `await Game.WaitFrameFinish()` 挂起，等到本帧所有 Update 执行完后再继续。常见用途：等待所有 ECS System 处理完毕后，做跨系统数据汇总。

## 时间管理

```csharp
public static void Update(float deltaTime, float realtimeSinceStartup)
{
    Game.realtimeSinceStartup = realtimeSinceStartup;
    Game.deltaTime = deltaTime;
    Game.gameDeltaTime = deltaTime * Game.gameTimeSpeedUp;
    Game.GameTime += deltaTime;
    Update();
}
```

`Game` 统一持有：
- `deltaTime`：真实帧间隔（Unity 传入）
- `gameDeltaTime`：经过游戏速度倍率缩放后的逻辑帧间隔
- `gameTimeSpeedUp`：倍速系数，支持慢动作/快进效果
- `FixedTime`（TrueSync.FP 定点数）：确定性物理帧时间，帧同步关键字段

## 关闭流程

```csharp
public static void Close()
{
    while (singletons.Count > 0)
    {
        ISingleton iSingleton = singletons.Pop();
        iSingleton.Destroy();
    }
    singletonTypes.Clear();
}
```

Stack 反序 Pop，保证先注册的系统最后销毁。例如：日志系统（首个注册）在所有业务系统销毁后才关闭，确保销毁过程中的报错信息可以正常输出。

### 定义一个带 Update 的单例

```csharp
public class AudioManager : Singleton<AudioManager>, ISingletonAwake, ISingletonUpdate
{
    private AudioSource[] pool;

    public void Awake()
    {
        pool = new AudioSource[16];
        // 初始化音频池
    }

    public void Update()
    {
        // 每帧检查音频状态，回收已播放完毕的 AudioSource
    }

    public override void Dispose()
    {
        // 释放音频资源
    }
}
```

注册：

```csharp
Game.AddSingleton<AudioManager>();
// Awake 自动调用，Update 自动注册进调度队列
```

## 设计总结

| 设计点 | 解决的问题 |
|--------|------------|
| `[StaticField]` 标注静态字段 | HybridCLR 热重载时可自动清理静态状态 |
| Stack 反序销毁 | 保证依赖关系安全：后依赖先销毁 |
| 快照计数调度 | 遍历时安全注册/销毁，不影响当前帧 |
| 按需实现生命周期接口 | 零成本抽象，没有无谓的虚函数调用 |
| Game.WaitFrameFinish | 异步等待帧尾，跨系统同步数据 |
| gameDeltaTime 倍速 | 一行代码支持慢动作/快进，所有系统统一读这个值 |

这套单例 + 主循环系统是整个游戏框架的"心跳"，理解它的设计原则，是读懂框架其他部分的基础。
