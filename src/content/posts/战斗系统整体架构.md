# 01 战斗系统整体架构

> 面向刚入行的游戏开发毕业生，本文将带你从整体视角理解这套战斗系统是如何设计、组织和运行的。

---

## 1. 系统概述

本项目的战斗系统是一套**回合制 + 实时动作**混合的战斗引擎，运行在 Unity 热更新层（Hotfix）之上，采用 **帧同步（LockStep）** 保证多端一致，使用 **ET 框架**（Entity-Component 模型）组织业务对象，并通过 **UniScript 可视化脚本**将战斗逻辑与代码解耦，由策划直接编辑战斗流程。

### 1.1 战斗系统的核心目标

| 目标 | 说明 |
|------|------|
| 一致性 | 多客户端、服务端的战斗逻辑结果完全一致（帧同步保证） |
| 可扩展性 | 技能、Buff、AI 等均通过可视化图（UniScript）配置，策划可独立迭代 |
| 高性能 | 确定性定点数（TrueSync）运算，避免浮点误差；对象池复用减少GC |
| 可维护性 | 清晰的分层架构：数据层、逻辑层、表现层分离 |

### 1.2 整体技术栈

```
┌─────────────────────────────────────────────────┐
│            Unity 渲染 & 表现层                    │
│  动画 / Timeline / 特效 / UI / 音效（Wwise）       │
├─────────────────────────────────────────────────┤
│           战斗逻辑层（Hotfix 热更新）               │
│  Unit / FSM / Skill / Buff / AI / Collider       │
│  UniScript 可视化脚本驱动战斗决策                  │
├─────────────────────────────────────────────────┤
│           帧同步基础设施                           │
│  LockStep / VKey 输入 / 确定性随机 / Hash校验      │
├─────────────────────────────────────────────────┤
│           ET 框架（ECS 模型）                      │
│  Entity / Component / System / EventSystem       │
├─────────────────────────────────────────────────┤
│           TrueSync 定点数库                        │
│  FP / TSVector / TSQuaternion / TSMatrix4x4      │
└─────────────────────────────────────────────────┘
```

## 2. 架构设计

### 2.1 分层架构

战斗系统分为四个大层：

#### 数据配置层
- 由 Luban 工具生成 `cfg.*` 命名空间下的只读配置类（如 `CUnit`、`CSubSkill`）
- 策划在 Excel 中填写数值，程序无需修改代码即可调整游戏平衡

#### 战斗逻辑层（核心）
- 路径：`Assets/Scripts/Hotfix/Battle/Model/`
- 使用 ET 框架的 Entity-Component 模式
- 每个战斗实体（`Unit`）挂载多个组件（FSM、Skill、Buff、Collider 等）
- 通过 `IFixedUpdate` 接口在固定帧（通常 30fps）驱动逻辑

#### 可视化脚本层（UniScript）
- 基于自研 UniScript 框架，提供 FlowCanvas（流程图）和 BehaviorTree（行为树）两种图
- 策划/技术美术在编辑器中拖拽节点配置技能流程
- 运行时序列化为 JSON，热更新不依赖 C# 编译

#### 表现层
- 监听逻辑层事件，驱动动画、特效、UI
- 与逻辑层通过 `EventSystem.Instance.Publish` 事件解耦

### 2.2 核心子系统关系

```
战斗场景(Scene)
│
├── UnitComponent（单位管理器）
│     ├── Unit（玩家角色、敌人）
│     │     ├── FSMComponent         ← 状态机，驱动角色状态流转
│     │     ├── SkillComponent       ← 技能状态，持有当前技能图
│     │     ├── BuffComponent        ← Buff 列表管理
│     │     ├── AttackComponent      ← 攻击命中记录
│     │     ├── ColliderComponent    ← 攻击/受击碰撞体
│     │     ├── NumericComponent     ← 数值（血量/攻击力等）
│     │     ├── BlackboardComponent  ← 运行时黑板（临时变量）
│     │     └── BTComponent          ← AI 行为树（AI 单位）
│     └── TeamEntity（队伍）
│           └── List<Unit> TeamMember
│
└── 帧同步驱动（LockStep）
      └── VKey 输入 → FixedUpdate → 逻辑推进
```

### 2.3 帧驱动流程

```
每帧（FixedUpdate）：
1. LockStep 接收 VKey 输入
2. 各 Unit 依次执行 FixedUpdate
   ├── FSMComponent.FixedUpdate  → 驱动状态机
   ├── SkillComponent.FixedUpdate → 推进技能 Timeline
   └── ColliderComponent.LateFixedUpdate → 碰撞检测
3. EventSystem 派发事件（命中、死亡等）
4. 表现层响应事件更新渲染
```

### 2.4 ET 框架基础概念

初学者必须理解 ET 框架的三个核心概念：

**Entity（实体）**：所有游戏对象的基类，每个 Entity 有唯一 Id，可以挂载 Component，也可以成为另一个 Entity 的子级。

**Component（组件）**：具有特定功能的数据+逻辑单元。通过 `[ComponentOf(typeof(XXX))]` 属性标记归属实体。

**System（系统）**：通过实现 `IAwake`、`IFixedUpdate`、`IDestroy` 等接口，ET 框架自动调用对应生命周期方法。

```csharp
// 示例：Unit 是一个 Entity，可以获取挂载的组件
var fsm = unit.GetComponent<FSMComponent>();
var skill = unit.GetComponent<SkillComponent>();
var buff = unit.GetComponent<BuffComponent>();
```

## 4. 设计亮点

### 4.1 Entity-Component 完全解耦
每个功能模块（FSM、技能、Buff 等）都是独立的 Component，Unit 本身只持有基础数据（坐标、队伍、暂停状态等）。增加新功能只需新增 Component，不修改 Unit 本体，完全符合开闭原则。

### 4.2 定点数确保帧同步一致性
所有坐标、速度、数值计算使用 `TrueSync` 库的 `FP`（Fixed Point，定点数）类型，彻底避免跨平台浮点数误差，是帧同步战斗的基石。

### 4.3 热更新友好的架构
战斗逻辑全部在 Hotfix 层（ILRuntime/HybridCLR 热更新），策划配置的 UniScript 图序列化为 JSON，均可不发包热更。

### 4.4 事件驱动降低耦合
模块间通过 `EventSystem.Instance.Publish` 传递事件（如 `NumericChange`、`Evt_UnitWrap`），发布者和订阅者完全解耦，可以随时增删监听者。

### 4.5 多层暂停机制
Unit 的暂停使用 `HashSet<EPauseFlag>` 标志位集合，支持多种暂停原因同时存在（如战斗暂停 + 演出暂停），只有所有暂停标志都清除才真正恢复。

## 5. 常见问题与最佳实践

### Q1：为什么坐标要用 TSVector 而不是 UnityEngine.Vector3？
**A**：`TSVector` 使用定点数（ Fixed Point ），在不同平台（iOS、Android、Windows）上的计算结果完全一致。`Vector3` 使用 IEEE 浮点数，在不同硬件/编译器优化下可能有微小差异，帧同步积累误差后会导致多端状态分叉（不同步）。

### Q2：什么时候使用 FixedUpdate，什么时候用 Update？
**A**：战斗逻辑全部放在 `IFixedUpdate` 中，固定帧率执行，确保帧同步。表现层（动画插值、UI 刷新）才用 `Update`，因为表现层不参与同步计算。

### Q3：如何新增一种 Unit 身上的属性？
**A**：推荐方式：
1. 在 `ENumericId` 枚举中添加新的属性 ID（如 `Speed`）
2. 在 Luban 配置表中配置初始值
3. 通过 `NumericComponent` 的 `AddModifier` 方法动态修改

### Q4：Entity 和 Component 的生命周期是怎样的？
**A**：创建时调用 `IAwake` 接口，每帧调用 `IFixedUpdate`/`IUpdate`，销毁时调用 `IDestroy`，重置时调用 `IReset`（对象池复用场景）。

### Q5：DeterministicDictionary 和普通 Dictionary 有什么区别？
**A**：`DeterministicDictionary` 的遍历顺序是确定的（按插入顺序），而 `Dictionary` 的遍历顺序在不同运行时可能不同。帧同步中若遍历顺序不一致会导致计算结果差异，因此用 `DeterministicDictionary` 替代。

                              │
                    ┌─────────┴────────┐
                    ▼                  ▼
              BuffComponent      ColliderComponent
              （增益/减益）        （碰撞检测）
                    │                  │
                    ▼                  ▼
