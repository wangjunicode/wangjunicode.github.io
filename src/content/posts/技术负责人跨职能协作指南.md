## 技术负责人的跨职能挑战

```
刚晋升为 TL 时最大的困惑：

"策划老是改需求，程序来不及做怎么办？"
"产品说两周必须上线，但技术上评估需要六周，怎么谈？"
"美术说这个特效一定要有，但性能不允许，怎么处理？"
"需求不清楚但老板要求先开发，风险怎么控制？"
```

这些问题没有标准答案，但有**系统化的思路**。

## 一、需求评估的方法论

### 1.1 理解"需求背后的需求"

```
表面需求 vs 真实需求的差异：

策划提：
  "登录界面要有粒子特效，要炫酷"
  
真实需求可能是：
  "游戏上线后第一印象不够好，需要提升质感"
  
当你理解了真实需求，就有更多方案可选：
  方案A：粒子特效（原始需求，工期：3天，性能代价高）
  方案B：高质量UI动画（更可控，工期：1天，性能开销小）
  方案C：换一张更有质感的背景图（工期：0.5天，零性能代价）
  
  可以提出："我建议先用方案C快速验证视觉效果，
  如果不够，再迭代做方案B，这样我们可以在明天就看到效果。"
```

### 1.2 技术需求评估框架

```
收到需求后的评估步骤：

1. 理解目标（Why）
   - 这个功能解决什么用户问题？
   - 有没有更简单的方案可以达到同样目标？

2. 明确边界（What）
   - 核心功能是什么？（必须有）
   - 附加功能是什么？（可以有）
   - 超出范围是什么？（明确不做）

3. 识别依赖（Dependencies）
   - 需要哪些前置条件？
   - 是否依赖其他系统/团队？

4. 评估风险（Risk）
   - 技术不确定性在哪里？
   - 需要原型验证哪些假设？

5. 工期估算（Timeline）
   - 开发时间（包含测试）
   - 上下游联调时间
   - Buffer（通常 20~30%）
```

## 二、工期估算

### 2.1 估算陷阱

```
常见的估算失误：

1. 只估算"编码时间"，忘记了：
   - 理解需求（20%）
   - 单元测试（20%）
   - 联调（15%）
   - 代码 Review 和修改（10%）
   - Bug 修复（15%）
   
2. 以"理想状态"估算
   实际上：会议、协作消耗约 30% 的工作时间
   
3. 没有留 Buffer
   项目上有什么是 100% 能准时完成的？没有。
   
4. 忘记了依赖项的不确定性
   "等设计稿出来就开始" → 设计稿晚了 3 天
```

### 2.2 三点估算法

```
三点估算：
  O（Optimistic）：一切顺利的情况，最短时间
  M（Most likely）：正常情况，最可能的时间
  P（Pessimistic）：遇到意外的情况，最长时间

期望 E = (O + 4M + P) / 6

例：
  实现网络同步功能
  O = 5天（一切顺利，接口清晰，测试也快）
  M = 10天（正常开发节奏）
  P = 20天（网络协议有问题，需要多次联调）
  
  E = (5 + 4×10 + 20) / 6 = 65 / 6 ≈ 11天

标准差 σ = (P - O) / 6 = (20 - 5) / 6 ≈ 2.5天

→ 对外承诺：11天，内部心理预期：11~14天
```

### 2.3 如何处理"强压工期"

```
场景：老板说"两周必须上线，没有商量余地"

步骤1：先别反驳，先理解
  "老板，能说说为什么两周这个时间点很重要吗？"
  （可能是某个商业机会，也可能只是拍脑袋）
  
步骤2：呈现事实（不是对抗，是透明）
  "我做了评估，这个功能完整实现需要 5 周。
   我来说明一下为什么..."
  
步骤3：提出选项（给决策者选择，而不是只说不行）
  方案A：完整版（5周）
  方案B：精简版（2周），暂不包含 [具体功能]
  方案C：2周先上核心版本，后续迭代增加功能
  
  "我建议方案C，核心价值两周可以交付。你选哪个？"
  
步骤4：记录决策
  无论结果如何，邮件记录：
  "根据今天的讨论，我们选择了方案C，
   两周上线核心功能，[某某功能]放入下期迭代。"
```

## 三、技术风险的沟通

### 3.1 风险可视化

```
不要用技术语言谈风险，用"影响"来谈

❌ 技术人的说法（对方听不懂）：
  "这个方案有内存碎片问题，GC 压力会导致性能抖动"

✅ 对非技术人员的说法：
  "如果按这个方案，玩家每隔30秒左右可能出现一次
   约0.1秒的卡顿，高端手机可能感知不明显，
   中低端手机会比较明显。预计会影响约30%的玩家。
   
   我们可以花额外1天做优化消除这个问题，
   或者接受现状先上线观察。你觉得怎么处理？"
```

### 3.2 技术债务的沟通

```markdown
## 技术债务汇报模板

**背景**：
当前 [模块名] 的架构设计于 [时间]，当时 [限制条件]。
随着业务发展，[具体问题]。

**风险**：
如果不处理：
  - 短期（1个月内）：[具体影响]
  - 长期（半年后）：[具体影响]
  - 量化数据：[开发效率下降 X%，Bug 率增加 Y%]

**解决方案**：
  工期：X 个工程师天
  方案：[具体怎么做]
  收益：[量化预期收益]

**建议**：
在 [Q2 迭代] 中安排 [X 人日] 解决此问题，
预计可以减少 [Y 个功能] 的开发时间。
```

## 四、策划与程序的协作模式

### 4.1 建立策划友好的架构

```
策划最大的痛点：
  "为什么改个数值要程序帮忙？"
  "为什么加个技能要等程序两周？"

解决方案：数据驱动架构

技能配置表（Excel/JSON），策划自己填：
  skill_id, name, damage_base, damage_ratio, 
  cooldown, range, buff_ids, animation_name

程序只处理：
  解析配置、执行逻辑、触发表现

效果：
  策划添加新技能 = 填一行配置表（不需要程序）
  策划调整数值 = 修改配置表，热更（不需要发版）
  
这才是"策划友好"的架构
```

### 4.2 需求变更管理

```
需求变更是正常的，但无节制的变更是灾难

建立变更流程：

1. 冻结期（Feature Freeze）
   迭代后期（最后1/4的时间）不接受新需求
   只修 Bug，不加功能

2. 变更代价透明化
   每次变更都要评估：影响哪些已完成工作？
   多少工期？交由决策者权衡
   
3. "加一个功能同时取消一个功能"原则
   总工期不变的情况下，可以增加需求，
   但同时要从已有需求中移除等量工作

4. 用数据记录变更
   用 TAPD/Jira 记录所有需求变更历史
   复盘时可以分析：变更率多少？主要变更在哪个阶段？
```

## 五、高效会议

### 5.1 会议预防

```
最好的会议是不需要开的会议

判断是否需要会议：
  □ 决策需要多人参与？
  □ 信息有歧义需要当面澄清？
  □ 问题无法通过文档/异步沟通解决？

可以替代会议的方式：
  - 明确的文档（让人看完文档再讨论）
  - 异步评论（Confluence/飞书文档评论）
  - 一对一快速对齐（5分钟 > 1小时会议）
```

### 5.2 高效会议技巧

```
每次会议必须有：
  1. 明确的目的：这个会议要做出什么决定/结论？
  2. 参与者名单：只邀请真正需要参与的人
  3. 会前资料：提前发出，要求提前阅读
  4. 时间限制：30/45/60 分钟，设好时钟
  5. 行动项：结束前明确：谁在什么时间完成什么

会后必须有：
  会议纪要（5分钟内发出）：
  - 决定了什么
  - 行动项：[责任人] [截止日期] [任务描述]
  
没有会议纪要 = 这个会议没有开过
```

## 总结

技术负责人的跨职能协作要点：

```
与产品/策划协作：
  ✅ 理解需求背后的目标
  ✅ 用影响而非技术语言谈风险
  ✅ 提供选项，让对方做决策
  ✅ 用数据支撑技术债务的决策

工期管理：
  ✅ 三点估算，留足 Buffer
  ✅ 工期压力下提供精简方案选项
  ✅ 记录决策，避免"反正是你说可以的"

会议文化：
  ✅ 可以不开的会议就不开
  ✅ 每个会议都有结论和行动项
  ✅ 5分钟内发出会议纪要
```

**核心认知**：技术负责人不是"技术最强的程序员"，而是**连接技术与业务的桥梁**。能够让技术服务于业务目标，同时在技术债务积累到危险程度前及时干预，才是 TL 的核心价值。
