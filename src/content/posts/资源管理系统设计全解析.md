## 资源管理的核心矛盾

游戏资源管理要解决的根本矛盾：

```
内存有限 vs 资源无限大
加载速度要快 vs 磁盘/网络读取慢
资源要复用 vs 不用时要释放
```

一个好的资源管理系统，需要在这三个矛盾中找到最优解。

---

## 一、AssetBundle 基础与原理

### 1.1 AssetBundle 的本质

```
AssetBundle = 一个或多个资源的压缩包 + 资源索引

内部结构：
  header:     包含资源列表、依赖关系、校验信息
  data block: 实际资源数据（可以 LZMA/LZ4 压缩）

加载流程：
  1. 加载 AB 包（解压头部，建立索引）
  2. 加载具体资源（AssetBundle.LoadAsset<T>）
  3. 使用资源（实例化等）
  4. 释放资源（Resources.UnloadAsset 或 Destroy）
  5. 卸载 AB 包（ab.Unload(false/true)）
```

### 1.2 手动管理 AssetBundle 的痛点

```csharp
// 手动管理的代码（充满陷阱）
AssetBundle LoadWithDependencies(string bundleName)
{
    // 问题1：需要手动维护依赖关系
    var manifest = // 从哪里读 manifest？
    var deps = manifest.GetAllDependencies(bundleName);
    
    foreach (var dep in deps)
    {
        if (!_loadedBundles.ContainsKey(dep))
        {
            // 问题2：循环依赖会导致无限递归
            LoadWithDependencies(dep);
        }
        // 问题3：引用计数错误会导致提前卸载或内存泄漏
        _refCounts[dep]++;
    }
    
    return AssetBundle.LoadFromFile(bundleName);
}

// ab.Unload(false) vs ab.Unload(true) 的陷阱：
// Unload(false)：卸载 AB 包本身，已实例化的对象继续存在，但失去了重新加载的能力
// Unload(true)：卸载 AB 包 + 所有已加载的对象（包括场景中的！）
// 错误使用会导致材质/纹理变成粉红色
```

## 二、Addressables：现代资源管理

### 2.1 Addressables 的核心概念

```
Addressable Asset：
  给资源一个逻辑地址（如 "Characters/Hero_Prefab"）
  与实际文件路径解耦

AsyncOperationHandle<T>：
  异步操作的句柄
  必须手动调用 Addressables.Release() 释放
  未释放 = 内存泄漏

IResourceLocator：
  资源地址到实际位置的映射
  支持从本地/远端 CDN 动态切换

ResourceManager：
  内部引用计数管理
  自动处理依赖 AB 包的加载/卸载
```

### 2.2 正确的资源生命周期管理

```csharp
/// <summary>
/// 资源引用安全包装器
/// 确保资源释放不被遗漏
/// </summary>
public sealed class AssetRef<T> : IDisposable where T : UnityEngine.Object
{
    private AsyncOperationHandle<T> _handle;
    private bool _disposed;
    
    public T Asset { get; private set; }
    public bool IsValid => !_disposed && _handle.IsValid() && Asset != null;
    
    private AssetRef() { }
    
    public static async UniTask<AssetRef<T>> LoadAsync(string address, CancellationToken ct = default)
    {
        var handle = Addressables.LoadAssetAsync<T>(address);
        
        try
        {
            await handle.WithCancellation(ct);
        }
        catch
        {
            if (handle.IsValid()) Addressables.Release(handle);
            throw;
        }
        
        if (handle.Status != AsyncOperationStatus.Succeeded)
        {
            Addressables.Release(handle);
            throw new Exception($"Failed to load asset: {address}");
        }
        
        return new AssetRef<T>
        {
            _handle = handle,
            Asset = handle.Result
        };
    }
    
    public void Dispose()
    {
        if (_disposed) return;
        _disposed = true;
        Asset = null;
        
        if (_handle.IsValid())
            Addressables.Release(_handle);
    }
}

// 使用示例
public class WeaponSystem : MonoBehaviour
{
    private AssetRef<GameObject> _currentWeaponRef;
    private GameObject _currentWeaponInstance;
    
    public async UniTask EquipWeapon(string weaponAddress, CancellationToken ct)
    {
        // 卸载旧武器
        UnequipCurrentWeapon();
        
        // 加载新武器
        _currentWeaponRef = await AssetRef<GameObject>.LoadAsync(weaponAddress, ct);
        
        // 实例化
        _currentWeaponInstance = Instantiate(_currentWeaponRef.Asset, weaponSlot);
    }
    
    private void UnequipCurrentWeapon()
    {
        if (_currentWeaponInstance != null)
        {
            Destroy(_currentWeaponInstance);
            _currentWeaponInstance = null;
        }
        
        _currentWeaponRef?.Dispose(); // 释放资源
        _currentWeaponRef = null;
    }
    
    void OnDestroy()
    {
        UnequipCurrentWeapon(); // 确保销毁时释放
    }
}
```

### 2.3 场景加载与资源管理

```csharp
/// <summary>
/// 场景加载管理器：确保场景卸载时正确释放资源
/// </summary>
public class SceneManager : MonoBehaviour
{
    private AsyncOperationHandle<SceneInstance> _currentSceneHandle;
    private readonly List<IDisposable> _sceneResources = new();
    
    public async UniTask LoadSceneAsync(string sceneAddress, CancellationToken ct)
    {
        // 1. 卸载当前场景（释放所有场景资源）
        await UnloadCurrentScene();
        
        // 2. 加载新场景
        _currentSceneHandle = Addressables.LoadSceneAsync(
            sceneAddress, 
            UnityEngine.SceneManagement.LoadSceneMode.Single);
        
        await _currentSceneHandle.WithCancellation(ct);
        
        if (_currentSceneHandle.Status != AsyncOperationStatus.Succeeded)
            throw new Exception($"Failed to load scene: {sceneAddress}");
        
        // 3. 场景加载完成后初始化
        OnSceneLoaded(_currentSceneHandle.Result.Scene);
    }
    
    private async UniTask UnloadCurrentScene()
    {
        // 释放场景持有的所有资源
        foreach (var resource in _sceneResources)
            resource.Dispose();
        _sceneResources.Clear();
        
        // 卸载 Addressable 场景
        if (_currentSceneHandle.IsValid())
        {
            await Addressables.UnloadSceneAsync(_currentSceneHandle).Task;
        }
    }
    
    private void OnSceneLoaded(Scene scene)
    {
        // 场景内的对象可以在这里注册它们的资源
        // 确保场景卸载时这些资源会被释放
    }
    
    // 提供给场景内对象注册资源的接口
    public void RegisterSceneResource(IDisposable resource)
    {
        _sceneResources.Add(resource);
    }
}
```

## 三、资源分组策略

### 3.1 按加载时机分组

```
资源分组原则：
  同一时间加载的资源放在一起 = 最小化下载等待

分组建议：
┌─────────────────────────────────────────────────┐
│ Startup Group（随启动加载，必须完成才进游戏）      │
│   - 公共 UI 图集                                  │
│   - 字体                                          │
│   - 音频管理器配置                                 │
├─────────────────────────────────────────────────┤
│ Common Group（游戏大厅时加载）                     │
│   - 主界面资源                                    │
│   - 公共特效                                      │
│   - 背景音乐                                      │
├─────────────────────────────────────────────────┤
│ Battle_XXX Group（进战斗时加载）                   │
│   - 关卡地图                                      │
│   - 战斗角色                                      │
│   - 战斗 UI                                       │
├─────────────────────────────────────────────────┤
│ Event_YYY Group（活动时下载）                      │
│   - 活动专属资源                                  │
│   - 活动 UI                                       │
└─────────────────────────────────────────────────┘
```

### 3.2 避免依赖跨组引用的问题

```
问题：A 组的 Prefab 引用了 B 组的 Texture
  加载 A 时，Addressables 会自动加载 B（为了解析依赖）
  但卸载 A 时，Addressables 不会自动卸载 B
  → 如果很多资源都引用 B 组的资源，B 会一直留在内存中

解决方案：
  方案1：共享资源放到专门的 Shared Group，永不卸载
  方案2：将高频引用的资源内嵌到引用它的 Group 中（增大包体但管理简单）
  方案3：使用 Addressables 的 Dependencies 工具分析引用关系，手动优化

实践建议：
  定期运行 Analyze 工具（Addressables → Analyze）
  检查 "Check Duplicate Bundle Dependencies"
  将重复引用的资源提取到 Shared Group
```

## 四、预加载策略

### 4.1 预测性预加载

```csharp
/// <summary>
/// 预加载管理器：在玩家进入某场景之前，提前加载资源
/// </summary>
public class PreloadManager
{
    private readonly Dictionary<string, List<AsyncOperationHandle>> _preloadedHandles = new();
    
    /// <summary>
    /// 预加载一组资源（后台加载，不阻塞游戏）
    /// </summary>
    public async UniTaskVoid PreloadGroupAsync(string groupKey, IEnumerable<string> addresses, 
                                                CancellationToken ct)
    {
        if (_preloadedHandles.ContainsKey(groupKey)) return; // 已在预加载
        
        var handles = new List<AsyncOperationHandle>();
        _preloadedHandles[groupKey] = handles;
        
        foreach (var address in addresses)
        {
            // 只下载到缓存，不加载到内存
            var handle = Addressables.DownloadDependenciesAsync(address, false);
            handles.Add(handle);
            
            // 不 await，让它在后台运行
            // 加载时会从本地缓存读取，速度快很多
        }
    }
    
    /// <summary>
    /// 释放预加载的句柄
    /// </summary>
    public void ReleaseGroup(string groupKey)
    {
        if (!_preloadedHandles.TryGetValue(groupKey, out var handles)) return;
        
        foreach (var handle in handles)
        {
            if (handle.IsValid()) Addressables.Release(handle);
        }
        
        _preloadedHandles.Remove(groupKey);
    }
}

// 使用场景：
// 玩家在大厅时，预加载当前关卡的资源（预测玩家会进入）
// 玩家进入关卡时，资源已在本地缓存，加载速度大幅提升
public class LobbyController : MonoBehaviour
{
    private PreloadManager _preloadManager;
    
    void Start()
    {
        // 玩家进入大厅，预加载第一关
        _preloadManager.PreloadGroupAsync(
            "Level_1", 
            new[] { "Levels/Level_1", "Characters/Hero", "UI/BattleHUD" },
            destroyCancellationToken
        ).Forget();
    }
}
```

### 4.2 加载进度显示

```csharp
/// <summary>
/// 聚合多个异步操作的进度
/// </summary>
public class LoadingProgress
{
    private readonly List<AsyncOperationHandle> _handles = new();
    
    public float Progress
    {
        get
        {
            if (_handles.Count == 0) return 1f;
            
            float total = 0f;
            foreach (var handle in _handles)
                total += handle.PercentComplete;
            
            return total / _handles.Count;
        }
    }
    
    public bool IsDone => _handles.All(h => h.IsDone);
    
    public void Add(AsyncOperationHandle handle) => _handles.Add(handle);
    
    public void Clear()
    {
        foreach (var handle in _handles)
            if (handle.IsValid()) Addressables.Release(handle);
        _handles.Clear();
    }
}

// 加载界面
public class LoadingScreen : MonoBehaviour
{
    [SerializeField] private Slider _progressSlider;
    [SerializeField] private Text _progressText;
    
    private LoadingProgress _progress;
    
    public async UniTask LoadSceneWithProgressAsync(string sceneAddress, CancellationToken ct)
    {
        _progress = new LoadingProgress();
        
        // 添加要追踪的操作
        var sceneHandle = Addressables.LoadSceneAsync(sceneAddress);
        _progress.Add(sceneHandle);
        
        // 更新进度
        while (!_progress.IsDone)
        {
            _progressSlider.value = _progress.Progress;
            _progressText.text = $"{(int)(_progress.Progress * 100)}%";
            await UniTask.NextFrame(ct);
        }
        
        _progressSlider.value = 1f;
        _progressText.text = "100%";
    }
}
```

## 五、内存泄漏排查

### 5.1 常见的内存泄漏场景

```csharp
// ❌ 场景1：AsyncOperationHandle 未释放
public class ProblemCode : MonoBehaviour
{
    async void Start()
    {
        var handle = Addressables.LoadAssetAsync<Texture2D>("Icons/Health");
        await handle.Task;
        
        _icon.texture = handle.Result;
        // 忘记 Addressables.Release(handle)！
        // 每次打开这个 UI，就多一份内存
    }
}

// ❌ 场景2：实例化的 Prefab 销毁但 handle 未释放
public class ProblemCode2 : MonoBehaviour
{
    private AsyncOperationHandle<GameObject> _prefabHandle;
    
    async void Spawn()
    {
        _prefabHandle = Addressables.LoadAssetAsync<GameObject>("Enemies/Dragon");
        await _prefabHandle.Task;
        
        var instance = Instantiate(_prefabHandle.Result);
        Destroy(instance); // 销毁实例，但 _prefabHandle 仍持有对 Prefab 的引用
        // AB 包因为 _prefabHandle 而无法被卸载
    }
}

// ✅ 正确做法：使用 InstantiateAsync（Addressables 管理引用计数）
public class CorrectCode : MonoBehaviour
{
    private AsyncOperationHandle<GameObject> _instanceHandle;
    
    async void Spawn()
    {
        // InstantiateAsync 内部管理 Prefab 的引用计数
        _instanceHandle = Addressables.InstantiateAsync("Enemies/Dragon", transform);
        await _instanceHandle.Task;
        
        // 不需要 Instantiate，直接使用
        // 销毁时：
    }
    
    void OnDestroy()
    {
        // 正确释放：自动减少引用计数，可能触发 AB 卸载
        if (_instanceHandle.IsValid())
            Addressables.ReleaseInstance(_instanceHandle);
    }
}
```

### 5.2 Memory Profiler 工作流

```
1. 进入游戏，拍摄基准快照（Snapshot A）

2. 执行目标操作（如：打开背包 → 关闭背包）

3. 执行 GC.Collect()，等待 1~2 秒

4. 拍摄快照（Snapshot B）

5. 对比 A 和 B：
   - 纹理内存增加了？找到对应的 Texture 对象，查看 References
   - AssetBundle 没有卸载？说明有未释放的 handle

6. 关注点：
   - "Leaked Managed Shell"：托管层持有已销毁的 Native 对象
   - AssetBundle 数量：操作前后应该相等
   - 纹理总数：界面关闭后，专属纹理应该被释放
```

## 六、构建优化

### 6.1 减少 AssetBundle 数量

```
原则：AB 包数量越多，加载时的 I/O 请求越多
     AB 包越大，首次加载时间越长

平衡点：
  - 太碎（每个资源一个 AB）：大量 HTTP 请求，移动端有连接数限制
  - 太大（所有资源一个 AB）：更新时需要重新下载整个包

建议：
  - 单个 AB 包大小：2~5 MB（移动端最优）
  - 相关性强的资源放同一 AB（如同一关卡的资源）
  - 公共资源单独一个 AB（避免重复打包）
```

### 6.2 压缩格式选择

```
LZ4（默认）：
  - 压缩率：约 40%
  - 解压速度：快（实时解压）
  - 适合：移动端实时加载的资源

LZMA：
  - 压缩率：约 60%（更小）
  - 解压速度：慢（需要完整解压才能读取）
  - 适合：下载包（下载后解压为 LZ4 缓存）

None（不压缩）：
  - 适合：已经高度压缩的资源（如 ASTC 纹理）
  - 二次压缩收益很小，但增加 CPU 开销

在 Addressables 中设置：
  AddressableAssetSettings → Bundle Mode → Build Bundle Layout
  每个 Group 可以单独设置压缩格式
```

## 总结

资源管理的核心设计原则：

1. **所有加载必须有对应的释放**：使用 RAII 模式（IDisposable），确保资源不泄漏
2. **异步优先**：避免同步加载阻塞主线程，影响帧率
3. **按需加载**：用到时加载，不用时释放，配合预加载减少等待
4. **分组合理**：同时用到的资源放一起，减少依赖复杂度

资源管理问题难以发现（平时看不出来，积累到一定程度突然崩溃），因此需要：
- 在 Code Review 中重点检查资源的释放逻辑
- 定期用 Memory Profiler 进行内存检查
- 建立自动化的内存监控告警
