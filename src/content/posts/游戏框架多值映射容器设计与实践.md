# 游戏框架多值映射容器设计与实践

在大型游戏项目中，**一个 Key 对应多个 Value** 的场景极为普遍：计时器按触发时间分桶、事件系统按事件类型分组、AI 行为树按状态分类挂接回调……标准的 `Dictionary<K, List<V>>` 虽然能解决问题，但每次 `Add/Remove` 都需要大量模板代码，且频繁创建 `List<V>` 会给 GC 带来压力。本文结合框架源码，深入解析两种专为游戏设计的多值映射容器：`MultiMap<T, K>` 与 `UnOrderMultiMap<T, K>`。

---

## 一、为什么需要专用的多值映射容器

### 1.1 标准写法的痛点

```csharp
// 原始写法：大量模板代码
var dict = new Dictionary<int, List<string>>();

void Add(int key, string val) {
    if (!dict.TryGetValue(key, out var list)) {
        list = new List<string>();
        dict[key] = list;
    }
    list.Add(val);
}

void Remove(int key, string val) {
    if (dict.TryGetValue(key, out var list)) {
        list.Remove(val);
        if (list.Count == 0)
            dict.Remove(key);
    }
}
```

每个用到多值映射的地方都要重复这段逻辑，且忘记清理空列表会造成内存浪费。

### 1.2 游戏场景的额外需求

- **计时器**：`MultiMap<long, long>` 按触发时间戳存放 TimerId，每帧需要高效遍历有序的触发时间
- **事件系统**：按事件类型存放多个监听回调，注册/注销频繁
- **连击系统**：按 BuffId 存放多个叠加层，同一 Id 可以有多个来源

## 二、MultiMap：有序键 + 对象池优化

### 2.1 继承设计

```csharp
public class MultiMap<T, K> : SortedDictionary<T, List<K>>
```

**关键点**：继承自 `SortedDictionary`，而不是普通 `Dictionary`。这意味着所有 Key 会自动按自然顺序排列。在计时器场景中，Key 是时间戳（`long`），遍历时天然从最近的时间开始，无需额外排序。

### 2.2 对象池减少 GC

这是 `MultiMap` 最重要的设计亮点：

```csharp
private readonly int maxPoolCount;
private readonly Queue<List<K>> pool;

public MultiMap(int maxPoolCount = 0)
{
    this.maxPoolCount = maxPoolCount;
    this.pool = new Queue<List<K>>(maxPoolCount);
}

private List<K> FetchList()
{
    if (this.pool.Count > 0)
        return this.pool.Dequeue();
    return new List<K>(10);   // 预分配10个容量，减少扩容
}

private void Recycle(List<K> list)
{
    if (list == null) return;
    if (this.pool.Count == this.maxPoolCount) return;  // 池满则丢弃
    list.Clear();
    this.pool.Enqueue(list);
}
```

当一个 Key 下的所有 Value 都被移除时，内部的 `List<K>` 不会被 GC 回收，而是归还到 `pool` 中，下次有新 Key 需要容器时直接复用。

**效果**：在计时器高频添加/移除的场景下，大幅减少 `List<K>` 的内存分配，GC 压力显著降低。

### 2.3 完整接口实现

```csharp
// 添加 Key-Value 对
public void Add(T t, K k)
{
    List<K> list;
    this.TryGetValue(t, out list);
    if (list == null)
    {
        list = this.FetchList();  // 从对象池取
        this.Add(t, list);
    }
    list.Add(k);
}

// 移除特定 Value
public bool Remove(T t, K k)
{
    List<K> list;
    this.TryGetValue(t, out list);
    if (list == null) return false;
    if (!list.Remove(k)) return false;
    if (list.Count == 0)
        this.Remove(t);  // Key 下无 Value 则清除整个 Key
    return true;
}

// 移除整个 Key（同时回收 List）
public new bool Remove(T t)
{
    List<K> list;
    this.TryGetValue(t, out list);
    if (list == null) return false;
    this.Recycle(list);  // 回收到对象池
    return base.Remove(t);
}
```

### 2.4 安全访问器

```csharp
// 返回内部 List 引用（不要在遍历时修改！）
public new List<K> this[T t]
{
    get
    {
        this.TryGetValue(t, out List<K> list);
        return list ?? Empty;  // 不存在时返回静态空列表，避免 null 检查
    }
}

// 返回拷贝数组（线程安全访问）
public K[] GetAll(T t)
{
    List<K> list;
    this.TryGetValue(t, out list);
    if (list == null) return Array.Empty<K>();
    return list.ToArray();
}

// 获取第一个 Value
public K GetOne(T t)
{
    List<K> list;
    this.TryGetValue(t, out list);
    if (list != null && list.Count > 0)
        return list[0];
    return default;
}
```

## 三、UnOrderMultiMap：无序版的轻量选择

### 3.1 设计差异

```csharp
public class UnOrderMultiMap<T, K> : Dictionary<T, List<K>>
```

`UnOrderMultiMap` 继承自普通 `Dictionary`，**不保证 Key 顺序**，但哈希查找性能更好（O(1) vs O(log n)）。

| 特性 | MultiMap | UnOrderMultiMap |
|------|----------|-----------------|
| 底层结构 | SortedDictionary（红黑树） | Dictionary（哈希表） |
| Key 遍历顺序 | 升序有序 | 无序 |
| 查找复杂度 | O(log n) | O(1) 均摊 |
| 对象池 | ✅ 支持 | ❌ 无 |
| 适用场景 | 计时器、需要有序遍历 | 事件系统、高频查找 |

### 3.2 简洁实现

```csharp
public void Add(T t, K k)
{
    List<K> list;
    this.TryGetValue(t, out list);
    if (list == null)
    {
        list = new List<K>();
        base[t] = list;
    }
    list.Add(k);
}

public bool Remove(T t, K k)
{
    List<K> list;
    this.TryGetValue(t, out list);
    if (list == null) return false;
    if (!list.Remove(k)) return false;
    if (list.Count == 0)
        this.Remove(t);
    return true;
}
```

逻辑与 `MultiMap` 相同，但没有对象池，代码更简单，适合注册/注销不那么频繁的场景。

## 四、计时器系统中的实战应用

`MultiMap` 最典型的应用场景就是游戏计时器：

```csharp
public class TimerComponent : Singleton<TimerComponent>, ISingletonUpdate
{
    // Key: 触发时间戳（毫秒），Value: 该时刻需要触发的 TimerId 列表
    private readonly MultiMap<long, long> TimeId = new();
    
    private readonly Queue<long> timeOutTime = new();
    private readonly Queue<long> timeOutTimerIds = new();
    private readonly Dictionary<long, TimerAction> timerActions = new();
    
    // 缓存最小触发时间，避免每次遍历
    private long minTime = long.MaxValue;

    public void Update()
    {
        if (this.TimeId.Count == 0) return;

        long timeNow = TimeHelper.ClientFrameTime();
        
        // 快速排除：当前时间早于最小触发时间
        if (timeNow < this.minTime) return;

        // 收集所有已到期的时间点（SortedDictionary 保证从小到大遍历）
        foreach (KeyValuePair<long, List<long>> kv in this.TimeId)
        {
            long k = kv.Key;
            if (k > timeNow)
            {
                this.minTime = k;  // 更新最小时间缓存
                break;
            }
            this.timeOutTime.Enqueue(k);
        }

        // 分两个 Queue 分别收集时间点和 TimerId，避免遍历中修改集合
        while (this.timeOutTime.Count > 0)
        {
            long time = this.timeOutTime.Dequeue();
            var list = this.TimeId[time];
            for (int i = 0; i < list.Count; ++i)
                this.timeOutTimerIds.Enqueue(list[i]);
            this.TimeId.Remove(time);  // 触发后从 MultiMap 移除，内部 List 回收到对象池
        }

        while (this.timeOutTimerIds.Count > 0)
        {
            long timerId = this.timeOutTimerIds.Dequeue();
            if (!this.timerActions.Remove(timerId, out TimerAction timerAction))
                continue;
            this.Run(timerAction);
        }
    }
}
```

**为什么用两个 Queue？**

直接在 `foreach TimeId` 里调用 `TimeId.Remove` 会抛出 `InvalidOperationException`（集合被修改）。用 `timeOutTime` 先收集需要处理的时间点，再统一移除，是标准的"先收集再处理"模式。

### 4.1 注册定时器

```csharp
private void AddTimer(TimerAction timer)
{
    long tillTime = timer.StartTime + timer.Time;
    this.TimeId.Add(tillTime, timer.Id);      // 同一时刻可以有多个 Timer
    this.timerActions.Add(timer.Id, timer);
    if (tillTime < this.minTime)
        this.minTime = tillTime;              // 更新最小时间缓存
}

// 创建一次性定时器（异步等待版本）
public async ETTask WaitAsync(long time, ETCancellationToken cancellationToken = null, bool bWaitFrame = false)
{
    if (time == 0 && !bWaitFrame) return;
    
    long timeNow = TimeHelper.ClientFrameTime();
    ETTask tcs = ETTask.Create(true);
    TimerAction timer = TimerAction.Create(this.GetId(), TimerClass.OnceWaitTimer, timeNow, time, 0, tcs);
    this.AddTimer(timer);
    long timerId = timer.Id;

    void CancelAction()
    {
        if (this.Remove(timerId))
            tcs.SetResult();  // 取消时也要完成 Task，否则 await 永远挂起
    }

    try
    {
        cancellationToken?.Add(CancelAction);
        await tcs;
    }
    finally
    {
        cancellationToken?.Remove(CancelAction);
    }
}
```

使用方式极其简洁：

```csharp
// 等待 1000ms
await TimerComponent.Instance.WaitAsync(1000);

// 等待下一帧
await TimerComponent.Instance.WaitFrameAsync();

// 等待 5 帧
await TimerComponent.Instance.WaitFramesAsync(5);

// 可取消的等待
var cts = new ETCancellationToken();
await TimerComponent.Instance.WaitAsync(3000, cts);
// 其他地方可以 cts.Cancel() 中断
```

## 五、三种定时器模式对比

```csharp
public enum TimerClass
{
    None,
    OnceTimer,       // 一次性：触发后不重新注册，通过 EventSystem.Invoke 回调
    OnceWaitTimer,   // 一次性：触发后完成 ETTask，用于 await 等待
    RepeatedTimer,   // 重复：触发后重新注册自身，实现周期性回调
}
```

```csharp
private void Run(TimerAction timerAction)
{
    switch (timerAction.TimerClass)
    {
        case TimerClass.OnceTimer:
            EventSystem.Instance.Invoke(timerAction.Type, new TimerCallback { Args = timerAction.Object });
            timerAction.Recycle();  // 回收到对象池
            break;
            
        case TimerClass.OnceWaitTimer:
            ETTask tcs = timerAction.Object as ETTask;
            tcs.SetResult();        // 唤醒 await
            timerAction.Recycle();
            break;
            
        case TimerClass.RepeatedTimer:
            long timeNow = TimeHelper.ClientFrameTime();
            timerAction.StartTime = timeNow;
            this.AddTimer(timerAction);  // 重新注册！
            EventSystem.Instance.Invoke(timerAction.Type, new TimerCallback { Args = timerAction.Object });
            break;
    }
}
```

**实际使用建议**（源码注释原话）：
- `WaitAsync/WaitTillAsync`：等待时间**短**且逻辑需要连贯时使用（优点：代码可读性好）
- `NewOnceTimer/NewRepeatedTimer`：等待时间**长**或不需要逻辑连贯时使用（优点：支持热更新）

## 六、性能优化总结

### 6.1 MultiMap 性能收益分析

假设游戏同时存在 500 个计时器，每秒平均新增 100 个、移除 80 个：

| 方案 | 每秒 List 分配次数 | GC 压力 |
|------|-------------------|---------|
| 裸 Dictionary<K, List<V>> | ~100 次 | 高 |
| MultiMap（maxPoolCount=50） | ~50 次（池命中 50%） | 中 |
| MultiMap（maxPoolCount=200） | ~5 次（池命中 95%） | 极低 |

### 6.2 `minTime` 缓存的意义

每帧 Update 时，如果没有 `minTime` 缓存，需要调用 `SortedDictionary.First()` 获取最小键，时间复杂度 O(log n)。通过缓存最小时间戳，99% 的帧（没有计时器触发）直接一次比较返回，达到 O(1)。

### 6.3 双 Queue 的零分配遍历

`timeOutTime` 和 `timeOutTimerIds` 都是预分配的 `Queue<long>`，存放值类型，不涉及装箱。整个 Update 过程在正常帧（无触发）下零堆分配。

## 八、总结

`MultiMap` 和 `UnOrderMultiMap` 是游戏框架中高频使用的基础容器，它们通过继承标准集合并添加对象池等优化，既保留了熟悉的 API 风格，又显著降低了 GC 开销。理解这两个容器的实现，对于读懂框架的事件系统、计时器系统、AI 系统等核心模块至关重要。

**选择原则**：
- 需要有序遍历 Key（如计时器按时间触发）→ `MultiMap`（SortedDictionary）
- 只需要快速查找（如事件回调分组）→ `UnOrderMultiMap`（Dictionary）
- 高频添加删除，对 GC 敏感 → 优先 `MultiMap` 并设置合适的 `maxPoolCount`
