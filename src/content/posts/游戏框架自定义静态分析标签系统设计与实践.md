## 前言

在大型游戏框架中，光靠运行时检查往往来不及——等到 Bug 爆发已经是线上问题了。**编译期约束**才是最低成本、最高可靠性的防护手段。

ET 框架在 `Core/Analyzer` 目录下设计了一套自定义 `Attribute`，通过配合 Roslyn 分析器，在编译阶段就能检测出 ECS 实体/组件关系错误、常量 ID 重复、静态字段滥用等问题。

本文将逐一解析这些 Attribute 的设计意图与使用规范，帮助你理解"用标签约束代码结构"这一工程化思路。

---

## 一、ECS 结构约束 Attribute 系列

ET 框架基于 ECS 架构，Entity 之间有严格的父子和组件关系：

- **父子关系（Child）**：某个 Entity 只能作为特定类型 Entity 的子节点
- **组件关系（Component）**：某个 Component 只能挂载在特定类型的 Entity 上

如果这些关系在代码层面没有约束，开发者很容易挂错父节点或把 Component 挂到不该挂的 Entity 上，运行时才爆出空指针或逻辑错误。

### 1.1 ChildOfAttribute — 声明 Entity 的父节点类型

```csharp
[AttributeUsage(AttributeTargets.Class)]
public class ChildOfAttribute : Attribute
{
    public Type type;

    public ChildOfAttribute(Type type = null)
    {
        this.type = type;
    }
}
```

**用法：**

```csharp
// 声明 PlayerEntity 只能是 RoomEntity 的子节点
[ChildOf(typeof(RoomEntity))]
public class PlayerEntity : Entity
{
    // ...
}

// 不限制父节点类型（任意 Entity 都可以是父）
[ChildOf]
public class BuffEntity : Entity
{
    // ...
}
```

**设计意图：**

`type = null` 时表示"不约束父节点类型"；指定 `type` 时，Roslyn 分析器可以检测调用 `entity.AddChild<PlayerEntity>()` 的地方，确认调用方的 entity 类型是 `RoomEntity` 或其子类。

这直接消灭了一类运行时 Bug：**把 PlayerEntity 错误地添加为 BattleEntity 的子节点**，这类错误如果没有分析器，只有运行时才能发现。

### 1.2 ComponentOfAttribute — 声明 Component 的宿主实体类型

```csharp
/// <summary>
/// 组件类父级实体类型约束
/// 父级实体类型唯一的 标记指定父级实体类型 [ComponentOf(typeof(parentType))]
/// 不唯一则标记 [ComponentOf]
/// </summary>
[AttributeUsage(AttributeTargets.Class)]
public class ComponentOfAttribute : Attribute
{
    public Type Type;

    public ComponentOfAttribute(Type type = null)
    {
        this.Type = type;
    }
}
```

**用法：**

```csharp
// HpComponent 只能挂在 CharacterEntity 上
[ComponentOf(typeof(CharacterEntity))]
public class HpComponent : Entity
{
    public int MaxHp;
    public int CurrentHp;
}

// InputComponent 可以挂在任意 Entity 上
[ComponentOf]
public class InputComponent : Entity
{
    // ...
}
```

**与 ChildOfAttribute 的区别：**

| 标签 | 约束对象 | 典型场景 |
|---|---|---|
| `[ChildOf(typeof(X))]` | Entity 的父节点类型 | 场景树层级关系 |
| `[ComponentOf(typeof(X))]` | Component 的宿主 Entity 类型 | 数据组件挂载约束 |

在 ET 框架中，Component 和 Entity 本质上都是继承自 `Entity` 的类，通过 `AddChild` 和 `AddComponent` API 区分语义。这两个标签分别约束了这两套 API 的合法调用范围。

### 1.3 EnableAccessEntiyChildAttribute — 许可内部访问 Entity 子节点

```csharp
/// <summary>
/// 当方法或属性内需要访问 Entity 类的 child 和 component 时使用此标签
/// 仅供必要时使用，大多数情况推荐通过 Entity 的子类访问
/// </summary>
[AttributeUsage(AttributeTargets.Method | AttributeTargets.Property)]
public class EnableAccessEntiyChildAttribute : Attribute
{
}
```

**背景：**

ET 框架中，通过 `entity.GetChild<T>()` 或 `entity.GetComponent<T>()` 访问子节点/组件，通常被分析器认为是"不安全的通用访问"——因为这绕过了 `[ChildOf]` / `[ComponentOf]` 的类型约束，任何人都可以拿到任意子节点。

正常情况下，推荐的访问方式是：

```csharp
// 推荐：通过具体类型的成员属性访问
public HpComponent Hp => this.GetComponent<HpComponent>(); // 在 CharacterEntity 内部

// 不推荐：外部直接通用查询
HpComponent hp = someEntity.GetComponent<HpComponent>(); // 分析器会警告
```

当确实需要在方法内通用访问时，加上 `[EnableAccessEntiyChild]` 告诉分析器"这里是知情的豁免"：

```csharp
[EnableAccessEntiyChild]
public static void DumpEntityTree(Entity root)
{
    // 这里需要通用遍历所有 child，是合理的豁免场景
    foreach (var child in root.Children.Values)
    {
        Log.Debug($"  child: {child.GetType().Name}");
        DumpEntityTree(child);
    }
}
```

### 1.4 FriendOfAttribute — 声明"友元"访问权限

```csharp
[AttributeUsage(AttributeTargets.Class, AllowMultiple = true, Inherited = false)]
public class FriendOfAttribute : Attribute
{
    public Type Type;

    public FriendOfAttribute(Type type)
    {
        this.Type = type;
    }
}
```

这是对 **C++ friend 关键字**的 C# 模拟。C# 没有 friend 的语言支持，但通过 Roslyn 分析器可以实现类似效果。

**用法：**

```csharp
// 声明 BattleSystem 是 CharacterEntity 的"友元"
// BattleSystem 可以访问 CharacterEntity 的受保护成员（由分析器放行）
[FriendOf(typeof(CharacterEntity))]
public class BattleSystem
{
    public void DealDamage(CharacterEntity target, int damage)
    {
        // 分析器允许 BattleSystem 访问 target 的非公开成员
        target._internalHp -= damage; // 假设 _internalHp 是 internal 字段
    }
}
```

**为什么需要这个？**

在 ECS 框架中，Entity 的数据字段理论上只应该被特定 System 修改，但 C# 的访问修饰符太粗粒度（public/protected/internal/private），无法精确表达"只有 BattleSystem 可以修改 _hp"这种语义。

`[FriendOf]` + Roslyn 分析器实现了**比语言更细粒度的访问控制**：

- 非友元类访问受保护成员 → 分析器报错
- 友元类访问受保护成员 → 分析器放行

`AllowMultiple = true` 支持一个类声明多个友元：

```csharp
[FriendOf(typeof(BattleSystem))]
[FriendOf(typeof(SkillSystem))]
[FriendOf(typeof(HealSystem))]
public class CharacterEntity : Entity { ... }
```

## 二、UniqueIdAttribute — 常量 ID 唯一性检查

```csharp
/// <summary>
/// 唯一 Id 标签
/// 使用此标签标记的类，会检测类内部的 const int 字段成员是否唯一
/// 可以指定唯一 Id 的最小值/最大值区间
/// </summary>
[AttributeUsage(AttributeTargets.Class, Inherited = false)]
public class UniqueIdAttribute : Attribute
{
    public int Min;
    public int Max;

    public UniqueIdAttribute(int min = int.MinValue, int max = int.MaxValue)
    {
        this.Min = min;
        this.Max = max;
    }
}
```

### 2.1 解决的问题

游戏框架中充斥着大量"常量 ID"：事件类型 ID、计时器回调 ID、消息类型 ID……这些 ID 通常定义在同一个类的多个 `const int` 字段中：

```csharp
public static class TimerCoreInvokeType
{
    public const int GameTimer        = 1;
    public const int FrameTimer       = 2;
    public const int NetworkTimer     = 3;
    // ... 很多开发者维护，非常容易出现重复值
    public const int BattleSkillTimer = 2; // ← 不小心和 FrameTimer 重复了！
}
```

ID 重复是一类极其难排查的 Bug：逻辑上运行正常，但事件会被错误地路由到两个处理器，现象是偶发的、不可复现的。

### 2.2 使用方式

```csharp
[UniqueId(1, 1000)] // 要求所有 const int 值在 1~1000 范围内，且互不重复
public static class TimerCoreInvokeType
{
    public const int GameTimer        = 1;
    public const int FrameTimer       = 2;
    public const int NetworkTimer     = 3;
    public const int BattleSkillTimer = 4; // 分析器在编译时检查，重复立即报错
}

[UniqueId] // 不限范围，只检查唯一性
public static class EventType
{
    public const int PlayerLogin  = 10001;
    public const int PlayerLogout = 10002;
}
```

分析器会在编译时：
1. 收集类内所有 `const int` 字段的值
2. 检查是否有重复值 → 报错
3. 检查是否有值超出 `[Min, Max]` 范围 → 报错

**将运行时 Bug 提前到编译期捕获，零成本、零遗漏。**

## 三、StaticFieldAttribute — 静态字段生命周期管理

```csharp
/// <summary>
/// 静态字段需加此标签
/// valueToAssign: 初始化时的字段值
/// assignNewTypeInstance: 从默认构造函数初始化
/// </summary>
[AttributeUsage(AttributeTargets.Field | AttributeTargets.Property)]
public class StaticFieldAttribute : Attribute
{
    public readonly object valueToAssign;
    public readonly bool assignNewTypeInstance;

    public StaticFieldAttribute() { }

    public StaticFieldAttribute(object valueToAssign)
    {
        this.valueToAssign = valueToAssign;
        this.assignNewTypeInstance = false;
    }

    public StaticFieldAttribute(bool assignNewTypeInstance)
    {
        this.valueToAssign = null;
        this.assignNewTypeInstance = assignNewTypeInstance;
    }
}
```

### 3.1 解决的问题

游戏框架中，静态字段的生命周期管理是一个棘手问题：

**问题1：静态字段在 Domain Reload 时不重置（Unity 编辑器）**

Unity 编辑器在 Play/Stop 之间不一定完全重置静态字段（取决于 Domain Reload 设置）。如果一个系统的静态缓存在上次播放结束时有脏数据，下次播放会直接使用，导致莫名 Bug。

**问题2：静态字段难以统一初始化**

如果静态字段需要在游戏启动时初始化为某个特定值（而不是 C# 默认的零值），通常需要手动在初始化代码里逐一设置，容易遗漏。

### 3.2 使用方式

```csharp
public static class EventSystem
{
    // 标记为静态字段，Domain Reload 时自动重置为 null
    [StaticField]
    private static EventSystem _instance = null;

    // 标记为静态字段，初始化时赋值为 false
    [StaticField(false)]
    public static bool IsInitialized;

    // 标记为静态字段，初始化时通过默认构造函数创建新实例
    [StaticField(true)]
    private static Dictionary<Type, List<object>> _systems = new Dictionary<Type, List<object>>();
}
```

框架的初始化工具（如 `ReflectUtil`）可以通过反射扫描所有带 `[StaticField]` 的字段，在合适时机批量重置/初始化，而不需要每个类手动维护。

## 四、EnableMethodAttribute — 许可特殊 Entity 声明方法

```csharp
/// <summary>
/// 对于特殊实体类，允许类内部声明方法的标签
/// </summary>
[AttributeUsage(AttributeTargets.Class, Inherited = false)]
public class EnableMethodAttribute : Attribute
{
}
```

### 4.1 背景

ET 框架强制推行**数据与逻辑分离**：Entity 类只存数据，所有逻辑写在 `EntitySystem`（ComponentSystem）中：

```csharp
// 推荐：数据在 Entity，逻辑在 System
public class HpComponent : Entity { public int Hp; }

public class HpComponentSystem : SystemObject<HpComponent>
{
    public static void TakeDamage(HpComponent self, int damage)
    {
        self.Hp -= damage;
    }
}
```

分析器会警告：**在 Entity 子类中直接声明方法**（除了属性访问器外）。

但有些特殊 Entity（如 `Game` 单例、工具类 Entity）确实需要内部方法，这时加 `[EnableMethod]` 豁免：

```csharp
[EnableMethod]
public class Game : Singleton<Game>
{
    // 允许在这里直接声明方法，不触发分析器警告
    public static T AddSingleton<T>() where T : ASingleton, new() { ... }
    public static void Close() { ... }
}
```

## 五、整体设计哲学总结

| Attribute | 约束目标 | 防止的错误 |
|---|---|---|
| `[ChildOf]` | Entity 父节点类型 | 实体挂错父级 |
| `[ComponentOf]` | Component 宿主类型 | 组件挂错实体 |
| `[EnableAccessEntiyChild]` | 通用子节点访问豁免 | 无意绕过类型约束 |
| `[FriendOf]` | 跨类访问权限 | 非授权访问内部成员 |
| `[UniqueId]` | 常量 ID 唯一性 | ID 重复导致事件路由错误 |
| `[StaticField]` | 静态字段生命周期 | 静态缓存脏数据 |
| `[EnableMethod]` | Entity 方法声明豁免 | 数据逻辑混杂 |

这套 Attribute 系统体现了一个核心工程理念：

> **把本可以在运行时发现的 Bug，用编译期工具提前拦截；把开发者的经验和约定，用标注代码化为机器可检查的规则。**

对于刚入行的开发者，这里有三点实操建议：

1. **先理解规则，再用豁免**：`[EnableAccessEntiyChild]`、`[EnableMethod]` 都是豁免标签，能不用就不用。理解为什么有这条规则，比记住如何豁免更重要
2. **`[UniqueId]` 是零成本保险**：凡是有一堆 `const int` 常量定义在同一个类里的，立刻加上 `[UniqueId]`，防止未来某天被人悄悄添加了重复值
3. **`[StaticField]` 配合框架初始化系统使用**：单独加这个标签没有效果，需要框架的初始化代码通过反射扫描并批量重置，才能发挥价值

## 六、总结

ET 框架的 `Analyzer` 目录虽然代码量很少（每个文件只有十几行），但背后是精心设计的**静态分析驱动的约束体系**。配合 Roslyn 分析器，这些简单的 Attribute 可以：

- 在编译时检测 ECS 结构错误（父子关系、组件挂载关系）
- 在编译时检测常量 ID 重复
- 标注静态字段以便框架统一生命周期管理
- 精细化控制跨类访问权限（友元机制）

这种"用标注来编码规则、用工具来执行规则"的思路，是构建大型团队项目的重要工程实践，值得每一位游戏开发者深入学习。
