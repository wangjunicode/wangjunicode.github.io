## Code Review 的真正价值

> Code Review 不是找 Bug 的工具，是**知识传播和技术共识建立**的手段。

```
价值排序（从高到低）：
1. 知识共享：资深成员的思路传递给新成员
2. 及早发现问题：发现架构问题比测试阶段发现便宜10倍
3. 代码质量：风格统一、文档完整
4. 安全保障：防止安全漏洞和性能问题进入代码库
5. Bug 发现：（这不是主要目的，但也很有价值）
```

## 一、Author（提交者）的职责

### 1.1 提交好的 PR

```markdown
## PR 模板示例

### 改动描述
本 PR 实现了技能冷却系统的优化：
- 将 SkillCooldown 从轮询检查改为事件驱动
- 减少了每帧约 3000 次的不必要计算

### 改动类型
- [x] 功能优化
- [ ] 新功能
- [ ] Bug 修复
- [ ] 重构

### 影响范围
修改了 SkillManager、CooldownComponent，
其他系统通过事件接口访问，无需改动。

### 测试情况
- [x] 单元测试：CooldownTests.cs（新增 5 个 Test Case）
- [x] 手动测试：在 5 个不同技能场景下验证正常

### Reviewer 注意点
CooldownComponent.cs 第 87 行有个特殊处理：
对于被动技能，不应该触发 CooldownStart 事件，
这里我用了 flag 来区分，想听听你的意见。

### 关联
- Closes #342 (技能冷却频繁更新导致帧率下降)
- Related: ADR-0015 (技能系统事件化改造)
```

### 1.2 小 PR 原则

```
大 PR 的代价：
  1000 行改动 → Reviewer 看 2 小时，精力耗尽，Review 质量差
  讨论时无法精确定位问题
  合并冲突风险高

小 PR 的好处：
  100~200 行改动 → 30分钟完成 Review
  Reviewer 专注，发现问题多
  更容易回滚

如何控制 PR 大小：
  - 功能 + 测试 = 一个 PR
  - 重构与新功能拆分（先重构，再在新结构上实现功能）
  - 数据结构变更单独一个 PR
  
例外：大规模重构确实需要大 PR 时：
  - 先做更细致的设计评审
  - PR 描述更详细
  - 安排专门的 Review 时间段
```

## 二、Reviewer（审查者）的技巧

### 2.1 Review 的优先级

```
不要平均分配注意力，按优先级 Review：

P1（必须处理）：
  - 逻辑错误（Bug）
  - 安全漏洞
  - 资源泄露（没有释放的资源/事件订阅）
  - 性能严重问题（O(n²)、热路径 GC）
  - 接口破坏性变更（影响其他系统）

P2（强烈建议）：
  - 未处理的边界条件（null、空列表、越界）
  - 缺失的单元测试
  - 注释不清晰的复杂逻辑
  - 违反已有设计规范

P3（建议/讨论）：
  - 更优雅的实现方式
  - 命名改进
  - 代码组织优化
  
P4（风格偏好，不强制）：
  - 空格/换行偏好
  - 私人编码习惯
```

### 2.2 好的 Review 评论

```
Review 评论的质量决定了它的价值

规则：评论必须清晰表达：
  1. 问题是什么
  2. 为什么是问题
  3. 建议的解决方向

❌ 差的评论（无行动指导）：
  "这里有问题"
  "写得不好"
  "建议重构"

✅ 好的评论：
```

```markdown
<!-- 例1：指出具体问题 -->
这里的循环在每帧都调用 `FindObjectsOfType<Enemy>()`，
这个方法会遍历所有 GameObject，在场景对象较多时会有明显性能开销。
建议在 Awake/Start 时缓存，或维护一个 EnemyManager 注册表。

<!-- 例2：提供改进建议 -->
`CalculateDamage` 方法目前有 5 个 if/else 分支，
随着技能类型增加，这里会越来越难维护。
可以考虑用策略模式（在 DamageCalculator 接口上注册不同策略），
参考 SkillFactory.cs 的做法。

<!-- 例3：提出问题（不是否定，是讨论） -->
这里的空值检查 `if (target == null)` 对 Unity Object 有效，
但如果 target 是普通 C# 对象，这里的行为是？
我想确认你预期的行为是什么。

<!-- 例4：指出风险 -->
这个异步方法中没有传入 CancellationToken。
如果 MonoBehaviour 被 Destroy，方法还在执行，
访问 this.gameObject 时会抛 MissingReferenceException。
建议传入 this.destroyCancellationToken。
```

### 2.3 给予正面反馈

```
Code Review 不只是找问题的，也要：
  ✅ 认可好的设计："这个错误处理做得很完整，
     特别是把 backup 逻辑放在 finally 里，安全！"
  ✅ 分享知识："顺便说一句，Unity 2022 的 
     Collections 包里有 NativeHashSet，
     这里可以用来替代 HashSet + lock，性能更好。"
  ✅ 提问而非断言："这里用了单例，
     是否考虑过依赖注入？我好奇有没有什么原因坚持用单例。"

正面反馈的价值：
  - 让 Author 知道哪些做法是对的（不改的要强化）
  - 降低 Review 的心理压力
  - 建立良好的团队氛围
```

## 三、Code Review Checklist

### 3.1 通用 Checklist

```markdown
## Reviewer Checklist（游戏客户端项目）

### 逻辑正确性
- [ ] 主逻辑流程是否正确？（想几个典型场景走读一遍）
- [ ] 边界条件：空值/空集合/最大值是否处理？
- [ ] 错误处理：异常是否被捕获？是否向上传递？

### Unity 特有问题
- [ ] 是否有在 Update/热路径 中调用 GetComponent/Find？
- [ ] 事件订阅是否有对应的取消订阅（OnDisable/OnDestroy）？
- [ ] Coroutine/UniTask 是否有取消机制？
- [ ] 是否有对 Destroyed 的 Unity Object 进行操作？

### 性能
- [ ] 热路径（每帧调用）中是否有不必要的内存分配？
- [ ] 是否有可以 Cache 的反复计算？
- [ ] 大数据集的算法复杂度是否合理？

### 可维护性
- [ ] 命名是否清晰表达意图？
- [ ] 复杂逻辑是否有注释说明 Why？
- [ ] 是否有重复代码可以提取？

### 测试
- [ ] 新增逻辑是否有对应的单元测试？
- [ ] 测试是否覆盖了主要分支？
```

## 四、常见 Review 反模式

### 4.1 Reviewer 的反模式

```
❌ 橡皮图章（Rubber Stamp Review）
  直接 Approve，没有认真看
  后果：PR 没有价值，引入 Bug

❌ 评论洪水（Comment Bombing）
  发出几十条评论，大部分是风格偏好
  后果：Author 感到被攻击，重要问题被淹没

❌ 迟到的 Review（SLA 超时）
  PR 提交后三天没有响应
  后果：开发节奏被打断，积累排队

❌ 只评论不对话
  提出问题但不解释为什么
  后果：Author 无法学习，也无从判断是否要改

✅ 好的 Review 行为：
  24小时内响应，P1 问题必须改，P3 问题供参考
  先理解作者意图，再评估实现
```

### 4.2 Author 的反模式

```
❌ 防御性回应
  "这样写是因为XXX，你不懂"
  后果：关闭讨论，氛围恶化

❌ 大 PR 炸弹
  一次提交2000行改动
  后果：Review 质量极低，最终只是走形式

❌ 不说明背景
  PR 描述为空或极其简单
  后果：Reviewer 浪费时间猜意图

❌ 拒绝所有建议
  不管 Reviewer 说什么都不改
  后果：Review 成为无意义的摩擦

✅ 好的 Author 行为：
  对每条评论都回复（即使只是 "Done" 或 "解释原因不改"）
  对争议点充分讨论，最终由 TL 拍板
```

## 五、建立 Code Review 文化

### 5.1 从 TL 开始

```
技术 Leader 如何带头建立 Review 文化：

1. TL 自己的代码也要接受 Review
   "平等的 Review 文化：级别不决定代码质量"
   
2. TL 要做示范性 Review
   写出团队认可的高质量 Review 评论
   让新成员学习什么是好的 Review

3. TL 要保证 SLA
   "24小时内响应 Review" 的规则，TL 自己带头遵守

4. 在团队会议中分享好的 Review 案例
   "上周 XX 的 Review 很有价值，一起看看学习"
```

### 5.2 Review 文化成熟度评估

```
Level 1（基础）：
  - 所有代码都经过 Review 才能合并
  - PR 模板存在且被使用

Level 2（规范）：
  - 有明确的 Review Checklist
  - SLA 被执行（24小时内响应）
  - Review 评论质量较高

Level 3（学习型）：
  - Review 发现了架构级别的问题
  - 通过 Review 传播了团队知识
  - 新成员通过 Review 快速成长

Level 4（优秀）：
  - Author 在提交 PR 前就已经内化了 Review 标准
  - Review 成为乐趣而非负担
  - Bug 逃逸率极低
```

## 总结

Code Review 的本质是**团队的集体智慧应用于每一行代码**。

关键行动项：
1. **建立模板**：PR 模板 + Review Checklist（今天就可以做）
2. **设置 SLA**：24小时内响应的规则（需要团队约定）
3. **以身作则**：TL 做出好的 Review 示范
4. **正向激励**：在 Review 中认可好的设计，不只找问题
5. **持续改进**：每季度回顾 Review 流程，找出可以提升的地方

> 最好的代码文化，是团队成员把高标准内化为习惯，而不是被规则约束。
