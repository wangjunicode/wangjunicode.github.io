# 游戏框架线程同步与反射工具系统解析

Unity 本质上是单线程的游戏引擎——几乎所有 Unity API 都必须在主线程调用。然而现代游戏框架为了提升性能，会在子线程处理网络收发、文件 I/O、逻辑运算等耗时操作。这就带来了一个核心问题：**如何安全地将子线程的结果转交给主线程执行？**

本文结合框架源码，解析两个实用工具：`ThreadSynchronizationContext`（线程同步上下文）和 `ReflectUtil`（反射工具），深入理解它们在多线程与热更新场景下的设计价值。

---

## 一、ThreadSynchronizationContext：主线程任务队列

### 1.1 问题背景

游戏框架中有大量异步操作：
- 网络层在 Socket 线程接收消息，收到后需要派发给逻辑层（主线程）
- `ETTask` 异步任务在 `.ConfigureAwait(false)` 后可能在线程池恢复
- 资源加载完成后需要回调主线程更新 UI

如果这些回调直接在子线程执行，会触发 Unity 的线程安全检查，轻则报错，重则崩溃。

### 1.2 源码实现

```csharp
using System;
using System.Collections.Concurrent;
using System.Threading;

namespace ET
{
    public class ThreadSynchronizationContext : SynchronizationContext
    {
        // 无锁并发队列：支持多生产者（各种子线程）单消费者（主线程 Update）
        private readonly ConcurrentQueue<Action> queue = new ConcurrentQueue<Action>();

        private Action a;

        // 由主线程的 Update 循环调用，消费队列中的所有待执行 Action
        public void Update()
        {
            while (true)
            {
                if (!this.queue.TryDequeue(out a))
                    return;

                try
                {
                    a();
                }
                catch (Exception e)
                {
                    Log.Error(e);  // 捕获异常，防止单个 Action 异常阻断后续任务
                }
            }
        }

        // SynchronizationContext 标准接口，.NET 异步机制会自动调用
        public override void Post(SendOrPostCallback callback, object state)
        {
            this.Post(() => callback(state));
        }

        // 自定义入队方法，直接投递 Action
        public void Post(Action action)
        {
            this.queue.Enqueue(action);
        }
    }
}
```

### 1.3 继承 SynchronizationContext 的意义

`SynchronizationContext` 是 .NET 的线程调度抽象基类。当你在 `async/await` 代码中这样写：

```csharp
await SomeThing();
// await 之后的代码会在哪个线程恢复？
DoSomethingWithUnityApi();
```

答案取决于 **当前线程的 `SynchronizationContext`**。如果主线程装配了 `ThreadSynchronizationContext`，那么 `await` 后的续体（continuation）会被 `Post` 到队列，在下一次主线程 `Update` 时执行，从而保证 Unity API 调用的线程安全。

### 1.4 安装到主线程

```csharp
// 框架启动时（主线程）
public class Entry : MonoBehaviour
{
    private ThreadSynchronizationContext context;

    void Awake()
    {
        context = new ThreadSynchronizationContext();
        // 将自定义上下文设置为当前（主）线程的同步上下文
        SynchronizationContext.SetSynchronizationContext(context);
    }

    void Update()
    {
        // 每帧消费队列
        context.Update();
    }
}
```

### 1.5 ConcurrentQueue 的选择

这里用 `ConcurrentQueue<Action>` 而非 `Queue<Action>` + `lock`，理由如下：

| 特性 | `Queue<Action>` + `lock` | `ConcurrentQueue<Action>` |
|------|--------------------------|--------------------------|
| 线程安全 | 依赖锁 | 无锁算法（CAS） |
| 高并发争用 | 存在锁竞争 | 基本无竞争 |
| 内存可见性 | 需要 Memory Barrier | 内置保障 |
| 实现复杂度 | 简单 | 稍复杂（.NET 封装好了） |

网络层和逻辑层在高负载下可能同时有多个子线程向队列投递 Action，`ConcurrentQueue` 的无锁设计避免了锁竞争造成的性能损耗。

### 1.6 异常隔离的重要性

```csharp
try
{
    a();
}
catch (Exception e)
{
    Log.Error(e);  // 关键：只记录错误，继续处理下一个
}
```

如果没有这层 `try/catch`，队列中某一个 Action 抛出异常，会导致本帧剩余所有 Action 都不被执行——网络消息丢失、UI 回调丢失。异常隔离确保了单点故障不会雪崩。

## 二、ReflectUtil：跨程序集的反射工具

### 2.1 热更新带来的反射困境

使用 `HybridCLR` 热更新后，游戏代码被分割到多个程序集：

- `Assembly-CSharp.dll`：主程序集（不可热更）
- `HotfixCode.dll`：热更程序集（运行时动态加载）

标准 `Type.GetType("SomeClass")` 只在当前程序集查找，热更类型无法被找到。框架用 `ReflectUtil` 解决了这个问题：

### 2.2 跨程序集类型查找

```csharp
public class ReflectUtil
{
    private static List<Type> types;

    // 初始化：收集所有已加载程序集的所有类型
    public static void InitTypes()
    {
        types = AppDomain.CurrentDomain.GetAssemblies()
                    .SelectMany(a => a.GetTypes())
                    .ToList();
    }

    // 先用 Type.GetType（快速路径），失败则遍历所有程序集
    public static Type GetType(string typeName)
    {
        var type = Type.GetType(typeName);
        if (type != null) return type;

        foreach (var a in AppDomain.CurrentDomain.GetAssemblies())
        {
            type = a.GetType(typeName);
            if (type != null) return type;
        }
        return null;
    }
}
```

**编辑器钩子**：
```csharp
#if UNITY_EDITOR
[UnityEditor.InitializeOnLoadMethod]
#endif
public static void InitTypes()
{
    // ...
}
```

`[InitializeOnLoadMethod]` 确保在 Unity 编辑器重新编译后自动刷新类型缓存，避免开发时使用过期的类型列表。

### 2.3 动态类型转换

```csharp
// 泛型版本：类型安全，编译期检查
public static T ChangeType<T>(object obj)
{
    var type = typeof(T);
    if (type.IsPrimitive)
        return (T)Convert.ChangeType(obj, type);  // int, float 等基本类型
    return (T)obj;                                 // 引用类型直接转
}

// 非泛型版本：运行时动态转换，处理所有复杂场景
public static object ChangeType(object value, Type type)
{
    if (value == null) return null;
    if (type == value.GetType()) return value;  // 类型相同，直接返回
    
    // 枚举转换
    if (type.IsEnum)
    {
        if (value is string)
            return Enum.Parse(type, value as string);    // "Attack" -> AttackType.Attack
        return Enum.ToObject(type, value);               // 0 -> AttackType.None
    }

    // 泛型类型（如 Nullable<int>）
    if (!type.IsInterface && type.IsGenericType)
    {
        var innerType = type.GetGenericArguments()[0];
        var innerValue = ChangeType(value, innerType);
        return Activator.CreateInstance(type, innerValue);
    }

    // Guid 和 Version 特殊处理
    if (value is string && type == typeof(Guid)) return new Guid(value as string);
    if (value is string && type == typeof(Version)) return new Version(value as string);
    
    // 其余 IConvertible 类型（int/long/double...）
    if (!(value is IConvertible)) return value;
    return Convert.ChangeType(value, type);
}
```

**应用场景**：配置表反序列化时，Excel 中的字符串 `"100"` 需要转换为 `int` 字段；策略模式中，根据字符串动态创建枚举值。

### 2.4 泛型约束验证

```csharp
public static bool AreTypeArgumentsValid(Type genericTypeDefinition, Type typeArgument)
{
    if (!genericTypeDefinition.IsGenericTypeDefinition)
        throw new ArgumentException("必须传入泛型类型定义");

    var genericArguments = genericTypeDefinition.GetGenericArguments();
    var genericArgument = genericArguments[0];
    var attributes = genericArgument.GenericParameterAttributes;

    // 检查 class 约束
    if (attributes.HasFlag(GenericParameterAttributes.ReferenceTypeConstraint) 
        && typeArgument.IsValueType)
        return false;

    // 检查 struct 约束
    if (attributes.HasFlag(GenericParameterAttributes.NotNullableValueTypeConstraint) 
        && !typeArgument.IsValueType)
        return false;

    // 检查 new() 约束
    if (attributes.HasFlag(GenericParameterAttributes.DefaultConstructorConstraint) 
        && typeArgument.GetConstructor(Type.EmptyTypes) == null)
        return false;

    // 检查接口/基类约束
    var constraints = genericArgument.GetGenericParameterConstraints();
    foreach (var constraint in constraints)
    {
        // 处理开放泛型约束（如 IEquatable<T>）
        if (constraint.IsGenericType && constraint.ContainsGenericParameters)
        {
            var concreteConstraint = constraint.GetGenericTypeDefinition()
                                               .MakeGenericType(typeArgument);
            if (!concreteConstraint.IsAssignableFrom(typeArgument))
                return false;
        }
        else
        {
            if (!constraint.IsAssignableFrom(typeArgument))
                return false;
        }
    }
    return true;
}
```

**使用场景**：可视化脚本（UniScript）或编辑器工具在运行时动态构建泛型类型时，先验证约束是否满足，避免 `MakeGenericType` 抛出 `ArgumentException`。

### 2.5 运行时成员读写

```csharp
// 读取字段或属性（统一接口，自动向上查找基类）
public static T GetMember<T>(object instance, string memberName)
{
    var type = instance.GetType();
    return GetMember<T>(type, instance, memberName);
}

// 内部实现：优先 Field，其次 Property，自动遍历基类继承链
private static bool GetMemberInfo(Type type, string memberName, 
    out FieldInfo fieldInfo, out PropertyInfo propertyInfo)
{
    fieldInfo = null;
    propertyInfo = null;

    fieldInfo = type.GetField(memberName,
        BindingFlags.Public | BindingFlags.NonPublic | 
        BindingFlags.Static | BindingFlags.Instance);
    if (fieldInfo != null) return true;

    propertyInfo = type.GetProperty(memberName,
        BindingFlags.Public | BindingFlags.NonPublic | 
        BindingFlags.Static | BindingFlags.Instance);
    if (propertyInfo != null) return true;

    // 向上查找基类
    if (type.BaseType != null)
        return GetMemberInfo(type.BaseType, memberName, out fieldInfo, out propertyInfo);
    return false;
}
```

**使用示例**：

```csharp
// 读取私有字段
var hp = ReflectUtil.GetMember<int>(unit, "m_hp");

// 写入字段
ReflectUtil.SetMember(unit, "m_attackPower", 150);

// 调用私有方法
var result = ReflectUtil.Invoke<bool>(unit, "CanAttack", new object[] { target });
```

## 三、两个工具的协同应用

在热更新框架中，这两个工具常常配合使用：

```csharp
// 场景：热更代码收到网络消息后，回调给主线程处理
// （运行在 Socket 接收线程）
void OnMessageReceived(byte[] data)
{
    var msgType = ReflectUtil.GetType("ET.C2S_LoginRequest");  // 热更类型
    var msg = Deserialize(data, msgType);
    
    // 通过 ThreadSynchronizationContext 投递给主线程
    ThreadSynchronizationContext.Instance.Post(() => {
        // 主线程安全地处理消息
        EventSystem.Instance.Publish(msg);
    });
}
```

## 四、性能注意事项

### 4.1 反射的性能成本

反射操作（`GetField`、`Invoke` 等）性能约是直接调用的 **10~100 倍**慢。在性能敏感路径（如每帧 Update、战斗计算）应避免反射。

**推荐策略**：
- 启动时用反射**初始化一次**，缓存 `FieldInfo`/`MethodInfo`
- 运行时用缓存的 `FieldInfo.GetValue()` 替代 `ReflectUtil.GetMember()`

```csharp
// 正确做法：缓存反射结果
private static FieldInfo s_hpField = typeof(UnitComponent).GetField("m_hp", 
    BindingFlags.NonPublic | BindingFlags.Instance);

// 每帧安全调用
var hp = (int)s_hpField.GetValue(unit);
```

### 4.2 ThreadSynchronizationContext 的队列积压

如果主线程帧率下降（如 GC Spike），队列中积压的 Action 数量会增加。框架当前的实现是"每帧处理完所有积压"，这可能加剧帧率抖动。

生产项目中可以考虑**限制每帧最大处理数量**：

```csharp
public void Update(int maxCount = 100)
{
    int count = 0;
    while (count++ < maxCount)
    {
        if (!this.queue.TryDequeue(out a)) return;
        try { a(); }
        catch (Exception e) { Log.Error(e); }
    }
}
```

## 五、总结

| 工具 | 解决的核心问题 | 关键设计 |
|------|-------------|---------|
| `ThreadSynchronizationContext` | 子线程结果安全回调主线程 | `ConcurrentQueue` + `SynchronizationContext.Post` |
| `ReflectUtil` | 热更程序集跨域类型访问 | 遍历所有 `AppDomain.GetAssemblies()` |

**学习要点**：
1. Unity 单线程模型 + 异步框架 → 必须有线程同步机制
2. 热更新 = 多程序集 → 标准 `Type.GetType` 失效 → 需要遍历所有程序集
3. 反射是"万能钥匙"，但有性能成本，用缓存降低开销
4. `ConcurrentQueue` 是高并发生产者/单消费者场景的最佳选择
