# 实体场景类型系统与脚本属性标注机制的工程设计解析

在 ECS 框架中，"场景"（Scene）是组织实体与组件的顶层容器。要让各类 Scene 实例在运行时被正确识别、创建和路由，就需要一套类型系统来描述场景的身份；同时，系统类（ComponentSystem）的自动发现与注册依赖属性标注（Attribute）机制。本文深入剖析这两个相互协作的子系统：SceneType / SceneHelper 场景类型体系，以及 EntitySystemAttribute 驱动的系统自动注册流程。

---

## 一、SceneType：场景身份的枚举编码

### 1.1 为什么不用字符串或纯整数

场景类型在运行时频繁用于分支判断、日志输出和配置查找。字符串比较开销大且容易拼写错误；裸整数可读性差，维护时难以对应含义。枚举提供了类型安全的整数编码：

```csharp
/// <summary>
/// 所有 Scene 实例的类型标识，用于创建时的路由与运行时的身份识别
/// </summary>
public enum SceneType : int
{
    // ─── 框架级场景 ───────────────────────────────────────────
    None      = 0,   // 未初始化占位
    Process   = 1,   // 进程根节点（单例 Root 直接持有）
    Client    = 2,   // 客户端主场景（包含 UI、声音、网络等全局组件）

    // ─── 逻辑玩法场景 ─────────────────────────────────────────
    Main      = 3,   // 主城（大厅）场景
    Battle    = 4,   // 战斗场景
    Room      = 5,   // 房间/匹配等待场景

    // ─── 服务端场景（如果 C/S 共用框架）─────────────────────
    Gate      = 100, // 网关服务
    Map       = 101, // 地图服务
    Realm     = 102, // 大区服务
}
```

**枚举段划分惯例**：
- 0–99：客户端场景与框架场景
- 100+：服务端场景（C/S 共用代码时不会冲突）

这使得一次简单的范围比较就能区分代码运行在哪一侧：`sceneType >= SceneType.Gate` → 服务端语境。

### 1.2 SceneType 与 Scene 实例的绑定

```csharp
public class Scene : Entity
{
    public SceneType SceneType { get; private set; }

    // 工厂方法内绑定，创建后不可变更
    internal static Scene Create(SceneType sceneType, ...)
    {
        Scene scene = ObjectPool.Fetch<Scene>(parentScene);
        scene.SceneType = sceneType;
        // ...
        return scene;
    }
}
```

`SceneType` 在工厂方法中一次性赋值，之后只读——场景的"身份"在创建时确定，不允许运行时变更，这避免了大量基于类型假设的系统出现不一致。

## 二、SceneHelper：场景操作的语义封装

### 2.1 设计动机

直接操作 `Scene.Create()` 需要了解所有参数（parent、sceneType、name 等），调用处代码冗长，且一旦修改参数签名，所有调用点都要改。SceneHelper 将"创建特定类型场景"的完整语义封装为高层方法：

```csharp
public static class SceneHelper
{
    /// <summary>
    /// 创建客户端主场景（自动挂到 Process 下）
    /// </summary>
    public static Scene CreateClientScene(string name = "Client")
    {
        Scene process = Root.Instance.Get<Scene>();
        return SceneFactory.Create(process, SceneType.Client, name);
    }

    /// <summary>
    /// 创建战斗场景
    /// </summary>
    public static Scene CreateBattleScene(Scene parent, long battleId)
    {
        string name = $"Battle_{battleId}";
        return SceneFactory.Create(parent, SceneType.Battle, name);
    }

    /// <summary>
    /// 查找第一个指定类型的子场景
    /// </summary>
    public static Scene FindScene(Scene parent, SceneType type)
    {
        foreach (Entity child in parent.Children.Values)
        {
            if (child is Scene s && s.SceneType == type)
                return s;
        }
        return null;
    }

    /// <summary>
    /// 断言当前场景类型（调试期间快速发现错误）
    /// </summary>
    public static void AssertSceneType(Scene scene, SceneType expected)
    {
#if UNITY_EDITOR || DEVELOPMENT_BUILD
        if (scene.SceneType != expected)
            throw new Exception($"场景类型断言失败：期望 {expected}，实际 {scene.SceneType}（{scene.Name}）");
#endif
    }
}
```

### 2.2 工厂 vs Helper 的职责边界

| 类 | 职责 | 感知层 |
|----|------|--------|
| `SceneFactory` | 分配 InstanceId、入池、触发 Awake | 框架内部 |
| `SceneHelper` | 业务语义封装、参数校验、便捷查询 | 业务外部 |

SceneFactory 是底层，SceneHelper 是门面。业务代码只应调用 SceneHelper，保持对底层工厂实现的隔离。

### 2.3 典型调用链

```
游戏初始化
    ↓
SceneHelper.CreateClientScene()
    ↓
SceneFactory.Create(process, SceneType.Client, "Client")
    ↓
ObjectPool.Fetch<Scene>(process)         // ECS 池取出实例
    ↓
scene.SceneType = SceneType.Client       // 绑定类型
scene.InstanceId = process.GenId()       // 分配实体 ID
    ↓
EntitySystemSingleton.Awake(scene)       // 触发所有 SceneAwakeSystem
    ↓
SceneAwakeSystem.Awake(scene)            // 初始化 UI 根、网络组件...
```

## 三、EntitySystemAttribute：系统自动注册的锚点

### 3.1 系统注册问题

ECS 框架的精髓之一是**解耦**：Entity 不知道自己被哪些 System 处理，System 也不依赖具体 Entity 类。但要让 EntitySystemSingleton 知道"哪些 System 需要处理哪种 Entity"，就必须有一个发现机制。

两种方案对比：

**方案 A：手动注册（命令式）**
```csharp
// 初始化时逐一注册，繁琐且容易遗漏
entitySystemSingleton.Register(new BattleComponentAwakeSystem());
entitySystemSingleton.Register(new BattleComponentUpdateSystem());
// ... 100 个系统手动写 100 行
```

**方案 B：Attribute 扫描（声明式）**
```csharp
[EntitySystem]
public class BattleComponentAwakeSystem : AwakeSystem<BattleComponent>
{
    public override void Awake(BattleComponent self) { ... }
}
// 启动时一次性扫描程序集，自动发现所有打了 [EntitySystem] 的类
```

框架选择了方案 B，核心是 `EntitySystemAttribute`。

### 3.2 Attribute 定义

```csharp
/// <summary>
/// 标记一个类为 ECS 系统实现，框架启动时会自动发现并注册到 EntitySystemSingleton。
/// 继承自 BaseAttribute，统一被 AssemblyHelper 扫描。
/// </summary>
[AttributeUsage(AttributeTargets.Class, Inherited = false, AllowMultiple = false)]
public class EntitySystemAttribute : BaseAttribute
{
}
```

**设计细节：**
- `Inherited = false`：只标记直接打了 Attribute 的类，子类不自动继承，避免误标注
- `AllowMultiple = false`：同一系统类只需标注一次，防止重复注册
- 继承 `BaseAttribute`：让 `AssemblyHelper` 只需扫描 `BaseAttribute` 子类，无需枚举每种具体 Attribute

### 3.3 BaseAttribute 统一扫描锚点

```csharp
/// <summary>
/// 所有框架扫描 Attribute 的基类，AssemblyHelper 以此为过滤条件
/// </summary>
[AttributeUsage(AttributeTargets.Class)]
public abstract class BaseAttribute : Attribute
{
}
```

BaseAttribute 的存在使反射扫描高效：`AssemblyHelper` 只需遍历程序集中所有类，检查是否有继承自 BaseAttribute 的 Attribute，即可一次性发现所有框架级标注：

```csharp
public static class AssemblyHelper
{
    public static Dictionary<Type, List<Type>> GetAssemblyTypes(Assembly assembly)
    {
        var result = new Dictionary<Type, List<Type>>();
        foreach (Type type in assembly.GetTypes())
        {
            foreach (Attribute attr in type.GetCustomAttributes(inherit: false))
            {
                if (attr is BaseAttribute baseAttr)
                {
                    Type attrType = baseAttr.GetType();
                    if (!result.TryGetValue(attrType, out var list))
                    {
                        list = new List<Type>();
                        result[attrType] = list;
                    }
                    list.Add(type);
                }
            }
        }
        return result;
    }
}
```

扫描结果：
```
{
    EntitySystemAttribute → [BattleComponentAwakeSystem, BuffComponentAwakeSystem, ...],
    EventAttribute        → [BattleStartEventHandler, BuffAddEventHandler, ...],
    InvokeAttribute       → [RpcLoginHandler, RpcMatchHandler, ...],
    ...
}
```

### 3.4 EntitySystemSingleton 消费注册结果

```csharp
public class EntitySystemSingleton : Singleton<EntitySystemSingleton>
{
    // key = ISystemType 接口类型（如 IAwakeSystem<BattleComponent>）
    // value = 对应系统实例列表（支持多个系统处理同一组件）
    private readonly Dictionary<Type, List<ISystemType>> systems 
        = new Dictionary<Type, List<ISystemType>>();

    protected override void Init()
    {
        // 加载所有程序集（热更层 + 主程序）
        var allTypes = AssemblyHelper.GetAllTypes();
        
        // 仅取 EntitySystemAttribute 标记的类型
        if (!allTypes.TryGetValue(typeof(EntitySystemAttribute), out var systemTypes))
            return;

        foreach (Type systemType in systemTypes)
        {
            ISystemType system = (ISystemType)Activator.CreateInstance(systemType);
            
            // 获取系统实现的所有接口（可能同时实现 IAwakeSystem + IUpdateSystem）
            foreach (Type interfaceType in system.GetType().GetInterfaces())
            {
                if (!typeof(ISystemType).IsAssignableFrom(interfaceType)) continue;
                if (!systems.TryGetValue(interfaceType, out var list))
                {
                    list = new List<ISystemType>();
                    systems[interfaceType] = list;
                }
                list.Add(system);
            }
        }
    }
}
```

整个流程：**Assembly 扫描 → 类型归类 → 系统实例化 → 接口索引** — 仅在框架初始化时执行一次，运行时零反射开销。

## 四、热更新场景下的再注册

HybridCLR 热更代码加载后，System 类存在于热更程序集，不在主工程中。框架通过**多次调用注册接口**支持热更：

```csharp
// 热更代码加载完成后触发
public static void RegisterHotfixAssembly(Assembly hotfixAssembly)
{
    var hotfixTypes = AssemblyHelper.GetAssemblyTypes(hotfixAssembly);
    EntitySystemSingleton.Instance.AddSystems(hotfixTypes);
    // 热更 System 追加到已有 systems 字典，不覆盖主工程 System
}
```

这一设计要求 `EntitySystemAttribute` 在主工程和热更层都可见（两者共享接口定义程序集），是多程序集 ECS 架构的关键扩展点。

## 五、SceneType、SceneHelper 与 EntitySystem 的协作

三者在实际运行中的协作关系：

```
SceneHelper.CreateBattleScene(parent, battleId)
    │
    ├─ SceneFactory.Create(..., SceneType.Battle, ...)
    │       ├─ scene.SceneType = SceneType.Battle
    │       └─ EntitySystemSingleton.Awake(scene)
    │
    └─ EntitySystemSingleton 查找 IAwakeSystem<Scene>
            └─ 找到 [EntitySystem] BattleSceneAwakeSystem
                    └─ Awake(scene) 内部：
                            ├─ scene.AssertSceneType(SceneType.Battle)  // SceneHelper 断言
                            ├─ scene.AddComponent<BattleComponent>()
                            └─ scene.AddComponent<ColliderComponent>()
```

`EntitySystemAttribute` 让 BattleSceneAwakeSystem 自动被框架发现；`SceneType` 让 Awake 实现可以安全断言当前场景身份；`SceneHelper` 提供便捷的工厂方法，业务代码无需了解底层细节。

## 六、工程建议

### 6.1 新增场景类型的标准流程

1. 在 `SceneType` 枚举添加新值（并更新文档注释）
2. 在 `SceneHelper` 添加对应的工厂方法
3. 实现 `XXXSceneAwakeSystem : AwakeSystem<Scene>`，打上 `[EntitySystem]`
4. 在 Awake 内第一行调用 `SceneHelper.AssertSceneType(self, SceneType.XXX)`
5. 如有需要，在 `EntitySystemSingleton.Destroy` 实现对应清理系统

### 6.2 常见问题排查

| 现象 | 原因 | 修复 |
|------|------|------|
| Awake 没有触发 | 忘记打 `[EntitySystem]` | 加上 Attribute |
| 热更系统不生效 | 热更程序集未调用 RegisterHotfixAssembly | 检查热更加载完成回调 |
| 场景类型断言失败 | 用错了工厂方法创建了不同类型的场景 | 检查创建处调用的 SceneHelper 方法 |
| System 被注册了两次 | 主工程和热更层都定义了同名 System 类 | 保持系统类只在一个程序集中定义 |

## 七、总结

- **SceneType 枚举**：以整数编码场景身份，类型安全、高效比较，段划分区分客户端/服务端
- **SceneHelper**：场景创建的语义封装门面，屏蔽底层工厂细节，提供业务友好的 API 和调试断言
- **EntitySystemAttribute**：声明式系统注册的锚点，继承 BaseAttribute 统一被扫描，支持主工程与热更层双重注册
- **三者协作**：SceneType 给 Scene 打标签 → SceneHelper 封装创建语义 → EntitySystemAttribute 确保正确的 System 在正确的时机被触发

这套设计将"场景是什么"（SceneType）、"如何创建场景"（SceneHelper）、"谁处理场景"（EntitySystem）三个关注点分离，是 ECS 框架可维护性的重要来源。
