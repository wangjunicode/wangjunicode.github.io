# 实体场景类型系统与脚本属性标注机制的工程设计解析

在 ECS 框架中，每个 Entity 都归属于某个 `Scene`，而每个 `Scene` 都有一个 `SceneType`。这套场景类型系统不仅决定了 Entity 的生命周期管理范围，还通过与 `Attribute` 标注机制的结合，实现了"系统按场景类型自动路由"的核心能力。本文深入解析这套设计的实现原理与工程价值。

---

## 一、Scene 与 Entity 的归属关系

### 1.1 树形层级结构

```
Game（静态入口）
└── World（根 Scene）
    ├── Scene A（战斗场景）
    │   ├── PlayerEntity
    │   ├── EnemyEntity
    │   └── BattleComponent
    └── Scene B（主城场景）
        ├── PlayerEntity（不同实例）
        └── UIComponent
```

每个 `Scene` 维护一个 `Dictionary<long, Entity>` 存储其下所有实体，`InstanceId` 由所在 `Scene` 分配，保证在该 Scene 内唯一。

### 1.2 Entity 与 Scene 的绑定

```csharp
public abstract class Entity
{
    // Entity 归属的 Scene（通过 Parent 链向上查找到第一个 Scene 类型节点）
    public Scene Scene
    {
        get
        {
            Entity current = this;
            while (current != null)
            {
                if (current is Scene scene) return scene;
                current = current.Parent;
            }
            throw new InvalidOperationException("Entity has no parent Scene.");
        }
    }

    // InstanceId 由所在 Scene 分配
    public long InstanceId { get; internal set; }
}
```

## 二、SceneType 枚举的设计

### 2.1 枚举定义

```csharp
public enum SceneType : int
{
    None    = 0,
    All     = 1,   // 特殊值：匹配所有场景类型
    Process = 2,   // 进程级 Scene（最顶层）
    Main    = 3,   // 主城场景
    Battle  = 4,   // 战斗场景
    Login   = 5,   // 登录场景
    Lobby   = 6,   // 大厅场景
    // ... 按项目需要扩展
}
```

### 2.2 SceneType 的作用域

```csharp
public class Scene : Entity
{
    public SceneType SceneType { get; private set; }

    public static Scene Create(SceneType sceneType, string name, Scene parent = null)
    {
        Scene scene = new Scene();
        scene.SceneType = sceneType;
        scene.Name = name;
        // 将 Scene 注册到父节点
        parent?.AddChild(scene);
        return scene;
    }
}
```

## 三、Attribute 标注驱动的系统注册

### 3.1 System 的 Attribute 声明

框架中的所有 System（Awake、Update、Destroy 等）都通过 Attribute 声明自己的作用场景，无需手动注册：

```csharp
// 只在战斗场景生效的 AwakeSystem
[ObjectSystem]
[FriendOfAttribute(typeof(BattleComponent))]
public class BattleComponentAwakeSystem : AwakeSystem<BattleComponent>
{
    protected override void Awake(BattleComponent self)
    {
        self.HP = self.Config.MaxHP;
        self.AttackPower = self.Config.BaseAttack;
    }
}

// 在所有场景生效的 DestroySystem
[ObjectSystem]
public class LogComponentDestroySystem : DestroySystem<LogComponent>
{
    protected override void Destroy(LogComponent self)
    {
        self.Flush();
    }
}
```

### 3.2 ObjectSystemAttribute 的扫描机制

```csharp
[AttributeUsage(AttributeTargets.Class)]
public class ObjectSystemAttribute : Attribute { }

// EventSystem 启动时扫描所有带 [ObjectSystem] 的类
public void Load(Dictionary<string, Type> allTypes)
{
    foreach (var (_, type) in allTypes)
    {
        // 跳过没有 ObjectSystemAttribute 的类
        if (!type.IsDefined(typeof(ObjectSystemAttribute), false)) continue;
        // 跳过抽象类
        if (type.IsAbstract) continue;

        // 判断是哪种 System 并注册
        var obj = Activator.CreateInstance(type);
        if (obj is IAwakeSystem awakeSystem)
            this.awakeSystems.Register(awakeSystem);
        if (obj is IUpdateSystem updateSystem)
            this.updateSystems.Register(updateSystem);
        // ...
    }
}
```

### 3.3 FriendOfAttribute —— 访问私有字段的声明

```csharp
// 告知框架：BattleComponentAwakeSystem 可以访问 BattleComponent 的内部字段
[AttributeUsage(AttributeTargets.Class, AllowMultiple = true)]
public class FriendOfAttribute : Attribute
{
    public Type Type { get; }
    public FriendOfAttribute(Type type) => this.Type = type;
}
```

这个特性来自 Roslyn Source Generator：在编译期生成一个 `partial` 类，将 Component 中 `private` 的字段对对应的 System 可见，保持封装性的同时允许 System 直接操作内部状态。

## 四、生命周期的场景隔离

### 4.1 Update 的场景过滤

```csharp
// EventSystem.Update 中按 SceneType 过滤
public void Update(Scene scene)
{
    // 只更新当前 Scene 中注册了 IUpdateSystem 的 Entity
    foreach (var entity in scene.GetEntities())
    {
        if (!this.updateSystems.TryGet(entity.GetType(), out var systems)) continue;
        foreach (var system in systems)
        {
            try { system.Run(entity); }
            catch (Exception e) { Log.Error(e); }
        }
    }
}
```

### 4.2 场景销毁时的级联清理

```csharp
public class Scene : Entity
{
    protected override void Dispose()
    {
        // 1. 先销毁所有子 Entity
        foreach (var child in this.Children.Values.ToArray())
            child.Dispose();

        // 2. 清空实体注册表
        this.entities.Clear();

        // 3. 清空 SceneType 关联的对象池
        ObjectPool.ClearScene(this);

        // 4. 触发自身 Destroy 系统
        EntitySystemSingleton.Instance.Destroy(this);
    }
}
```

## 五、实践中的典型模式

### 5.1 按场景创建专属 Component

```csharp
// 登录场景：只有 LoginScene 才有 LoginComponent
Scene loginScene = Scene.Create(SceneType.Login, "Login");
loginScene.AddComponent<LoginComponent>();

// 战斗场景：只有 BattleScene 才有 BattleComponent
Scene battleScene = Scene.Create(SceneType.Battle, "Battle");
battleScene.AddComponent<BattleComponent>();
battleScene.AddComponent<BulletPoolComponent>();
```

### 5.2 场景切换时的生命周期流转

```csharp
public async ETTask SwitchToBattle()
{
    // 1. 销毁主城场景（级联 Destroy 所有 Entity 和 Component）
    mainScene.Dispose();

    // 2. 创建战斗场景（级联 Awake 所有 Entity 和 Component）
    battleScene = await CreateBattleScene();

    // 3. 主循环切换到战斗场景更新
    currentScene = battleScene;
}
```

## 六、设计取舍分析

### 优势

| 特点 | 工程价值 |
|------|----------|
| Scene 作为生命周期边界 | 场景切换等同于整个对象图销毁+重建，内存干净 |
| SceneType 过滤 | 同一 System/Event 在不同场景自动路由，无需 if/else |
| Attribute 自动注册 | 新增 System 只需标注，无需修改注册代码 |
| InstanceId 按 Scene 分配 | 跨 Scene 不会 ID 冲突，EntityRef 失效检测精确 |

### 局限

- **跨 Scene 通信需显式处理**：父 Scene 调用子 Scene 需要保存引用，或通过 Publish 事件
- **Scene 层级不宜太深**：过深的 Parent 链会让 `entity.Scene` 的查找开销增大

---

## 七、小结

实体场景类型系统是 ECS 框架"隔离性"的核心保障：

1. **Scene 是 Entity 的生命周期归属单位**：创建、销毁、对象池都以 Scene 为边界
2. **SceneType 是业务逻辑的路由维度**：System 和 Event 按 SceneType 自动分发
3. **Attribute 标注驱动注册**：消除了手动维护注册表的模板代码，让扩展零侵入

理解这套设计，是读懂框架中"为什么要传 Scene 参数"以及"为什么切换场景后状态完全干净"的关键所在。
