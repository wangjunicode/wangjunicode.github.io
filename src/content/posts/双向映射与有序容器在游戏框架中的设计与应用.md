## 前言

在游戏框架开发中，我们经常会遇到"普通字典不够用"的场景：需要从值反查键，需要保证遍历顺序，或者需要一个键对应多个不重复的值。这时候，就需要更专门的容器来解决问题。

本文深入分析游戏框架 Core 层中三个自定义容器：**DoubleMap**（双向映射）、**OrderedDictionary**（有序字典）、**UnOrderMultiMapSet**（无序多值集合映射），从设计动机、实现原理到实际应用场景，给出一套完整的技术视角。

## 一、DoubleMap：双向映射，正反都能查

### 1.1 为什么需要双向映射？

在游戏逻辑中，大量存在"一一映射"的场景：

- **技能 ID ↔ 技能实例对象**：通过 ID 找实例，通过实例找 ID
- **角色 UID ↔ 网络连接**：根据连接断开时快速清理 UID 记录
- **配置 Key ↔ 枚举值**：双向查找避免遍历

标准 `Dictionary<K, V>` 只支持 Key → Value 的单向查找。如果想反查，要么遍历（O(n) 复杂度），要么维护两个字典（容易不同步）。

`DoubleMap<K, V>` 封装了这种"维护两个字典"的逻辑，保证原子性和一致性。

### 1.2 源码分析

```csharp
public class DoubleMap<K, V>
{
    private readonly Dictionary<K, V> kv = new Dictionary<K, V>();
    private readonly Dictionary<V, K> vk = new Dictionary<V, K>();

    public void Add(K key, V value)
    {
        if (key == null || value == null || kv.ContainsKey(key) || vk.ContainsKey(value))
        {
            return;  // 任何一方已存在，直接拒绝
        }
        kv.Add(key, value);
        vk.Add(value, key);
    }

    public V GetValueByKey(K key)   => kv.TryGetValue(key, out var v) ? v : default;
    public K GetKeyByValue(V value) => vk.TryGetValue(value, out var k) ? k : default;

    public void RemoveByKey(K key)
    {
        if (!kv.TryGetValue(key, out V value)) return;
        kv.Remove(key);
        vk.Remove(value);  // 同步移除另一侧
    }

    public void RemoveByValue(V value)
    {
        if (!vk.TryGetValue(value, out K key)) return;
        kv.Remove(key);
        vk.Remove(value);
    }
}
```

**关键设计点：**

1. **拒绝重复**：`Add` 方法在 key 或 value 任意一方已存在时直接返回，保证双向唯一映射的语义。
2. **同步删除**：无论从哪一侧删除，另一侧都同步清理，不会出现"僵尸记录"。
3. **null 安全**：所有查找和删除操作都做了 null 检查，不会因空引用崩溃。

### 1.3 ForEach 遍历设计

```csharp
public void ForEach(Action<K, V> action)
{
    if (action == null) return;
    foreach (K key in kv.Keys)
    {
        action(key, kv[key]);
    }
}
```

只暴露 `kv` 方向的遍历（避免二次遍历 `vk`），简洁实用。

### 1.4 典型使用场景

**场景一：网络连接管理**

```csharp
// 玩家 UID ↔ 网络会话 ID
private DoubleMap<long, int> _playerSessionMap = new DoubleMap<long, int>();

// 玩家上线
_playerSessionMap.Add(playerUID, sessionId);

// 收到数据包（已知 sessionId），找对应 UID
long uid = _playerSessionMap.GetKeyByValue(sessionId);

// 玩家断线（已知 sessionId），清理记录
_playerSessionMap.RemoveByValue(sessionId);
```

**场景二：枚举与字符串互查（配置系统）**

```csharp
var roleTypeMap = new DoubleMap<string, RoleType>();
roleTypeMap.Add("warrior", RoleType.Warrior);
roleTypeMap.Add("mage",    RoleType.Mage);

// 解析配置时，字符串 → 枚举
RoleType type = roleTypeMap.GetValueByKey("warrior");

// 序列化时，枚举 → 字符串
string typeName = roleTypeMap.GetKeyByValue(RoleType.Mage);
```

## 二、OrderedDictionary：顺序可预期的字典

### 2.1 问题来源

C# 的 `Dictionary<K, V>` 不保证遍历顺序（哈希表的本质）。在某些场景中，顺序是业务逻辑的一部分：

- **技能优先级队列**：按照添加顺序处理
- **UI 列表元素**：需要按插入顺序展示
- **状态机状态**：状态必须按定义顺序遍历检查

C# 的 `SortedDictionary` 按 Key 排序，而不是插入顺序。`OrderedDictionary` 是填补"按插入顺序的字典"这个空缺的标准解法。

### 2.2 源码分析

```csharp
public class OrderedDictionary<TKey, TValue> : IDictionary<TKey, TValue>
{
    private Dictionary<TKey, TValue> _dictionary;  // 哈希查找
    private List<TKey>   _keys;                    // 维护插入顺序
    private List<TValue> _values;                  // 与 _keys 对齐的值列表

    public void Add(TKey key, TValue value)
    {
        _dictionary.Add(key, value);  // 哈希层
        _keys.Add(key);               // 顺序层
        _values.Add(value);
    }

    public TValue this[TKey key]
    {
        set
        {
            RemoveFromLists(key);  // 先从顺序层移除旧位置
            _dictionary[key] = value;
            _keys.Add(key);        // 重新追加到末尾（更新 = 移到最后）
            _values.Add(value);
        }
    }

    public IEnumerator<KeyValuePair<TKey, TValue>> GetEnumerator()
    {
        int i = 0;
        foreach (TKey key in _keys)    // 按 List<TKey> 顺序遍历
        {
            yield return new KeyValuePair<TKey, TValue>(key, _values[i]);
            i++;
        }
    }
}
```

**双层架构：**

| 层次 | 结构 | 职责 |
|------|------|------|
| 哈希层 | `Dictionary<K, V>` | O(1) 快速查找 |
| 顺序层 | `List<K>` + `List<V>` | 维护插入顺序 |

**更新语义：**  
`this[key] = value` 会先移除旧记录，再追加到末尾。这意味着更新后该 key 的顺序会变化（移到最后）。这是一个需要注意的细节。

### 2.3 性能分析

- **Add**：O(1) 平摊（List.Add + Dict.Add）
- **查找（ContainsKey/TryGetValue）**：O(1)
- **Remove**：O(n)，因为 `List.IndexOf` 是线性扫描
- **遍历**：O(n)，按插入顺序

Remove 是瓶颈，适合"插入多、删除少"的场景。

### 2.4 典型使用场景

**场景：技能书顺序管理**

```csharp
var skillBook = new OrderedDictionary<int, SkillConfig>();
skillBook.Add(1001, skillConfigA);
skillBook.Add(1002, skillConfigB);
skillBook.Add(1003, skillConfigC);

// 遍历时按照添加顺序处理技能
foreach (var kv in skillBook)
{
    ApplySkill(kv.Value);
}
```

## 三、UnOrderMultiMapSet：一键对多值（去重版）

### 3.1 与 MultiMap 的对比

前面分析过 `MultiMap<T, K>`（一个 Key 对应多个 Value，Value 允许重复）。`UnOrderMultiMapSet<T, K>` 则是"一个 Key 对应多个不重复的 Value"，底层用 `HashSet<K>` 代替 `List<K>`。

| 容器 | 底层 Value 存储 | 重复处理 |
|------|-----------------|---------|
| `MultiMap<T, K>` | `List<K>` | 允许重复 |
| `UnOrderMultiMapSet<T, K>` | `HashSet<K>` | 自动去重 |

### 3.2 源码分析

```csharp
public class UnOrderMultiMapSet<T, K> : Dictionary<T, HashSet<K>>
{
    public void Add(T t, K k)
    {
        if (!this.TryGetValue(t, out HashSet<K> set))
        {
            set = new HashSet<K>();
            base[t] = set;
        }
        set.Add(k);  // HashSet.Add 自动去重
    }

    public bool Remove(T t, K k)
    {
        if (!this.TryGetValue(t, out HashSet<K> set)) return false;
        if (!set.Remove(k)) return false;
        if (set.Count == 0)
            this.Remove(t);  // 自动清理空 HashSet，避免内存泄漏
        return true;
    }

    public bool Contains(T t, K k)
    {
        return this.TryGetValue(t, out HashSet<K> set) && set.Contains(k);
    }

    // 重写 Count：返回所有 HashSet 的元素总数
    public new int Count
    {
        get
        {
            int count = 0;
            foreach (var kv in this) count += kv.Value.Count;
            return count;
        }
    }
}
```

**关键特性：**

1. **继承 `Dictionary<T, HashSet<K>>`**：直接复用字典的哈希查找能力
2. **自动清理**：当某个 Key 对应的 HashSet 变空时，自动从字典中移除该 Key，防止累积空条目
3. **`Count` 语义重定义**：返回全局元素总数，而非 Key 的数量

### 3.3 典型使用场景

**场景一：事件系统中的监听者管理**

```csharp
// 每个事件 ID 对应多个监听者 EntityID（不允许重复注册）
private UnOrderMultiMapSet<int, long> _eventListeners = new UnOrderMultiMapSet<int, long>();

// 注册监听（自动防重）
_eventListeners.Add(EventId.PlayerDead, entity.Id);

// 触发事件，遍历所有监听者
if (_eventListeners.TryGetValue(EventId.PlayerDead, out var listeners))
{
    foreach (long entityId in listeners)
        NotifyEntity(entityId, EventId.PlayerDead);
}

// 实体销毁时，注销所有监听
_eventListeners.Remove(EventId.PlayerDead, entity.Id);
```

**场景二：标签系统（Tag System）**

```csharp
// 实体 → 标签集合
private UnOrderMultiMapSet<long, string> _entityTags = new UnOrderMultiMapSet<long, string>();

_entityTags.Add(entity.Id, "Boss");
_entityTags.Add(entity.Id, "Immune");
_entityTags.Add(entity.Id, "Boss");  // 重复添加无效

bool isBoss = _entityTags.Contains(entity.Id, "Boss");
```

## 四、三者横向对比

| 维度 | DoubleMap | OrderedDictionary | UnOrderMultiMapSet |
|------|-----------|-------------------|--------------------|
| 映射关系 | 1:1（双向） | 1:1（有序） | 1:N（去重） |
| 正向查找 | O(1) | O(1) | O(1) |
| 反向查找 | O(1) | ❌ | ❌ |
| 顺序保证 | ❌ | ✅ 插入顺序 | ❌ |
| Value 重复 | 不允许 | 允许 | 不允许 |
| 删除复杂度 | O(1) | O(n) | O(1) |
| 内存占用 | 2× Dictionary | 1× Dict + 2× List | 1× Dict + N× HashSet |

**选型指引：**

- 需要"通过值找键" → `DoubleMap`
- 需要"遍历顺序与插入顺序一致" → `OrderedDictionary`
- 需要"一键对多值，且值不能重复" → `UnOrderMultiMapSet`
- 需要"一键对多值，且值可以重复" → `MultiMap`（另一个容器）

## 五、实现细节中的工程智慧

### 5.1 DoubleMap 的完整性约束

`Add` 中对 `kv.ContainsKey(key) || vk.ContainsKey(value)` 的检查，保证了双向映射的数学完整性：每个 K 最多对应一个 V，每个 V 也最多对应一个 K。违反约束时静默返回（而非抛异常），适合游戏这种"不能因容器逻辑崩溃"的运行时环境。

### 5.2 OrderedDictionary 的 Indexer 陷阱

更新操作（`dict[key] = newValue`）会将 key 移到末尾，这与直觉不符（直觉上是原地更新）。在需要"顺序不因更新改变"的场景下，需要注意这个行为差异。

### 5.3 UnOrderMultiMapSet 的空 Set 清理

游戏长时间运行时，实体不断创建销毁，若不清理空的 HashSet，内存会持续增长。`Remove` 方法中 `if (set.Count == 0) this.Remove(t)` 这一行，是典型的"防内存泄漏"设计模式，值得在所有类似容器中复用。

## 六、总结

游戏框架的 Core 层之所以需要这些定制容器，是因为标准库的泛型容器只解决"通用场景"，而游戏业务有很多超出通用场景的约束：双向查找、插入顺序、一键多值去重……

设计自定义容器的原则：
1. **明确语义**：容器的名字就是契约（DoubleMap = 双向，OrderedDictionary = 有序）
2. **最小接口**：只暴露必要的操作，避免使用方绕过不变量
3. **防御性实现**：null 检查、空 Set 清理、重复添加静默忽略
4. **性能权衡**：明确哪个操作是瓶颈（如 OrderedDictionary 的 Remove 是 O(n)），在文档/注释中标注

熟练驾驭这些容器，是写出高质量游戏框架代码的基础能力之一。
