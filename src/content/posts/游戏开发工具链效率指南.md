## 工具链的价值

```
一名高级工程师和初级工程师的差距：
  不只是技术深度，还有工具使用效率
  
高级工程师：
  - 用 Profiler 10分钟定位性能问题
  - Git 操作流畅，不会因合并冲突浪费时间

```

## 二、性能分析工具清单

### 2.1 Unity 内置工具

```
Unity Profiler（必须掌握）：
  Window → Analysis → Profiler
  
  用途：
  - CPU 性能（哪个方法最耗时）
  - GPU 性能（渲染哪个阶段最慢）
  - 内存（每帧 GC 分配量）
  - 音频、物理等各模块
  
  关键技巧：
  - 在真实设备上 Profile（不要只在 Editor 里）
  - 连接到设备：Build Settings → Development Build + Autoconnect Profiler
  - 使用 Profiler.BeginSample/EndSample 标记自定义代码段

Frame Debugger（DrawCall 分析）：
  Window → Analysis → Frame Debugger
  
  用途：
  - 查看每一帧的每个 DrawCall
  - 找出合批失败的原因
  - 分析渲染 Pass 的顺序
  
Memory Profiler（内存泄露）：
  Package Manager 安装 Memory Profiler 包
  Window → Analysis → Memory Profiler
  
  用途：
  - 截取内存快照
  - 对比两次快照发现内存泄露
  - 查看哪种对象占用最多内存
```

### 2.2 第三方性能工具

```
RenderDoc（GPU 帧分析）：
  下载：renderdoc.org（免费开源）
  用途：
  - 截取一帧，分析每个 DrawCall 的详细信息
  - 查看 Shader 的编译输出
  - 分析纹理、网格数据
  - 深度分析渲染管线
  
  使用方法：
  Unity 运行 → RenderDoc 注入 → F12 截帧 → 分析

Snapdragon Profiler（Android/高通 GPU）：
  下载：developer.qualcomm.com（免费）
  用途：
  - 高通芯片（骁龙）的 GPU 性能分析
  - GPU 周期数、带宽、管线利用率
  - 针对移动端 GPU 特性的优化
  
ARM Mobile Studio（Android/Mali GPU）：
  适用联发科/三星/海思芯片设备
  类似 Snapdragon Profiler

Xcode Instruments（iOS）：
  随 Xcode 附带
  GPU Frame Debugger、Metal System Trace
```

## 三、调试辅助工具

### 3.1 Unity 实用插件

```
Odin Inspector（强烈推荐，付费）：
  功能：
  - 强大的自定义 Inspector（用特性代替 Editor 脚本）
  - 序列化任意类型（Dictionary、接口等）
  - 内置验证（特性标记数值范围、非空等）
  - 组织 Inspector 分组、Tab 等
  
  价值：节省大量 Editor GUI 开发时间

NaughtyAttributes（免费替代）：
  Unity Package Manager：git+https://github.com/dbrizov/NaughtyAttributes
  提供基础的 Inspector 增强特性

Debug Draw（调试可视化）：
  Unity Debug.DrawLine / DrawRay / DrawCube 等
  在 Scene/Game 窗口显示辅助调试图形
  
  封装工具：
  - 推荐：RuntimeDebugDraw（Asset Store）
  - 可以在运行时 Game 窗口显示调试信息
```

### 3.2 自定义调试系统

```csharp
/// <summary>
/// 游戏内调试信息显示
/// 仅在 Development Build 中显示
/// </summary>
public class DebugOverlay : MonoBehaviour
{
    private readonly StringBuilder _sb = new(2048);
    private GUIStyle _style;
    
    // 注册调试信息的回调
    private readonly List<Func<string>> _providers = new();
    
    public static DebugOverlay Instance { get; private set; }
    
    void Awake()
    {
        Instance = this;
        _style = new GUIStyle
        {
            fontSize = 14,
            normal = { textColor = Color.white }
        };
    }
    
    public void Register(Func<string> infoProvider) => _providers.Add(infoProvider);
    
    void OnGUI()
    {
        if (!Debug.isDebugBuild) return;
        
        _sb.Clear();
        foreach (var provider in _providers)
            _sb.AppendLine(provider());
        
        GUI.Label(new Rect(10, 10, 400, 600), _sb.ToString(), _style);
    }
}

// 注册调试信息（在各个系统的 Start 中）
void Start()
{
    DebugOverlay.Instance?.Register(() => 
        $"HP: {_health.Current}/{_health.Max}\n" +
        $"Pos: {transform.position:F1}\n" +
        $"FPS: {(int)(1f / Time.unscaledDeltaTime)}"
    );
}
```

### 4.2 分支策略

```
推荐使用 Git Flow 的简化版：

main：生产分支（只有经过测试的代码）
develop：开发分支（日常开发合并到这里）
feature/xxx：功能分支（从 develop 创建，完成后合并回 develop）
hotfix/xxx：紧急修复（从 main 创建，修复后合并回 main 和 develop）

日常工作流程：
  1. 从 develop 创建 feature/my-feature
  2. 开发、提交
  3. 提 PR → Code Review → 合并到 develop
  4. develop 定期合并到 main（发版）

团队规范：
  - 提交信息格式：feat/fix/refactor/docs: 描述
  - 例：feat: 添加技能冷却系统，fix: 修复登录超时问题
  - 每次提交只做一件事（Small Commit Principle）
```

### 4.3 Unity YAML Merge Tool

```bash
# 配置 Smart Merge（Unity 的 YAML 合并工具）
# 在 .git/config 或全局 git config 中添加：

[merge "unityyamlmerge"]
    name = Unity SmartMerge
    driver = "UnityYAMLMerge" merge -p %O %B %A %P

# 确保 UnityYAMLMerge.exe 在 PATH 中
# Windows: C:\Program Files\Unity\Hub\Editor\[版本]\Editor\Data\Tools\UnityYAMLMerge.exe
# macOS: /Applications/Unity/Hub/Editor/[版本]/Unity.app/Contents/Tools/UnityYAMLMerge
```
