## 一个残酷的真相

很多工程师成为技术 Leader 后，面临的第一个困境是：

> "我现在做的事情，好像不如以前有价值了。"

以前作为工程师，写代码是直接产出，成就感清晰。变成 Leader 后，开会、沟通、文档、审查代码……感觉自己没有在"干活"。

这是一个根本性的**角色认知错误**。

---

## 一、重新定义"产出"

### 技术 Leader 的核心产出不是代码

```
工程师的产出：
  代码行数 × 质量 = 个人产出

技术 Leader 的产出：
  团队的整体技术质量 × 交付速度 = 团队产出
```

一个 10 人团队，如果 Leader 能让每个人的效率提升 20%，
这相当于团队多了 2 个人，而且这 2 个人会随着时间积累越来越强。

**这是杠杆效应**，是任何个人贡献都无法比拟的价值。

### 你的时间应该花在哪里

```
工程师时间分配（理想状态）：
  70% 写代码
  20% 设计与评审
  10% 其他

技术 Leader 时间分配（健康状态）：
  30% 核心技术攻坚（保持技术敏感）
  25% 代码评审与技术指导
  20% 架构设计与技术方案
  15% 跨部门沟通与对齐
  10% 工程效能建设

常见错误分配（技术 Leader 的陷阱）：
  60% 写代码（应该留给团队，自己只做最关键的部分）
  40% 其他（效能建设投入不够）
```

## 二、核心职责的第一性原理

### 2.1 澄清目标：最被忽视的工作

技术 Leader 最重要的工作之一，是**让团队清楚地知道在做什么、为什么做、做到什么程度**。

```
糟糕的状态（你见过多少次？）：
  策划说：做一个技能系统
  程序理解：和上个项目一样的技能系统
  实现后：策划说这不是我想要的...
  
根因：没有在开发前对齐理解

好的做法：
  1. 收到需求后，先问清楚"为什么"
     - 这个技能系统支持多少种技能？
     - 技能之间可以组合吗？
     - 需要支持热更新吗？
     - 性能目标是什么？
     
  2. 写技术设计文档，让策划确认理解是否一致
  
  3. 制定可测量的完成标准
     - 功能完成标准：所有用例通过
     - 性能完成标准：技能释放 < 16ms
     - 质量完成标准：单测覆盖率 > 70%
```

### 2.2 降低协作成本：被低估的高价值工作

```
团队 10 人，每人每天因为沟通不畅浪费 30 分钟
= 5 人·天/周 = 20 人·天/月 ≈ 2 个工程师

技术 Leader 通过建立规范，将沟通浪费降低到 10 分钟
= 增加了约 1.3 个工程师的效能

这是"系统性提效"，比自己写代码更有价值

降低协作成本的手段：
  - 制定代码规范（减少 code review 中的风格争论）
  - 建立接口文档（减少"这个函数是干嘛的"的提问）
  - 明确模块边界（减少"我以为你会处理这个"的摩擦）
  - 定期同步进度（减少"我不知道他在做什么"的焦虑）
```

### 2.3 识别并移除障碍

```
技术 Leader 的重要职责之一：发现并清除阻碍团队的因素

类型一：技术障碍
  "这个第三方 SDK 的文档太差了，大家都卡在接入上"
  → Leader 主动攻坚，或找到能解决的人

类型二：流程障碍
  "提交 PR 要等 3 天才有人 review"
  → 建立 review 轮班制，当天必须响应

类型三：信息障碍
  "我不知道那个模块的接口是怎么设计的"
  → 建立接口文档，主动共享知识

类型四：决策障碍
  "这个需求不合理，但我不敢和策划说"
  → 帮助团队成员与外部对齐，做决策的"护盾"
```

## 三、代码审查：技术传播的最佳载体

代码审查（Code Review）是技术 Leader 影响团队技术质量的最高效手段。

### 3.1 代码审查的正确目标

```
代码审查不是：
  ❌ 找茬和挑错（让作者难堪）
  ❌ 证明自己技术更强
  ❌ 确保代码按自己的风格写

代码审查是：
  ✅ 共享知识（让整个团队了解这个变更）
  ✅ 发现潜在 Bug 和设计问题
  ✅ 保持代码库的一致性和可维护性
  ✅ 引导作者思考更好的实现
```

### 3.2 高质量代码审查的实践

```markdown
## Code Review 评论分级

[BLOCKER] 必须修改，否则不合并
  - 明确的 Bug
  - 安全漏洞
  - 破坏架构边界的设计
  
[MINOR] 建议修改，但不阻断合并
  - 可以改进但不必须的优化
  - 更清晰的命名方案
  
[NIT] 微小建议，完全可选
  - 风格偏好（在规范已统一的前提下）
  
[QUESTION] 提问，不是评论
  - "这里为什么选择 X 而不是 Y？"
  - 帮助自己理解，也可能引发作者重新思考
  
[PRAISE] 赞美好的代码
  - "这个设计很巧妙！学到了"
  - 正向激励，让作者知道什么是好的
```

**高效 Review 的工作流**：

```
1. 先看整体（5分钟）
   - 这个 PR 解决了什么问题？
   - 设计思路是什么？
   - 变更范围是否合理？

2. 再看细节（10~30分钟）
   - 逻辑是否正确？
   - 边界情况处理了吗？
   - 有没有潜在的性能问题？

3. 给出综合评价
   - "总体设计很好，有几个小问题请关注"
   - 避免只挑问题，不肯定做得好的地方

响应时效承诺：
  收到 PR → 4小时内响应（不一定完成，但要回应"我看到了，今天/明天看"）
  简单 PR → 当天完成
  复杂 PR → 24小时内完成
```

## 四、技术方案决策

### 4.1 决策的成本

```
决策的成本不是做决策本身，而是：
  - 错误决策的修复成本
  - 在错误路径上浪费的时间

所以好的技术决策 = 在信息不充分的情况下，做出错误代价最小的选择
```

### 4.2 决策框架：逆向工作法

```
不要从"我能做什么"出发，而是从"玩家体验/业务目标"出发

案例：需要设计一个聊天系统

错误做法：
  "我熟悉 WebSocket，就用 WebSocket 好了"

正确做法（逆向）：
  1. 目标：玩家能流畅实时聊天
  
  2. 约束条件：
     - 同时在线 10 万玩家
     - 每条消息延迟 < 500ms
     - 服务器成本预算 X 元/月
  
  3. 评估方案：
     - WebSocket：实时性好，但服务器维持长连接，成本高
     - HTTP 轮询：成本低，延迟高（每 1-2 秒轮询一次）
     - MQTT：适合大量设备的消息广播，运营成本低
  
  4. 结合约束选择：
     - 10 万在线、500ms 延迟 → WebSocket（轮询延迟不够）
     - 但需要优化连接池降低成本
```

### 4.3 避免过度工程（Over-engineering）

```
过度工程的典型表现：
  "我们未来可能需要支持 100 种不同的 XXX，所以现在设计一个超级灵活的框架"

真实情况：
  - "未来可能"的需求，80% 不会出现
  - 过于灵活的框架，维护成本远高于简单实现
  - YAGNI（You Ain't Gonna Need It）原则

判断标准：
  需要这个功能的概率 × 未来添加的成本 > 现在设计的额外成本
  才值得现在设计扩展点
  
案例：
  ✅ 接入了 2 个平台，可能加第 3 个 → 设计插件接口
  ❌ 技能系统现在只有 3 个技能，"将来可能有 1000 个" → 用简单实现先走
```

## 五、团队建设

### 5.1 识别团队成员的发展阶段

```
四个象限：能力 × 意愿

高能力 + 高意愿 = 明星员工
  → 给挑战性任务，授权，减少干预

高能力 + 低意愿 = 可能遇到了瓶颈或不满
  → 了解原因，给新方向，或者帮助找到重新燃起动力的事

低能力 + 高意愿 = 有潜力的新人
  → 多指导，多 pair programming，快速成长

低能力 + 低意愿 = 需要重点关注
  → 私下深入谈话，了解根因，制定改进计划
```

### 5.2 有效的技术分享文化

```
为什么很多公司的技术分享流于形式？
  - "为了分享而分享"，没有真实问题驱动
  - 只讲理论，不讲实践
  - 没有讨论和互动，变成单向演讲

建立高质量分享文化的方法：

1. 问题驱动：分享应该源于真实遇到的问题
   "我们项目上遇到了 DrawCall 过高的问题，是这样解决的..."
   比"今天我来介绍一下 DrawCall 优化"更有价值

2. 实战优先：展示真实代码，而非理论图表

3. 留时间讨论：分享后 15 分钟问答
   "你们有没有遇到类似的问题？有没有更好的方案？"

4. 记录存档：分享内容写成文档，沉淀为团队知识

5. 鼓励质疑：最好的分享是能被提问挑战的分享
```

### 5.3 如何给出有建设性的反馈

```
SBI 模型（Situation-Behavior-Impact）

情境（Situation）：描述具体情况，不笼统
行为（Behavior）：描述观察到的具体行为，不猜测动机
影响（Impact）：描述这个行为造成的影响

❌ 糟糕的反馈：
  "你的代码写得很差，需要改进"
  → 模糊，对方不知道改什么

✅ 建设性反馈（SBI）：
  S: "在上周的战斗系统开发中（Situation）"
  B: "你提交了一个没有写单测的 PR，直接合并了（Behavior）"
  I: "导致我们后来出现了一个因为边界情况没有覆盖到的 Bug，
      修复花了半天时间（Impact）"

  然后探讨改进：
  "我们能不能约定，对核心逻辑的 PR 必须包含单测？
   你觉得什么样的测试覆盖率是合理的？"
```

## 六、技术 Leader 的自我管理

### 6.1 防止技术脱节

```
技术 Leader 的最大陷阱：管理工作越来越多，技术越来越久没碰，
逐渐失去技术判断力，变成一个纯管理者。

保持技术敏感的方法：

1. 每个版本必须亲自参与一个核心技术模块的开发
   不是全部，但要有。
   
2. 参与代码审查（不只是走流程，要深度理解）
   每周至少认真审查 3~5 个 PR

3. 定期做技术探索
   每季度花 2~3 天研究一个新技术方向
   不求立即落地，保持视野

4. 与团队成员结对编程（Pair Programming）
   既能了解团队在做什么，也能发现工程问题
```

### 6.2 学会做艰难的决策

```
技术 Leader 必须做的艰难决策：

1. 重构 vs 继续堆功能
   技术债到达临界点时，必须停下来还债
   即使短期影响交付速度，长期更健康
   
2. 拒绝不合理的需求
   "这个功能在当前架构下实现会带来很大的技术风险"
   要敢于说不，并提供替代方案

3. 对低绩效成员的处理
   给机会、给指导，但如果持续没有改进，
   对团队其他成员不公平，需要果断处理
   
4. 技术选型的大转变
   "我们现在用 X，但 Y 明显更适合我们，值得切换"
   这类决策涉及大量迁移成本，需要仔细评估，
   但不能因为惧怕变化而永远维持现状
```

## 七、晋升的底层逻辑

### 7.1 晋升不是奖励，是契约

```
晋升 = 公司认为你在更高职级上能创造更大价值

所以晋升准备不是：
  "我已经做了这么多工作，应该晋升了"
  
而是：
  "我能证明自己在 T+1 级别上持续创造价值"

两者的区别：
  前者是付出视角（已经做了什么）
  后者是价值视角（能创造什么）
```

### 7.2 晋升答辩的准备

```
常见问题和应答框架：

Q: "说说你最有挑战性的项目"
A: STAR 模型
   Situation: 背景是什么
   Task: 面对的挑战是什么
   Action: 你具体做了什么（重点！要有细节）
   Result: 结果如何（量化！"性能提升 60%"比"性能显著提升"有说服力）

Q: "如果重来，你会做什么不一样的决定"
A: 诚实地反思，展示成长
   不要说"没有什么想改变的"（不真实）
   要说"我会更早 XXX，因为 YYY"

Q: "你对团队/业务的影响是什么"
A: 超出个人边界
   不只说自己做的，要说对团队、对项目的贡献
   "我设计了 XXX 框架，让全组 5 个人的开发效率提升了..."
```

### 7.3 技术影响力的建立

```
在公司内建立技术影响力：

1. 输出：写文档、做分享，让更多人知道你在想什么
   不是炫技，是知识共享

2. 解决公共问题：主动帮助解决跨团队的公共技术问题
   "这个 SDK 接入问题好几个组都遇到了，我来整理一下通用解决方案"

3. 标准建设：推动制定和维护技术标准
   "我们来制定一个项目的 C# 编码规范，这样以后 review 效率更高"

4. 技术选型参与：当公司有重要技术选型时，积极参与评估
   展示你的判断力和全局视角
```

## 结语

成为优秀的技术 Leader，是一条需要持续修炼的路。

没有完美的 Leader，只有不断反思、不断改进的 Leader。

最后送给每一位在这条路上的人一句话：

> **你的代码会老化，但你帮助成长的人会永远记得你。**

技术可以被替代，影响不会。
