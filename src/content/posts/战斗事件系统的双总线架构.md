# 战斗事件系统的双总线架构

在复杂的战斗系统中，模块之间有着千丝万缕的联系：受击时需要通知 UI 更新血条、通知音效播放、通知特效系统、通知统计系统……如果直接调用，会形成紧密耦合的"蜘蛛网"。

本文解析一套"双总线"事件系统架构，理解如何在大型游戏中优雅地解耦模块间通信。

---

## 第一性原理：为什么需要事件系统？

假设受击时需要触发以下行为：

```csharp
// 紧密耦合的写法
public static void TakeDamage(Unit defender, FP damage)
{
    // 核心逻辑
    defender.GetComponent<NumericComponent>().Modify(ENumericId.RealTeamHp, -damage);
    
    // 直接调用其他系统（糟糕）
    UIManager.UpdateHpBar(defender);
    AudioManager.PlayHurtSound();
    EffectManager.PlayHurtEffect(defender);
    StatisticManager.RecordDamage(damage);
    AchievementManager.CheckDamageAchievement(defender, damage);
}
```

问题：
- `TakeDamage` 需要知道所有依赖系统
- 新增系统就要修改 `TakeDamage`
- 无法独立测试核心逻辑

**事件系统的解法**：

```csharp
public static void TakeDamage(Unit defender, FP damage)
{
    defender.GetComponent<NumericComponent>().Modify(ENumericId.RealTeamHp, -damage);
    // 发布事件，不需要知道谁在订阅
    EventSystem.Instance.Publish(scene, new Evt_TakeDamage { Unit = defender, Damage = damage });
}

// 各系统独立订阅，各自处理
[EventSystem]
public class OnTakeDamage_UI : AEvent<Evt_TakeDamage> { ... }
public class OnTakeDamage_Audio : AEvent<Evt_TakeDamage> { ... }
public class OnTakeDamage_Statistic : AEvent<Evt_TakeDamage> { ... }
```

## 双总线架构：全局总线 vs 局部分发器

系统提供了两种事件机制：

### 全局事件总线（EventSystem）

```csharp
// 发布（全局广播）
EventSystem.Instance.Publish(scene, new Evt_HurtHitEffect
{
    Attacker = attacker,
    Defender = defender,
    jumpTextType = jumpTextType
});

// 订阅（注解声明）
[EventSystem]
public class OnHurtHitEffect : BEventSystem<Evt_HurtHitEffect>
{
    protected override void Run(Scene scene, Evt_HurtHitEffect args)
    {
        // 处理受击特效
    }
}
```

`EventSystem` 是全局单例，通过 `[EventSystem]` 注解自动注册到系统中。`Publish` 找到所有订阅了该事件类型的处理器并调用。

### 局部事件分发器（EventDispatcherComponent）

```csharp
// 发布（针对特定 Unit 的局部事件）
defender.GetComponent<EventDispatcherComponent>().FireEvent(new Evt_DefenderHitState
{
    HitState = defenderState
});

// 订阅（只在特定 Unit 上订阅）
dispatcher.RegisterEvent<Evt_DefenderHitState>(OnHitState);

// 取消订阅
dispatcher.UnRegisterEvent<Evt_DefenderHitState>(OnHitState);
```

`EventDispatcherComponent` 是挂载在 Unit 上的局部分发器，只处理该 Unit 的事件。

---

## 两种总线的适用场景对比

| 特性 | EventSystem（全局总线） | EventDispatcherComponent（局部分发） |
|------|----------------------|--------------------------------------|
| 作用域 | 全场景广播 | 单个 Entity |
| 订阅方式 | 注解 `[EventSystem]` | 手动 `RegisterEvent` |
| 取消订阅 | 自动（对象销毁时） | 手动 `UnRegisterEvent` |
| 性能 | 遍历所有订阅者 | 只通知该 Entity 的订阅者 |
| 适用 | 跨系统通知（UI、音效） | Entity 内部状态通知 |

**使用全局总线**的场景：
- 血量变化 → 通知 UI 更新血条
- 角色死亡 → 通知摄像机、统计系统、结算系统
- 战斗结束 → 通知所有相关系统清理

**使用局部分发器**的场景：
- Unit A 的状态变化 → 只通知 A 自己的技能图节点
- Unit A 受击 → 只通知 A 的技能图（不需要通知场景中所有 Unit）
- Unit 的自定义生命周期事件

## 事件定义：轻量结构体

事件定义使用结构体而非类，减少 GC 压力：

```csharp
// 受击特效事件
public struct Evt_HurtHitEffect
{
    public Unit Attacker;
    public Unit Defender;
    public JumpTextType jumpTextType;
}

// 位置变化事件
public struct ChangePosition
{
    public Unit Unit;
    public TSVector NewPos;
}

// 破防事件
public struct Evt_BreakDefense
{
    public Unit atker;
    public Unit target;
}
```

结构体在栈上分配，不产生 GC 对象，适合高频事件。

但注意：**包含引用类型字段的结构体**（如 `Unit Attacker`）并不能完全避免 GC——字段引用的对象仍然在堆上，但事件本身是栈上数据，开销小于创建 class 对象。

## 帧循环事件：驱动整个战斗更新

```csharp
// LogicComponent（Unity 生命周期适配器）
private static void FixedUpdate(this LogicComponent self)
{
    // 将 Unity FixedUpdate 转化为自定义事件
    EventSystem.Instance.Publish(self.ClientScene(), new Evt_FixedUpdate());
}

private static void LateFixedUpdate(this LogicComponent self)
{
    EventSystem.Instance.Publish(self.ClientScene(), new Evt_LateFixedUpdate());
}

private static void Update(this LogicComponent self)
{
    EventSystem.Instance.Publish(self.ClientScene(), new Evt_UnityUpdate());
}
```

`LogicComponent` 是 Unity 生命周期的"适配器"——它接收 Unity 的 `FixedUpdate`、`Update` 等回调，转化为自定义事件广播出去。

战斗中的各个系统不再直接实现 Unity 的 MonoBehaviour 接口，而是订阅这些事件：

```csharp
// WorldComponent 订阅帧更新事件
private static void Awake(this WorldComponent self)
{
    self.RootDispatcher().RegisterEvent<Evt_FixedUpdate>(self.EnterFrame);
    self.RootDispatcher().RegisterEvent<Evt_Update>(self.Update);
}
```

这种设计的好处：
- 系统可以是纯 C# 类（不需要挂载在 GameObject 上）
- 更容易测试（可以手动发布事件而不依赖 Unity 运行时）
- 更容易控制更新顺序（不依赖 Script Execution Order）

---

## 事件的时序保证

`LogicComponent` 确保了以下时序：

```
Unity FixedUpdate
    → Evt_FixedUpdate → 所有逻辑帧更新（战斗核心）

Unity LateFixedUpdate（自定义，调用时机固定）
    → Evt_LateFixedUpdate → 帧末清理

Unity Update
    → Evt_UnityUpdate → 渲染帧更新（位置插值）

Unity LateUpdate
    → Evt_PreLateUpdate → 渲染后处理
    → Evt_CamerCompLateUpdate → 摄像机更新
    → Evt_LateUpdate → 最后的渲染更新
```

摄像机更新放在 `CamerCompLateUpdate`，保证它在所有 Unit 位置更新完成后才更新摄像机位置，避免摄像机抖动。

## 被动技能事件系统（PSkillEvent）

被动技能有一套专用的事件机制 `Evt_PSkillEvent`，用于触发"X条件下Y效果"：

```csharp
// 触发被动技能事件
EventSystem.Instance.Publish(scene, new Evt_PSkillEvent
{
    owner = attacker,           // 触发者
    pSkillEvent = EPSkillEventTag.OnAttack,  // 事件类型（攻击时）
    skillID = skillId,
    level = level,
    slotType = slotType
});

// 被动技能图中的节点监听
[EventSystem]
public class TG_PSkillEventHandler : AFlowNodeHandler<TG_PSkillEvent>
{
    public override void OnPostGraphStarted(TG_PSkillEvent flowNode)
    {
        // 监听 Evt_PSkillEvent 事件
        ownerPSkill.GetComponent<EventDispatcherComponent>()
            .RegisterEvent<Evt_PSkillEvent>(flowNode.OnFireEvent);
    }
}
```

`EPSkillEventTag` 枚举定义了所有可能触发被动技能的时机：

```csharp
public enum EPSkillEventTag
{
    OnAttack,       // 发动攻击时
    OnHit,          // 命中时
    OnKill,         // 击杀时
    OnDodge,        // 成功闪避时
    OnParry,        // 格挡时
    OnBreakDefense, // 破防时
    OnReceiveHeal,  // 受到治疗时
    // ...
}
```

这套机制让被动技能配置非常灵活：策划只需要配置"触发条件 + 效果"，不需要硬编码每种情况。

## 设计总结

| 特性 | 设计选择 |
|------|---------|
| 全局广播 | EventSystem + 注解自动注册 |
| 局部通知 | EventDispatcherComponent + 手动订阅 |
| 事件类型 | 结构体（减少 GC） |
| 生命周期 | 局部分发器手动管理订阅 |
| Unity 适配 | LogicComponent 转发 Unity 生命周期 |
| 被动技能触发 | 专用 Evt_PSkillEvent + Tag 系统 |

事件系统是大型游戏代码的"高速公路"——它让各个模块可以独立开发、独立测试，同时保持整体的协同运作。理解了这套双总线架构，你就掌握了中大型游戏项目代码组织的核心技术之一。
