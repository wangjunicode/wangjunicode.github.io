# 从面向对象到可视化脚本：战斗逻辑编程范式演进全解析

> 很多人学了节点图、行为树、可视化脚本，却说不清楚"为什么要这样设计"。本文从最朴素的 `if/else` 出发，一步步推导出主流战斗可视化脚本方案的来龙去脉——每一次演进都是被真实痛点逼出来的。

---

## 第一阶段：最朴素的起点——顺序 + 条件 + 循环

### 1.1 三大基础结构

任何程序语言都只有三种基本控制结构：

```
顺序（Sequence）：  A → B → C
条件（Selection）：  if 条件 then A else B
循环（Repetition）： while 条件 do A
```

用这三种结构，你可以描述世界上任何算法（这是图灵完备性的核心含义）。

### 1.2 用三大结构写一个"技能释放"

假设我们要实现"玩家按Q键释放火球技能"：

```csharp
void Update()
{
    // ① 顺序：每帧执行以下步骤
    
    // ② 条件：检测输入
    if (Input.GetKeyDown(KeyCode.Q))
    {
        // ③ 条件：检查 MP 是否足够
        if (playerMP >= 20)
        {
            // ④ 条件：检查 CD 是否冷却
            if (skillCooldown <= 0)
            {
                // ⑤ 顺序：执行技能逻辑
                playerMP -= 20;
                skillCooldown = 3.0f;
                
                // 生成火球
                GameObject fireball = Instantiate(fireballPrefab, 
                    transform.position + transform.forward, 
                    transform.rotation);
                fireball.GetComponent<Rigidbody>().velocity = 
                    transform.forward * 15f;
                
                // 播放动画
                animator.SetTrigger("CastSpell");
                
                // 播放音效
                AudioSource.PlayClipAtPoint(castSound, transform.position);
            }
        }
    }
    
    // ⑥ 循环（隐式）：CD 倒计时
    if (skillCooldown > 0)
    {
        skillCooldown -= Time.deltaTime;
    }
}
```

**这段代码完全可以运行。**对于一个简单的 Demo，它没什么问题。

### 1.3 第一个痛点出现：多技能时的灾难

游戏需要 5 个技能：Q/W/E/R/F，每个技能有各自的 MP 消耗、CD、效果……

```csharp
void Update()
{
    // Q 技能
    if (Input.GetKeyDown(KeyCode.Q))
    {
        if (playerMP >= 20 && skillQCooldown <= 0)
        {
            playerMP -= 20;
            skillQCooldown = 3.0f;
            // ... Q 技能逻辑（50行）
        }
    }
    
    // W 技能
    if (Input.GetKeyDown(KeyCode.W))
    {
        if (playerMP >= 35 && skillWCooldown <= 0)
        {
            playerMP -= 35;
            skillWCooldown = 5.0f;
            // ... W 技能逻辑（80行）
        }
    }
    
    // E 技能（100行）
    // R 技能（120行）
    // F 技能（60行）
    
    // 5个CD倒计时
    if (skillQCooldown > 0) skillQCooldown -= Time.deltaTime;
    if (skillWCooldown > 0) skillWCooldown -= Time.deltaTime;
    // ...
}
```

`Update()` 变成了一个 500 行的怪物。**核心问题**：

| 问题 | 表现 |
|------|------|
| **重复代码** | "检查MP → 检查CD → 扣MP → 重置CD"在每个技能都重复 |
| **难以维护** | 改一个技能要在 500 行里找对应位置 |
| **难以扩展** | 加第 6 个技能，继续往下堆 |
| **难以复用** | "检查CD"逻辑无法在其他地方复用 |

## 第二阶段：面向对象——用封装消灭重复

### 2.1 提取公共逻辑：Skill 基类

发现规律：**每个技能都有 MP 消耗、CD、释放条件检查、释放逻辑**。把这个"模板"抽象成基类：

```csharp
/// <summary>
/// 技能基类：封装所有技能共有的 MP/CD/条件检查逻辑
/// </summary>
public abstract class SkillBase
{
    public int    mpCost;        // MP 消耗
    public float  cooldown;      // CD 时长（秒）
    public float  currentCD;     // 当前剩余 CD
    public KeyCode triggerKey;   // 触发按键

    // ── 公共行为（所有技能共用）────────────────────────────────

    /// <summary>检查是否可以释放</summary>
    public bool CanCast(float currentMP)
    {
        return currentMP >= mpCost && currentCD <= 0;
    }

    /// <summary>尝试释放技能（检查通过后调用 OnCast）</summary>
    public bool TryCast(ref float currentMP)
    {
        if (!CanCast(currentMP)) return false;
        
        currentMP    -= mpCost;
        currentCD     = cooldown;
        OnCast();     // 子类实现具体效果
        return true;
    }

    /// <summary>每帧更新 CD</summary>
    public void Update(float deltaTime)
    {
        if (currentCD > 0) currentCD -= deltaTime;
    }

    // ── 抽象行为（子类必须实现）────────────────────────────────

    /// <summary>技能释放效果（子类实现）</summary>
    protected abstract void OnCast();
}

// ── 具体技能实现 ────────────────────────────────────────────────
public class FireballSkill : SkillBase
{
    private Transform _casterTransform;
    private GameObject _fireballPrefab;

    public FireballSkill(Transform caster, GameObject prefab)
    {
        mpCost        = 20;
        cooldown      = 3.0f;
        triggerKey    = KeyCode.Q;
        _casterTransform = caster;
        _fireballPrefab  = prefab;
    }

    protected override void OnCast()
    {
        var fb = GameObject.Instantiate(_fireballPrefab,
                     _casterTransform.position + _casterTransform.forward,
                     _casterTransform.rotation);
        fb.GetComponent<Rigidbody>().velocity = _casterTransform.forward * 15f;
    }
}

public class HealSkill : SkillBase
{
    private PlayerStats _stats;

    public HealSkill(PlayerStats stats)
    {
        mpCost     = 30;
        cooldown   = 8.0f;
        triggerKey = KeyCode.W;
        _stats     = stats;
    }

    protected override void OnCast()
    {
        _stats.hp = Mathf.Min(_stats.hp + 50, _stats.maxHp);
    }
}
```

### 2.2 用 SkillManager 统一管理

```csharp
public class SkillManager : MonoBehaviour
{
    private List<SkillBase> _skills = new();
    private float _currentMP = 100f;

    void Start()
    {
        var stats = GetComponent<PlayerStats>();
        _skills.Add(new FireballSkill(transform, fireballPrefab));
        _skills.Add(new HealSkill(stats));
        // 添加更多技能...
    }

    void Update()
    {
        foreach (var skill in _skills)
        {
            skill.Update(Time.deltaTime);

            if (Input.GetKeyDown(skill.triggerKey))
            {
                skill.TryCast(ref _currentMP);
            }
        }
    }
}
```

**`Update()` 从 500 行变成了 10 行！** 面向对象的封装 + 多态完美解决了重复代码问题。

### 2.3 OOP 的新痛点：逻辑固化在代码里

游戏上线后，策划说：
- "Q 技能在玩家有 Buff_X 时，额外增加 50% 伤害"
- "W 技能在 Boss 战期间 CD 减少 2 秒"
- "E 技能加一个新的前置条件：必须在地面才能释放"

每次改动都需要：**修改 C# 代码 → 重新编译 → 重新打包**。

策划的每一个想法都要等程序实现，项目迭代速度被严重拖慢。

// 通用技能类：读取配置执行逻辑
public class GenericSkill : SkillBase
{
    private SkillConfig _config;

    {
        mpCost   = config.mpCost;
        cooldown = config.cooldown;
        _config  = config;
    }

    protected override void OnCast()
    {
        // 根据 skillType 分发到不同的处理逻辑
        switch (_config.skillType)

        }
    }

**收益**：策划可以在编辑器或表格里修改数值，不需要等程序。

### 3.2 数据驱动的新瓶颈：`switch` 爆炸

随着技能类型越来越多，`switch` 语句变成了新的 500 行怪物：

```csharp
switch (_config.skillType)
{
    case "Projectile":      ExecuteProjectile(); break;
    case "Heal":            ExecuteHeal();       break;
    case "AOE":             ExecuteAOE();        break;
    case "Dash":            ExecuteDash();       break;
    case "Summon":          ExecuteSummon();     break;
    case "Shield":          ExecuteShield();     break;
    case "ProjectileChain": ExecuteProjectileChain(); break;
    case "HealOverTime":    ExecuteHealOverTime(); break;
    // ... 50个case
}
```

更关键的问题：**技能的效果开始变得"组合化"**。一个技能可能既发射子弹、又触发AOE、又给自己加Buff、还播放Timeline——用单一的 `skillType` 字段无法描述这种**多步骤、多效果的组合逻辑**。

## 第四阶段：命令模式——把"行为"变成对象

### 4.1 核心思想：行为也是数据

GoF 设计模式中的**命令模式（Command Pattern）**提供了一个关键洞察：

> 把一个"操作"封装成一个对象，就可以像数据一样传递、存储、组合、撤销它。

```csharp
/// <summary>
/// 技能效果命令接口：每个"效果"是一个独立的命令对象
/// </summary>
public interface ISkillEffect
{
    void Execute(SkillContext ctx); // ctx 包含施法者、目标、位置等信息
}

// ── 具体效果命令 ────────────────────────────────────────────────
public class SpawnProjectileEffect : ISkillEffect
{
    public string prefabName;
    public float  speed;

    public void Execute(SkillContext ctx)
    {
        var prefab = Resources.Load<GameObject>(prefabName);
        var obj    = GameObject.Instantiate(prefab, ctx.caster.position, ctx.caster.rotation);
        obj.GetComponent<Rigidbody>().velocity = ctx.caster.forward * speed;
    }
}

public class HealEffect : ISkillEffect
{
    public int amount;

    public void Execute(SkillContext ctx)
    {
        ctx.caster.GetComponent<PlayerStats>().hp += amount;
    }
}

public class AddBuffEffect : ISkillEffect
{
    public int buffId;
    public int duration;

    public void Execute(SkillContext ctx)
    {
        ctx.target.GetComponent<BuffSystem>().AddBuff(buffId, duration);
    }
}

public class PlayAnimationEffect : ISkillEffect
{
    public string animName;

    public void Execute(SkillContext ctx)
    {
        ctx.caster.GetComponent<Animator>().SetTrigger(animName);
    }
}
```

### 4.2 技能 = 效果列表

```csharp
/// <summary>
/// 技能定义：一个技能就是一个"效果命令"的有序列表
/// </summary>
[Serializable]
public class SkillDefinition
{
    public int              skillId;
    public int              mpCost;
    public float            cooldown;
    public List<ISkillEffect> effects;  // ← 核心！技能效果是一个列表
}

// 运行时执行：按顺序执行所有效果命令
public void CastSkill(SkillDefinition skill, SkillContext ctx)
{
    foreach (var effect in skill.effects)
    {
        effect.Execute(ctx);
    }
}
```

现在，一个技能可以组合任意数量的效果：

```
技能：闪现斩
  effects:
    ① DashToTargetEffect   (冲向目标)
    ② PlayAnimationEffect  (播放斩击动画)
    ③ DealDamageEffect     (造成伤害)
    ④ AddBuffEffect        (给自己加攻击力Buff)
    ⑤ SpawnVFXEffect       (生成特效)
```

**收益**：技能效果可以自由组合，新增效果类型不需要修改现有代码（开闭原则）。

### 4.3 命令模式的局限：无法表达"流程控制"

命令模式解决了"技能效果组合"的问题，但遇到**带条件分支的流程**就力不从心了：

```
技能：地刺术
  如果目标在地面 → 触发地刺，造成击飞
  如果目标在空中 → 改为向下砸击，造成下落攻击
  
  击飞后等待 1 秒
  如果目标仍然被击倒 → 触发追加伤害
```

这种"**条件分支 + 等待 + 后续判断**"的结构，用命令列表根本无法描述——它需要**流程图**。

## 第五阶段：有限状态机（FSM）——管理"状态"的变化

### 5.1 为什么需要状态机

战斗中的角色在任意时刻只处于一种状态：`Idle / Running / Attacking / Hurt / Dead`。

状态之间的转换有明确的条件（触发器）。这天然适合用**有限状态机（FSM）**描述：

```
            按攻击键
  Idle ─────────────→ Attacking
    ↑                      │
    │    动画结束           │ 受到伤害
    └──────────────────────┤
                           ↓
                         Hurt
                           │
                           │ 硬直结束
                           ↓
                          Idle
```

### 5.2 手写 FSM 的代码结构

```csharp
public enum CharacterState { Idle, Running, Attacking, Hurt, Dead }

public class CharacterFSM : MonoBehaviour
{
    public CharacterState CurrentState { get; private set; } = CharacterState.Idle;

    void Update()
    {
        // 处理当前状态的逻辑
        switch (CurrentState)
        {
            case CharacterState.Idle:
                UpdateIdle();
                break;
            case CharacterState.Attacking:
                UpdateAttacking();
                break;
            case CharacterState.Hurt:
                UpdateHurt();
                break;
        }
    }

    void UpdateIdle()
    {
        if (Input.GetKeyDown(KeyCode.J))
            TransitionTo(CharacterState.Attacking);
    }

    void UpdateAttacking()
    {
        // 等待攻击动画结束
        if (_attackAnimFinished)
            TransitionTo(CharacterState.Idle);
    }

    void UpdateHurt()
    {
        if (_hurtDurationTimer <= 0)
            TransitionTo(CharacterState.Idle);
    }

    void TransitionTo(CharacterState newState)
    {
        // 退出旧状态
        OnExitState(CurrentState);
        CurrentState = newState;
        // 进入新状态
        OnEnterState(newState);
    }

    void OnEnterState(CharacterState state)
    {
        switch (state)
        {
            case CharacterState.Attacking:
                animator.SetTrigger("Attack");
                _attackAnimFinished = false;
                break;
            case CharacterState.Hurt:
                _hurtDurationTimer = hurtDuration;
                animator.SetTrigger("Hurt");
                break;
        }
    }
}
```

### 5.3 FSM 的局限：状态爆炸与层级问题

对于一个动作游戏角色，状态可能多达 30-50 个（站立攻击1/2/3/4、空中攻击1/2、奔跑攻击、蹲伏攻击、冲刺攻击……），每两个状态之间都可能需要转换条件，连线数量达到 `N²` 级别——完全无法维护。

同时，很多状态有**共同行为**（例如"所有攻击状态都能被击打打断"），用扁平 FSM 无法表达这种层级关系。

这推动了**分层状态机（HFSM）**和**行为树（BT）**的出现。

## 第六阶段：行为树（Behavior Tree）——用树结构替代状态爆炸

### 6.1 行为树的核心思想

行为树将 AI 决策描述为一棵树：

- **叶节点（Leaf）**：具体的行为（移动到目标、播放动画、造成伤害）
- **组合节点（Composite）**：控制子节点的执行顺序和条件
  - `Sequence`（顺序）：从左到右执行，**全部成功才算成功**（AND 语义）
  - `Selector`（选择）：从左到右执行，**第一个成功即返回**（OR 语义）
- **装饰节点（Decorator）**：修改子节点的行为（取反、重复、限时）

```
Root
└── Selector（优先选择第一个可行方案）
    ├── Sequence（方案一：如果受到攻击，则格挡）
    │   ├── Condition: IsBeingAttacked?
    │   └── Action: PlayBlockAnimation
    │
    ├── Sequence（方案二：如果目标在范围内，则攻击）
    │   ├── Condition: IsTargetInRange?
    │   └── Action: AttackTarget
    │
    └── Action: Patrol（兜底：巡逻）
```

### 6.2 代码实现

```csharp
/// <summary>行为树节点基类</summary>
public abstract class BTNode
{
    public enum Status { Success, Failure, Running }
    public abstract Status Tick(BTContext ctx);
}

/// <summary>Sequence：子节点全部成功才返回 Success</summary>
public class Sequence : BTNode
{
    private List<BTNode> _children;
    private int _currentIndex = 0;

    public Sequence(params BTNode[] children) 
        => _children = new List<BTNode>(children);

    public override Status Tick(BTContext ctx)
    {
        while (_currentIndex < _children.Count)
        {
            var status = _children[_currentIndex].Tick(ctx);
            if (status == Status.Failure) { _currentIndex = 0; return Status.Failure; }
            if (status == Status.Running) { return Status.Running; }
            _currentIndex++;
        }
        _currentIndex = 0;
        return Status.Success;
    }
}

/// <summary>Selector：第一个成功的子节点即返回 Success</summary>
public class Selector : BTNode
{
    private List<BTNode> _children;

    public Selector(params BTNode[] children) 
        => _children = new List<BTNode>(children);

    public override Status Tick(BTContext ctx)
    {
        foreach (var child in _children)
        {
            var status = child.Tick(ctx);
            if (status != Status.Failure) return status;
        }
        return Status.Failure;
    }
}

/// <summary>Condition 节点：检查条件</summary>
public class ConditionNode : BTNode
{
    private Func<BTContext, bool> _condition;
    public ConditionNode(Func<BTContext, bool> condition) => _condition = condition;
    public override Status Tick(BTContext ctx) 
        => _condition(ctx) ? Status.Success : Status.Failure;
}

/// <summary>Action 节点：执行动作</summary>
public class ActionNode : BTNode
{
    private Func<BTContext, Status> _action;
    public ActionNode(Func<BTContext, Status> action) => _action = action;
    public override Status Tick(BTContext ctx) => _action(ctx);
}

// ── 组装行为树（代码方式）──────────────────────────────────────
public BTNode BuildEnemyAI()
{
    return new Selector(
        new Sequence(
            new ConditionNode(ctx => ctx.self.IsBeingAttacked()),
            new ActionNode   (ctx => { ctx.self.PlayBlock(); return BTNode.Status.Success; })
        ),
        new Sequence(
            new ConditionNode(ctx => ctx.self.IsTargetInRange(3f)),
            new ActionNode   (ctx => { ctx.self.AttackTarget(); return BTNode.Status.Success; })
        ),
        new ActionNode(ctx => { ctx.self.Patrol(); return BTNode.Status.Running; })
    );
}
```

### 6.3 行为树的本质突破

| 对比项 | FSM | 行为树 |
|--------|-----|--------|
| **结构** | 状态 + 转换连线 | 树形层级 |
| **状态爆炸** | 严重（N² 连线）| 无（层级复用）|
| **可读性** | 复杂后难读 | 树形结构直观 |
| **复用** | 困难 | 子树可复用 |
| **添加新行为** | 需要改连线 | 添加叶节点即可 |
| **调试** | 难以追踪转换路径 | 可高亮当前执行路径 |

## 第七阶段：流程图（FlowGraph）——把"时序逻辑"可视化

### 7.1 行为树不能描述的问题

行为树擅长描述**每帧做什么决策**，但不擅长描述**技能释放的完整时序流程**：

```
技能：炎爆术
①  播放前摇动画（等待 0.3 秒）
②  检查前方是否有敌人（条件分支）
    ├─ 有敌人 → 生成爆炸特效 + 造成伤害
    └─ 无敌人 → 生成哑弹
③  播放后摇动画（等待 0.5 秒）
④  技能结束
```

这是一个**带等待、带条件分支的有序流程**。用行为树可以描述，但不直观；用代码写协程可以，但又绕回了"逻辑固化"的老问题。

**流程图（FlowGraph）**是为此而生的：

```
[Enter]
   │
   ▼
[PlayAnimation "Cast_Begin"]  ←─── 动作节点
   │ 动画完成（0.3s）
   ▼
[CT_CheckEnemyInRange]  ←────────── 条件节点
   │ 真                   │ 假
   ▼                      ▼
[SpawnExplosion]     [SpawnDud]
   │                      │
   └──────────┬───────────┘
              ▼
[PlayAnimation "Cast_End"]
              │ 动画完成（0.5s）
              ▼
[FlowOutput "Complete"]
```

### 7.2 流程图节点的代码结构

```csharp
/// <summary>流程图节点基类</summary>
public abstract class FlowNode
{
    public List<FlowPort> flowOutputs = new();
    
    // 节点执行，返回下一个要执行的端口（null 表示结束）
    public abstract IEnumerator Execute(FlowContext ctx, 
                                        Action<FlowPort> onNext);
}

/// <summary>动作节点：播放动画并等待完成</summary>
public class PlayAnimNode : FlowNode
{
    public string animName;

    public override IEnumerator Execute(FlowContext ctx, Action<FlowPort> onNext)
    {
        ctx.caster.GetComponent<Animator>().SetTrigger(animName);
        
        // 等待动画结束（通过事件回调）
        bool finished = false;
        ctx.caster.OnAnimFinish += () => finished = true;
        while (!finished) yield return null; // 逐帧等待
        
        onNext(flowOutputs[0]); // 触发唯一出口
    }
}

/// <summary>条件分支节点</summary>
public class ConditionNode : FlowNode
{
    // flowOutputs[0] = True 出口
    // flowOutputs[1] = False 出口
    
    public override IEnumerator Execute(FlowContext ctx, Action<FlowPort> onNext)
    {
        bool result = EvaluateCondition(ctx);
        onNext(result ? flowOutputs[0] : flowOutputs[1]);
        yield break; // 条件节点不等待
    }

    protected virtual bool EvaluateCondition(FlowContext ctx) => false;
}

/// <summary>流程图执行引擎（协程驱动）</summary>
public class FlowGraphExecutor : MonoBehaviour
{
    public void Execute(FlowGraph graph, FlowContext ctx)
    {
        StartCoroutine(RunFlow(graph.entryNode, ctx));
    }

    private IEnumerator RunFlow(FlowNode node, FlowContext ctx)
    {
        if (node == null) yield break;

        FlowPort nextPort = null;
        yield return node.Execute(ctx, port => nextPort = port);

        if (nextPort?.connectedNode != null)
            yield return RunFlow(nextPort.connectedNode, ctx);
    }
}
```

## 第八阶段：可视化脚本——把流程图"画"出来

### 8.1 从代码到画布的关键一跳

到目前为止，我们的流程图依然是"在代码里组装节点"。但策划不会写代码。

**可视化脚本的核心思想**：

> 把流程图的"节点和连线"用图形界面表示，让非程序员直接用鼠标拖拽来编写游戏逻辑。

这需要解决三个工程问题：

| 问题 | 解决方案 |
|------|---------|
| **如何画节点图** | Unity GraphView API（UI Toolkit）|
| **如何序列化保存** | JSON 存储节点位置 + 连线关系 |
| **如何运行时执行** | 反序列化 JSON → 构建节点图 → 驱动执行 |

### 8.2 可视化脚本的完整数据流

```

      │
      ▼
拖拽节点 + 连线
      │
      ▼
序列化为 JSON（存储在 Assets / 热更包中）
      │
      ▼
运行时加载 JSON
      │
      ▼
NodeTypeRegistry 多态反序列化 → 构建 FlowGraph 对象
      │
      ▼
GraphExecutor 驱动执行
      │
      ▼
按节点连线顺序调用 Execute()
```

### 9.2 每次演进的"逼迫力量"总结

每次范式演进，都不是"为了设计而设计"，而是被真实的工程痛点**逼出来**的：

| 演进 | 逼迫力量 |
|------|---------|
| `if/else` → OOP | 代码重复到无法维护 |
| OOP → 数据驱动 | 改参数要重新编译，策划等不起 |
| 数据驱动 → 命令模式 | 技能效果需要自由组合 |
| 命令模式 → FSM | 角色状态需要清晰的边界和转换 |
| FSM → 行为树 | 状态爆炸，N² 连线无法维护 |
| 行为树 → 流程图 | 技能时序逻辑无法用决策树描述 |
| 流程图 → 可视化脚本 | 策划需要直接参与逻辑迭代 |

## 第十阶段：现代战斗可视化脚本的工程实现要点

### 10.1 节点类型系统

可视化脚本的精髓在于**类型安全**。错误的连线在编辑器里就应该被拒绝，而不是在运行时崩溃：

```csharp
/// <summary>
/// 端口类型兼容性检查
/// 编辑器连线时调用，不兼容则拒绝连接
/// </summary>
public static bool CanConnect(Type outputType, Type inputType)
{
    // 精确匹配
    if (outputType == inputType) return true;
    
    // 继承关系（子类可以连接到父类端口）
    if (inputType.IsAssignableFrom(outputType)) return true;
    
    // 注册的隐式转换（如 int 可以连接到 float 端口）
    if (ImplicitConverterRegistry.Has(outputType, inputType)) return true;
    
    return false;
}
```

### 10.2 热更新支持：JSON 序列化优先

可视化脚本的最大优势之一是**热更新友好**。实现要点：

```csharp
/// <summary>
/// 节点字段使用短键序列化（减少 JSON 体积）
/// [fsSerializeAs("d")] 表示序列化键名为 "d" 而非 "damage"
/// </summary>
[Serializable]
public class DealDamageNode : FlowNode
{
    [fsSerializeAs("d")] public int    damage    = 100;
    [fsSerializeAs("t")] public string damageType = "Physical";
    [fsSerializeAs("r")] public float  range     = 3.0f;
    
    // 节点逻辑...
}
```

**热更新流程**：
```
策划修改技能图（编辑器）
    │
    ▼
导出 JSON 文件
    │
    ▼
上传至热更新 CDN
    │
    ▼
客户端下载新 JSON（无需重新打包）
    │
    ▼
运行时 JSON → FlowGraph 对象 → 执行
```

### 10.3 对象池：解决反序列化性能

JSON 反序列化在运行时很耗时，尤其是一个技能可能每秒释放多次。通过**对象池**复用已反序列化的图实例：

```csharp
public class SkillGraphPool
{
    // graphId → 可用实例队列
    private Dictionary<string, Queue<FlowGraph>> _pools = new();
    
    // graphId → 图定义 JSON（只解析一次）
    private Dictionary<string, string> _jsonCache = new();

    /// <summary>租用一个图实例（从池中取或新建）</summary>
    public FlowGraph Rent(string graphId)
    {
        if (_pools.TryGetValue(graphId, out var pool) && pool.Count > 0)
        {
            var instance = pool.Dequeue();
            instance.Reset(); // 清除上次执行的状态
            return instance;
        }

        // 首次创建：从 JSON 反序列化
        var json  = _jsonCache[graphId];
        return FlowGraph.FromJson(json);
    }

    /// <summary>归还图实例到池中</summary>
    public void Return(string graphId, FlowGraph graph)
    {
        if (!_pools.ContainsKey(graphId))
            _pools[graphId] = new Queue<FlowGraph>();
        
        _pools[graphId].Enqueue(graph);
    }
}
```

### 10.4 帧同步支持：定点数替代浮点数

如果游戏需要帧同步（多端逻辑完全一致），可视化脚本节点中的所有数值运算**必须使用定点数**：

```csharp
// ❌ 错误：浮点数在不同平台计算结果可能不同
public class DealDamageNode : FlowNode
{
    public float damage = 100.0f;  // 不能用 float！
    
    public override void Execute(Context ctx)
    {
        // float 运算不具确定性
        float finalDmg = damage * ctx.attackMultiplier; 
    }
}

// ✅ 正确：定点数保证跨平台一致
public class DealDamageNode : FlowNode
{
    public FP damage = FP.FromInt(100);  // 定点数
    
    public override void Execute(Context ctx)
    {
        FP finalDmg = damage * ctx.attackMultiplier; // 定点数运算，结果确定
        ctx.target.TakeDamage(finalDmg);
    }
}
```

### 10.5 运行时调试可视化

可视化脚本的一大优势是**运行时可以直观看到执行路径**，这需要在执行引擎中埋入回调：

```csharp
public class FlowGraphExecutor
{
    // 调试回调（编辑器订阅，Runtime 下为 null）
    public event Action<string> OnNodeEnter;  // 参数：节点 GUID
    public event Action<string> OnNodeExit;

    private IEnumerator RunFlow(FlowNode node, FlowContext ctx)
    {
        if (node == null) yield break;

        OnNodeEnter?.Invoke(node.guid);  // ← 通知编辑器高亮此节点

        FlowPort nextPort = null;
        yield return node.Execute(ctx, port => nextPort = port);

        OnNodeExit?.Invoke(node.guid);   // ← 通知编辑器取消高亮

        if (nextPort?.connectedNode != null)
            yield return RunFlow(nextPort.connectedNode, ctx);
    }
}
```

## 总结：为什么可视化脚本是演进的终点（之一）

可视化脚本并不是"最高级"的方案，而是在特定约束下**多种演进的最优交汇点**：

```
代码可读性  ──────────────────────────────→ 越高越好
                    ▲
                    │
      可视化脚本（甜蜜点）
                    │
           面向对象 + 数据驱动
                    │
            命令 + 状态机
                    │
              纯 if/else
                    │
──────────────────────────────────────────→ 灵活性/可热更性
```

它同时满足了：

| 需求 | 满足方式 |
|------|---------|
| **策划可以独立迭代** | 节点图直接编辑，无需程序介入 |
| **热更新** | 图序列化为 JSON，随资源包更新 |
| **运行时可调试** | 执行路径高亮，断点暂停 |
| **类型安全** | 编辑器连线时做类型检查 |
| **性能可接受** | 对象池 + 缓存，避免重复反序列化 |
| **程序可扩展** | 新节点类型只需继承基类 + 编译 |

从最朴素的 `if/else` 到今天你在商业游戏引擎里看到的节点图编辑器，每一步都是被真实的工程痛点**逼出来**的——理解这个演进路径，比背会最终答案更有价值。
