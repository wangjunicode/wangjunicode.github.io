## 背景：Unity 的单线程渲染约束

Unity 引擎的绝大多数 API（GameObject 操作、组件访问、场景管理等）只能在**主线程**调用。然而现代游戏框架中大量使用了异步任务、Job System、网络 IO 等多线程操作。当这些子线程需要将结果"还给主线程执行"时，就需要一套线程安全的调度桥梁。

`MainThreadSynchronizationContext` 正是这座桥梁。

---

## 核心源码解析

```csharp
public class MainThreadSynchronizationContext : Singleton<MainThreadSynchronizationContext>, ISingletonUpdate
{
    private readonly ConcurrentQueue<Action> queue = new ConcurrentQueue<Action>();
    private Action a;

    public void Update()
    {
        while (true)
        {
            if (!this.queue.TryDequeue(out a))
            {
                return;
            }
            try
            {
                a();
            }
            catch (Exception e)
            {
                Log.Error(e);
            }
        }
    }

    public void Post(SendOrPostCallback callback, object state)
    {
        this.Post(() => callback(state));
    }

    public void Post(Action action)
    {
        queue.Enqueue(action);
    }
}
```

代码极其精简，却涵盖了生产者-消费者模式的完整精髓。

## 设计剖析

### 1. 无锁并发队列：ConcurrentQueue

`ConcurrentQueue<T>` 是 .NET 提供的**线程安全、无锁**先进先出队列，多个线程可同时调用 `Enqueue` 而无需显式加锁。与传统 `lock + Queue<T>` 方案相比：

| 特性 | `lock + Queue<T>` | `ConcurrentQueue<T>` |
|------|-------------------|----------------------|
| 线程安全 | ✅ 有锁 | ✅ 无锁（CAS 操作） |
| 性能 | 竞争时退化 | 高竞争下更优 |
| GC 压力 | 中等 | 略高（内部分段） |
| 适用场景 | 低频调度 | 高频多生产者场景 |

对游戏服务器或带有大量子线程的客户端框架来说，无锁队列避免了频繁加锁导致的线程饥饿问题。

---

### 2. 生产者端：Post 方法

```csharp
public void Post(Action action)
{
    queue.Enqueue(action);
}
```

任何线程（网络回调线程、Job 完成回调、Timer 回调等）都可以安全调用 `Post`，将一段逻辑"投递"到主线程队列。`Post` 命名来自 .NET 标准的 `SynchronizationContext.Post` 语义——**异步投递，不等待执行**。

重载版本兼容了标准 `SendOrPostCallback` 签名，方便与 .NET 的异步基础设施（`TaskScheduler`、`async/await` 的 `SynchronizationContext` 捕获机制）无缝集成。

---

### 3. 消费者端：Update 方法

```csharp
public void Update()
{
    while (true)
    {
        if (!this.queue.TryDequeue(out a))
        {
            return;
        }
        try { a(); }
        catch (Exception e) { Log.Error(e); }
    }
}
```

`Update` 在**主线程游戏循环**中被调用（通过 `ISingletonUpdate` 接口注入），每帧**排干队列**——将所有积压的跨线程任务全部执行完毕。

关键设计决策分析：

**① 排干而非限量**

每次 `Update` 不是只取一个或固定数量，而是一次性取完所有待执行项。这确保了**零积压延迟**：网络包在同帧处理完毕，不会推迟到下帧。

代价是：若某帧瞬间涌入大量任务，可能造成该帧卡顿。实际项目可加入帧时间预算控制（如每帧最多消费 N ms）。

**② 异常隔离**

```csharp
try { a(); }
catch (Exception e) { Log.Error(e); }
```

单个任务抛异常不会中断整个消费循环。这是生产级代码的重要保障——一条网络消息处理失败不应导致后续所有消息无法处理。

**③ 字段复用而非局部变量**

```csharp
private Action a; // 字段，非局部变量
```

这是一个**零 GC 优化细节**：将 `a` 声明为字段而非 `while` 循环内的局部变量，避免了循环内反复在栈上分配变量的开销（尽管现代 JIT 通常会优化，但框架层显式规避更稳妥）。

## 与 Unity SynchronizationContext 的关系

Unity 自带了一个 `UnitySynchronizationContext`，用于在 `async/await` 中捕获主线程上下文。本框架的 `MainThreadSynchronizationContext` 是**框架自研替代品**，原因如下：

1. **热更新兼容性**：Unity 原版在 HybridCLR/ILRuntime 环境下行为不一致
2. **可控性**：自研版可在 `Update` 中精确控制执行时机
3. **与 ETTask 集成**：框架的异步任务系统（ETTask）可直接向此队列 `Post` 延续（continuation）

## 典型使用场景

### 场景一：网络线程回调切回主线程

```csharp
// 网络接收线程中
void OnMessageReceived(byte[] data)
{
    MainThreadSynchronizationContext.Instance.Post(() =>
    {
        // 这里在主线程执行，可安全操作 Unity API
        Game.EventSystem.Publish(new NetworkMessageEvent { Data = data });
    });
}
```

### 场景二：与 C# async/await 集成

通过将框架的 `SynchronizationContext.Current` 设置为此实例，`await` 之后的代码自动回到主线程：

```csharp
// 设置同步上下文（启动时调用一次）
SynchronizationContext.SetSynchronizationContext(
    new FrameworkSynchronizationContext(MainThreadSynchronizationContext.Instance)
);

// 之后所有 await 都会回到主线程继续执行
async ETTask LoadAndApply()
{
    var data = await NetworkManager.FetchData(); // 子线程加载
    // await 后自动回主线程
    ApplyDataToUI(data); // 安全操作 UI
}
```

### 场景三：Job System 结果回传

```csharp
// JobSystem 完成回调中
void OnJobComplete(NativeArray<float> results)
{
    var snapshot = results.ToArray(); // 复制数据（NativeArray 需在主线程释放）
    MainThreadSynchronizationContext.Instance.Post(() =>
    {
        ProcessPathfindingResult(snapshot);
    });
}
```

## 性能考量与优化建议

### 1. 帧时间预算保护

当游戏存在大量异步任务时，建议加入时间预算：

```csharp
public void Update()
{
    var deadline = DateTime.UtcNow.AddMilliseconds(2); // 每帧最多 2ms
    while (DateTime.UtcNow < deadline)
    {
        if (!this.queue.TryDequeue(out a)) return;
        try { a(); }
        catch (Exception e) { Log.Error(e); }
    }
}
```

### 2. 监控队列深度

```csharp
public int PendingCount => queue.Count; // ConcurrentQueue 的 Count 是近似值，但够用
```

在性能面板中暴露此值，可及时发现某模块过度向主线程投递任务的问题。

### 3. 避免闭包 GC

频繁 `Post` 的热路径应预分配 Action，避免每次 lambda 都产生新对象：

```csharp
// ❌ 每次 new Action
void HandleTick() => MainThreadSynchronizationContext.Instance.Post(() => OnTick());

// ✅ 复用委托
private readonly Action _onTickAction;
void Init() => _onTickAction = OnTick;
void HandleTick() => MainThreadSynchronizationContext.Instance.Post(_onTickAction);
```

## 与其他框架方案对比

| 方案 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| 本框架 | `ConcurrentQueue + Update` | 无锁、可控、透明 | 依赖游戏循环 |
| UniRx | `MainThreadScheduler` | 响应式风格 | 学习成本高 |
| UniTask | `PlayerLoopHelper` | 与 Unity 深度集成 | 依赖 Unity 版本 |
| Channel (System.Threading) | `Channel<T>` | .NET 原生 | 需要额外消费逻辑 |

框架自研方案的最大优势：**可预测性**。开发者清楚地知道任务在哪一帧的哪个阶段被执行。

## 总结

`MainThreadSynchronizationContext` 虽只有数十行代码，却体现了游戏框架设计中几个重要原则：

1. **显式优于隐式**：跨线程调度行为完全可见，不依赖运行时魔法
2. **无锁优于有锁**：`ConcurrentQueue` 在多生产者场景下性能更优
3. **隔离优于崩溃**：异常不扩散，单任务失败不影响全局
4. **零 GC 意识**：字段复用、委托预分配等细节贯穿框架设计

这种"简单但正确"的设计哲学，是大型游戏框架在长期演进中沉淀下来的宝贵经验。
