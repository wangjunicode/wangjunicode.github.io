## 代码质量的经济学

```
发现 Bug 的成本：

  开发阶段：$1（几分钟修复）
  代码审查：$5（需要沟通和重新实现）
  测试阶段：$25（需要定位、修复、验证）
  上线后：$250+（需要紧急热更，影响玩家体验，品牌损失）

投资代码质量就是在降低 Bug 的修复成本
```

好的代码质量体系包含三层防线：
1. **静态分析**：在写代码时就发现问题（最早）
2. **自动化测试**：在提交前验证逻辑（最快）
3. **代码审查**：通过人工检查发现深层问题（最全）

## 一、Roslyn 静态分析：自定义规则

### 1.1 什么是 Roslyn 分析器

```
Roslyn 是 C# 编译器平台，提供代码分析 API。
通过编写分析器，可以：
  - 在 IDE 中实时显示代码问题（红/黄线）
  - 在编译时检查规范违反
  - 自动提供代码修复建议

Unity 游戏中的典型自定义规则：
  ❌ 在游戏循环中使用 string.Format（应该用 StringBuilder 或直接拼接）
  ❌ 在 Update 中使用 GetComponent（应该缓存）
  ❌ MonoBehaviour 中使用了静态状态（多场景加载问题）
  ❌ 比较 null 与 Unity Object（应该用 == null 而非 is null）
```

### 1.2 编写自定义 Roslyn 分析器

```csharp
// 项目结构：
// GameAnalyzers/
//   GameAnalyzers.csproj  (Roslyn 分析器项目)
//   Analyzers/
//     GetComponentInUpdateAnalyzer.cs
//     StringFormatInLoopAnalyzer.cs

using Microsoft.CodeAnalysis;
using Microsoft.CodeAnalysis.CSharp;
using Microsoft.CodeAnalysis.CSharp.Syntax;
using Microsoft.CodeAnalysis.Diagnostics;
using System.Collections.Immutable;

/// <summary>
/// 检测在 Update/LateUpdate/FixedUpdate 中调用 GetComponent
/// </summary>
[DiagnosticAnalyzer(LanguageNames.CSharp)]
public class GetComponentInUpdateAnalyzer : DiagnosticAnalyzer
{
    public static readonly DiagnosticDescriptor Rule = new DiagnosticDescriptor(
        id: "GAME001",
        title: "避免在 Update 中使用 GetComponent",
        messageFormat: "在 '{0}' 中调用了 GetComponent，应该在 Awake/Start 中缓存结果",
        category: "Performance",
        defaultSeverity: DiagnosticSeverity.Warning,
        isEnabledByDefault: true,
        description: "GetComponent 每次调用都会进行反射查找，在高频调用的 Update 中使用会造成性能问题。"
    );
    
    public override ImmutableArray<DiagnosticDescriptor> SupportedDiagnostics 
        => ImmutableArray.Create(Rule);
    
    public override void Initialize(AnalysisContext context)
    {
        context.ConfigureGeneratedCodeAnalysis(GeneratedCodeAnalysisFlags.None);
        context.EnableConcurrentExecution();
        
        // 注册语法节点分析
        context.RegisterSyntaxNodeAction(
            AnalyzeInvocation, 
            SyntaxKind.InvocationExpression
        );
    }
    
    private static readonly HashSet<string> UPDATE_METHODS = new()
    {
        "Update", "LateUpdate", "FixedUpdate"
    };
    
    private void AnalyzeInvocation(SyntaxNodeAnalysisContext context)
    {
        var invocation = (InvocationExpressionSyntax)context.Node;
        
        // 检查是否是 GetComponent 调用
        if (!IsGetComponentCall(invocation, context.SemanticModel))
            return;
        
        // 检查是否在 Update/LateUpdate/FixedUpdate 方法中
        var containingMethod = invocation.Ancestors()
            .OfType<MethodDeclarationSyntax>()
            .FirstOrDefault();
        
        if (containingMethod == null) return;
        
        string methodName = containingMethod.Identifier.ValueText;
        if (!UPDATE_METHODS.Contains(methodName)) return;
        
        // 报告诊断
        var diagnostic = Diagnostic.Create(
            Rule, 
            invocation.GetLocation(),
            methodName
        );
        
        context.ReportDiagnostic(diagnostic);
    }
    
    private bool IsGetComponentCall(InvocationExpressionSyntax invocation, SemanticModel model)
    {
        if (invocation.Expression is MemberAccessExpressionSyntax memberAccess)
        {
            string name = memberAccess.Name.Identifier.ValueText;
            if (name.StartsWith("GetComponent")) return true;
        }
        
        if (invocation.Expression is IdentifierNameSyntax identifier)
        {
            string name = identifier.Identifier.ValueText;
            if (name.StartsWith("GetComponent")) return true;
        }
        
        return false;
    }
}
```

## 二、单元测试最佳实践

### 2.1 什么值得测试

```
游戏开发中的可测试部分：

✅ 高价值测试：
  - 伤害计算公式（数值系统）
  - 技能配置解析
  - 物品合成/分解逻辑
  - 存档序列化/反序列化
  - AI 决策逻辑
  - 经济系统（货币计算）

❌ 低价值测试：
  - Unity 内置功能（不需要测 Instantiate 能工作）
  - 纯 UI 显示（很难自动化）
  - 动画和特效

原则：测试"业务逻辑"，不测"胶水代码"
```

### 2.2 好的单元测试的标准

```
FIRST 原则：
  Fast（快速）：每个测试应该 < 1ms（毫秒级）
  Independent（独立）：测试之间互不依赖
  Repeatable（可重复）：每次运行结果相同
  Self-validating（自验证）：Pass/Fail 自动判断
  Timely（及时）：写代码时同步写测试
```

### 2.3 测试用例设计模式

```csharp
[TestFixture]
public class CraftingSystemTests
{
    private CraftingSystem _craftingSystem;
    private Inventory _inventory;
    
    [SetUp]
    public void Setup()
    {
        // 每个测试前重新初始化（保证独立）
        _inventory = new Inventory(capacity: 20);
        _craftingSystem = new CraftingSystem(_inventory);
    }
    
    // 等价类划分：覆盖正常情况
    [Test]
    public void Craft_ValidRecipe_WithSufficientMaterials_Succeeds()
    {
        // Arrange
        _inventory.AddItem(ItemId.Wood, 3);
        _inventory.AddItem(ItemId.Iron, 1);
        var recipe = RecipeDatabase.Get(RecipeId.WoodenSword);
        
        // Act
        var result = _craftingSystem.Craft(recipe);
        
        // Assert
        Assert.AreEqual(CraftingResult.Success, result.Status);
        Assert.IsTrue(_inventory.Contains(ItemId.WoodenSword, 1));
        Assert.AreEqual(0, _inventory.GetCount(ItemId.Wood)); // 材料被消耗
    }
    
    // 边界值：材料恰好足够
    [Test]
    public void Craft_ExactlyEnoughMaterials_Succeeds()
    {
        _inventory.AddItem(ItemId.Wood, 3); // 恰好需要 3 个
        var recipe = RecipeDatabase.Get(RecipeId.WoodenSword);
        
        var result = _craftingSystem.Craft(recipe);
        
        Assert.AreEqual(CraftingResult.Success, result.Status);
    }
    
    // 边界值：材料不足
    [Test]
    public void Craft_InsufficientMaterials_Fails()
    {
        _inventory.AddItem(ItemId.Wood, 2); // 需要 3 个，只有 2 个
        var recipe = RecipeDatabase.Get(RecipeId.WoodenSword);
        
        var result = _craftingSystem.Craft(recipe);
        
        Assert.AreEqual(CraftingResult.InsufficientMaterials, result.Status);
        // 确认材料没有被消耗（合成失败不应消耗材料）
        Assert.AreEqual(2, _inventory.GetCount(ItemId.Wood));
    }
    
    // 背包已满时合成
    [Test]
    public void Craft_InventoryFull_Fails()
    {
        // 填满背包
        for (int i = 0; i < 20; i++)
            _inventory.AddItem(ItemId.Wood, 1);
        
        var recipe = RecipeDatabase.Get(RecipeId.WoodenSword);
        var result = _craftingSystem.Craft(recipe);
        
        Assert.AreEqual(CraftingResult.InventoryFull, result.Status);
    }
    
    // 参数化测试：多组输入
    [TestCase(ItemId.Wood, 3, ItemId.WoodenSword, 1)]
    [TestCase(ItemId.Iron, 5, ItemId.IronSword, 1)]
    [TestCase(ItemId.Gold, 10, ItemId.GoldSword, 1)]
    public void Craft_DifferentWeapons_ProducesCorrectOutput(
        ItemId material, int materialCount, ItemId expectedOutput, int expectedCount)
    {
        _inventory.AddItem(material, materialCount);
        var recipe = RecipeDatabase.GetByOutput(expectedOutput);
        
        _craftingSystem.Craft(recipe);
        
        Assert.AreEqual(expectedCount, _inventory.GetCount(expectedOutput));
    }
}
```

### 2.4 Mock 对象：隔离外部依赖

```csharp
// 使用接口和 Mock 对象隔离测试
public interface ITimeProvider
{
    float CurrentTime { get; }
}

public class SkillCooldownManager
{
    private readonly ITimeProvider _time;
    private readonly Dictionary<int, float> _lastCastTimes = new();
    
    public SkillCooldownManager(ITimeProvider time) => _time = time;
    
    public bool CanCast(int skillId, float cooldown)
    {
        if (!_lastCastTimes.TryGetValue(skillId, out float lastCast))
            return true;
        return _time.CurrentTime - lastCast >= cooldown;
    }
    
    public void OnCast(int skillId)
    {
        _lastCastTimes[skillId] = _time.CurrentTime;
    }
}

// Mock 时间提供者（控制时间流逝）
public class MockTimeProvider : ITimeProvider
{
    public float CurrentTime { get; set; } = 0f;
}

[TestFixture]
public class SkillCooldownTests
{
    private MockTimeProvider _time;
    private SkillCooldownManager _cooldownManager;
    
    [SetUp]
    public void Setup()
    {
        _time = new MockTimeProvider { CurrentTime = 0f };
        _cooldownManager = new SkillCooldownManager(_time);
    }
    
    [Test]
    public void CanCast_BeforeCooldownExpires_ReturnsFalse()
    {
        const int skillId = 1001;
        const float cooldown = 3f;
        
        _cooldownManager.OnCast(skillId);
        _time.CurrentTime = 2f; // 2秒后
        
        Assert.IsFalse(_cooldownManager.CanCast(skillId, cooldown));
    }
    
    [Test]
    public void CanCast_AfterCooldownExpires_ReturnsTrue()
    {
        const int skillId = 1001;
        const float cooldown = 3f;
        
        _cooldownManager.OnCast(skillId);
        _time.CurrentTime = 3.1f; // 3.1秒后
        
        Assert.IsTrue(_cooldownManager.CanCast(skillId, cooldown));
    }
}
```

## 三、代码审查最佳实践

### 3.1 代码审查清单

```markdown
## 代码审查清单（每次 Review 必过的关卡）

### 正确性
- [ ] 逻辑是否正确？（不只看代码，要思考业务场景）
- [ ] 边界条件是否处理？（空值、最大值、最小值）
- [ ] 错误处理是否完整？（异常捕获、返回错误码检查）
- [ ] 并发安全？（如果涉及多线程）

### 性能
- [ ] 热路径（Update/帧循环）中是否有不必要的分配？
- [ ] 是否有 O(n²) 或更差的算法可以优化？
- [ ] 是否有可以缓存的重复计算？
- [ ] Unity 对象操作是否有性能问题？（GetComponent、Find 等）

### 可维护性
- [ ] 命名是否清晰表达意图？
- [ ] 单个函数/类的职责是否单一？
- [ ] 是否有重复代码可以提取？
- [ ] 是否有 Magic Number 需要命名为常量？

### 资源管理
- [ ] 所有加载的资源是否有对应的释放？
- [ ] 事件订阅是否有对应的取消订阅？
- [ ] IDisposable 对象是否被正确释放？

### 安全性
- [ ] 是否有 SQL 注入、XSS 等安全漏洞？（网络游戏）
- [ ] 是否暴露了不应该暴露的内部状态？
```

### 3.2 Reviewer 的职责

```
Reviewer 要做的事：
  1. 理解这个 PR 要解决什么问题（不理解就问）
  2. 检查实现方式是否合理（有没有更好的方案）
  3. 找出潜在 Bug（不只看代码，要想业务场景）
  4. 分享知识（"这里可以用 X 模式，更清晰"）
  5. 给予鼓励（"这个设计很好！"）

Reviewer 不要做的事：
  ❌ 只看风格不看逻辑
  ❌ 因为不是自己的写法就要求改变（无功能影响的风格偏好）
  ❌ 不给出具体建议（"这里不对"vs"这里在 X 情况下会有 Y 问题，可以考虑 Z"）
  ❌ 拖延超过 24 小时不响应
```

## 四、技术债务量化管理

### 4.1 技术债务的类别

```
类型A：故意的技术债务（有意识地做的临时方案）
  例："这个配置读取先 hardcode，下个迭代完善"
  
类型B：无意的技术债务（当时不知道更好的方案）
  例：写了一个后来发现有问题的架构决策
  
类型C：腐烂的技术债务（代码随时间老化）
  例：当初的最佳实践，现在已经有更好的方案

处理策略：
  类型A：有计划地偿还（记录在 Backlog 中）
  类型B：通过 Code Review 和学习减少
  类型C：在需求触达时顺手重构
```

### 4.2 建立技术债务看板

```markdown
# 技术债务登记表

| ID | 描述 | 模块 | 影响 | 偿还成本 | 优先级 | 负责人 |
|----|------|------|------|---------|--------|--------|
| TD-001 | UIManager 使用 Find 查找对象，应该改为注册模式 | UI | 中 | 2天 | 高 | TBD |
| TD-002 | 战斗系统中的 Magic Number，需要提取到配置 | Battle | 低 | 0.5天 | 中 | TBD |
| TD-003 | 资源加载没有错误处理，容易崩溃 | Resource | 高 | 1天 | 紧急 | TBD |

优先级判断：
  Impact × Urgency - Effort = 优先级分数
  
  - Impact：这个债务对项目的影响程度
  - Urgency：这个债务是否在增长（随着项目变大，影响越来越大？）
  - Effort：偿还这个债务需要的工作量
```

## 总结

代码质量工程的核心原则：

1. **自动化优先**：静态分析和自动化测试可以无成本地持续运行
2. **早发现早修复**：在 IDE 中发现 > 提交前发现 > Code Review 发现 > 测试发现 > 线上发现
3. **量化技术债务**：让技术债务"可见"，才能有计划地管理
4. **文化比工具重要**：代码审查是文化，不是流程。团队氛围决定了 Review 的质量

**技术 Leader 的任务**：建立这套体系，并通过榜样作用推动团队执行。最好的代码质量来自团队的内化认同，而不是强制要求。
